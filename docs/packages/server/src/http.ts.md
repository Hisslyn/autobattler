# packages/server/src/http.ts

**Path & purpose** — `packages/server/src/http.ts`. The minimal HTTP API layer, sharing the same TCP port as the WebSocket server: guest auth registration, leaderboard, profile read/update, match history. CORS-enabled (any origin), pure `Repository`-backed handlers, no framework (raw Node `http` request/response objects).

**Responsibility** — Owns every non-WebSocket HTTP endpoint the client uses: `POST /auth/guest` (account creation/lookup), `GET /leaderboard` (public top-N by MMR), `GET /profile` + `GET /history` (authenticated, Bearer-token-protected reads), `PATCH /profile` (authenticated name change, name-validated). This is the bridge between raw HTTP and the `Repository` interface — no game/match logic lives here, only account/profile/persistence-facing endpoints.

**Exports**
- `validateName(raw: unknown): string | null` — the canonical name-validation function: returns `null` if `raw` isn't a string; trims it; rejects (returns `null`) if trimmed length is outside `[NAME_MIN=2, NAME_MAX=16]`; rejects if it doesn't match `NAME_PATTERN = /^[A-Za-z0-9 _-]+$/` (letters, digits, spaces, underscore, hyphen only); otherwise returns the TRIMMED name. This is the single source of truth for "what names are valid" — used by `PATCH /profile`, and presumably reusable by any other future name-accepting path.
- `createHttpHandler(repo: Repository)` — a factory returning the actual `(req, res) => Promise<void>` HTTP request handler, closing over the given `repo` instance. This handler is meant to be passed to Node's raw `http.createServer` (or attached alongside the `ws` server on the same port, per the doc comment "sharing the WS port").

**Key behavior — route by route**
- **CORS**: every response (success or error) carries `Access-Control-Allow-Origin: *`, `Access-Control-Allow-Headers: Content-Type, Authorization`, `Access-Control-Allow-Methods: GET, POST, PATCH, OPTIONS` (set inside `sendJson`, so EVERY response path gets these headers uniformly) — `OPTIONS` requests (CORS preflight) are answered immediately with `204` and an empty body, before any route matching.
- **`POST /auth/guest`**: reads+JSON-parses the body (400 on invalid JSON); requires `deviceId` to be a non-empty string (400 otherwise); optionally accepts `name` (must be a non-empty-after-trim string, then SILENTLY TRUNCATED to 24 chars via `.slice(0,24)` — notably this 24-char truncation is DIFFERENT from `validateName`'s 16-char max — `POST /auth/guest`'s name handling does NOT go through `validateName` at all, it has its own looser ad-hoc truncation instead of validation/rejection); calls `repo.createGuest(deviceId, name)` (idempotent per device, per `Repository`'s contract), then fetches the resulting profile and returns `{accountId, token, profile}` with `200`.
- **`GET /leaderboard`**: reads `?n=` query param (default 50), clamps to `[1, LEADERBOARD_MAX=200]` via `Math.min(200, Math.max(1, Number(n) || 50))` (the `|| 50` fallback catches `NaN` from a non-numeric query value, defaulting back to 50 rather than propagating `NaN` into the clamp), returns `{leaderboard: Profile[]}` — NO AUTHENTICATION required (public endpoint).
- **`PATCH /profile`**: requires a valid Bearer token (`bearerToken(req)` extracts it from the `Authorization` header, must start with `"Bearer "`); resolves it via `repo.findByToken` — `401 UNAUTHENTICATED` if missing/invalid; parses the JSON body (400 on parse failure); runs the `name` field through `validateName` (NOT the looser ad-hoc truncation used in `/auth/guest` — this is the STRICT path) — `400 INVALID_NAME` if it fails validation; otherwise calls `repo.updateProfile(accountId, {name})` and returns the FRESH profile (`200 {profile}`).
- **`GET /profile` / `GET /history`** (same auth-gate block, branching on pathname after auth): both require a valid Bearer token (`401 UNAUTHENTICATED` otherwise, same pattern as PATCH). `/profile` returns `{profile: await repo.getProfile(accountId)}`. `/history` reads `?limit=` (default 20, clamped `[1, HISTORY_MAX=100]`, same NaN-safe pattern as leaderboard's `n`) and returns `{history: await repo.matchHistory(accountId, limit)}`.
- **Fallback**: any unmatched method+pathname combination → `404 {error: "not found"}`.
- **Error handling**: the ENTIRE route-matching body is wrapped in one `try`/`catch` — any uncaught exception (e.g. a `Repository` method rejecting) is logged (`console.error("[http] error handling", method, pathname, err)`) and converted to a generic `500 {error: "internal error"}` — no internal error detail/stack leaks to the client response.

**Invariants & constraints**
- `MAX_BODY = 4096` bytes — `readBody` actively rejects (destroys the request, rejects the promise) any request body exceeding this size DURING streaming (checked on every `data` chunk, not just at the end) — a basic DoS/abuse guard against oversized payloads on the tiny JSON bodies these endpoints actually need.
- The TWO DIFFERENT name-handling paths (`/auth/guest`'s loose ad-hoc trim+24-char-truncate-no-rejection vs `/profile` PATCH's strict `validateName` 2-16-char+charset gate) are a deliberate asymmetry: a brand-new guest's auto-generated/initial name is more permissive (anything non-empty, truncated rather than rejected, presumably because the CLIENT controls what it sends at registration time and a hard rejection there would be an awkward UX moment during account bootstrap), while an explicit user-initiated rename via `PATCH /profile` is strictly validated and CAN be rejected with a typed error (`INVALID_NAME`) since the user can see and correct the validation failure in a settings UI. A reader should NOT assume `/auth/guest`-created names are always `validateName`-conformant — they might exceed 16 chars or contain characters outside `NAME_PATTERN` (anything up to 24 chars that isn't empty-after-trim) until/unless the user later runs them through a `PATCH /profile` rename.
- `bearerToken` requires the EXACT prefix `"Bearer "` (case-sensitive, single space) — any other auth scheme or malformed header silently yields `null` (treated as no token, leading to `401`), not a distinct error.
- The `Number(x) || 50` / `Number(x) || 20` NaN-fallback pattern has a subtle footgun: if the query param literally parses to `0` (e.g. `?n=0`), `Number("0") || 50` evaluates to `50` (since `0` is falsy) — meaning a caller explicitly asking for `n=0` results in `0`'s the FALLBACK default, not an actual zero-length result clamped by `Math.max(1, ...)` (which would ALSO force a minimum of 1 anyway) — in practice both paths converge on "at least 1 result requested", so this isn't a functional bug, just a slightly confusing double-correction (the `|| 50` already prevents 0, then `Math.max(1, ...)` would have caught it anyway even without the `||` fallback).

**Depends on** — Node's `node:http` (`IncomingMessage`, `ServerResponse` types only — no runtime import, the actual server creation happens in `index.ts`); `./db/index.js` (`Repository` type).

**Used by** — `packages/server/src/index.ts` (presumably calls `createHttpHandler(repo)` once and attaches the returned handler to the shared HTTP server, per CLAUDE.md's "HTTP API + ws on the same port"); the client's `auth.ts` (`fetchProfile`/`fetchHistory`/`patchName`/`fetchLeaderboard` helpers, per CLAUDE.md's client internals, call these exact endpoints).

**Notes** — This file has NO routing framework/library dependency — every route is a hand-written `if (req.method === X && url.pathname === Y)` block in sequence, appropriate given the small, fixed endpoint set (5 routes total) — a reader extending this with new endpoints should follow the same pattern (early-return on each match, fall through to `404` at the end) and remember the implicit endpoint inventory this represents: `/auth/guest` (POST), `/leaderboard` (GET), `/profile` (GET, PATCH), `/history` (GET) — exactly five method+path combinations, matching CLAUDE.md's exact description of `http.ts`'s internals.
