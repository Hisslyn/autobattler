# packages/server/src/room.ts

**Path & purpose** — `packages/server/src/room.ts`. The authoritative match-room orchestrator: owns one `MatchState` per match, drives the PLANNING → COMBAT → RESOLUTION → PLANNING phase loop on real wall-clock timers, applies validated player commands, runs AI for bot/AFK seats, broadcasts every state-changing event over WebSocket, and persists the match result via `recorder.ts` at match end. This is the single largest, most central file in `packages/server` — nearly everything else (matchmaker, session, index, recorder, mmr) exists to feed into or be driven by this file.

**Responsibility** — Owns: (1) the `Room` data structure (one per active match: `MatchState` + seat↔session mapping + seat metadata + phase timer + room-local PRNG); (2) the real-time phase clock (30s planning via `setTimeout`, configurable resolution pause via `economy.json`'s `resolutionSeconds`); (3) command validation + application (delegating actual legality/effects to `@autobattler/rules`'s `applyCommand`); (4) AI backfill for bot seats AND for AFK human seats (both treated identically — `applyAiCommands` runs for any seat that's `null` (bot) OR whose session `.afk` is `true`); (5) all outbound serialization (public vs private view splitting per seat) and broadcast; (6) match-end persistence orchestration (calling `recorder.ts`, building the `MATCH_END` payload) and room cleanup.

**Exports**
- `PLANNING_MS: number` — the real-time duration of the PLANNING phase. Defaults to `30_000` (30s) but can be overridden via the `PLANNING_MS_OVERRIDE` env var (must parse to a finite positive number) — explicitly described in a comment as existing so `room.test.ts`'s fake-timer integration harness can shorten real-wall-clock planning, since READY no longer skips planning (see `handlePlayerReady` below).
- `interface Room { id, matchId, state: MatchState, seats: (Session|null)[], seatAccounts: (string|null)[], seatNames: string[], phaseTimer, prng: ()=>number }` — `seats[i] === null` means seat `i` is a bot (no session ever occupies it); `seatAccounts`/`seatNames` are captured ONCE at room creation and never change for the lifetime of the match (even across reconnects, since reconnect only restores a `Session` reference into `seats[i]`, it doesn't touch `seatAccounts`/`seatNames`).
- `setRoomRepository(r: Repository): void` — module-level dependency injection: swaps the `Repository` instance every `finalizeMatch` call will persist through. Defaults to a fresh `MemoryRepository()` so tests driving rooms directly need zero setup; `index.ts` calls this once at boot with the real `createRepository()` result.
- `createRoom(sessions: Session[], seedOverride?: number): Room` — the matchmaker's entry point into this module. Builds a brand-new `Room`: generates a match seed (`seedOverride ?? Date.now() ^ (Math.random() * 0xffffffff)` — note this top-level match seed generation legitimately uses `Math.random`/`Date.now`, since `packages/server` is I/O-permitted; the seed itself, once chosen, drives the ENTIRELY deterministic `createMatch`/`mulberry32` simulation downstream), creates the room-local `prng` (`mulberry32(seed)`) and the initial `MatchState` (`createMatch(seed, gameData)`), pre-fills all 8 seats as bots (`seats`=null array, `seatNames`=`"Bot N"` placeholders), then overwrites the first `sessions.length` seats with the real sessions (capturing their `accountId` and `name ?? "Player N"`), registers the room, stamps `session.roomId`/`seatIndex` on each real session, registers a SEAT TOKEN per human (`registerSeatToken(session.token, roomId, i)` — enabling later `RECONNECT`), sends each human a `MATCH_FOUND {roomId, token, seatIndex}` then an initial `STATE_SNAPSHOT`, and finally calls `startPlanning(room)`.
- `getRoom(roomId: string): Room | undefined` — simple registry lookup (the module-level `rooms: Map<string, Room>`).
- `handlePlayerCommand(room, seatIndex, rawCmd): void` — the per-CMD-message entry point from `index.ts`. Rejects with `WRONG_PHASE` if not currently `PLANNING`; rejects with `INVALID_MESSAGE` if `validateCommand` can't structurally parse `rawCmd` into a typed `Command`; otherwise calls `applyCommand(state, seatIndex, cmd, room.prng, gameData)` — on rejection (`result.ok === false`), sends `COMMAND_REJECTED` with the rules-layer's typed error string; on success, calls `broadcastDelta(room, seatIndex)` to push the resulting state change to everyone.
- `handlePlayerReady(room, _seatIndex): void` — checks whether ALL SEATS are "ready" (a seat counts as ready if it's a bot (`null`), AFK, or has explicitly sent `READY` this phase — `isReady`); if so AND the current phase is `RESOLUTION`, calls `finishResolution(room)` early. Explicitly a no-op for `PLANNING` (a comment clarifies: "Planning runs its full duration; READY is a no-op for PLANNING... READY still skips the RESOLUTION pause") — meaning ready-up can only shortcut the RESOLUTION pause, never the 30s planning timer itself.
- `markReady(room, seatIndex): void` — records this seat as ready (in the per-room `readySet` WeakMap-backed Set) then immediately calls `handlePlayerReady` to check if that completes the all-ready condition.
- `reconnectSession(room, session, seatIndex): void` — restores a fresh `Session` object into `room.seats[seatIndex]`, re-stamps `session.roomId`/`seatIndex`, clears `session.afk`, and sends a fresh `STATE_SNAPSHOT` — the full mechanism behind the `RECONNECT` C2S message handled in `index.ts`.

**Key behavior — the phase loop**
- **`startPlanning(room)`**: clears the ready-set, broadcasts `PHASE_CHANGE {phase: PLANNING, round, endsAt}` (an absolute wall-clock deadline, letting clients render a countdown without needing tight clock sync), and schedules `advanceCombat(room)` via `setTimeout(..., PLANNING_MS)` — this is the ONLY way `advanceCombat` is reached; `markReady`/`handlePlayerReady` explicitly CANNOT shortcut planning (per the comment in `handlePlayerReady`).
- **`advanceCombat(room)`**: clears the phase timer; guards against double-invocation (`if (room.state.phase !== "PLANNING") return`); for every seat that's a bot OR AFK (and whose `MatchState.players[i]` is still `alive`), calls `applyAiCommands(state, i, room.prng, gameData)` — this is where ALL bot AND afk-human planning decisions get made, using the room's OWN seeded `prng` (not a fresh one per call, so calls accumulate PRNG state across the whole match, consistent with one continuous deterministic stream). Then calls `advancePhase(room.state, gameData)` from `@autobattler/rules` — this single call internally transitions PLANNING→COMBAT, runs the actual combat simulation(s) for this round, and populates `state.lastPairings`/`lastRoundSeed`/`lastOpponentBoards`/`lastCombatResults`/etc. Afterward, `room.ts` builds `opponentSnapshots` (one per player who was in a pairing, using `state.lastOpponentBoards` — "exactly what the server simulated, even if the opponent was eliminated this round," per the comment) and broadcasts `COMBAT_START {pairings, opponentSnapshots, roundSeed}` followed immediately by `COMBAT_RESULT {results: serializeCombatResults(room)}` (built from `state.lastCombatResults`, a `Map<id, CombatResult>` converted to a plain object). If `isMatchOver(state)`, calls `finalizeMatch(room)` (fire-and-forget `void`, since this function is itself synchronous but `finalizeMatch` is async) and returns; otherwise calls `startResolution(room)`.
- **`startResolution(room)`**: clears the ready-set; computes `resolutionMs = gameData.economy.resolutionSeconds * 1000`. If this round `isPveRound`, sends each non-AFK human their OWN `LOOT {round, orbs}` message PRIVATELY (from `state.lastLootOrbs.get(seatIndex)`, defaulting to `[]`) — explicitly BEFORE the phase-change broadcast, "so the client has them when it triggers the reveal on RESOLUTION." Then, REGARDLESS of PvE/PvP, sends each non-AFK human their OWN `ROUND_RESULT {round, result}` PRIVATELY (from `state.lastRoundResult.get(seatIndex)`, only if present) — same ordering rationale ("before the phase change so the resolution screen can read it as it opens"). Then broadcasts the public `PHASE_CHANGE {phase: RESOLUTION, round, endsAt}` to everyone, sends a fresh `STATE_SNAPSHOT` to every non-AFK human (reflecting post-combat HP/eliminations), and schedules `finishResolution(room)` via `setTimeout(..., resolutionMs)`.
- **`finishResolution(room)`**: clears the timer; guards against double-invocation (phase must still be `RESOLUTION`); calls `advancePhase(room.state, gameData)` AGAIN — this second call (RESOLUTION→PLANNING this time) handles income distribution, round increment, and shop refresh internally (per the comment); sends fresh snapshots to every non-AFK human; calls `startPlanning(room)` to begin the next round's cycle.
- **`finalizeMatch(room)` (async)**: builds the final `placements` array by spreading `state.placements` (elimination order, 1=first-out) then appending the one remaining `alive` player (the winner, who never got a placement entry from elimination since they never got eliminated). Builds `seatResults: SeatResult[]` — one per player, converting the rules-layer's "elimination order" numbering into "final standing" numbering: a still-`alive` player gets `placement: 1` (the winner); an eliminated player's stored `p.placement` (1=first eliminated) is converted via `playerCount - placement + 1` (so e.g. in an 8-player match, the FIRST player eliminated, `p.placement=1`, becomes final standing `8 - 1 + 1 = 8`, i.e. LAST place — and the player eliminated 7th, `p.placement=7`, becomes `8 - 7 + 1 = 2`, second place) — this inversion is the precise mapping between rules' internal elimination-order bookkeeping and the human-facing "1st place...8th place" semantics. Calls `recordMatchResult(repo, room.matchId, seatResults)` inside a `try/catch` (logging, not throwing, on failure — a persistence failure does NOT prevent the match-end broadcast from happening; `mmr` simply stays `{}` if it fails). Builds `names` (seat→`seatNames[i]`) and `stats` (seat→`MatchStats{roundWins, roundLosses, totalDamageTaken, totalDamageDealt}`, read directly off each player's final accumulated `PlayerState` fields — explicitly NOT persisted, payload-only, per the inline comment). Broadcasts `MATCH_END {placements, mmr, names, stats}` to everyone, THEN calls `clearRoomSeatTokens(room.id)` (invalidating all RECONNECT capability for this now-finished match) and `rooms.delete(room.id)` (the room ceases to exist — any further `getRoom` calls for this id return `undefined`).

**Key behavior — command validation & serialization**
- `validateCommand(raw)` — a hand-written structural type-guard converting an untyped `Record<string,unknown>` into a typed `Command` or `null`: switches on `raw.type`, checking each variant's required field types (`BUY` needs numeric `shopSlotIndex`; `SELL`/`MOVE`/`EQUIP` need numeric `unitUid`; `MOVE` additionally needs boolean `toBench` + numeric `toIndex`; `EQUIP` additionally needs string `itemId`; `REROLL`/`BUY_XP` need no extra fields) — any unrecognized `type` or missing/mistyped field returns `null` (treated as `INVALID_MESSAGE` by the caller). **Note**: this validator does NOT include `UNEQUIP` or `COMBINE_ITEMS` (both documented as real `Command` variants in `@autobattler/rules`'s `commands.ts` per CLAUDE.md) — meaning as written, this file's network-facing command surface is INCOMPLETE relative to the full rules-layer command set; a `raw.type === "UNEQUIP"` or `"COMBINE_ITEMS"` message would fall through to the `default: return null` branch and be rejected as `INVALID_MESSAGE`, even though `@autobattler/rules`'s `applyCommand` itself supports them. This is a real functional gap worth flagging for any future work involving item un-equipping or component combination over the network (Practice/LocalDriver would be unaffected, since it presumably calls `applyCommand` directly without going through this validator).
- `serializePlayerPublic(p, name)` — the PUBLIC view of a player: `id, name, hp, level, xp, winStreak, loseStreak, board, bench, alive, placement` — notably `board`/`bench` (and therefore each unit's full stats/items) ARE public (visible to every other seat, enabling opponent inspection per CLAUDE.md's client scout feature), but `gold`/`shop`/`items` (the unequipped inventory) are NOT included here — those are PRIVATE, only ever sent to the OWNING seat (consistent with CLAUDE.md: "Public player fields... carry name... PvE loot and round results are private per-seat").
- `serializeOpponentSnapshot(state, recipientId, opponentId)` — builds the COMBAT_START opponent view for one recipient: pulls the board from `state.lastOpponentBoards.get(recipientId)` (the rules-captured snapshot, NOT a live lookup of `state.players[opponentId].board` — this matters because the opponent might have moved on/been eliminated by the time this serializes, but the snapshot freezes exactly what was fought against) and basic opponent identity fields (falling back to zeroed/null values + `id: -1` if `opponentId` is `null`, i.e. a ghost pairing).
- `serializeState(state, seatIndex, names)` — the FULL `STATE_SNAPSHOT` payload: `round, phase, me: state.players[seatIndex]` (the RAW, unredacted `PlayerState` for the recipient's own seat — including gold/shop/items, since it's their own data), `players: [...serializePlayerPublic(...)]` (the redacted public view for ALL seats, including the recipient's own — so `me` and the corresponding `players[seatIndex]` entry both exist with different levels of detail), `lastPairings`.
- `serializeDelta(state, changedSeat, recipientSeat, names)` — the `STATE_DELTA` payload: always includes the full `players: [...serializePlayerPublic(...)]` array (so every recipient can see public changes to ANY seat), and CONDITIONALLY spreads in `me: state.players[changedSeat]` ONLY IF `changedSeat === recipientSeat` (i.e. you get your own private full state only when YOU were the one who just acted) — implementing the "private fields go only to the acting player's own seat" comment.
- `broadcastDelta(room, changedSeat)` — sends a `STATE_DELTA` to EVERY non-AFK human seat (not just the actor), each one separately serialized via `serializeDelta` with that recipient's own `recipientSeat` — meaning every command produces up to 8 separate per-seat-tailored message sends (not one shared broadcast payload), since the "is this `me`" decision differs per recipient.
- `broadcastAll(room, msg)` — encodes the message ONCE (`encode(msg)`, from `@autobattler/protocol`) and sends the SAME encoded string to every seat with an OPEN websocket — used for messages with no per-seat-private content (`PHASE_CHANGE`, `COMBAT_START`, `COMBAT_RESULT`, `MATCH_END`) — contrast with `LOOT`/`ROUND_RESULT`/`STATE_SNAPSHOT`/`STATE_DELTA`, which are always built+sent individually per recipient via `send()` (from `session.ts`) since they carry per-seat-private data.

**Invariants & constraints**
- **The room's `prng` is one continuous seeded stream for the WHOLE match** — both bot/AFK AI decisions (`applyAiCommands`) and the room's own seed derivation draw from it; this is essential for reproducibility/determinism (though note the room's OWN seed (`createRoom`'s `Date.now() ^ Math.random()*0xffffffff`) is itself non-deterministic across real matches — determinism is only about "given a fixed seed, the WHOLE match is reproducible," not "every match is identical").
- A seat with `seats[i] === null` (bot) is functionally IDENTICAL, for AI-driving purposes, to a seat with `seats[i].afk === true` (disconnected human) — both get `applyAiCommands` called for them every round. This means an AFK human's bot takes over seamlessly and indefinitely until/unless they `RECONNECT`.
- `handlePlayerReady`/`markReady` can ONLY affect the `RESOLUTION` phase's wait, never `PLANNING`'s — a reader implementing any "ready check" UI logic must know the 30s planning timer is NEVER skippable via READY (despite what CLAUDE.md's older client-doc text says about "READY from all human seats skips the 30s wait" — **this is a discrepancy worth flagging**: the CLAUDE.md `packages/server` internals section states "READY from all human seats skips the 30s wait," but the actual code's `handlePlayerReady` comment explicitly says the OPPOSITE — "Planning runs its full duration... READY is a no-op for PLANNING" — only RESOLUTION is skippable. A reader should trust the CODE (and this doc) over that specific CLAUDE.md line, which appears to describe an EARLIER behavior that was since changed (the comment in the code itself references this explicitly: "READY no longer skips it").
- `finalizeMatch`'s persistence failure is swallowed (logged, not re-thrown) — a database outage at match-end does NOT prevent players from seeing their `MATCH_END` results (though `mmr` would be empty in that case, meaning no MMR change is shown — a degraded-but-graceful failure mode).
- Room cleanup (`rooms.delete`) happens ONLY in `finalizeMatch` — a room that never reaches match-over (e.g. all 8 players disconnect mid-match without an explicit forfeit mechanism) would remain in the `rooms` Map indefinitely, continuing to run its phase timers (bots playing out the match alone) until it naturally concludes — there's no separate "abandon room" cleanup path visible in this file.
- `seatNames`/`seatAccounts` are captured ONCE at `createRoom` time and never updated — if a human's profile name changes mid-match (via `PATCH /profile`), the room continues broadcasting their ORIGINAL name for the rest of that match (a deliberate "names are public via the leaderboard... set once per match" design per the comment, not a bug).

**Depends on** — `@autobattler/data` (`gameData`); `@autobattler/protocol` (`encode`, `S2CMessage`, `MatchStats` types); `@autobattler/rules` (`createMatch`, `advancePhase`, `isMatchOver`) plus deep-path imports `@autobattler/rules/src/commands.js` (`applyCommand`, `Command` type), `@autobattler/rules/src/ai.js` (`applyAiCommands`), `@autobattler/rules/src/rounds.js` (`isPveRound`), `@autobattler/rules/src/state.js` (`MatchState`, `PlayerState` types); `@autobattler/sim/src/prng.js` (`mulberry32`); `node:crypto` (`randomUUID`, for `matchId`); `./session.js` (`Session` type, `send`, `registerSeatToken`, `clearRoomSeatTokens`); `./db/index.js` (`Repository` type, `MemoryRepository` default); `./recorder.js` (`recordMatchResult`, `SeatResult` type).

**Used by** — `packages/server/src/matchmaker.ts` (`createRoom`); `packages/server/src/index.ts` (`getRoom`, `handlePlayerCommand`, `markReady`, `reconnectSession`, `setRoomRepository`); `packages/server/tests/room.test.ts` / `integration.test.ts` (presumably drive rooms directly via `createRoom` + fake timers, using `PLANNING_MS_OVERRIDE` to shorten real-time waits).

**Notes**
- The "deep import" pattern (`@autobattler/rules/src/commands.js` instead of the package's public barrel export) suggests `Command`/`applyAiCommands`/`isPveRound` aren't (or weren't, at the time this file was written) re-exported from `@autobattler/rules`'s top-level index — worth checking the rules package's `index.ts` if a future refactor wants to clean this up to use the public API uniformly.
- The CLAUDE.md vs. code discrepancy on READY-skipping-planning (noted above) is the most actionable finding in this file for a future agent — anyone relying on the CLAUDE.md prose description of "READY skips 30s wait" for either client or server work should re-verify against this file's actual `handlePlayerReady` logic, since the code (and its own comment) directly contradicts that specific claim.
- `validateCommand`'s missing `UNEQUIP`/`COMBINE_ITEMS` cases is the second actionable finding — if the ONLINE item-combine/unequip UX doesn't work in practice while Practice mode's does, this function is the first place to check.
