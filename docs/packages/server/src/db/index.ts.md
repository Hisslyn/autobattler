# packages/server/src/db/index.ts

**Path & purpose** ‚Äî `packages/server/src/db/index.ts`. The `db/` subpackage's barrel/entry module: re-exports the `Repository` interface and its two concrete implementations, plus the `createRepository()` factory that picks the right one at server startup based on environment.

**Responsibility** ‚Äî Owns the single decision point for "which persistence backend does this server instance use" ‚Äî every other server module (`room.ts`, `recorder.ts`, `http.ts`, etc.) should obtain its `Repository` instance via `createRepository()` rather than importing `MemoryRepository`/`PostgresRepository` directly and choosing themselves, keeping the env-based selection logic centralized.

**Exports**
- Re-exported types: `Repository` (the persistence interface ‚Äî `createGuest`, `findByToken`, `getProfile`, `updateProfile`, `recordMatch`, `leaderboard`, `matchHistory`, `close`), `Profile` (`{accountId, name, mmr}`), `MatchPlayerRecord` (`{accountId: string|null, seat, placement, mmrBefore: number|null, mmrAfter: number|null}` ‚Äî `accountId`/`mmrBefore`/`mmrAfter` are `null` for bots, which are never persisted as accounts), `MatchHistoryEntry` (`{matchId, dataVersion, endedAt (epoch ms), seat, placement, mmrBefore, mmrAfter}`) ‚Äî all defined in `./repo.js`, just re-exported here for a single import surface.
- `MemoryRepository` (re-exported class from `./memory.js`) ‚Äî the default/dev/test in-process implementation.
- `PostgresRepository` (re-exported class from `./postgres.js`) ‚Äî the Postgres-backed implementation used when `DATABASE_URL` is configured.
- `createRepository(): Promise<Repository>` ‚Äî the factory: reads `process.env["DATABASE_URL"]`; if set, constructs a `new PostgresRepository(url)`, calls `repo.ensureSchema()` (applying `schema.sql` idempotently) and awaits it, logs `"[db] using postgres"`, and returns it; if unset, logs `"[db] DATABASE_URL not set, using in-memory repository"` and returns a fresh `new MemoryRepository()`.

**Key behavior** ‚Äî This is the ONLY place in the codebase that decides Postgres-vs-memory based on environment; the choice is made ONCE at server startup (whoever calls `createRepository()`, presumably `index.ts`'s server bootstrap) and the resulting `Repository` instance is threaded through the rest of the server's lifetime. `ensureSchema()` is called eagerly on Postgres selection ‚Äî meaning a `PostgresRepository` is never returned to the caller without its schema already having been applied (no separate manual migration step needed at runtime; `schema.sql` is designed to be idempotent per CLAUDE.md, safe to re-run on every server start).

**Invariants & constraints** ‚Äî The console logging (`"[db] using postgres"` / `"[db] DATABASE_URL not set..."`) is the only operational visibility into which backend is active ‚Äî useful for a reader debugging a deployment to check server startup logs rather than needing to inspect environment variables directly. Both branches return a `Promise<Repository>` (the function itself is `async`), so callers must `await createRepository()` ‚Äî relevant since `PostgresRepository`'s `ensureSchema()` involves real I/O (a query against the database) that must complete before the repository is considered ready.

**Depends on** ‚Äî `./repo.js` (the `Repository` interface + its type re-exports ‚Äî type-only import of `Repository` itself, runtime + type for the rest via the `export type {...}` / structural re-export); `./memory.js` (`MemoryRepository`); `./postgres.js` (`PostgresRepository`); `process.env["DATABASE_URL"]`.

**Used by** ‚Äî `packages/server/src/index.ts` (presumably calls `createRepository()` once at server bootstrap to obtain the `Repository` instance passed into `room.ts`/`recorder.ts`/`http.ts`); any other server module importing `Repository`-related types likely imports from this barrel (`./db/index.js`) rather than reaching into `./db/repo.js` directly, per the barrel-module convention.

**Notes** ‚Äî Since `packages/server` is one of the few I/O-permitted packages in the system (CLAUDE.md restricts pure I/O permission narrowly, though `server` is inherently I/O-heavy by its nature as the network-facing process ‚Äî this is distinct from the STRICTER "packages/balance is the ONLY package permitted I/O" rule which applies specifically among `sim`/`rules`/`balance`), this module's environment-variable read and console logging are unremarkable/expected here, unlike if similar code appeared in `sim` or `rules`.
