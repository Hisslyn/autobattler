# packages/server/src/db/repo.ts

**Path & purpose** — `packages/server/src/db/repo.ts`. Defines the `Repository` interface — the single abstraction every persistence concern in `packages/server` must go through — plus its supporting data shapes (`Profile`, `MatchPlayerRecord`, `MatchHistoryEntry`). Pure type declarations, no implementation.

**Responsibility** — Owns the persistence CONTRACT that both `MemoryRepository` and `PostgresRepository` implement identically (modulo the subtle tie-break/token-durability differences documented on those files) — this is the abstraction boundary that lets the rest of the server (`room.ts`, `recorder.ts`, `http.ts`) remain agnostic to which backend is active, per CLAUDE.md's "all persistence writes go through the Repository interface" hard invariant.

**Exports**
- `interface Profile { accountId: string; name: string; mmr: number }` — a player's persistent identity: their account id, display name, and current MMR (Elo-like rating).
- `interface MatchPlayerRecord { accountId: string | null; seat: number; placement: number; mmrBefore: number | null; mmrAfter: number | null }` — one seat's outcome from a just-finished match, as written via `recordMatch`. `accountId`/`mmrBefore`/`mmrAfter` are `string|null`/`number|null` SPECIFICALLY to represent bots (`null`) vs real humans (non-null) — bots occupy a seat and get a `placement`, but never an account or MMR delta. `placement: 1 = winner` (explicit doc comment — placement is 1-indexed, best=1).
- `interface MatchHistoryEntry { matchId: string; dataVersion: string; endedAt: number; seat: number; placement: number; mmrBefore: number | null; mmrAfter: number | null }` — one row of a player's match history, as returned by `matchHistory()`; `endedAt` is epoch milliseconds (explicit doc comment); `dataVersion` records which `packages/data` content version was active when this match was played (the wider system's "DATA_VERSION recorded on every persisted match" rule from `packages/data`'s loader).
- `interface Repository` — the full persistence contract:
  - `createGuest(deviceId: string, name?: string): Promise<{accountId: string, token: string}>` — doc comment: "Idempotent per deviceId: returns the existing account if one exists."
  - `findByToken(token: string): Promise<{accountId: string} | null>` — resolves a previously-issued token back to its owning account, or `null` if unrecognized.
  - `getProfile(accountId: string): Promise<Profile | null>`.
  - `updateProfile(accountId: string, patch: {name?: string, mmr?: number}): Promise<void>` — a partial patch; omitted fields are left unchanged (both implementations honor this via different mechanisms — JS field-presence checks in `MemoryRepository`, SQL `COALESCE` in `PostgresRepository`).
  - `recordMatch(matchId: string, dataVersion: string, players: MatchPlayerRecord[]): Promise<void>` — the interface-level doc comment specifies: "recordMatch persists the match and applies every non-bot player's mmrAfter to their profile atomically (single transaction in the postgres impl)" — explicitly calling out that ATOMICITY is a REQUIRED property of any conforming implementation, not just an implementation detail of one of them.
  - `leaderboard(n: number): Promise<Profile[]>` — top `n` profiles (by MMR, descending, per both implementations, though the interface itself doesn't literally specify the sort — that's an implicit contract inferred from "leaderboard" semantics and confirmed by both implementations' actual code).
  - `matchHistory(accountId: string, limit: number): Promise<MatchHistoryEntry[]>` — most-recent-first (again, an implicit but consistently-implemented ordering).
  - `close(): Promise<void>` — releases any held resources (a connection pool for Postgres; a no-op for in-memory).
- The doc comment directly above the interface also states the system-wide invariant: "bots (accountId null) never get accounts/profiles" — this is the canonical place that rule is declared as a TYPE-LEVEL contract (via the `string | null` typing on `MatchPlayerRecord.accountId`/`mmrBefore`/`mmrAfter`), not just a prose note elsewhere.

**Key behavior** — N/A (pure interface/type declarations; no runtime logic in this file at all).

**Invariants & constraints** — This file is the canonical SOURCE OF TRUTH for what "the Repository interface" means throughout the codebase — any reader needing to know the exact shape of a `Profile`/`MatchPlayerRecord`/`MatchHistoryEntry`, or what operations a `Repository` must support, should read THIS file rather than inferring from one implementation (since the implementations could, in principle, diverge subtly — as already noted for `leaderboard`'s tie-break ordering between `Memory`/`Postgres`). Any future repository backend (e.g. a third implementation) MUST implement every method here with the exact same async signatures to be usable via `createRepository()`'s return type.

**Depends on** — Nothing (no imports) — a self-contained type-definition file.

**Used by** — `packages/server/src/db/memory.ts` (`MemoryRepository implements Repository`); `packages/server/src/db/postgres.ts` (`PostgresRepository implements Repository`); `packages/server/src/db/index.ts` (re-exports all these types + uses `Repository` as `createRepository`'s return type); `packages/server/src/recorder.ts` (presumably takes a `Repository` parameter, per CLAUDE.md's `recordMatchResult(repo, matchId, seats)` signature); `packages/server/src/http.ts` (uses `Repository` methods for the guest-auth/profile/leaderboard/history endpoints); `packages/server/tests/repo.test.ts` (the shared contract test suite running against both concrete implementations via this common interface).

**Notes** — The doc comments embedded directly in this interface (the idempotency note on `createGuest`, the atomicity note on `recordMatch`, the bot-exclusion note above the interface) are NOT just casual code comments — they function as the BEHAVIORAL SPECIFICATION that both implementations are tested against (the shared contract suite). A reader implementing a new `Repository` backend, or auditing an existing one for compliance, should treat every doc comment here as a hard requirement, not a suggestion.
