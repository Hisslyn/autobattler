# packages/server/src/db/memory.ts

**Path & purpose** — `packages/server/src/db/memory.ts`. The in-process, in-memory `Repository` implementation — the default persistence backend when `DATABASE_URL` is unset (dev/tests), holding all accounts/profiles/match history purely in JS `Map`/array fields that vanish on process restart.

**Responsibility** — Implements the full `Repository` contract (`./repo.js`) without any external database, so the server (and its test suite) can run end-to-end without provisioning Postgres. Functionally mirrors `PostgresRepository`'s observable behavior (used by the shared contract test suite, per CLAUDE.md: "contract test suite runs against both") but with simple synchronous data structures wrapped in `async` methods to satisfy the interface.

**Exports**
- `class MemoryRepository implements Repository` — the only export. Private fields: `accountsByDevice: Map<deviceId, accountId>`, `tokens: Map<token, accountId>`, `profiles: Map<accountId, Profile>`, `matches: MatchRow[]` (an internal type `{matchId, dataVersion, endedAt, players: MatchPlayerRecord[]}`, append-only).

**Key behavior**
- `createGuest(deviceId, name?)`: IDEMPOTENT per `deviceId` — if `accountsByDevice` already has an entry for this device, returns the EXISTING `accountId` + a FRESH `signToken(existing)` call (tokens aren't cached/reused per call, but since `signToken` is a pure deterministic function of `accountId`, re-signing produces the identical token string anyway) — and if a `name` was passed on a repeat call, it OVERWRITES the existing profile's name (so calling `createGuest` again with a new name effectively renames the guest, a side effect worth knowing about for any caller). On first call for a device: generates a fresh `accountId` via `randomUUID()`, signs a token, registers both the device→account and token→account mappings, and creates a new `Profile` with `mmr: gameData.economy.mmrStart` (the data-driven starting MMR, NOT a hardcoded literal — consistent with the project's data-driven economy invariant) and `name` defaulting to `"Guest"` if none provided.
- `findByToken(token)`: a simple `Map` lookup — returns `{accountId}` or `null`. Notably this does NOT call `verifyToken`/HMAC-validate the token itself — it trusts that the token exists in its OWN `tokens` map (populated only by `createGuest`), meaning this repository implicitly trusts any caller already validated the token's signature upstream (consistent with `auth.ts`'s `verifyToken` being called separately by `http.ts`/session logic BEFORE consulting the repository, OR this lookup serves as an independent double-check that the token was actually issued by THIS repository instance, not just any HMAC-valid string).
- `getProfile(accountId)` / `updateProfile(accountId, patch)`: straightforward map lookups; `getProfile` returns a SHALLOW COPY (`{...p}`) to prevent external mutation of the internal store via the returned object; `updateProfile` is a no-op (silently returns) if the accountId doesn't exist — does NOT throw or signal failure.
- `recordMatch(matchId, dataVersion, players)`: appends a new immutable-ish `MatchRow` (player records SHALLOW-COPIED via `.map(p => ({...p}))`, `endedAt: Date.now()`) to the `matches` array, THEN for every player with a non-null `accountId` AND non-null `mmrAfter` (i.e. real human accounts, never bots), updates that profile's `mmr` to the new value — this is the "atomic" MMR-update-plus-match-record behavior described in CLAUDE.md's repo.ts note, though here "atomic" is trivially true since it's all synchronous in-process JS with no concurrent access (unlike the Postgres version, which needs an explicit transaction).
- `leaderboard(n)`: returns the top `n` profiles by descending `mmr`, each shallow-copied — a fresh sort+slice on every call (no caching), fine for in-memory/test scale.
- `matchHistory(accountId, limit)`: walks `matches` in REVERSE insertion order (most recent first, since `matches` is append-only and this iterates from the end), finds this account's `MatchPlayerRecord` within each match (if present — i.e. this account participated), and collects up to `limit` `MatchHistoryEntry` objects — stops early once `limit` is reached (`out.length < limit` in the loop condition), so this isn't a full O(n) scan if the account was in a recent match and the limit is small.
- `close()`: a no-op `async` stub — nothing to release for an in-memory store, but satisfies the `Repository` interface's `close(): Promise<void>` contract uniformly with `PostgresRepository` (which presumably closes a real connection pool).

**Invariants & constraints**
- ALL STATE IS LOST ON PROCESS RESTART — by design, this is the dev/test fallback, never intended for any deployment needing persistence across restarts.
- Bots (per the system's "bots never persist" invariant) are naturally excluded from MMR updates here since `recordMatch` explicitly checks `accountId !== null` — a caller passing a `MatchPlayerRecord` with `accountId: null` (a bot) simply skips the profile-mmr-update branch, and since bots never call `createGuest`, they never have a `profiles` entry to update anyway.
- `findByToken` trusting its own map (rather than re-verifying HMAC) means a MemoryRepository instance's tokens are tied to ITS OWN in-memory `tokens` map — restarting the server (which creates a fresh `MemoryRepository`) invalidates every previously-issued token from the caller's perspective (the map is empty again), even though `verifyToken`'s HMAC check alone would still pass (since `AUTH_SECRET` is env-based, not instance-based) — this is a meaningful behavioral difference from `PostgresRepository`, which presumably persists tokens (or accountId mappings) durably and would still recognize old tokens after a server restart.

**Depends on** — Node's `node:crypto` (`randomUUID`); `@autobattler/data` (`gameData`, specifically `economy.mmrStart`); `../auth.js` (`signToken`); `./repo.js` (`Repository` interface + its supporting types, type-only imports).

**Used by** — `packages/server/src/db/index.ts`'s `createRepository()` (the fallback branch when `DATABASE_URL` is unset); the shared repository contract test suite (`packages/server/tests/repo.test.ts`, run against both `MemoryRepository` and `PostgresRepository` per CLAUDE.md).

**Notes** — The `findByToken`-trusts-own-map behavior (rather than re-deriving via `verifyToken`) is a subtle design choice worth flagging for any future agent comparing `MemoryRepository` and `PostgresRepository` behavior for parity — if `PostgresRepository` ALSO independently tracks issued tokens (rather than relying purely on HMAC verification), the two implementations are consistent; if `PostgresRepository` instead re-derives via `verifyToken` and looks up by the decoded accountId, there could be a SUBTLE divergence in behavior after a process restart (Postgres "remembers" tokens issued before a restart via the HMAC trick, while in-memory does not) — worth checking `postgres.ts`'s `findByToken` implementation directly if this distinction ever matters for a bug investigation.
