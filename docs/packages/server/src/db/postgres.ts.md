# packages/server/src/db/postgres.ts

**Path & purpose** — `packages/server/src/db/postgres.ts`. The Postgres-backed `Repository` implementation, used when `DATABASE_URL` is set — persists accounts/profiles/match history durably via a `pg.Pool`, with `recordMatch` and the guest-creation path wrapped in real SQL transactions.

**Responsibility** — Implements the full `Repository` contract against a real Postgres database, providing durable persistence across server restarts (unlike `MemoryRepository`). Owns the schema-application bootstrapping (`ensureSchema`) and all SQL query construction for every repository operation.

**Exports**
- `class PostgresRepository implements Repository` — constructor takes a `connectionString: string` and creates a `pg.Pool({connectionString})` (connection pooling, not a single persistent connection — concurrent requests get separate pooled connections).
- `ensureSchema(): Promise<void>` — an EXTRA method not part of the `Repository` interface itself (interface-implementers can have additional members): reads `schema.sql` (resolved relative to this file's own directory via `import.meta.url`→`fileURLToPath`→`dirname`, NOT relative to `process.cwd()` — robust regardless of where the server process is launched from) and runs the ENTIRE file content as one `pool.query(sql)` call. Per CLAUDE.md, `schema.sql` consists of idempotent `CREATE ... IF NOT EXISTS` statements, so calling this repeatedly (e.g. on every server start) is safe.

**Key behavior — method by method**
- `createGuest(deviceId, name?)`: first queries `accounts` by `device_id`; if a row exists, returns its `account_id`+`token` UNCHANGED (the token is NOT re-signed on each call, unlike `MemoryRepository` — Postgres stores the ORIGINAL token value durably in the `accounts.token` column rather than recomputing it), and if a `name` was passed, updates the profile's name via a separate `updateProfile` call. If no existing row: generates a fresh `accountId`+token, then INSERTS into both `accounts` (`account_id, device_id, token`) and `profiles` (`account_id, name, mmr` — `mmr` from the data-driven `gameData.economy.mmrStart`) inside an explicit `BEGIN`/`COMMIT` transaction (with `ROLLBACK` + re-throw on any error) using a CHECKED-OUT client (`pool.connect()`, manually `release()`d in a `finally`) — ensures the account row and profile row are created atomically (never one without the other).
- `findByToken(token)`: a direct `SELECT account_id FROM accounts WHERE token = $1` — trusts its OWN stored token column (durable across restarts, unlike `MemoryRepository`'s in-process map which resets on restart) — confirms the earlier hypothesis from `memory.ts`'s doc: Postgres durably remembers issued tokens via the `accounts` table, so a token issued before a server restart STILL resolves correctly after restart (a real behavioral difference from `MemoryRepository`, where restarting wipes the token map even though `verifyToken`'s HMAC check alone would still pass).
- `getProfile(accountId)`: simple `SELECT` by `account_id`, mapping the snake_case row to the camelCase `Profile` shape; `null` if no row.
- `updateProfile(accountId, patch)`: early-returns (no query at all) if BOTH `patch.name` and `patch.mmr` are undefined (avoids a no-op UPDATE round-trip); otherwise runs an `UPDATE ... SET name = COALESCE($2, name), mmr = COALESCE($3, mmr), updated_at = now()` — the `COALESCE` pattern means passing `undefined` for either field (mapped to SQL `null` via `?? null`) leaves that column UNCHANGED rather than nulling it out — a single flexible query handles "update name only," "update mmr only," or "update both" without branching SQL.
- `recordMatch(matchId, dataVersion, players)`: a full transaction (`BEGIN`/`COMMIT`/`ROLLBACK`-on-error, same checked-out-client pattern as `createGuest`): inserts one `matches` row (`match_id, data_version`), then for EACH player inserts one `match_players` row (`match_id, seat, account_id, placement, mmr_before, mmr_after` — `account_id`/`mmr_before`/`mmr_after` can be SQL `null` for bots, since `MatchPlayerRecord.accountId` is typed `string | null`), and IMMEDIATELY AFTER each insert, if that player is a real account with a non-null `mmrAfter`, runs an `UPDATE profiles SET mmr = ...` for that account — all within the SAME transaction, so the match record + every profile's MMR update commit or roll back together atomically (the "atomic" behavior CLAUDE.md's `recorder.ts` note depends on: "writes match + MMR through the repo" in one transaction).
- `leaderboard(n)`: `SELECT ... ORDER BY mmr DESC, account_id LIMIT $1` — the `, account_id` secondary sort key makes tie-breaking on equal MMR DETERMINISTIC (stable ordering for players with identical MMR, rather than leaving tie order to Postgres's unspecified default), a detail `MemoryRepository`'s plain `.sort((a,b) => b.mmr - a.mmr)` does NOT replicate (Array.sort's stability for equal elements depends on the JS engine, though V8 is stable — still, this is a SUBTLE divergence in tie-break ordering between the two repository implementations worth flagging for the contract test suite or any reader comparing their exact output for equal-MMR players).
- `matchHistory(accountId, limit)`: a JOIN between `match_players` and `matches` (matching on `match_id`), filtered to this account's `account_id`, ordered by `ended_at DESC`, limited — maps `ended_at` (a SQL `Date` driver value) to an epoch-ms number via `.getTime()` to match the `MatchHistoryEntry.endedAt: number` contract.
- `close()`: `await this.pool.end()` — drains/closes the connection pool, the real resource-release counterpart to `MemoryRepository`'s no-op `close()`.

**Invariants & constraints**
- Every multi-statement write path (`createGuest`'s insert-pair, `recordMatch`'s match+players+profile-updates) uses an explicit transaction with rollback-on-error — no write path here can leave the database in a partially-updated state from a single logical operation.
- `ensureSchema` re-running the WHOLE `schema.sql` file on every server start is safe ONLY because `schema.sql` is written with idempotent `CREATE ... IF NOT EXISTS` statements (per CLAUDE.md) — this file's correctness depends on that invariant holding in `schema.sql`, which this module doesn't itself enforce (it just executes whatever SQL it's given).
- A connecting client is always `.release()`d in a `finally` block for both `createGuest` and `recordMatch` — even on error (after the `ROLLBACK` + re-throw), the client is returned to the pool, preventing pool exhaustion from accumulated leaked connections after repeated errors.
- The `mmr_before`/`mmr_after` columns and the `MatchPlayerRecord` type both allow `null` specifically to represent bots (consistent with CLAUDE.md: "accountId null in match_players" for bots) — `recordMatch`'s per-player profile-mmr-update loop explicitly skips any player with `accountId === null`, so a bot's match-row is recorded (for historical/audit completeness) but never causes a profile lookup/update (since none exists for a bot).

**Depends on** — Node's `node:crypto` (`randomUUID`), `node:fs` (`readFileSync`), `node:path` (`dirname`, `resolve`), `node:url` (`fileURLToPath`); `pg` (the `Pool`/query client); `@autobattler/data` (`gameData.economy.mmrStart`); `../auth.js` (`signToken`); `./repo.js` (`Repository` + supporting types); `./schema.sql` (read as a file at `ensureSchema()` time, not imported as a module — a sibling file in the same directory).

**Used by** — `packages/server/src/db/index.ts`'s `createRepository()` (the Postgres branch, when `DATABASE_URL` is set); the shared repository contract test suite (`packages/server/tests/repo.test.ts`), gated to run only when `DATABASE_URL` is actually configured in the test environment (per CLAUDE.md: "the postgres repo contract tests run only when DATABASE_URL is set, skipped + logged otherwise").

**Notes** — The `leaderboard` tie-break-by-`account_id` detail (absent from `MemoryRepository`'s sort) is the one notable behavioral asymmetry between the two `Repository` implementations spotted in this pass — a future agent debugging a leaderboard-ordering discrepancy between dev (in-memory) and prod (Postgres) for players with identical MMR should check this first. `ensureSchema`'s path resolution via `import.meta.url` (rather than `process.cwd()`-relative) means `schema.sql` MUST physically live alongside the compiled/run `postgres.ts`/`postgres.js` (same directory) — moving `schema.sql` without updating this resolution would silently break schema application.
