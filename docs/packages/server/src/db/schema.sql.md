# packages/server/src/db/schema.sql

**Path & purpose** — `packages/server/src/db/schema.sql`. The complete, idempotent Postgres schema for the server's persistence layer — four tables (`accounts`, `profiles`, `matches`, `match_players`) plus two supporting indexes. Applied via `PostgresRepository.ensureSchema()` on every server start when `DATABASE_URL` is set.

**Responsibility** — Owns the canonical relational shape backing the `Repository` interface's Postgres implementation — every column referenced by `postgres.ts`'s SQL queries is defined here, and this file IS the migration system (CLAUDE.md: "migrations = idempotent schema.sql").

**Exports** — N/A (SQL DDL file, not a code module). Defines:
- `accounts (account_id UUID PRIMARY KEY, device_id TEXT NOT NULL UNIQUE, token TEXT NOT NULL UNIQUE, created_at TIMESTAMPTZ DEFAULT now())` — one row per guest account; `device_id` UNIQUE enforces `createGuest`'s idempotency-per-device at the DATABASE level (not just application logic — a duplicate-device insert would fail the unique constraint, though `postgres.ts`'s `createGuest` already checks for an existing row first and never attempts a duplicate insert in practice); `token` UNIQUE supports `findByToken`'s lookup efficiently and guarantees no two accounts ever share a token value.
- `profiles (account_id UUID PRIMARY KEY REFERENCES accounts(account_id), name TEXT NOT NULL, mmr INTEGER NOT NULL DEFAULT 1000, created_at, updated_at)` — one-to-one with `accounts` (same PK, foreign-keyed) holding the mutable display name + MMR; `mmr INTEGER` confirms MMR is stored as a plain integer (no fixed-point scaling needed here, unlike `sim`'s stat values — MMR is already integer-valued per `economy.json`'s `mmrK`/`mmrStart`/`mmrEloDivisor` constants); the column default `1000` is a SQL-level fallback distinct from (but consistent with) `economy.json`'s `mmrStart` constant — `postgres.ts`'s `createGuest` always explicitly supplies `gameData.economy.mmrStart` on insert, so this column default would only matter for some hypothetical insert path that omits the column entirely (not currently exercised by any code in this pass).
- `matches (match_id TEXT PRIMARY KEY, data_version TEXT NOT NULL, ended_at TIMESTAMPTZ DEFAULT now())` — one row per completed match; `data_version` records the `packages/data` content version active at match end (matching `MatchHistoryEntry.dataVersion`).
- `match_players (match_id TEXT REFERENCES matches, seat INTEGER, account_id UUID REFERENCES accounts NULLABLE, placement INTEGER, mmr_before INTEGER NULLABLE, mmr_after INTEGER NULLABLE, PRIMARY KEY (match_id, seat))` — one row per SEAT per match (composite PK on `match_id+seat`, since a seat number is only unique within a single match, not globally); `account_id`/`mmr_before`/`mmr_after` are explicitly NULLABLE (no `NOT NULL` constraint, unlike most other columns) with an inline comment `-- NULL for bots` — the schema-level encoding of the "bots never persist accounts/MMR" rule, matching `MatchPlayerRecord`'s `string|null`/`number|null` TypeScript typing exactly.
- Indexes: `idx_profiles_mmr ON profiles (mmr DESC)` (accelerates `leaderboard`'s `ORDER BY mmr DESC` query); `idx_match_players_account ON match_players (account_id)` (accelerates `matchHistory`'s `WHERE mp.account_id = $1` filter).

**Key behavior** — Every `CREATE TABLE`/`CREATE INDEX` statement uses `IF NOT EXISTS`, making the entire file safely RE-RUNNABLE on every server start (idempotent migrations, per CLAUDE.md) — no separate migration-versioning system, no migration history table; the schema's CURRENT desired state is just re-asserted every time, and Postgres no-ops on anything that already exists.

**Invariants & constraints**
- This idempotent-DDL approach means SCHEMA CHANGES (e.g. adding a column to an existing table) CANNOT be expressed as a simple edit to this file — `CREATE TABLE IF NOT EXISTS` won't add a missing column to an already-existing table. Any future schema evolution beyond "add a wholly new table/index" would need either a manual `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` statement appended here, or a more sophisticated migration system — this is a real constraint a future agent must respect: editing an EXISTING `CREATE TABLE` block in this file does NOT retroactively alter already-provisioned databases.
- Foreign keys (`profiles.account_id → accounts`, `match_players.match_id → matches`, `match_players.account_id → accounts`) enforce referential integrity at the database level — e.g. `match_players` can never reference a nonexistent match or (for non-bot rows) a nonexistent account.
- `match_players`'s PK is the COMPOSITE `(match_id, seat)`, NOT `account_id` — correctly modeling that a single match has up to 8 seats and a seat can be occupied by a bot (no account) — this is why `account_id` can't be part of the primary key (bots would collide on a `NULL` PK value, which isn't valid for a PK anyway).

**Depends on** — Nothing (pure SQL, no application code dependencies) — though its column/table names and types are tightly coupled to `postgres.ts`'s hand-written SQL queries (which use snake_case column names matching this file exactly) and to `repo.ts`'s TypeScript interfaces (camelCase equivalents).

**Used by** — `packages/server/src/db/postgres.ts`'s `ensureSchema()` (reads this file's full text and executes it as one query against the connected Postgres database); `docker compose up -d` (per CLAUDE.md, brings up a local `postgres:16` instance that this schema is then applied to via the server's own startup, not via a separate migration command — there's no standalone "run migrations" script, `ensureSchema()` IS the migration runner).

**Notes** — Every table/column here maps 1:1 to a `Repository` method's SQL in `postgres.ts` — a reader cross-referencing "what does `recordMatch` actually persist" should read `postgres.ts`'s `recordMatch` query alongside this file's `matches`/`match_players`/`profiles` definitions together, since neither file alone tells the full story (the application code decides WHEN/WHAT to write; this schema defines WHERE it can be written and what integrity constraints apply).
