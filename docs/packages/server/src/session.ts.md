# packages/server/src/session.ts

**Path & purpose** — `packages/server/src/session.ts`. The connection registry: defines the `Session` type (one per live WebSocket connection), tracks three separate token-keyed lookup maps (session tokens, seat-reconnect tokens), implements the per-connection CMD rate limiter, and provides the single `send()` helper for encoding+dispatching one `S2CMessage` to one session.

**Responsibility** — Owns connection-level (not match-level) state: who is connected, what room/seat they currently occupy (if any), their account identity (once authenticated), their AFK flag, and their command-rate bookkeeping. Also owns the SEAT-RECONNECT token system that lets a disconnected human resume their exact seat in an in-progress match from a brand-new WebSocket connection.

**Exports**
- `interface Session { id: string; ws: WebSocket; roomId: string | null; seatIndex: number | null; token: string | null; accountId: string | null; name: string | null; afk: boolean; cmdCount: number; cmdWindowStart: number }` — one connection's full state. `accountId`/`name` are explicitly commented "set on authenticated QUEUE_JOIN" (i.e. `null` until the player has successfully authenticated via a `QUEUE_JOIN` carrying a valid `authToken` — `index.ts`'s handler sets these). `roomId`/`seatIndex` are `null` until seated in a room (`createRoom`/`reconnectSession` in `room.ts` set them).
- `createSession(ws: WebSocket): Session` — allocates a new `Session` with an auto-incrementing id (`s1`, `s2`, ...), all fields at their "fresh connection" defaults (`afk: false`, counters zeroed, `cmdWindowStart: Date.now()`), registers it in the module-level `sessions: Map<string, Session>`, and returns it.
- `removeSession(id: string): void` — deletes a session from the registry; if it had a registered SESSION token, also removes that token's reverse mapping (`tokenToSessionId`) — but does NOT touch the SEAT token map (`tokenToSeat`), which is a deliberately separate lifecycle (cleared only via `clearRoomSeatTokens`, on match end, not on disconnect).
- `getSession(id: string): Session | undefined` — registry lookup by session id.
- `registerToken(sessionId: string, token: string): void` — associates a SESSION token with a session: if the session already had a different token registered, that old token's reverse mapping is removed first (so a session never has two simultaneously-valid session tokens); sets `session.token` and the `tokenToSessionId` reverse map.
- `findSessionByToken(token: string): Session | undefined` — reverse lookup, session token → `Session`.
- `interface SeatClaim { roomId: string; seatIndex: number }` — what a SEAT token resolves to (which room, which seat). A doc comment directly above this clarifies the seat-token lifecycle: "Seat tokens survive disconnects: token → seat mapping is kept until the match ends so RECONNECT can restore the seat from a fresh connection."
- `registerSeatToken(token: string, roomId: string, seatIndex: number): void` — registers (or overwrites) a seat-claim mapping in `tokenToSeat`.
- `findSeatByToken(token: string): SeatClaim | undefined` — lookup used by the `RECONNECT` C2S handler in `index.ts` to resolve a client-supplied token back to a specific room+seat.
- `clearRoomSeatTokens(roomId: string): void` — iterates the ENTIRE `tokenToSeat` map and deletes every entry whose `claim.roomId` matches — an O(total active seat tokens across ALL rooms) sweep, called once by `room.ts`'s `finalizeMatch` when a match ends (invalidating reconnect capability for that match's tokens specifically, leaving other in-progress matches' tokens untouched).
- `send(session: Session, msg: S2CMessage): void` — the single canonical way to deliver one message to one session: checks `session.ws.readyState === 1` (OPEN) before calling `session.ws.send(encode(msg))` — silently no-ops if the socket isn't open (no error thrown, no queuing/buffering — a message sent to a momentarily-disconnected or not-yet-open socket is simply DROPPED, never retried).
- `checkRateLimit(session: Session): boolean` — sliding-window-ish rate limiter (see Key behavior) — returns `true` if the command is allowed, `false` (after sending a typed `RATE_LIMITED` error AND forcibly terminating the WebSocket) if the limit is exceeded.

**Key behavior**
- **Rate limiting algorithm** (`checkRateLimit`): `CMD_RATE_LIMIT = 20` commands per `CMD_WINDOW_MS = 1000` ms window. This is a FIXED (not sliding) window: if `now - cmdWindowStart >= 1000ms`, the window resets (`cmdCount = 0`, `cmdWindowStart = now`) BEFORE incrementing — meaning the actual enforcement is "max 20 commands in any 1-second-aligned-to-first-command-in-window bucket," not a true sliding window (a burst straddling a window boundary could in principle allow up to ~40 commands across a sub-2-second span: up to 20 right before a window resets, then 20 more right after) — this is a standard fixed-window rate-limiter tradeoff, not a bug, but worth knowing precisely for anyone reasoning about abuse-resistance guarantees.
- On exceeding the limit: NOT a graceful "drop and warn" — the session is sent a `RATE_LIMITED` error AND its WebSocket is immediately `.terminate()`d (a hard disconnect, not a graceful close) — this is a strict punitive response, not a soft throttle. The caller (`index.ts`) only invokes `checkRateLimit` for `CMD`-type messages (per its own dispatch logic), so `PING`/`READY`/`QUEUE_JOIN`/etc. are never rate-limited by this mechanism.
- **Three separate token maps, three separate purposes and lifecycles**: (1) `tokenToSessionId` — a SESSION token (generated fresh per WS connection by `index.ts`'s `generateToken()`, `Math.random`-based) identifying a specific live connection, cleared on disconnect (`removeSession`); (2) `tokenToSeat` — a SEAT-RECONNECT token (registered by `room.ts` at `createRoom` time, using the SAME token VALUE as the session's current session-token — see `room.ts`'s `registerSeatToken(s.token!, roomId, i)`, reusing `s.token` rather than minting a new distinct token) mapping to a `{roomId, seatIndex}` claim, cleared only at match end via `clearRoomSeatTokens`; (3) the AUTH token (NOT managed by this file at all — that's `auth.ts`'s `signToken`/`verifyToken`, resolved via the `Repository`'s `findByToken`, identifying the player's ACCOUNT). A reader must not confuse these — `findSessionByToken` and `findSeatByToken` operate on DIFFERENT maps even though, in practice, the same literal token STRING value is used as both the session token and (once registered into a room) the seat-claim token for a given connection — they happen to share a value but serve conceptually distinct lookups.

**Invariants & constraints**
- All session/token state is held in plain in-process `Map`s — restarting the server process wipes EVERY session, EVERY session token, and EVERY seat-reconnect token (no persistence here at all — this is appropriate, since sessions are inherently per-process/per-connection constructs, unlike `Repository`'s account/profile data which IS meant to survive restarts).
- `removeSession` deliberately does NOT clear seat tokens — this is intentional: a human's WebSocket dropping (e.g. brief network blip) should NOT immediately forfeit their seat-reconnect capability; only an explicit match-end (`clearRoomSeatTokens`) revokes it. This is the mechanism underpinning AFK-then-reconnect continuity.
- `send()` never queues or retries — if a message is sent to a session whose socket isn't currently `OPEN` (e.g. mid-reconnect, or already disconnected but not yet cleaned up), that message is permanently lost for that session — there is no "catch up on missed messages" mechanism visible in this file; a reconnecting session gets a fresh full `STATE_SNAPSHOT` (via `room.ts`'s `reconnectSession`) rather than replaying anything it missed, which is the correct mitigation for this drop-on-closed-socket behavior.

**Depends on** — `ws` (`WebSocket` type only); `@autobattler/protocol` (`encode`, `S2CMessage` type).

**Used by** — `packages/server/src/index.ts` (`createSession`, `removeSession`, `send`, `checkRateLimit`, `registerToken`, `findSeatByToken` — the full connection lifecycle + dispatch); `packages/server/src/room.ts` (`send`, `registerSeatToken`, `clearRoomSeatTokens`, plus the `Session` type throughout); `packages/server/src/matchmaker.ts` (`Session` type only, for the queue array's element type).

**Notes** — `findSessionByToken` and `getSession` are exported but were NOT observed being called from `index.ts`'s message dispatch in this pass (the dispatch logic primarily uses `findSeatByToken` for RECONNECT) — they may be used elsewhere (e.g. future reconnect-by-session-token flows, or test harnesses) or could be currently-unused exports; worth a quick grep if a future cleanup pass wants to confirm dead exports.
