# packages/server/src/index.ts

**Path & purpose** — `packages/server/src/index.ts`. The server's entry point / bootstrap module: creates the shared HTTP+WebSocket listener on one port, wires up the `Repository`, and implements the top-level per-connection message dispatch (auth, queueing, reconnect, in-match commands, heartbeat). This is the file actually run by `npm run server` (`tsx src/index.ts`, per `package.json`).

**Responsibility** — Owns process bootstrap (repository creation, HTTP server creation, WS server creation, `listen()`), the WS connection lifecycle (session creation/teardown, ping/pong heartbeat, AFK marking on disconnect), and the C2S message switch — routing each decoded message type to the appropriate `session.ts`/`matchmaker.ts`/`room.ts` function. This is the top of the server's call graph — nearly every other server module is invoked, directly or indirectly, from this file's `ws.on("message", ...)` handler.

**Exports** — None (this is a script/entry-point module, not an importable library — it runs side effects at module-load time via top-level `await` and `httpServer.listen(...)`, and is never `import`ed by any other module in the codebase, including tests, which instead import the individual pieces like `room.ts`/`matchmaker.ts` directly to test logic without a real network listener).

**Key behavior**
- **Bootstrap**: `const repo = await createRepository()` (picks Postgres or in-memory per `DATABASE_URL`, from `db/index.ts`); `setRoomRepository(repo)` (injects the repository into `room.ts`'s module state — `room.ts` needs it at match-end to call `recordMatchResult`); `createServer((req,res) => void createHttpHandler(repo)(req,res))` builds a raw Node HTTP server whose handler is the `http.ts` factory output (the `void` discards the returned promise — fire-and-forget, the handler internally catches all its own errors so there's nothing to await here); `new WebSocketServer({server: httpServer})` attaches the `ws` library's WebSocket upgrade handling to the SAME HTTP server instance — this is the literal mechanism behind "HTTP API + ws on the same port" (one `httpServer.listen(PORT)` call serves both).
- **`generateToken()`**: produces an opaque per-connection session token via two concatenated `Math.random().toString(36)` slices — NOTE: this is a DIFFERENT token from the auth token (`signToken`/`verifyToken` in `auth.ts`, HMAC-based) — this `generateToken` produces the WS SESSION token (registered via `session.ts`'s `registerToken(session.id, token)`, mapping token→sessionId for in-process reconnect bookkeeping), unrelated to account authentication. **This token uses `Math.random()`**, which is fine here since it's a server-side-only transient connection identifier, not game-simulation randomness — it does NOT violate the `packages/sim` purity invariant (this file is in `packages/server`, which is explicitly allowed I/O and non-determinism).
- **Per-connection setup** (`wss.on("connection", ...)`): creates a `Session` (`createSession(ws)`), generates+registers its session token, and starts a heartbeat: `setInterval` every `HEARTBEAT_MS=5000` sends a WS ping frame; if the PREVIOUS interval's `alive` flag was never flipped back to `true` (no pong received, and no message received either — see below) since the last tick, the connection is forcibly `ws.terminate()`d. `alive` is reset to `true` on EITHER a `pong` event OR any successfully-processed `message` event (the `alive = true` at the bottom of the message handler) — so an active connection sending regular CMD/PING traffic is also considered alive even without explicit WS-protocol pong frames.
- **Message dispatch** (`ws.on("message", ...)`): decodes the raw payload via `decodeC2S` (from `@autobattler/protocol`) — on `null` (malformed/failed validation), replies with a typed `ERROR {code: "INVALID_MESSAGE"}` and returns early (no further processing, no rate-limit charge). Otherwise, ONLY `CMD`-type messages are rate-limited (`checkRateLimit(session)` — 20/sec from CLAUDE.md; the check happens BEFORE the switch, and the function returns early — without alive-flagging — if the limit is exceeded, silently dropping that message). The actual switch handles 6 message types:
  - `PING` → replies `PONG` echoing the client's `ts` plus a fresh `serverTs: Date.now()` (used by `NetClient`'s RTT/clock-offset measurement on the client side).
  - `QUEUE_JOIN` → rejects (`ALREADY_QUEUED`) if already seated in a room; rejects (`UNAUTHENTICATED`) if `authToken` is missing; otherwise asynchronously resolves the token via `repo.findByToken` — on failure, `UNAUTHENTICATED`; on success, sets `session.accountId` and fetches+sets `session.name` from the account's profile (defaulting to `null` if no profile, though in practice every account created via `createGuest` always has one), THEN calls `joinQueue(session)` — this is the ONLY message type whose handling is asynchronous/promise-based inside the switch (every other case is synchronous).
  - `RECONNECT` → looks up the claimed seat via `findSeatByToken(msg.token)` and the room via `getRoom(claim.roomId)`; if either is missing, `RECONNECT_FAILED`; otherwise `registerToken(session.id, msg.token)` (the FRESH connection's session now owns the OLD seat token — re-using `registerToken`, which is normally for the session-token map, to make this new session resolvable by the OLD seat-claim token too) and `reconnectSession(room, session, claim.seatIndex)` (delegates the actual seat-restoration logic to `room.ts`).
  - `QUEUE_LEAVE` → `leaveQueue(session)`.
  - `CMD` → validates the session is actually seated (`NOT_IN_MATCH` if not), the room exists (`NOT_IN_MATCH` if not), and — critically — that `room.seats[seatIndex] === session` (THIS EXACT session object, not just a matching seat index — `WRONG_SEAT` error otherwise; this guards against a stale/duplicate session somehow still holding a seatIndex after a reconnect replaced it with a different session object) — then delegates to `handlePlayerCommand(room, seatIndex, msg.cmd)`.
  - `READY` → silently no-ops (no error sent) if not seated or room missing (unlike `CMD`, which sends explicit errors) — otherwise `markReady(room, seatIndex)`.
  - After the switch (regardless of which case ran, even error cases — this line is OUTSIDE the switch, unconditional), `alive = true` — meaning ANY successfully-decoded message (even one met with a typed ERROR reply) counts as proof-of-life for heartbeat purposes, not just "successful" commands.
- **Disconnect** (`ws.on("close", ...)`): clears the heartbeat interval; if the session was seated in a room, marks `session.afk = true` (the room's own AFK-handling logic, elsewhere, presumably acts on this flag — NOT removed from the room immediately, allowing for the RECONNECT path to restore the same logical seat later); if NOT seated (e.g. still in queue or never joined one), calls `leaveQueue(session)` instead; either way, finally calls `removeSession(session.id)` (purges the session + its session-token mapping from `session.ts`'s registries — but NOTE this does NOT clear the SEAT token registered via `registerSeatToken`/looked up via `findSeatByToken` — that's a separate map cleared only by `clearRoomSeatTokens(roomId)`, presumably called by `room.ts` when the match actually ends, so seat-reconnect tokens intentionally OUTLIVE the disconnected session object itself).
- **`ws.on("pong", ...)`**: sets `alive = true` (standard WS heartbeat ack).
- **Listen**: `httpServer.listen(PORT, () => console.log(...))` — `PORT` from `process.env["PORT"]` or default `3001`.

**Invariants & constraints**
- This file is the ONLY place a `WebSocketServer`/raw HTTP server is actually constructed and listened on in the whole codebase — all other server modules are pure logic operating on `Session`/`Room`/`MatchState` objects, testable without a real socket.
- The `room.seats[seatIndex] !== session` check on every `CMD` is the critical guard preventing a STALE session (e.g. one that disconnected and was later superseded by a reconnect) from ever successfully issuing commands into a room it no longer legitimately occupies — any reader investigating "can two connections both control one seat" should look here first.
- Two DISTINCT token systems coexist and must not be confused: (1) the WS session token (`generateToken`/`registerToken`/`findSessionByToken`, `Math.random`-based, server-instance-local, used for... notably the code generates and registers one per connection but the only consumer visible in this file is the seat-token re-registration during RECONNECT) and (2) the seat-claim token (`registerSeatToken`/`findSeatByToken`, presumably created by `room.ts` when seating a player, used by the CLIENT-supplied `RECONNECT.token` to restore a specific seat) and (3) the AUTH token (`signToken`/`verifyToken`/HMAC, from `auth.ts`, carried in `QUEUE_JOIN.authToken`, identifies the player's ACCOUNT, unrelated to any in-match seat). A reader must keep these three separate token concepts straight — this file touches all three but they serve entirely different purposes (connection identity vs seat-reconnect capability vs account identity).
- Heartbeat: a connection is terminated if no `pong` AND no successfully-decoded message arrives within one `HEARTBEAT_MS` (5s) window — i.e. the death window is effectively up to ~2× `HEARTBEAT_MS` (one full interval where `alive` was already false before the NEXT tick's check fires the terminate) in the worst case.

**Depends on** — `node:http` (`createServer`); `ws` (`WebSocketServer`, `WebSocket` type); `@autobattler/protocol` (`decodeC2S`); `./session.js` (`createSession`, `removeSession`, `send`, `checkRateLimit`, `registerToken`, `findSeatByToken`); `./matchmaker.js` (`joinQueue`, `leaveQueue`); `./room.js` (`getRoom`, `handlePlayerCommand`, `markReady`, `reconnectSession`, `setRoomRepository`); `./db/index.js` (`createRepository`); `./http.js` (`createHttpHandler`).

**Used by** — Nothing imports this file (it's the process entry point, invoked directly by `npm run server` / `tsx src/index.ts`). Not used by any test (tests exercise `room.ts`/`matchmaker.ts`/etc. directly without going through a real socket).

**Notes** — The asynchronous `QUEUE_JOIN` handling (the only `await`-involving case in the switch) means a client could in principle send a SECOND message while the first `QUEUE_JOIN`'s `repo.findByToken`/`repo.getProfile` promise chain is still in flight — the code doesn't appear to guard against double-`QUEUE_JOIN` races explicitly beyond the `session.roomId` check (which wouldn't be set yet during the async gap) — a second `QUEUE_JOIN` sent before the first resolves could both proceed to `joinQueue(session)`, though `matchmaker.ts`'s own queue logic would need to be checked to know if that's actually problematic (e.g. if `joinQueue` is itself idempotent for an already-queued session). This is a candidate edge case worth flagging if a future bug report involves duplicate-queue-entry symptoms.
