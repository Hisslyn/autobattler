# packages/server/src/recorder.ts

**Path & purpose** — `packages/server/src/recorder.ts`. The single function that persists a finished match: looks up each account seat's current MMR, computes Elo deltas via `mmr.ts`, writes the match record + updated profile MMR through the `Repository`, and returns the per-seat before/after MMR for the `MATCH_END` payload.

**Responsibility** — Owns the end-to-end "match just ended, now persist it" orchestration step, bridging `room.ts`'s in-memory match-end moment to durable storage. This is the ONLY place `computeMmrDeltas` is actually invoked in production code (besides its own unit tests) and the only place a `Repository.recordMatch` call originates from.

**Exports**
- `interface SeatResult { seat: number; accountId: string | null; placement: number }` — one seat's match-end summary as input to `recordMatchResult`: `accountId` is `null` for bots (explicit comment), `placement` is 1-indexed (`1 = winner`, explicit comment).
- `recordMatchResult(repo: Repository, matchId: string, seats: SeatResult[]): Promise<Record<number, MmrChange>>` — the persistence entry point. Returns a map from SEAT NUMBER (not accountId) to `{before, after}` MMR — but ONLY for seats backed by a real account (bot seats are simply absent as keys in the returned record, matching `MmrChange`'s doc comment in `protocol/messages.ts`: "only seats backed by an account appear").

**Key behavior**
1. **Fetch `mmrBefore` for every seat** (in parallel via `Promise.all`): for a bot seat (`accountId === null`), uses the data-driven `gameData.economy.mmrStart` directly (no repository lookup needed — bots have no stored profile) — this is the literal mechanism behind CLAUDE.md's "bots count at mmrStart for the lobby average." For a real account seat, calls `repo.getProfile(accountId)` and uses its `.mmr`, falling back to `mmrStart` if the profile is somehow missing (`profile?.mmr ?? start` — a defensive fallback, not expected to trigger in normal operation since every account should have a profile).
2. **Compute deltas**: calls `computeMmrDeltas` ONCE for the whole lobby, mapping each seat to `{mmr: mmrBefore[i], placement: s.placement}` — i.e. bots' `mmrStart`-baseline MMR DOES feed into the Elo average-of-others calculation for every other seat (per `mmr.ts`'s behavior), even though the bot's own resulting delta is about to be thrown away.
3. **Build the `MatchPlayerRecord[]`** for the repository write: for bot seats, `mmrBefore`/`mmrAfter` are explicitly forced to `null` (even though `mmrBefore[i]` actually holds a numeric `mmrStart` value internally — that internal number is DISCARDED here, never written to the database for a bot) — only real account seats get their actual `mmrBefore[i]` and `mmrBefore[i] + deltas[i]` (the computed `mmrAfter`) persisted.
4. **Persist**: `await repo.recordMatch(matchId, DATA_VERSION, players)` — a single call into the `Repository` interface, which (per its own contract, documented on `repo.ts`) handles writing the match row + all `match_players` rows + applying each non-bot player's `mmrAfter` to their profile, ATOMICALLY (one transaction in the Postgres implementation).
5. **Build the return value**: iterates the just-built `players` array and, for every NON-BOT entry, populates `changes[seat] = {before: mmrBefore, after: mmrAfter}` (using non-null assertions `!` since these fields are statically known non-null for non-bot entries by construction) — bot seats are simply never added as keys.

**Invariants & constraints**
- **MMR is computed and persisted in the SAME call that writes the match record** — there's no separate "now update everyone's MMR" step elsewhere; this function is the sole authority for that side effect, and `room.ts` (per CLAUDE.md: "at match end persists via recorder before broadcasting MATCH_END (so payload MMR always matches repo state)") is explicitly ordered to call this BEFORE constructing the `MATCH_END` message, guaranteeing the broadcast payload's MMR numbers are never stale relative to what's actually durably stored.
- This function calls `gameData.economy.mmrStart`/`computeMmrDeltas` (which itself defaults to `gameData.economy.mmrK`/`mmrEloDivisor`) — all MMR tuning ultimately traces back to `economy.json`, none of it hardcoded here.
- `DATA_VERSION` (from `@autobattler/data`) is recorded on EVERY persisted match — per the project-wide rule that every match records which content version was active.
- The returned `Record<number, MmrChange>` is KEYED BY SEAT NUMBER, not accountId — a caller (room.ts) must map seat→whatever else it needs (e.g. which session/account that seat belonged to) using its own seat-tracking data structures; this function has no concept of sessions, only abstract seat numbers + accountIds.
- Bots are silently excluded from BOTH the persisted MMR fields (`null`) AND the returned `changes` map (absent key) — consistently, at every layer, reinforcing "bots never persist" project-wide.

**Depends on** — `@autobattler/data` (`gameData.economy.mmrStart`, `DATA_VERSION`); `./db/index.js` (`MatchPlayerRecord`, `Repository` types); `./mmr.js` (`computeMmrDeltas`); `@autobattler/protocol` (`MmrChange` type — `interface MmrChange { before: number; after: number }`, also documented as the value type of `S2C_MatchEnd.mmr?: Record<number, MmrChange>`).

**Used by** — `packages/server/src/room.ts` (calls `recordMatchResult` when a match ends, then uses its returned `Record<number, MmrChange>` to populate the `MATCH_END` message's `mmr` field, per CLAUDE.md: "persists via recorder before broadcasting MATCH_END... with per-seat stats"); `packages/server/tests/mmr.test.ts` and/or `persistence.test.ts` (likely exercise this function directly against a `MemoryRepository`).

**Notes** — The fact that a bot's `mmrBefore` value IS computed (as `mmrStart`) and DOES feed into `computeMmrDeltas`'s lobby-average math, yet is then explicitly nulled-out before persistence/return, is an important nuance: a reader investigating "why does adding a bot to a lobby slightly change everyone else's MMR delta" should know it's BECAUSE bots are treated as full-fledged `mmrStart`-rated entrants for the PURPOSES OF THE ELO MATH ONLY, never for storage. This mirrors exactly what `mmr.ts`'s own doc note says ("bots count at mmrStart for the lobby average but are never persisted") — this file is where that statement is actually implemented end-to-end.
