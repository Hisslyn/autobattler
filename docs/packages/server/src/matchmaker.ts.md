# packages/server/src/matchmaker.ts

**Path & purpose** — `packages/server/src/matchmaker.ts`. The FIFO matchmaking queue: collects sessions wanting a match, flushes them into a new room either once 8 are queued or after a 10-second timeout (whichever comes first), and broadcasts live queue-position updates to everyone still waiting.

**Responsibility** — Owns the single global queue of `Session`s awaiting a match and the decision of WHEN to actually start one (`createRoom`). Does not decide WHO ends up in which seat beyond simple FIFO order (first 8 queued get seats 0-7 of the new room, in queue order) and does not handle bot backfill itself — that's `room.ts`'s `createRoom`'s responsibility (this file just hands it whatever humans it has, even fewer than 8).

**Exports**
- `joinQueue(session: Session): void` — adds `session` to the queue (idempotent: a no-op if the session is ALREADY in the queue, via `queue.includes(session)` — meaning calling this twice for the same session in flight does not double-enqueue it, which directly resolves the "could a double QUEUE_JOIN race double-enqueue" question raised when documenting `index.ts`). After pushing, immediately broadcasts updated queue positions to everyone (including the new joiner) via `broadcastQueueStatus()`. If the queue has now reached `QUEUE_TARGET=8`, immediately `flushQueue()`s (starts a match right away, no waiting). Otherwise, if no timeout timer is already running, starts one (`QUEUE_TIMEOUT_MS=10_000` — 10 seconds) that will flush the queue (with however many are waiting, as long as `queue.length >= 1`) when it fires.
- `leaveQueue(session: Session): void` — removes `session` from the queue (a no-op, via `indexOf` returning `-1`, if it wasn't queued — safe to call unconditionally, e.g. from `index.ts`'s WS `close` handler even for a session that was never queued), then re-broadcasts updated positions to the remaining queued sessions.

**Key behavior**
- **Queue is a single global module-level array** (`const queue: Session[] = []`) — there is only ONE matchmaking queue for the entire server process (no regions/skill-tiers/game-modes to split across), consistent with CLAUDE.md's "FIFO queue; 8 players OR 10s timeout with ≥1 human; backfills remaining seats with AI bots" description.
- **`flushQueue()`**: clears any pending timeout timer (so a flush triggered by reaching 8 players doesn't ALSO leave a stale timer that fires later for an unrelated future queue state); takes the FIRST `QUEUE_TARGET=8` sessions off the front of the queue via `queue.splice(0, 8)` (if fewer than 8 are queued, takes all of them — `splice` clamps gracefully); broadcasts the queue status to whoever's LEFT in the queue (i.e. if more than 8 were somehow queued — not possible given `joinQueue`'s immediate-flush-at-8 behavior, but `flushQueue` is defensively correct regardless); then calls `createRoom(humans)` with exactly the spliced-off sessions — `room.ts`'s `createRoom` is responsible for padding any seats beyond `humans.length` up to the full 8-seat room with bots (confirmed by reading `room.ts`: `seats`/`seatAccounts`/`seatNames` arrays are pre-sized to `HUMAN_SEAT_COUNT` and only the first `sessions.length` indices are overwritten with real session data — the rest keep their bot defaults, e.g. `seatNames[i] = "Bot " + (i+1)`).
- **Timeout-triggered flush**: the `setTimeout` callback checks `queue.length >= 1` before flushing — meaning if the queue somehow became EMPTY before the timer fired (e.g. every queued session called `leaveQueue` in the interim), the stale timer simply does nothing (no room is created with zero players) — though notably the timer is never explicitly CLEARED by `leaveQueue` itself, only by `flushQueue` — so a timer started when the queue had 1+ players keeps ticking even if everyone leaves and a few new players join later (it doesn't get "reset" on each join, only the FIRST join starts it; this is intentional FIFO/bounded-wait behavior, not a bug — a player who joins right before the timer fires doesn't get a fresh 10s wait, they get whatever time is left on the EXISTING timer).
- **`broadcastQueueStatus()`**: iterates the FULL current queue and, for each session whose underlying WebSocket is OPEN (`readyState === 1`), sends a raw hand-constructed `QUEUE_STATUS` envelope (`{v:1, t:"QUEUE_STATUS", p:{type:"QUEUE_STATUS", position: i+1, size: queue.length}}`) DIRECTLY via `s.ws.send(JSON.stringify(...))` — notably this does NOT go through `session.ts`'s `send()` helper or `@autobattler/protocol`'s `encode()` function; the envelope shape is hand-assembled inline here, matching the protocol's envelope format by convention/duplication rather than by calling the shared encoder. `position` is 1-indexed (first in line = position 1).

**Invariants & constraints**
- `QUEUE_TARGET = 8` and `QUEUE_TIMEOUT_MS = 10_000` are the two literal tuning constants for matchmaking — changing match size or wait tolerance means editing this file directly (these are server operational parameters, not `packages/data` game content, so they correctly live in code rather than JSON).
- The matchmaker NEVER rejects/caps queue growth beyond 8 in a way that would leave more than 8 in the queue array at once in practice — `joinQueue` flushes IMMEDIATELY upon reaching 8, so the array essentially never holds more than `QUEUE_TARGET - 1` sessions between flushes under normal operation (the defensive over-8 handling in `flushQueue` exists for correctness but isn't expected to trigger).
- This file holds NO logic about WHICH bots get created, their names, stats, or how seats are assigned beyond simple positional order — all of that is `room.ts`'s `createRoom` responsibility; this module's only output is "start a room with these sessions" (a `Session[]`, possibly fewer than 8).
- The hand-rolled `QUEUE_STATUS` JSON construction is a notable INCONSISTENCY with the rest of the server's outbound-message pattern (which typically goes through `session.ts`'s `send()` + `@autobattler/protocol`'s `encode()`) — a future agent modifying the `QUEUE_STATUS` message shape must remember to update it in BOTH `protocol/src/messages.ts`'s type definition AND this hand-written literal here, since there's no shared encoder call enforcing they stay in sync structurally (TypeScript's structural typing on the literal object would catch a SHAPE mismatch at compile time only if this literal is typed against `S2CMessage`, which it does not appear to be — it's an inline untyped object literal).

**Depends on** — `./session.js` (`Session` type only); `./room.js` (`createRoom`).

**Used by** — `packages/server/src/index.ts` (`QUEUE_JOIN` → `joinQueue`, `QUEUE_LEAVE` → `leaveQueue`, and indirectly on WS `close` for an unseated session).

**Notes** — The "10s timeout with ≥1 human" backfill-with-bots design (per CLAUDE.md) means EVERY match guarantees at least 1 real human seat — a lone player queuing alone will wait the full 10 seconds and then play against 7 bots, never instantly. There is no "practice vs bots immediately" fast-path here (that's `LocalDriver`'s job entirely client-side, per CLAUDE.md's client internals — Practice mode never touches this matchmaker at all).
