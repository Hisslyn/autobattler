# packages/server/src/auth.ts

**Path & purpose** — `packages/server/src/auth.ts`. HMAC-signed opaque authentication token issuance/verification for guest accounts — the server-side primitive behind `POST /auth/guest` and Bearer-token-protected endpoints.

**Responsibility** — Owns the entire token scheme: encoding an `accountId` into a tamper-evident opaque string, and verifying a presented token's signature before trusting the `accountId` it claims to represent. This is the ONLY place the token format/secret logic lives — `http.ts`/`session.ts`/anything else needing to issue or check tokens calls into this module rather than reimplementing HMAC logic.

**Exports**
- `signToken(accountId: string): string` — produces a token of the form `base64url(accountId).base64url(hmac_sha256(accountId, SECRET))` — the accountId itself is NOT secret/encrypted, just base64url-encoded for safe transport (readable by anyone who decodes it), with the HMAC providing tamper-evidence (forging a token for an arbitrary accountId requires knowing `SECRET`).
- `verifyToken(token: string): string | null` — splits the token at the first `.`, base64url-decodes the first segment back to a candidate `accountId` (returns `null` immediately if decoding throws, e.g. malformed input), then RE-SIGNS that candidate accountId via `signToken` and checks if the result matches the ORIGINAL token string exactly — if so, returns the verified `accountId`; otherwise `null`. Used by any endpoint/handler needing to authenticate a Bearer token or a `QUEUE_JOIN` message's `authToken`.

**Key behavior** — The verification approach is "recompute and compare the WHOLE token", not "extract the HMAC segment and compare HMACs separately" — `verifyToken` calls `signToken(accountId)` again (recomputing both the base64url encoding AND the HMAC) and does a full string equality check against the original token. This is simpler to reason about (no separate HMAC-only comparison path that could have its own bugs) at the cost of a redundant base64url re-encode, which is computationally trivial.

**Invariants & constraints**
- `SECRET` defaults to the literal string `"dev-secret-change-me"` if `AUTH_SECRET` isn't set in the environment — explicitly flagged in-code as "dev default is fine for local play, not production" — any production deployment MUST set `AUTH_SECRET` or all issued tokens are forgeable by anyone who reads this source file (the default is public/known).
- Tokens are NOT expiring/time-limited — there's no embedded timestamp or expiry check; a signed token remains valid forever (until/unless `AUTH_SECRET` is rotated, which would invalidate ALL outstanding tokens at once, including for unrelated accounts — there's no per-token revocation mechanism).
- The string comparison `token === signToken(accountId)` is a plain `===`, NOT a constant-time comparison — theoretically vulnerable to a timing side-channel that could help an attacker recover the HMAC bit-by-bit, though in practice this is a low-stakes guest-auth scheme (no real money/PII at stake) for a casual game server, so this is a pragmatic, accepted simplification rather than a flagged TODO in the source.
- `accountId` is NOT validated/sanitized within this module — `verifyToken` will happily decode and re-sign ANY string that decodes from base64url without erroring (including malformed/garbage accountIds) — it's the CALLER's responsibility (e.g. `http.ts`/`session.ts` looking up the accountId in the `Repository`) to confirm the returned accountId actually corresponds to a real account; `verifyToken` only proves "this accountId string was signed by US at some point", not "this accountId currently exists."

**Depends on** — Node's `node:crypto` (`createHmac`); `process.env["AUTH_SECRET"]`.

**Used by** — `packages/server/src/http.ts` (issuing tokens on `POST /auth/guest`, verifying Bearer tokens on `GET /profile`, `GET /history`, `PATCH /profile`); `packages/server/src/index.ts` and/or `session.ts` (verifying the `authToken` carried in a `QUEUE_JOIN` C2S message, per CLAUDE.md's session.ts description "accountId + profile name set on authed QUEUE_JOIN" and index.ts's "QUEUE_JOIN without a valid authToken → typed ERROR UNAUTHENTICATED").

**Notes** — This is a minimal, dependency-free token scheme (no JWT library, no external auth provider) appropriate for a guest-identity system where the security bar is "prevent casual tampering/impersonation," not "withstand serious cryptographic attack" — consistent with the broader system design where bots/guests never hold real persistent value (CLAUDE.md: "Bots never persist"). A future agent looking to harden this (token expiry, constant-time comparison, secret rotation with grace period) would need to extend this module without breaking the existing `accountId`-in-token format that `http.ts`/`session.ts` already depend on.
