# packages/server/tests/integration.test.ts

**Path & purpose** — `packages/server/tests/integration.test.ts`. The ONLY full-stack, real-process, real-network test suite in `packages/server` — spawns the ACTUAL server (`tsx src/index.ts`) as a child process on a free port, drives it via real `WebSocket` connections + real `fetch` HTTP calls, and asserts on the wire-level message sequences a true client would observe. Everything else in `packages/server/tests/` presumably exercises individual modules in-process; this file is the end-to-end smoke/contract test.

**Responsibility** — Proves the full system (HTTP auth + WS matchmaking + room phase loop + PvE loot + round results + reconnect + rate limiting) works correctly when wired together exactly as production runs it, including catching integration bugs that unit tests of individual modules couldn't see (e.g. correct message ORDERING across the wire, correct per-seat privacy of LOOT/ROUND_RESULT, the real `PLANNING_MS_OVERRIDE` env var actually shortening real wall-clock waits).

**Exports** — None (a test file; `describe`/`it` blocks only, no importable symbols).

**Key behavior — test infrastructure helpers**
- `getFreePort()`: opens a throwaway `net.Server` on port 0, lets the OS assign a free port, reads it back, closes the throwaway listener, and resolves with that port number — used so concurrent test runs (or repeated CI runs) never collide on a fixed port.
- `waitForPort(port, timeoutMs=5000)`: polls (every 100ms, raw `net.Socket` connect attempts) until the given port accepts a TCP connection or the timeout elapses — used to know when the spawned server subprocess has finished booting (the boot sequence includes `await createRepository()` and synchronous schema setup, so it's not instantaneous).
- `launchServer(port)`: spawns `tsx <server's src/index.ts>` as a real child process via `node:child_process.spawn`, with `PORT` set to the given port AND **`PLANNING_MS_OVERRIDE: "50"`** — the inline comment explains why: "READY no longer skips PLANNING (planning runs its full duration), so shorten the planning window for the real-subprocess integration harness — otherwise a full match would take 30s/round of wall-clock time and blow the timeouts." This is a DIRECT, test-file-level confirmation of the exact same fact discovered in `room.ts`'s own doc: READY does NOT skip planning in the current implementation, and this override mechanism exists specifically because that's true — without it, this entire test file's timeouts (40s/50s/60s/70s) would be far too short for a full 8-round (ish) match at 30s/round.
- `sendRaw(ws, msg)`: hand-constructs and sends a raw protocol envelope `{v: PROTOCOL_VERSION, t: msg.type, p: msg}` directly via `JSON.stringify` — bypassing `@autobattler/protocol`'s `encode()` helper entirely (this test deliberately exercises the WIRE FORMAT a real client would send, including the envelope shape, rather than relying on the same encoder the server itself uses — though in practice it reconstructs that exact shape by hand, so a protocol envelope-shape regression could theoretically go undetected if BOTH this test's hand-rolled envelope AND the server's actual decoder changed in the same incompatible way — unlikely, but worth noting it's not literally calling the shared `encode`).
- `authGuest(port, deviceId, name?)`: a real `fetch` POST to `/auth/guest` — exercises the genuine HTTP API (`http.ts`) over the wire, returning `{accountId, token}`.
- `waitOpen(ws)` / `waitClose(ws)`: promise wrappers around the WS `open`/`close` events.
- `collectUntil(ws, stopType, timeoutMs)`: collects every successfully-`decodeS2C`-decoded message received on `ws` into an array, resolving EITHER when a message of type `stopType` arrives (immediately, clearing the timeout) OR when `timeoutMs` elapses (whichever first) — the core message-collection primitive used by every test in this file; note it resolves with WHATEVER was collected so far on timeout, not a rejection, so a test that doesn't receive `stopType` in time still gets a (likely incomplete) array back rather than failing fast with a clear "timed out" error — assertions on the returned array (e.g. `expect(end1).toBeDefined()`) are what actually surface that failure mode.

**Key behavior — the four test suites**
1. **"integration: 2 humans + 6 bots full match"**: two real guest accounts + two real WS connections both `QUEUE_JOIN`; both auto-`READY` on every `PHASE_CHANGE` to `PLANNING` or `RESOLUTION` (a comment clarifies this is now a "harmless no-op" for PLANNING specifically, only actually skipping the RESOLUTION pause); both collect messages until `MATCH_END`; asserts BOTH clients receive `MATCH_END`, both have an 8-entry `placements` array, and CRITICALLY both clients' `placements` arrays are byte-IDENTICAL (`toEqual`) — proving the server broadcasts the SAME canonical match outcome to every seat, not divergent per-client views.
2. **"integration: PvE loot + player names"**: two NAMED guest accounts ("Alice"/"Bob"); after a full match (same auto-ready pattern), asserts in detail: (a) every one of the 8 seats in a `STATE_SNAPSHOT` has a non-empty `name`, the two human seats show their REAL profile names, each human's snapshot shows the OTHER human's real name too (public visibility), and all 6 remaining seats show generated `"Bot N"` names; (b) round 1 (confirmed PvE) delivers EXACTLY ONE `LOOT` message per human for that round, each containing exactly 2 orbs (matching `loot.json`'s round-1 config), all `common` rarity, each orb's `reward.kind` being one of `gold`/`component`/`item` — and implicitly, since `loot1`/`loot2` are tracked SEPARATELY per socket and never cross-checked for shared content, this also implicitly verifies no orb LEAKAGE between the two humans' private channels (the comment makes this explicit: "never the other's (no broadcast leak)"); (c) every `ROUND_RESULT` received has a `status` in the valid set `won|lost|bye|pve`, round 1 specifically is `pve` with `damageTaken`/`damageDealt` both `0`, and CRUCIALLY the cross-field invariant "a loss is the only outcome with HP damage; a win is the only one dealing it" is checked for EVERY received round-result across both clients (`if status !== "lost", damageTaken === 0`; `if status !== "won", damageDealt === 0`) — this is the wire-level confirmation of `rounds.ts`'s `runCombatPhase` invariant (winner.damageDealt === loser.damageTaken) actually reaching the client correctly; (d) `MATCH_END.names` is present and correct for both humans plus exactly 6 bot-named entries; (e) `MATCH_END.stats` is present for all 8 seats with correctly-typed non-negative `roundWins`/`roundLosses`/`totalDamageTaken`/`totalDamageDealt` fields.
3. **"reconnect: token restores seat mid-match"**: client 1 joins, captures its `MATCH_FOUND` `{token, seatIndex}`; client 2 auto-readies to keep the match progressing; client 1 plays through one phase transition then closes its OWN socket (`ws1.close()`, a deliberate disconnect, not a server-initiated kick) and waits for the close event to fully complete; THEN opens a brand-new `WebSocket` and sends `RECONNECT {token}` using the ORIGINAL token from the first connection — asserts the resulting `STATE_SNAPSHOT`'s `state.me.id` equals the original `seatIndex`, proving the seat-token system (`session.ts`'s `registerSeatToken`/`findSeatByToken`, consumed by `room.ts`'s `reconnectSession`) correctly restores identity across a fully independent new TCP connection.
4. **"rate limit: flood disconnects client"**: sends a BARE `QUEUE_JOIN` with no `authToken` at all (deliberately — this connection is NEVER actually authenticated or seated in any room — confirming the rate limiter operates at the SESSION level regardless of match/auth state, not gated behind successful queueing); waits 200ms; then fires 25 `CMD {type: REROLL}` messages in a tight synchronous loop (exceeding `CMD_RATE_LIMIT=20`); asserts the connection eventually closes (`waitClose`) AND that an `ERROR {code: "RATE_LIMITED"}` was received before the close — the direct wire-level proof of `session.ts`'s `checkRateLimit`'s punitive terminate-on-exceed behavior.

**Invariants & constraints**
- This suite is SLOW and STATEFUL relative to the rest of the test suite — it spawns 4 separate real server subprocesses (one per `describe` block, each with its own `beforeAll`/`afterAll` lifecycle) and waits on real timers (even shortened to 50ms via `PLANNING_MS_OVERRIDE`, a full match still requires real wall-clock seconds to complete several rounds) — test timeouts here are explicitly generous (15s for server boot, 40-70s for the actual `it()` bodies) compared to typical fast unit tests elsewhere in the repo.
- Relies on `PLANNING_MS_OVERRIDE` being read by `room.ts` at MODULE LOAD time (computed once into the exported `PLANNING_MS` constant) — since this test spawns a FRESH subprocess per `describe` block (not the same long-lived process), the env var is correctly picked up fresh each time; this mechanism would NOT work if the server were a long-lived shared process across test files (the override is read once at process start, not dynamically).
- Every `describe` block independently allocates its own free port and spawns/kills its own subprocess — tests are fully isolated from each other (no shared server state, no test-ordering dependency), at the cost of repeated ~startup overhead per suite.
- The hand-rolled envelope construction in `sendRaw` must be kept in sync with `@autobattler/protocol`'s actual envelope shape (`{v, t, p}`) — if that shape ever changes, this helper needs a matching update (it won't get one "for free" by importing the encoder, since it deliberately doesn't use it).

**Depends on** — `vitest` (`describe/it/expect/afterAll/beforeAll`); `ws` (`WebSocket`); `node:child_process` (`spawn`, `ChildProcess` type); `node:net` (`createServer`, `Socket`, for free-port discovery + polling); `node:url`/`node:path` (resolving the server entry path relative to this test file); `@autobattler/protocol` (`decodeS2C`, `PROTOCOL_VERSION`, `S2CMessage` type); the GLOBAL `fetch` (Node's built-in, for the HTTP auth call); indirectly, the ENTIRE server stack (`index.ts` and everything it imports), since it spawns and exercises the real compiled/run server.

**Used by** — Run as part of `npm test` (vitest's default test discovery picks up this file alongside all other `*.test.ts` files); not imported by any other module (a leaf test file).

**Notes** — This is the single test file most likely to break from environment issues unrelated to actual code bugs (port allocation races, subprocess spawn failures in restricted sandboxes, slow CI machines blowing the wall-clock timeouts) — a reader debugging a failure here should first rule out environmental/timing flakiness before assuming a logic regression, though the specific assertions (placement consistency, loot privacy, damage-status cross-field invariants, reconnect identity, rate-limit enforcement) are all real behavioral contracts worth trusting once a flaky-environment explanation is ruled out. The PLANNING_MS_OVERRIDE-driven test design is itself a strong piece of EVIDENCE (independent of `room.ts`'s comments) that "READY skips the 30s planning wait" (as CLAUDE.md's prose states) is OUT OF DATE relative to the actual code — if READY truly skipped planning, this override mechanism would serve no purpose and likely wouldn't exist.
