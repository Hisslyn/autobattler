# packages/server/tests/mmr.test.ts

**Path & purpose** — `packages/server/tests/mmr.test.ts`. Unit tests for `mmr.ts`'s `computeMmrDeltas` — pins exact known-answer deltas for an equal-MMR 8-player lobby and checks directional correctness (winners gain, losers lose, higher-rated players lose more for an equally-bad placement).

**Responsibility** — Verifies the Elo-style FFA MMR formula in `mmr.ts` produces both mathematically-expected exact values (for the simplest equal-MMR case) and intuitively-correct relative behavior (for mixed-MMR scenarios), guarding against regressions in the formula's implementation.

**Key behavior — test by test**
- **"equal-MMR lobby, placements 1..8: known answers, symmetric around 0"**: builds an 8-entrant lobby where everyone has `mmr: 1000` and placements run `1..8` (one each). Independently recomputes the EXPECTED deltas via the closed-form simplification that applies ONLY when every entrant shares the same MMR: since all MMRs are equal, `expected` (the Elo win probability vs the lobby average) is exactly `0.5` for everyone (no rating gap to give anyone an edge), so `delta = round(K * (actual - 0.5))` where `actual = (8-placement)/7`. Computes this `known` array independently in the test itself and asserts `deltas` matches it via `toEqual`. ALSO pins the literal exact numeric output: `[20, 14, 9, 3, -3, -9, -14, -20]` (using the real `gameData.economy.mmrK` — i.e. this test is sensitive to `economy.json`'s actual `mmrK`/`mmrEloDivisor` values; a tuning change to either would need this literal array updated, OR the test would need converting to NOT hardcode literals — as written, it's pinned to whatever `K` currently is, multiplied through the formula). Additionally asserts: (a) the deltas SUM TO EXACTLY ZERO across the 8-player equal-MMR lobby (`reduce sum === 0`), and (b) the deltas are perfectly ANTI-SYMMETRIC pairwise (`deltas[i] === -deltas[7-i]` for `i=0..3`, i.e. placement 1's gain exactly cancels placement 8's loss, placement 2 cancels placement 7, etc.).
- **"winner gains, last place loses in an equal lobby"**: the same equal-MMR 8-placement-spread lobby; asserts `deltas[0] > 0` (1st place gains MMR) and `deltas[7] < 0` (8th/last place loses MMR) — a coarser, more readable sanity check duplicating part of what the first test already pins exactly.
- **"higher-MMR player placing 8th loses more than lower-MMR player placing 8th"**: builds two 8-entrant lobbies sharing the SAME 7 `mmr:1000` "bots" occupying placements 1-7, differing only in the 8th entrant: one lobby's 8th-place entrant has `mmr: 1400` (well above the lobby), the other's has `mmr: 600` (well below). Asserts the HIGH-MMR entrant's delta-for-placing-8th is MORE NEGATIVE (a bigger loss) than the LOW-MMR entrant's delta-for-placing-8th — and that the high-MMR entrant's delta is negative at all. This captures the intuitive Elo principle: a HIGHLY-RATED player was EXPECTED to do well, so placing dead last is a bigger surprise/penalty for them than for an already-low-rated player who was somewhat expected to underperform anyway.

**Invariants & constraints**
- **IMPORTANT CORRECTION to a claim made in `mmr.ts`'s own doc**: that doc states "the SUM of all deltas across a lobby is NOT guaranteed to be zero... unlike classical pairwise Elo" as a general property of this N-player generalization. This test's first case PROVES the sum IS exactly zero **in the specific case where every entrant shares the same starting MMR** (since `avgOthers` reduces to the SAME constant `1000` for every entrant when all MMRs are equal, making `expected = 0.5` uniformly, and the `actual` scores by construction already sum to exactly `(n-1)/2` over a full placement spread of `1..n`, which cancels symmetrically against the uniform `0.5` expected). The doc's broader claim ("not guaranteed in general") remains TRUE for MIXED-MMR lobbies (not exercised by a sum-check in this test file — no test here checks that a mixed-MMR lobby's deltas sum to zero, and in general they will NOT, since `avgOthers` differs per entrant) — but a reader should understand the equal-MMR case is the one well-tested EXCEPTION where zero-sum holds exactly, demonstrated here.
- These tests use the REAL `gameData.economy.mmrK` (not a hardcoded override) for the first two tests' literal-array assertion — meaning if `economy.json`'s `mmrK` ever changes, this test file's `[20, 14, 9, 3, -3, -9, -14, -20]` literal would need updating to match (the `known` array computed inline would still self-consistently match `computeMmrDeltas`'s output regardless of `K`'s value, but the SEPARATE hardcoded literal-array assertion would fail if `K` changes without updating it) — flagging this as a maintenance coupling point for anyone tuning `mmrK`.

**Depends on** — `vitest`; `@autobattler/data` (`gameData.economy.mmrK`); `../src/mmr.js` (`computeMmrDeltas`, the unit under test).

**Used by** — Run as part of `npm test`; not imported elsewhere.

**Notes** — This file is a good complement to `mmr.ts`'s own documentation: where that file's doc flags "sum isn't guaranteed zero in general" as a caveat, THIS test file is the concrete evidence for the boundary case where it actually DOES hold (equal-MMR lobbies) — a future agent reasoning about MMR drift across a population should treat "equal-MMR lobbies are exactly zero-sum, mixed-MMR lobbies are not" as the precise, test-backed statement of the invariant, rather than either extreme ("always zero-sum" or "never zero-sum").
