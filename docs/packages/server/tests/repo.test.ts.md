# packages/server/tests/repo.test.ts

**Path & purpose** — `packages/server/tests/repo.test.ts`. THE shared `Repository` CONTRACT test suite — a single parameterized `repoContract(name, makeRepo)` factory run against BOTH `MemoryRepository` (always) and `PostgresRepository` (only when `DATABASE_URL` is set), proving both implementations satisfy the exact same behavioral contract documented on `repo.ts`'s `Repository` interface.

**Responsibility** — Owns the canonical, backend-agnostic verification of every `Repository` method's behavior: account creation + idempotency, token resolution, profile patching, match recording with bot-exclusion, match-history ordering/limiting, and leaderboard ordering. This is the file that would catch a `PostgresRepository`/`MemoryRepository` BEHAVIORAL DIVERGENCE (the exact kind of divergence flagged when documenting both implementations earlier — e.g. the leaderboard tie-break ordering question).

**Key behavior**
- **`repoContract(name, makeRepo)`**: a reusable test-suite FACTORY (not a `describe`/`it` itself, but a function that GENERATES one) — takes a display name and an async repository-factory function, and produces a `describe(\`repository contract: ${name}\`, ...)` block containing 6 `it()` cases, each run against whatever `Repository` instance `makeRepo()` produces. `beforeAll` constructs the repo once per describe block; `afterAll` calls `repo.close()`. The comment above this function states the deliberate design: "every Repository impl must pass. Unique deviceIds/matchIds per run so the suite is safe against a shared postgres database" — i.e. the test data uses `randomUUID()`-suffixed device/match ids specifically so running this suite repeatedly against a PERSISTENT shared Postgres instance (not wiped between runs) never collides with leftover rows from a previous run.
- **The 6 contract tests** (run identically against whichever backend):
  1. **"createGuest creates account + profile at starting MMR"**: creates a guest with a name, asserts both `accountId`/`token` are truthy, then asserts `getProfile` returns EXACTLY `{accountId, name: "Alice", mmr: 1000}` — note this test HARDCODES the literal `1000` rather than reading `gameData.economy.mmrStart` — meaning if `economy.json`'s `mmrStart` were ever changed away from `1000`, this specific assertion would need updating (a coupling point, similar to `mmr.test.ts`'s hardcoded K-dependent literal array).
  2. **"createGuest is idempotent per deviceId"**: calls `createGuest` twice with the SAME `deviceId` (second call omits the name) — asserts both calls return the SAME `accountId` AND the SAME `token` — the core idempotency contract from `repo.ts`'s doc comment, verified identically for whichever backend is under test.
  3. **"findByToken resolves the account; unknown token is null"**: asserts a freshly-issued token resolves to `{accountId}`, and an arbitrary bogus string (`"bogus.token"`) resolves to `null`.
  4. **"updateProfile patches name and mmr"**: calls `updateProfile` TWICE in sequence — once with only `{name: "New"}`, once with only `{mmr: 1234}` — then asserts the FINAL profile has BOTH the new name AND the new mmr (`{accountId, name: "New", mmr: 1234}`) — proving each partial-patch call leaves the OTHER field untouched (the `COALESCE`/conditional-field-presence contract from both implementations).
  5. **"recordMatch persists players, applies MMR, skips bots"**: creates two real accounts (P1, P2) plus a bot row (`accountId: null, mmrBefore: null, mmrAfter: null`), calls `recordMatch` with all three as `MatchPlayerRecord`s, then asserts: P1's profile mmr is now `1020` (its `mmrAfter`), P2's is `980`, AND `matchHistory` for P1 returns exactly 1 entry matching the recorded fields (`matchId, dataVersion, seat, placement, mmrBefore, mmrAfter`) — this test does NOT separately assert anything about the BOT row beyond it being included in the `recordMatch` call without erroring (there's no `getProfile`/history check for the bot, since bots never get a profile at all — the absence of any error/crash IS the implicit pass condition for the bot-skip behavior).
  6. **"matchHistory returns newest first and respects limit"**: records 3 matches sequentially for one account (each with an incrementing `mmr`/`mmrAfter`), with a deliberate `await new Promise(r => setTimeout(r, 5))` PAUSE between each `recordMatch` call — the comment explains why: "ensure distinct ended_at ordering in pg (timestamp resolution)" — i.e. without this delay, Postgres's `TIMESTAMPTZ` column (or even `Date.now()` in the in-memory case, though that's less of a risk in JS) could record two matches with the SAME timestamp if they happen fast enough, making "newest first" ordering ambiguous/unstable; with the delay, each `ended_at` is guaranteed distinct. Then calls `matchHistory(accountId, 2)` (limit 2 of the 3 recorded) and asserts EXACTLY the 2 MOST RECENT match ids come back, in newest-first order (`[matchIds[2], matchIds[1]]` — the array built in CHRONOLOGICAL insertion order, so index 2 is the last/newest one created).
  7. **"leaderboard orders by MMR descending"**: creates a "Low" (mmr 900) and "High" (mmr 2000) account, fetches the FULL leaderboard (`leaderboard(1000)`, a generously large `n` to capture everyone), filters down to just these two accounts and asserts they appear in `[high, low]` order (descending), THEN does a GLOBAL check across the ENTIRE returned board: every consecutive pair satisfies `board[i].mmr <= board[i-1].mmr` (a full descending-sort verification, not just for the two newly-created accounts but for whatever else might already exist in a shared/reused repository instance) — this GLOBAL check is notably ROBUST to test-ordering/shared-state concerns (it doesn't assume it's the only test that's ever created accounts in this repo instance).

**Invariants & constraints**
- **`repoContract("memory", ...)` ALWAYS runs** (unconditionally, at module scope) — `MemoryRepository`'s contract compliance is checked on every single test run, everywhere, regardless of environment.
- **`repoContract("postgres", ...)` runs ONLY IF `process.env["DATABASE_URL"]` is set** — when unset, the postgres-specific suite is entirely skipped via `describe.skip` (with a single placeholder `it` whose body does nothing, just so the skip shows up with a descriptive name in test output) and a `console.log` explicitly states "DATABASE_URL not set: skipping postgres repository contract tests" — this is the literal mechanism behind CLAUDE.md's "the postgres repo contract tests run only when DATABASE_URL is set (skipped + logged otherwise)" statement.
- The Postgres factory function calls `repo.ensureSchema()` immediately after construction (before returning it to the contract-test harness) — meaning the schema is freshly (idempotently) applied at the START of every `repoContract("postgres", ...)` run, never assumed pre-existing.
- This suite is the SINGLE BEST PLACE to add a new assertion if a future change to `Repository`'s contract (e.g. a new method, or a tightened guarantee on an existing one) needs verifying identically across both backends — any new behavioral requirement added to `repo.ts`'s interface documentation should get a corresponding case added INSIDE `repoContract`, not as a backend-specific one-off test, to preserve the "both backends pass the same suite" guarantee.

**Depends on** — `vitest`; `node:crypto` (`randomUUID`, for collision-safe unique device/match ids); `../src/db/repo.js` (`Repository` type); `../src/db/memory.js` (`MemoryRepository`); `../src/db/postgres.js` (`PostgresRepository`).

**Used by** — Run as part of `npm test`; not imported elsewhere. The `postgres` half additionally requires `docker compose up -d` (a local Postgres instance) + `DATABASE_URL` exported in the test environment to actually execute (otherwise gracefully skipped, never failing CI/local runs that lack a database).

**Notes** — This file is the definitive answer to "do `MemoryRepository` and `PostgresRepository` behave identically" — anything NOT covered by one of these 6 cases (e.g. the leaderboard'S EXACT tie-break ordering for accounts with IDENTICAL mmr, which `postgres.ts`'s implementation breaks ties by `account_id` but `memory.ts`'s plain sort does not explicitly) is, by construction, UNVERIFIED by this contract suite — the global descending-sort check in case 7 would PASS either way for tied values (since `<=` doesn't care about tie-order), meaning the previously-flagged Memory-vs-Postgres tie-break asymmetry would slip through this suite undetected. A future agent wanting to lock in or fix that specific asymmetry should add a dedicated tied-MMR case here.
