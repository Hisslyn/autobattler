# packages/server/tests/room.test.ts

**Path & purpose** — `packages/server/tests/room.test.ts`. An IN-PROCESS (no real sockets, no subprocess) test of `room.ts`'s `Room` orchestration logic, using Vitest's fake timers (`vi.useFakeTimers()`) to deterministically drive the planning/combat/resolution phase loop, fake `Session` objects that capture sent messages via a stubbed `ws.send`, and a from-scratch CLIENT-SIDE re-simulation helper that proves a client (using ONLY the `COMBAT_START` payload + its own board) reproduces byte-identical combat to the server's authoritative `COMBAT_RESULT`.

**Responsibility** — The DEFINITIVE proof, at the unit-test level (not full-stack-integration), of: (1) the client-can-resimulate-and-match-server invariant central to the "server result is canon; client sim is presentation only" architecture; (2) the EXACT phase-transition timing semantics of `room.ts` (confirming, via actual fake-timer-driven assertions rather than just code comments, that PLANNING always runs its full `PLANNING_MS` duration and READY ONLY shortcuts RESOLUTION); (3) `STATE_DELTA`'s privacy guarantee (a foreign seat's broadcast delta never leaks gold/shop, only public scouting fields).

**Key behavior — test helpers**
- `readyAll(room)`: calls `markReady(room, 0)` and `markReady(room, 1)` — the file's own comment states explicitly: "READY no longer skips PLANNING: planning runs its full PLANNING_MS, advanced only by the room's planning timer. READY still skips the RESOLUTION pause." — this is the THIRD independent piece of code-level evidence (alongside `room.ts`'s own comment and both integration test files' `autoReady` comments) for the CRITICAL FINDING that CLAUDE.md's "READY from all human seats skips the 30s wait" line does not match the current implementation.
- `elapsePlanning()`: `vi.advanceTimersByTime(PLANNING_MS)` — fast-forwards the FAKE clock by exactly the real planning duration (imported directly from `room.ts`'s exported `PLANNING_MS` constant), triggering the room's internal `setTimeout`-based planning-to-combat transition synchronously within the test (no real wall-clock wait).
- `playFullRound(room)`: composes `elapsePlanning()` (PLANNING → combat → RESOLUTION, since planning always runs to completion) then `readyAll(room)` (RESOLUTION → next PLANNING, since READY DOES skip the resolution pause) — the single building block every multi-round test in this file uses to advance exactly one full round.
- `makeFakeSession(id)`: constructs a minimal fake `Session` whose `ws` stub has `readyState: 1` (OPEN) and a `send` that decodes the outgoing string via `decodeS2C` and pushes the resulting typed `S2CMessage` into a `messages` array — giving the test full typed visibility into every message a real client connection would receive, without any actual socket.
- `makeUnit(uid, defId)`: builds a fully-populated `UnitInstance` directly from a `units.json` def's base stats (1-star, no items, `attackCooldown: 0`, empty `statusEffects`/`items`) — a manual constructor mirroring what `rules`' board-placement logic would otherwise produce, used here to seed fixed test boards without going through the full `BUY`/`MOVE` command flow.
- `eventsHash(events)`: `sha256(JSON.stringify(events))` — collapses an entire `CombatResult.events` log into one comparable hash string, used to assert two independently-produced event logs are EXACTLY identical without doing a deep structural diff inline.
- `resimulate(seatId, myBoard, cs)`: THE core of this file's central test — a faithful, from-scratch reproduction of what `NetDriver` does client-side: given a `COMBAT_START` payload (`cs`, with `pairings`, `opponentSnapshots`, `roundSeed`) and the client's own locally-tracked board, it (a) finds which pairing entry involves this seat and which `side` (0 or 1) it occupies, (b) bails (`return null`) if it's a ghost-paired bye (`bId < 0`) since that's not covered by this helper, (c) looks up the opponent's board from `cs.opponentSnapshots[seatId]`, (d) converts both boards to sim `CombatState` via `rules`' `boardToCombatState` (oriented to the correct side), (e) derives the EXACT per-pairing seed via `rules`' `derivePairingSeed(cs.roundSeed, idx)`, and (f) calls `@autobattler/sim`'s `simulateCombat` directly with the two combat states + that seed + `gameData` — producing a `CombatResult` using ONLY public wire data, mirroring NetDriver's real reconciliation path exactly.

**Key behavior — test by test**
1. **"clients re-simulate from COMBAT_START and match COMBAT_RESULT event-log hashes (A-side and B-side)"**: creates a room with 2 fake human sessions + (implicitly) 6 bots, seeds BOTH human seats with an identical fixed 3-unit board (warrior/archer/mage) at level 3, then runs UP TO 12 rounds (`playFullRound` in a loop), each round snapshotting both seats' boards BEFORE the round resolves (since boards can change between rounds — though this loop doesn't actually issue any commands, so they stay static; the snapshot pattern matters because `room.state.players[*].board` mutates in place and could in principle differ from what was true at COMBAT_START time for a more complex test). It artificially keeps both human seats alive (`hp = 100` each round, overriding whatever the real combat resolution would have done) so their pairing persists across multiple rounds, allowing the loop to observe BOTH possible sides (A-side and B-side of a pairing) before stopping (`sidesCovered.size < 2`). For each round, it reads the FRESH (since-last-checked) messages for each seat, finds the `COMBAT_START`/`COMBAT_RESULT` pair, skips PvE rounds (`cs.pairings.length === 0`), calls `resimulate` to get the LOCAL client-side result, and asserts: the seat's own entry in the server's `COMBAT_RESULT.results` map is defined, `eventsHash(local) === eventsHash(server)` (BYTE-IDENTICAL event logs), and `local.result.winner === server.winner`. Finally asserts `sidesCovered` ended up containing BOTH `0` and `1` — i.e. the test actually exercised the resimulation logic from BOTH perspectives of a pairing (not just always-A or always-B), giving confidence the side-relative board orientation logic (`boardToCombatState`'s side parameter) is correct in both directions.
2. **"COMBAT_START carries the CURRENT round's pairings"**: plays 3 rounds and asserts each one's LATEST `COMBAT_START.pairings` is the empty array `[]` — confirming that with only 2 human seats + 6 bots, rounds 1-3 are PvE rounds (per the rules-layer `pveRounds` schedule) and crucially that `COMBAT_START` for a PvE round carries NO STALE pairings from a previous PvP round (a potential bug class this test specifically guards against — pairings data lingering/leaking across phase transitions). Then advances to round 4 (the first PvP round) and asserts its pairings array has length 4 (4 pairs covering 8 players) and that flattening all pairing entries yields exactly 8 UNIQUE ids (every seat appears in exactly one pairing, confirming `buildPairings`'s coverage).
3. **"holds a real RESOLUTION pause: phase sequence and just-finished round label"**: advances exactly one round's planning, asserts the room is now in `RESOLUTION` phase at `round: 1`, and asserts the EXACT sequence of `PHASE_CHANGE` messages received so far is `[{PLANNING,1}, {RESOLUTION,1}]` — confirming `RESOLUTION`'s `round` field labels the JUST-FINISHED round (not the upcoming one). Also asserts the `RESOLUTION` phase-change's `endsAt` timestamp is in the future (`> Date.now()`, even under fake timers — confirming the resolution-pause deadline is a real upcoming timestamp, not already-elapsed). Then calls `readyAll` and asserts the room advances to `PLANNING` round 2, with the phase-change sequence now `[{PLANNING,1}, {RESOLUTION,1}, {PLANNING,2}]` — directly demonstrating READY's effect IS scoped to ending the resolution pause, exactly matching this file's own `readyAll`/`playFullRound` doc comments.
4. **"STATE_DELTA privacy: another seat's delta never contains gold or shop"**: seat 0 gets gold and issues `REROLL` + `BUY_XP` commands (each of which broadcasts a `STATE_DELTA`); the test then inspects EVERY `STATE_DELTA` received by seat 1 (a DIFFERENT, non-acting seat) and asserts: `delta.me` is `undefined` (no private payload section AT ALL for a message about someone else's action), AND the full JSON-stringified delta contains NEITHER the literal substring `"gold"` NOR `"shop"` anywhere (a blunt but effective whole-payload leak check, not just a check of specific known fields — would catch a leak through any unexpected nested path) — while still asserting the PUBLIC scouting fields (`board`, `bench`, `xp`, `winStreak`) ARE present on the acting player's serialized public entry within seat 1's view. Finally confirms the ACTING seat (seat 0, viewing its OWN delta) DOES receive a defined `me` object containing `gold`/`shop` — proving the privacy gate is seat-relative, not a blanket suppression.

**Invariants & constraints**
- This is the test file most likely to break (and thus alert) if the client-resimulation/server-result reconciliation contract central to the "combat playback = pure reducer" / "server result is canon" architecture is ever violated by a future change to `rounds.ts`'s seed derivation, `boardToCombatState`'s board conversion, or `sim`'s `simulateCombat` itself — since it performs the EXACT resimulation a real `NetDriver` does, using only wire-visible data.
- Uses `vi.useFakeTimers()`/`vi.advanceTimersByTime` rather than real waits or subprocess spawning — making this file's tests run near-instantaneously compared to `integration.test.ts`/`persistence.test.ts`'s real-subprocess-plus-real-delay approach, while still exercising the REAL `room.ts` orchestration code (not a mock) — `afterEach` explicitly clears `room.phaseTimer` (if set) before restoring real timers, to avoid a dangling fake-timer-scheduled callback firing unexpectedly after the test ends.
- `resimulate`'s explicit early-return for a ghost pairing (`side === 0 && bId < 0`) means this test file does NOT exercise odd-player-count ghost-bye resimulation — that scenario is implicitly covered elsewhere (or not at all) rather than here.
- This is now the THIRD file (after `room.ts`'s own source comment and both `integration.test.ts`/`persistence.test.ts`'s `autoReady` comments) to explicitly assert/encode that READY does not skip PLANNING — readers should treat this as settled, well-corroborated behavior of the current codebase, in contradiction to CLAUDE.md's prose on this specific point.

**Depends on** — `vitest` (incl. `vi` fake timers); `node:crypto` (`createHash`, for the event-log hash); `@autobattler/data` (`gameData`, for unit defs); `@autobattler/protocol` (`decodeS2C`, `S2CMessage`, `S2C_CombatStart`, `S2C_CombatResult` types); `@autobattler/sim` (`simulateCombat`) + `@autobattler/sim/src/types.js` (`UnitInstance`, `CombatResult` — a direct deep-path import into `sim`'s internals rather than its public barrel, notably); `@autobattler/rules/src/rounds.js` (`derivePairingSeed`, `boardToCombatState` — also a deep-path import); `../src/room.js` (`createRoom`, `markReady`, `handlePlayerCommand`, `PLANNING_MS`, `Room` type); `../src/session.js` (`Session` type).

**Used by** — Run as part of `npm test`; not imported elsewhere.

**Notes** — The deep-path imports (`@autobattler/sim/src/types.js`, `@autobattler/rules/src/rounds.js`) bypass each package's normal public entry point, reaching directly into internal source files — this is a deliberate test-only convenience (these symbols may not be part of either package's intended public API surface) but means a refactor that moves/renames these internal files (even without changing their public-facing behavior) could break this test file specifically, unlike code that only imports from each package's barrel export.
