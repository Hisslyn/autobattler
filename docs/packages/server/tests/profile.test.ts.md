# packages/server/tests/profile.test.ts

**Path & purpose** — `packages/server/tests/profile.test.ts`. Tests `http.ts`'s `validateName` pure function directly, plus an HTTP-only (no WebSocket, no subprocess) integration test of the `PATCH /profile` endpoint against a REAL `http.createServer` wired to `createHttpHandler` and a real `MemoryRepository` instance.

**Responsibility** — Verifies the name-validation rules (length/charset) in isolation, and verifies the rename-flow's full HTTP behavior (auth gating, validation rejection, successful persistence, and that a persisted change is immediately visible via a subsequent `GET /profile`) — without the overhead of spawning a full server subprocess or opening WebSocket connections (unlike `integration.test.ts`/`persistence.test.ts`, this file runs the HTTP handler IN-PROCESS, directly inside the test's own Node process, just on a real-but-locally-bound `http.Server`).

**Key behavior — test by test**
- **`describe("validateName")`**: pure unit tests of `http.ts`'s exported `validateName(raw)`:
  - Accepts: `"Al"` (exactly the 2-char minimum, unchanged) → `"Al"`; `"  Spaced Name  "` (leading/trailing whitespace) → `"Spaced Name"` (trimmed); `"a_b-c 1"` (mixed allowed chars: letters, underscore, hyphen, space, digit) → unchanged; `"Sixteen_chars_16"` (exactly the 16-char maximum) → unchanged. Confirms BOTH boundary lengths (2 and 16) are INCLUSIVE.
  - Rejects (all → `null`): `"A"` (1 char, below minimum); `"seventeen_chars_x"` (17 chars, above maximum); `"bad!name"` (contains `!`, outside the allowed charset `[A-Za-z0-9 _-]`); `"emoji😀"` (contains a non-ASCII emoji character, outside the charset — confirming the regex correctly rejects Unicode outside its explicit allow-list, not just ASCII punctuation); `42` (a number, not a string — confirms the `typeof raw !== "string"` guard); `undefined` (confirms the same type guard handles a missing/absent value).
- **`describe("PATCH /profile integration")`**: `beforeAll` creates a fresh `MemoryRepository`, seeds ONE guest account directly via `repo.createGuest("patch-dev-1", "Original")` (bypassing the HTTP `/auth/guest` endpoint entirely — this test goes straight to the repository layer to set up its fixture, rather than exercising registration), wraps `createHttpHandler(repo)` in a real `http.createServer`, and listens on an OS-assigned port (`server.listen(0, ...)`) — `afterAll` closes both the HTTP server and the repository.
  - **"requires auth"**: `PATCH /profile` with NO `Authorization` header → asserts `res.status === 401`.
  - **"rejects invalid names with a typed error"**: `PATCH /profile` with a valid Bearer token but `{name: "x"}` (1 char, below `validateName`'s minimum) → asserts `res.status === 400` and the JSON body's `.error === "INVALID_NAME"`; ALSO asserts the underlying repository's stored name is UNCHANGED (`"Original"`) — confirming the rejection happens BEFORE any `repo.updateProfile` call, not as a rolled-back side effect.
  - **"persists a valid change, reflected in subsequent GET /profile"**: `PATCH /profile {name: "  Renamed  "}` (deliberately padded with whitespace, to exercise the trim behavior end-to-end) → asserts `200` status and the response body's `profile.name === "Renamed"` (trimmed, confirming the PATCH response itself reflects the validated/trimmed name, not the raw input); THEN performs a SEPARATE `GET /profile` call (a fresh HTTP request, not reusing any cached response) and asserts `profile.name === "Renamed"` there too — confirming the write is durably visible to a subsequent independent read, not just echoed back in the PATCH response.

**Invariants & constraints**
- This file runs ENTIRELY in-process (no `child_process.spawn`, no `tsx` subprocess) — making it MUCH faster and less flaky than `integration.test.ts`/`persistence.test.ts`, at the cost of not exercising the WebSocket layer, the matchmaker, or room/match logic at all — it is scoped purely to `http.ts` + `MemoryRepository`'s `PATCH /profile`/`GET /profile` round-trip.
- Confirms `validateName`'s charset rejection extends correctly to non-ASCII input (the emoji case) — a detail that could be easy to get wrong with a naive regex (e.g. one that only checks ASCII alphanumerics without anchoring `^...$`, which might incorrectly ACCEPT a string containing an emoji if the regex isn't fully anchored) — this test would catch such a regression.

**Depends on** — `vitest`; `node:http` (`createServer`, `Server` type — for the real HTTP server instance); `../src/http.js` (`createHttpHandler`, `validateName`); `../src/db/memory.js` (`MemoryRepository`); the global `fetch`.

**Used by** — Run as part of `npm test`; not imported elsewhere.

**Notes** — This file's "go straight to the repository to seed fixtures, bypass the HTTP registration endpoint" pattern (`repo.createGuest` called directly rather than via `POST /auth/guest`) is a useful pattern other server tests follow too where appropriate — it isolates the TEST's concern (verifying `PATCH /profile`) from an unrelated dependency (the registration endpoint's own correctness), even though both ultimately rely on the same `Repository` contract.
