# packages/server/tests/persistence.test.ts

**Path & purpose** — `packages/server/tests/persistence.test.ts`. Another full-stack, real-subprocess integration suite (sibling to `integration.test.ts`), focused specifically on PROVING the account/MMR/match-history persistence path end-to-end through real HTTP + WS traffic against a real (forced in-memory) `Repository` — verifies the `MATCH_END` payload's MMR numbers exactly match what's durably readable afterward via `GET /profile` and `GET /history`.

**Responsibility** — Closes the loop between `room.ts`'s `finalizeMatch` → `recorder.ts`'s `recordMatchResult` → the `Repository` → and back out through `http.ts`'s authenticated endpoints, confirming the WHOLE chain produces self-consistent numbers a real client (checking its post-match profile/history) would see. Also covers an unauthenticated-`QUEUE_JOIN`-rejection case and a queue-join/queue-leave smoke check using a real auth token replayed on a third connection.

**Key behavior — test infrastructure** — Shares the SAME helper pattern as `integration.test.ts` (`getFreePort`, `waitForPort`, `sendRaw`, `waitOpen`, `collectUntil` — byte-for-byte duplicated implementations, not imported from a shared module — a notable code-duplication point between the two files), plus two NEW helpers specific to this file:
- `getAuthed<T>(port, path, token)`: a generic authenticated `fetch` GET helper — sends `Authorization: Bearer <token>`, asserts `res.ok`, returns the parsed JSON body typed as `T`. Used for `/profile` and `/history` calls.
- `interface Profile { accountId, name, mmr }` / `interface HistoryEntry { matchId, dataVersion, endedAt, seat, placement, mmrBefore, mmrAfter }` — locally-declared mirrors of the server's `Repository`-layer `Profile`/`MatchHistoryEntry` shapes (matching `db/repo.ts`'s actual interfaces), used purely for typing the HTTP response bodies in this test file (not imported from the server package itself — another duplication point, presumably to avoid a test-to-internal-module coupling, though it does mean the two shapes could silently drift apart undetected by the type system alone).
- `authGuest(port, deviceId, name)`: like `integration.test.ts`'s version but REQUIRES a `name` argument (not optional) and asserts `res.ok` inline (rather than throwing on non-ok) — returns `{accountId, token, profile}` (note: this version ALSO returns the `profile`, unlike `integration.test.ts`'s narrower version that only returns `{accountId, token}` — each file's helper is tailored to its own needs).
- `autoReady(ws)`: identical pattern to `integration.test.ts`'s version (auto-sends `READY` 30ms after any `PLANNING`/`RESOLUTION` `PHASE_CHANGE`), with the SAME comment confirming "READY skips the RESOLUTION pause; planning now elapses on its own short (override) timer, so a PLANNING READY is a harmless no-op" — a second independent confirmation (beyond `integration.test.ts` and `room.ts` itself) that READY does NOT skip PLANNING in the current implementation.

**Key behavior — server launch** — The `beforeAll` spawns the server with an EXPLICIT `DATABASE_URL: ""` override (commented "Force the in-memory repository regardless of the host environment") — this is important: even if the environment running this test suite happens to have a real `DATABASE_URL` set (e.g. a developer's shell, or a CI step that exports it for OTHER test files), this specific suite FORCES the empty-string override so `createRepository()` always falls back to `MemoryRepository`, making this suite's behavior independent of whatever `Repository` backend happens to be configured elsewhere — ensuring its assertions about exact persisted values are deterministic and don't depend on a real Postgres instance being reachable.

**Key behavior — test by test**
1. **"unauthenticated QUEUE_JOIN is rejected with typed ERROR"**: opens a WS connection, sends a bare `QUEUE_JOIN` (no `authToken` field at all), and asserts the resulting `ERROR` message matches `{type: "ERROR", code: "UNAUTHENTICATED"}` — the direct wire-level proof of `index.ts`'s `!authToken` early-rejection branch.
2. **"2 authed clients + bots: match persisted, MMR per formula, MATCH_END deltas match repo"** — the suite's centerpiece, a long multi-part assertion chain:
   - Registers two real guest accounts ("Player1"/"Player2"), asserts both start at `gameData.economy.mmrStart` exactly (confirming `createGuest`'s `mmrStart` seeding).
   - Plays a full match to `MATCH_END` (auto-ready pattern, same as `integration.test.ts`).
   - Asserts BOTH clients' `MATCH_END.mmr` fields are defined and IDENTICAL to each other (`toEqual`) — the same "canonical broadcast, not divergent per-client" property checked in `integration.test.ts`, this time for the `mmr` field specifically.
   - Asserts `Object.keys(end1.mmr).map(Number).sort()` equals exactly `[0, 1]` — i.e. ONLY the two human seats (0 and 1, since they were the first two sessions assigned seats by `createRoom`) carry MMR changes; the 6 bot seats are correctly ABSENT as keys (per `recorder.ts`'s bot-exclusion logic).
   - For EACH human (`auth1`@seat0, `auth2`@seat1): asserts `change.before === start` (both began the match at `mmrStart`, confirmed via the `MATCH_END` payload itself); fetches the LIVE profile via `GET /profile` and asserts `profile.mmr === change.after` (**the critical cross-check**: the value the `MATCH_END` WebSocket message claimed as the post-match MMR is EXACTLY what's durably stored and re-readable via a totally separate HTTP call — proving `room.ts`'s "persists via recorder BEFORE broadcasting MATCH_END" ordering actually delivers a consistent payload, per CLAUDE.md's stated invariant); fetches `GET /history?limit=20` and asserts exactly 1 history entry exists, with `seat`/`mmrBefore`/`mmrAfter` all matching the `MATCH_END` payload's claims, and `placement` falling in the valid `[1,8]` range; INDEPENDENTLY recomputes the expected Elo delta using the closed-form equal-starting-MMR formula (`round(k * ((8-placement)/7 - 0.5))`, the SAME formula `mmr.test.ts` uses for its equal-MMR case, since both human accounts started at the identical `mmrStart`) and asserts it matches `change.after - change.before` EXACTLY — an independent, from-scratch verification that the FULL pipeline (room→recorder→mmr→repository→HTTP) reproduces the exact textbook Elo answer, not just an internally-self-consistent one; cross-checks the `MATCH_END.placements` array's positional encoding against the persisted `entry.placement` via `entry.placement === 8 - placements.indexOf(seat)` (confirming the elimination-order-to-final-standing conversion documented in `room.ts`'s `finalizeMatch` — `placements` is elimination order with the winner appended last, so index 7 (last array position) = the winner = placement 1, i.e. `8 - 7 = 1`; index 0 = first eliminated = placement 8, i.e. `8 - 0 = 8`).
   - Confirms BOTH humans' `/history` entries share the SAME `matchId` (they really were recorded as one shared match row, not two separate ones).
   - Finally, opens a THIRD WebSocket connection and replays `auth1`'s ORIGINAL token via a fresh `QUEUE_JOIN` (the SAME account, now AFTER its match has ended and been persisted) — asserts a `QUEUE_STATUS` message is received, confirming the account's auth token remains valid and reusable for a brand-new match after a previous one completed (then immediately `QUEUE_LEAVE`s to clean up without actually waiting for a new match).

**Invariants & constraints**
- This suite is the STRONGEST end-to-end evidence in the whole test base that the `MATCH_END.mmr` payload and the durable `Repository` state are NEVER inconsistent — any future regression in `room.ts`'s "persist before broadcast" ordering (e.g. an accidental reordering that broadcasts first) would likely be caught here via the `profile.mmr === change.after` cross-check.
- Like `integration.test.ts`, depends on `PLANNING_MS_OVERRIDE=50` to keep wall-clock test duration tractable — same caveat about READY not skipping PLANNING applies.
- Explicitly forces `DATABASE_URL: ""` to guarantee `MemoryRepository` regardless of the ambient test environment — a deliberate isolation choice distinguishing this suite from whatever `repo.test.ts`'s dual-backend contract suite does (that one presumably DOES exercise Postgres when `DATABASE_URL` is actually set, per CLAUDE.md).

**Depends on** — `vitest`; `ws`; `node:child_process`, `node:net`, `node:url`, `node:path` (subprocess + port plumbing, identical pattern to `integration.test.ts`); `@autobattler/protocol` (`decodeS2C`, `PROTOCOL_VERSION`, `S2CMessage`, `S2C_MatchEnd` types); `@autobattler/data` (`gameData.economy.mmrStart`, `mmrK`); the global `fetch`; indirectly the entire server stack (spawned as a real subprocess).

**Used by** — Run as part of `npm test`; not imported elsewhere.

**Notes** — The helper-function duplication between this file and `integration.test.ts` (`getFreePort`/`waitForPort`/`sendRaw`/`waitOpen`/`collectUntil` are textually identical in both) is a real opportunity for a shared test-utilities module, though as documentation-only work this is simply flagged, not acted upon. A future agent doing test-infrastructure cleanup could extract these into a shared `tests/helpers.ts` without changing any test's observable behavior.
