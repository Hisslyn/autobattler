# packages/data/tests/integrity.test.ts

**Path & purpose** — `packages/data/tests/integrity.test.ts`. Vitest suite that cross-validates ALL of `packages/data`'s JSON content against itself (referential integrity, schema invariants, content-count invariants) using the assembled `gameData` object from `loader.ts`. This is the regression guard that keeps the data package internally consistent as content is added/changed.

**Responsibility** — Owns verification (not creation) of data integrity: every cross-reference between units/traits/items/economy/loot resolves, every count invariant documented in CLAUDE.md actually holds, and every typed helper (`itemKind`, `itemTier`) behaves correctly against the real content. No production code lives here — pure test assertions against `gameData`.

**Exports** — None (test file; Vitest auto-discovers `describe`/`it` blocks). Local constants: `CONSUMABLE_EFFECTS` (`Set` of valid `consumableEffect` strings: `remove_item`, `reforge`, `radiant_upgrade`), `VALID_STATS` (`Set` of stat keys items/traits are allowed to buff: `hp`, `ad`, `as`, `armor`, `mr`, `range`, `mana`, `abilityDamage`).

**Key behavior** — Four `describe` blocks, each testing a layer of integrity:
- **`"data integrity"`**: every `unit.traits` entry resolves to a `traits.json` id; a generic recursive `scan()` walks all of `gameData.units` looking for any `items`/`itemId` field and checks it resolves against `gameData.items` (future-proofing — units carry no item refs today, comment notes this explicitly); every trait breakpoint's `effect.stat` and every item's `stats` keys are in `VALID_STATS`; every unit's `abilityDamage > 0`, `mana > 0`, `0 <= manaStart <= mana`; every tier present in `units.json` has a nonzero `economy.poolCounts` entry and a long-enough `shopOdds` row; tiers absent from `units.json` are tolerated ONLY if their shop odds are 0 at every level (so shops never roll an empty tier); every `shopOdds` row sums to exactly 100.
- **`"v1 content completeness"`**: exactly 50 units with tier distribution `{1:13, 2:13, 3:12, 4:8, 5:4}`; no duplicate ids within units, traits, or items (separately, not cross-collection); every unit has exactly one resolvable `origin` (trait `kind:"origin"`) and 1-2 resolvable `classes` (trait `kind:"class"`), AND `unit.traits` equals exactly `[origin, ...classes]`; every trait's breakpoint `count` sequence is exactly `[2,4,6]` truncated to its length (enforces the documented 2/4/6 cadence — no other cadence is tolerated) AND the top breakpoint's count is reachable by the actual number of units carrying that trait id; every unit's `ability.effect.kind` is one of the 5 engine-supported kinds and `ability.manaCost === unit.mana` (abilities always cast at full mana, no partial-mana cast exists); items split into exactly 9 components + 36 completed (completed = has a `recipe`, excludes eagerly-materialized radiants which share `itemKind() === "completed"` but lack a recipe), every completed item's recipe resolves to two valid component ids forming a UNIQUE unordered pair (no two completed items share a recipe), and there are exactly 36 distinct pairs (= C(9,2), full coverage); exactly 3 consumables each with a valid `consumableEffect`; all 5 tiers have a positive pool count and appear with nonzero odds at some shop level.
- **`"phase 3 artifacts + mythicals"`**: exactly 6 artifacts + 3 mythicals; both use only `VALID_STATS` keys with strictly positive values; neither carries a `recipe` or a truthy `component` flag (they're not part of the component/recipe item economy); `pairPassive` artifacts are mutually consistent — if A pairs with B, B must reference A back, both must be artifacts, the pair count must be even and nonzero, and the passive effect kind must be `burn` or `shield`; mythicals NEVER carry a `pairPassive`; `itemTier(id, items, economy)` returns `economy.itemTierArtifact` (3) for every artifact and `economy.itemTierMythical` (5) for every mythical; artifacts appear in the `loot.json` `"rare"` table (and explicitly NOT in `"legendary"`), mythicals appear in `"legendary"` (and explicitly NOT in `"rare"`).

**Invariants & constraints**
- This suite is the canonical enforcement point for several hard content rules that CLAUDE.md states as facts (50 units 13/13/12/8/4, 22 traits' 2/4/6 cadence, 45 items split 9/36, 6 artifacts/3 mythicals) — if any of these numbers change, this test file must be updated alongside the content, or the suite fails by design.
- The breakpoint-cadence assertion (`[2,4,6].slice(0, length)`) is strict: a trait CANNOT use a custom cadence like `[3,5]` — only a prefix of `2,4,6` is valid content.
- The "tiers with zero units" test guards against a latent footgun: if a tier is configured in `economy.json` (pool/odds) but has no units in `units.json` yet, shops must never roll it (odds must be exactly 0 at every level) or the shop would produce dead/empty slots.
- This file has no side effects; it imports `gameData` once (module-level singleton from `loader.ts`) and asserts against it — running it never mutates data.

**Depends on** — `packages/data/src/loader.ts` (`gameData`, `itemKind`, `itemTier` — the only imports); transitively exercises every JSON file in `packages/data/src` (units/traits/items/economy/loot, and indirectly gameplay/mobs/ranks via the same `gameData` object, though those aren't directly asserted on here).

**Used by** — Run by `npm test` (vitest) as part of the data package's test suite; no other code imports this file (it's a leaf test, not a reusable module). Acts as a CI gate: any content change to units/traits/items/economy/loot that violates these invariants fails `npm test`.

**Notes** — The generic `scan()` helper for item references in units is explicitly forward-looking (a comment notes "Units carry no item references today" — this exists so a future content addition of equippable-by-default items on units would be caught automatically rather than requiring a new test). The radiant-item exclusion logic in the "9 components + 36 completed" test (`itemKind(i) === "completed" && i.recipe`) is a deliberate workaround for `loader.ts`'s eager radiant-item materialization (see `loader.ts` docs) — radiants share the "completed" itemKind but have no recipe, so they'd otherwise inflate the completed-item count past 36.
