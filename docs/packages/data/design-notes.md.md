# Path & purpose

`packages/data/design-notes.md` — the canonical content-design rationale document for `packages/data`. Records WHY the JSON content is shaped the way it is and explicitly lists deferred/future behaviors that must NOT be encoded in JSON because the engine doesn't support them yet. All actual tuning numbers live in the JSON files themselves; this file is prose-only and never read by code.

# Responsibility

Owns the design intent and historical/structural rationale behind: the 50-unit roster split, the 22-trait breakpoint derivation, the 5 engine-supported ability effects, the two-regime PvE/PvP round-stage formula, the mobs.json schema, the starting-bench-unit mechanic, the full item-system phases 2-3 design (consumables, artifacts, mythicals, pair-passives), and an explicit "future: do not implement yet" backlog. It is the most detailed and most CURRENT design source in the repo for the item system — substantially more detailed than CLAUDE.md's item-system summary, which does not mention consumables, artifacts, mythicals, or pair-passives at all.

# Exports

Not code — a markdown document, read by humans/agents only, never imported.

# Key behavior (content map)

**Roster**: 50 units, tiers split 13/13/12/8/4. Each unit has exactly one `origin` + 1-2 `classes`; `traits` field is the flattened `[origin, ...classes]` array the sim/rules resolve against. The original 12 units (pre-expansion) kept their exact stats/ids/order so old determinism fixtures stay valid.

**Traits**: 12 origins + 10 classes = 22 total. Breakpoints derived per-trait from unit count carrying it: `[2]`, `[2,4]`, or `[2,4,6]`, chosen so the top breakpoint is always reachable. `knight` is the one exception keeping its hand-authored armor curve (+200/+500/+800).

**Ability effects** (the complete engine-supported set — mirrors CLAUDE.md's sim section but stated from the content-design side): `magic_damage` (single-target nearest-enemy magic dmg via `abilityDamage`), `burn` (magic dmg + true-damage DoT), `shield` (self absorb shield for a duration), `buff` (self `ad` buff for a duration, reverted on expiry), `stealth` (start-of-combat untargetable window). Item passives reuse `burn` (on-hit) and `shield` (start-of-combat) — no separate primitive set for items.

**Stage formula** (implemented in `packages/rules/src/rounds.ts` as `stageForRound`/`isPveRound`; JSON only holds the camp/breakpoint tuning, the formula itself is structural code): Stage 1 = rounds 1-3, ALL PvE. Stage 2+ each spans exactly 7 rounds; within a stage, `roundInStage` 4 and 7 are PvE, the other 5 are PvP. `stageForRound(round)`: rounds 1-3 → `{stage:1, roundInStage:round}`; round 4+ → `stage = 2 + floor((round-4)/7)`, `roundInStage = ((round-4)%7)+1`. `isPveRound(round)` is true for all of stage 1 and for `roundInStage` 4/7 in stage 2+; accepts an ignored optional 2nd arg for call-site back-compat.

**mobs.json schema**: `stages[]` entries are keyed by `{stage, roundInStage, name, campType?, units}`, NOT a flat `round` number — `pveStageForRound` derives `{stage, roundInStage}` from the absolute round number and looks up by those two fields. `campType` is a PURELY COSMETIC display tag (shown in the combat header) with zero gameplay effect. Mob defs carry `isMob: true`, are never drawn from the unit pool, and never contribute to/receive player trait bonuses.

**Camp assignment table** (9 named camps across stages 1-4): Stage 1 rounds 1/2/3 = Wandering Pack / Raiding Party / Den Guardians (no campType tag, tier-2 boss on round 3). Stage 2 round 4/7 = Krugs/Krug Camp (1-star stone golems) / Wolves/Wolf Pack (1-star fast packs). Stage 3 round 4/7 = Raptors/Raptor Flock (high-attack birds) / Dragons/Dragon Brood (burn+magic dmg). Stage 4 round 4/7 = Krugs/Krug Warband (2-star) / Dragons/Ancient Dragon (2-star `ember_drake`). Dragon camps use only engine-supported abilities: `ember_drake` (tier 4) has `burn`, `dragon_whelp` (tier 3) has `magic_damage`. Difficulty scales via mob tier + star in `stages[].units`; stage 4 uses 2-star mobs for late-game PvE challenge.

**Default starting bench unit**: every player starts with one 1-star unit (id from `gameplay.startingUnitId`, NOT hardcoded in `match.ts`) drawn from the real unit pool via `drawFromPool` in `createMatch`, placed on bench slot 0. This counts against pool conservation (`totalInMatch` = pool + all bench/board/shop copies still equals `initialTotal()` at match start) — the 8 starting draws are a real pool deficit, not a separate free unit. If the pool is exhausted for that tier (edge case), the draw loop simply breaks and no unit is placed. In practice tier-1 `warrior` (the current starting unit) has 29 pool copies; 8 simultaneous draws leave 21.

**Consumables (item-system phase 2)** — a third `ItemDataDef.kind` alongside `component`/`completed` (an item with neither `component:true` nor a `recipe` defaults to `completed` for back-compat per loader notes). Consumables are NOT equippable (never occupy an item slot), NOT shop-purchasable, NOT craftable — the ONLY acquisition path is PvE loot. Used via the `USE_CONSUMABLE` rules command against a held inventory id; single-use (removed from inventory on resolution); always targets a unit. Three consumables, discriminated by `consumableEffect`:
- `item_remover` (`remove_item`): strips EVERY item equipped on the target unit, returns each to inventory unchanged (mirrors `UNEQUIP` looped over all slots). No randomness. Zero items equipped = normal no-op SUCCESS (not a typed error) — consumable not consumed, no state change.
- `reforger` (`reforge`): reforges EVERY equipped item, each independently replaced with a different random item of its OWN SAME TIER (component→1, completed→2, artifact→3, radiant→4, mythical→5), excluding its own id. Each slot gets its own prng draw in equipped-slot order off the match's seeded prng (determinism preserved). Consumed exactly once regardless of slot count. Tier-restricted so reforge never changes an item's tier. Zero items = no-op success. ANY equipped item with no same-tier alternative → whole command fails `NO_ALTERNATIVE_ITEM` (no partial reforge — all-or-nothing). Scarcity note: with only 6 artifacts/3 mythicals, targeting a unit holding one of those frequently hits `NO_ALTERNATIVE_ITEM` — this is INTENDED scarcity (reforger deliberately less reliable at high tiers); future tuning lever is expanding the roster, not changing the mechanic.
- `radiant_enhancer` (`radiant_upgrade`): targets exactly ONE named completed (tier-2) item via explicit required `targetItemId`, upgrading it to its tier-4 "radiant" variant — every stat in the base item's bundle scaled by `economy.radiantStatMultiplier` (1750 = ×1.75 fixed-point). Radiant items are NEVER hand-authored in items.json; the loader derives `radiant_<baseId>` lazily/once, the same pattern as `recipeResult` (pure, deterministic, generated from base item data). Zero tier-2 items equipped → `NO_TIER_2_ITEMS_EQUIPPED`. `targetItemId` present but not a tier-2 completed item (component/consumable/artifact/mythical/already-radiant) → `NOT_TIER_2_ITEM`.

**Rounding rule** (applies everywhere a stat is scaled, e.g. radiant multiply): `Math.round((statValue * multiplier) / 1000)` — NEAREST integer (ties away from zero), explicitly NOT truncation like `sim/fixed.ts`'s `fmul`/`fdiv` (which truncate for per-tick combat math). Item stat derivation is a one-time data transform, not hot combat math, so nearest-rounding avoids visibly shaving stats.

Selling/UNEQUIP never decomposes a completed/radiant item back into components — items always return to inventory as the single id they are. `SELL` now ALSO returns the unit's equipped items to inventory before removing the unit (previously items were destroyed with the unit) — folds in the same transfer `UNEQUIP` already does.

Client UI wiring (already implemented, not deferred): consumable chips render distinct in the inventory bar (own fill/rim + per-effect glyph, never the gilded equippable rim), drag only onto a unit. `item_remover`/`reforger` send `USE_CONSUMABLE` with no `targetItemId`; the renderer infers "changed" vs no-op purely from a state diff (renderer-is-dumb — never pre-inspects items to decide). `radiant_enhancer` opens a tier-2 item picker if the unit holds completed items (pick→send with `targetItemId`; dismiss→cancel, chip returns), else sends anyway and lets the server reject with `NO_TIER_2_ITEMS_EQUIPPED`.

**Artifacts and Mythicals (item-system phase 3)** — two more loot-only kinds. Tier overview table: tier 1 component (shop/loot), tier 2 completed (crafted/loot, 36 combos), tier 3 artifact (loot only, 6 hand-authored, may carry pair-passive), tier 4 radiant (derived via `radiant_enhancer`, never hand-authored), tier 5 mythical (loot only, 3 hand-authored). Artifacts/mythicals are NOT shop-purchasable, NOT craftable (no `recipe` field) — PvE loot only. Both ARE equippable (subject to the normal 3-slot cap) and eligible for `item_remover`/`reforger`, but NOT for `radiant_enhancer` (not tier-2 completed; targeting one → `NOT_TIER_2_ITEM`).

Artifact roster (6, power benchmark ~1.5× a typical tier-2 item's primary stat): `warblade` (ad 220, as 200, pairs with `warplate`), `warplate` (armor 420, hp 600, pairs with `warblade`), `voidstaff` (abilityDamage 280, mana 40, pairs with `voidmantle`), `voidmantle` (mr 320, hp 500, pairs with `voidstaff`), `stormrider` (as 300, ad 120, mr 150, unpaired), `titan_heart` (hp 1100, armor 250, unpaired).

Mythical roster (3, power benchmark ~2× a typical tier-2 primary stat): `eclipse_crown` (ad 250, abilityDamage 350, as 180), `undying_bulwark` (hp 1800, armor 450, mr 350), `arcane_engine` (abilityDamage 500, mana 80, as 200).

**Pair-passive mechanic**: two named pairs (Warblade+Warplate, Voidstaff+Voidmantle). When a unit holds BOTH artifacts of a pair, each item's `pairPassive` activates at combat start, reusing existing primitives (no new engine code) — Warblade+Warplate grants on-hit burn (value 40, duration 50 ticks); Voidstaff+Voidmantle grants start-of-combat shield (value 600, duration 120 ticks, STACKS with other shield sources). Gating is purely the `partnerId` equip check at combat start: if the partner id is present in the unit's `items` array, the passive fires; base stats always apply regardless. Both items in a pair reference each other via `pairPassive.partnerId`, so the effect is naturally applied once per item — for burn, the engine processes pairPassives in slot order so the second item's `onHitBurn` entry overwrites the first (same value, no practical difference); for shield, EACH item's pairPassive independently adds 600 shield, so both equipped = 1200 total shield (intentional, by design). Only artifacts carry `pairPassive` — mythicals never have one (field absent, no null/false sentinel for "no pair passive"); unpaired artifacts (`stormrider`, `titan_heart`) also lack it. Design intent: equipping both costs 2 of 3 slots for the bonus, creating a real slot-budget decision; counterplay is scouting + prioritizing burst/silence to kill the paired unit before it benefits over multiple rounds.

**Loot table placement**: artifacts in the `rare` rarity table (weight 3 each, 18 total new weight out of ~100+); mythicals in `legendary` (weight 6 each, 18 total new weight). Other entries reduced proportionally; `rare` gold entries reduced 18+10→14+8, `legendary` gold reduced 14→10, to keep total weight coherent.

**Reforge scarcity**: 6 artifacts (5 alternatives when reforging one) vs 3 mythicals (only 2 alternatives) — makes `NO_ALTERNATIVE_ITEM` a frequent, realistic outcome at mythical tier. Balance lever if it becomes a frustration point: expand the mythical roster (add 1-2), NOT relax the tier-restriction.

**Future: deferred, NOT implemented, do NOT encode in JSON** (each needs new engine primitives/event types, intentionally out of v1 scope): area-of-effect/cone/line damage; heals and resurrection (negative damage / revive events); crowd control (stun/root/knockback needing movement-lock state); lifesteal/omnivamp on-hit healing; mana-burn and ability-silence; recipe combination AT EQUIP TIME (currently rules equip any item id directly; recipes are data-only lookups, not an equip-time auto-combine beyond what `commands.ts`'s EQUIP auto-combine already does in place); full scripted tutorial match (only dismissable coachmarks exist, in client `onboarding.ts`); pair-passive on mythical items (currently artifact-only); named pair-passive interactions beyond burn/shield (would need new effect kinds like haste/omnivamp/mana-restore).

# Invariants & constraints

- This is a prose source-of-truth for WHY, never read by any runtime code — if it drifts from the actual JSON/engine behavior, nothing breaks mechanically, but agents relying on it for understanding intent would be misled. Cross-check against `items.json`/`loot.json`/`mobs.json`/`gameplay.json`/`economy.json` and `packages/rules/src/commands.ts` for current ground truth on exact values.
- The "future:" section is a hard boundary — these behaviors must NOT be encoded in JSON content even if tempting, because the engine has no primitive to execute them; encoding them would create dead/no-op content.
- CLAUDE.md's item-system description is INCOMPLETE relative to this file — it does not mention consumables (`USE_CONSUMABLE`, `item_remover`/`reforger`/`radiant_enhancer`), artifacts, mythicals, radiant items, or pair-passives at all. This design-notes.md file is the only place that fully documents item-system phases 2 and 3.
