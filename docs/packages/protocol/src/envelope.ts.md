# packages/protocol/src/envelope.ts

**Path & purpose** — `packages/protocol/src/envelope.ts`. Defines the wire `{v, t, p}` envelope shape, `encode`/`decode` functions for both directions (server→client and client→server), and every runtime field validator used to reject malformed messages before they reach application logic.

**Responsibility** — Owns the boundary between "raw bytes off the wire" and "typed, validated protocol message." Every message that crosses the WS connection passes through this file's encode/decode in both directions; nothing downstream (server room logic, client driver) needs to re-validate message shape.

**Exports**
- `PROTOCOL_VERSION = 1` — the envelope's `v` field; bumping this is the mechanism for breaking wire-format changes (not currently checked/enforced anywhere beyond being stamped on encode — decode does not reject a mismatched version).
- `isValidName(value): value is string` — type guard: a valid display name is a string of length 1-32 (`NAME_MAX_LEN`). Used wherever a name crosses the wire (also reused conceptually by the server's `validateName` for profile patches, though that's a separate, stricter implementation in `packages/server/src/http.ts`).
- `validateNameMap(value): Record<number, string> | null` — validates a seat→name map (used by `MATCH_END.names`): every value must pass `isValidName`; returns `null` (not throw) on any invalid entry.
- `validateLootOrbs(value): LootOrbWire[] | null` — validates an array of loot orbs (used by `LOOT.orbs`): each orb needs a `rarity` in `LOOT_RARITIES` (`common`/`uncommon`/`rare`/`legendary`) and a `reward` object shaped as `{kind:"gold", amount: number}` | `{kind:"component"|"item", id: string}`; any other `reward.kind` or missing field invalidates the whole array (returns `null`, not partial).
- `validateRoundResult(value): RoundResultWire | null` — validates a single round-result payload (used by `ROUND_RESULT.result`): `status` must be in `ROUND_RESULT_STATUSES` (`won`/`lost`/`bye`/`pve`), `damageTaken`/`damageDealt` must both be numbers.
- `validateMatchStatsMap(value): Record<number, MatchStats> | null` — validates a seat→stats map (used by `MATCH_END.stats`): every entry must have numeric `roundWins`/`roundLosses`/`totalDamageTaken`/`totalDamageDealt`.
- `validateUseConsumableCmd(cmd): boolean` — field-level validator for the `USE_CONSUMABLE` command payload: `consumableId` must be a string, `targetUnitId` must be a number, `targetItemId` if present must be a string; additionally `radiant_enhancer` (a specific consumableId) REQUIRES `targetItemId` to be present (it targets exactly one tier-2 item), while other consumable kinds (`remove_item`, `reforge`) now act on every item on the unit and tolerate `targetItemId` being absent (kept optional for client back-compat if a client still sends it — it's ignored).
- `interface Envelope { v: number; t: string; p: unknown }` — the wire shape every message is wrapped in.
- `encode(msg: S2CMessage): string` — wraps a server message in the envelope (`{v: PROTOCOL_VERSION, t: msg.type, p: msg}`) and `JSON.stringify`s it. Server-side only (S2C direction).
- `decodeEnvelope(raw: string): Envelope | null` — parses raw JSON text and validates the outer envelope shape (`v` is a number, `t` is a string, `p` key exists); returns `null` (never throws) on any parse error or shape mismatch.
- `decodeC2S(raw: string): C2SMessage | null` — full client→server decode pipeline: envelope-decode, then `validateC2S(env.p)`. Used by the server to validate incoming WS frames.
- `decodeS2C(raw: string): S2CMessage | null` — full server→client decode pipeline: envelope-decode, then checks `p.type` is a string, then runs message-specific field validators for the payloads that need them (`LOOT.orbs`, `ROUND_RESULT.result`, `MATCH_END.names` if present, `MATCH_END.stats` if present) — any failing validator rejects the WHOLE message (`null`), not just the bad field. Used by the client to validate incoming WS frames (defense against a buggy/malicious server or protocol drift).
- `validateC2S(p: unknown): C2SMessage | null` — the core client→server message validator/discriminator: switches on `p.type` and either constructs (and narrows) the typed message or returns `null`. Handles `QUEUE_JOIN` (optional `authToken: string`), `QUEUE_LEAVE`, `READY` (no payload), `PING` (`ts: number`), `RECONNECT` (`token: string`), and `CMD` (`cmd: {type: string, ...}` — additionally runs `validateUseConsumableCmd` if `cmd.type === "USE_CONSUMABLE"`). Any unrecognized `type` or shape mismatch returns `null`.

**Key behavior**
- All decode functions are total and exception-safe: malformed/garbage input always returns `null`, never throws — this is what lets the server treat "decode failed" as a normal, typed rejection path (e.g. responding with a typed `ERROR`) rather than crashing the connection handler.
- `decodeS2C`'s extra per-type field checks exist specifically because three message types carry compound/array payloads that a structurally-valid-but-semantically-wrong object could slip through a shallow type check (e.g. `orbs` being present but containing entries with an invalid `rarity` string) — those three (`LOOT`, `ROUND_RESULT`, `MATCH_END`) are explicitly listed in CLAUDE.md as the ones `decodeS2C` "rejects... with malformed orbs / a bad status or missing damage / malformed names or stats."
- `CMD`'s inner `cmd` object is only checked for having a string `type` at the envelope layer — deeper per-command-type field validation (other than `USE_CONSUMABLE`) is deferred to `packages/rules/src/commands.ts`'s `applyCommand`, which is the actual legality gate server-side. This file's job is shape-validity, not game-rule legality.
- `validateUseConsumableCmd`'s special-case for `radiant_enhancer` requiring `targetItemId` reflects a real asymmetry in the consumable system: `radiant_upgrade`-effect consumables target exactly one item, while `remove_item`/`reforge`-effect consumables act on a unit's full item set and don't need a specific target.

**Invariants & constraints**
- Decode is one-way trust: this file NEVER assumes the wire payload is well-typed just because it parsed as JSON — every field that matters is explicitly type-checked before being cast.
- The envelope's `v` (protocol version) is stamped on encode but NOT currently validated on decode (no version-mismatch rejection logic exists here) — a version bump would need new logic added to actually enforce compatibility; today it's purely informational.
- `validateC2S`'s `default: return null` means any message `type` not in its switch is silently rejected — adding a new C2S message type to `messages.ts` requires also adding a case here, or it will never decode.
- `decodeS2C` casts the final payload `p as unknown as S2CMessage` after only PARTIAL validation (full structural validation only for the 3 flagged types) — message types other than `LOOT`/`ROUND_RESULT`/`MATCH_END` get only the `type: string` check, not deep field validation. This is an intentional trust boundary: the server is the canonical source for those other types, so the client trusts their shape more than it trusts itself reconstructing loot/stats logic.
- Zero runtime dependencies (consistent with the package's zero-deps invariant) — uses only `JSON.parse`/`JSON.stringify` and plain object/array checks.

**Depends on** — `./messages.js` (compiled-extension import, but resolves to `messages.ts` in this TS-direct-execution monorepo) for the `C2SMessage`/`S2CMessage` union types, `LootOrbWire`/`RoundResultWire`/`MatchStats` payload types, and the `LOOT_RARITIES`/`ROUND_RESULT_STATUSES` const arrays used as the validators' enum sources.

**Used by**
- `packages/server` — `decodeC2S` validates every incoming WS frame before any room/session logic runs; `encode` serializes every outgoing S2C message (STATE_SNAPSHOT, PHASE_CHANGE, LOOT, ROUND_RESULT, MATCH_END, etc.).
- `packages/client/src/net.ts` (`NetClient`) — `decodeS2C` validates every incoming WS frame from the server before the driver/scene layer consumes it; outgoing C2S messages are constructed as typed objects and presumably encoded similarly (mirror path, though `encode` here is typed specifically for `S2CMessage` — the client's outgoing encode would use its own thin wrapper or the same envelope shape manually, since this file's `encode` is one-directional toward `S2CMessage`).
- `packages/protocol/tests/protocol.test.ts` — exercises every validator and the encode/decode round-trip directly.

**Notes** — `encode` is typed to accept only `S2CMessage`, meaning the protocol package provides no symmetric `encode` for `C2SMessage` — client-side C2S encoding must construct the `{v,t,p}` envelope shape itself (or via a thin local helper) rather than reusing this file's `encode`. This asymmetry mirrors the fact that only the server is expected to need a generic "encode any outgoing typed message" helper; the client's C2S message surface is small (6 types) and hand-constructed at each call site (see `net.ts`).
