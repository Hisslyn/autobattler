# packages/protocol/tests/protocol.test.ts

**Path & purpose** — `packages/protocol/tests/protocol.test.ts`. Vitest suite exercising the full protocol package: encode/decode round-trips for every S2C message type, field-level rejection of malformed payloads, every `validateC2S` message-type branch, and the standalone validators (`validateRoundResult`, `validateMatchStatsMap`, `validateLootOrbs`, `validateNameMap`/`isValidName`, `validateUseConsumableCmd`).

**Responsibility** — Owns regression coverage for the entire wire-protocol contract: this is what guarantees `encode`+`decodeS2C`/`decodeC2S` are mutual inverses for well-formed messages, AND that every documented rejection case (malformed loot rarity, bad round-result status, missing damage, malformed names/stats, bad USE_CONSUMABLE shape, missing envelope fields, malformed JSON, unknown message type) is actually caught.

**Exports** — None (test file). Imports `encode`, `decodeC2S`, `decodeS2C`, `validateC2S`, `validateLootOrbs`, `validateNameMap`, `validateRoundResult`, `validateMatchStatsMap`, `validateUseConsumableCmd`, `isValidName`, `PROTOCOL_VERSION` plus the `S2C_Loot`/`S2C_MatchEnd`/`S2C_RoundResult` types, all from `../src/index.js` (the public barrel, not the individual source files — this test exercises the package's PUBLIC surface, not internals).

**Key behavior**
- **`"encode/decodeS2C round-trip"`**: for `PONG`, `QUEUE_STATUS`, `MATCH_FOUND`, `MATCH_END` (bare, with `names`, with `stats`), `LOOT` (3 different reward kinds: gold/item/component across different rarities), and `ROUND_RESULT` (looped over all 4 statuses `won`/`lost`/`bye`/`pve` with status-appropriate damage values) — every case asserts `decodeS2C(encode(msg))` returns a value deep-equal to the original `msg`. This is the core round-trip guarantee.
- **`"S2C field validation (reject malformed)"`**: hand-constructs raw envelope JSON strings (bypassing `encode`, to simulate a malicious/buggy server) and asserts `decodeS2C` returns `null` for: a `LOOT` orb with rarity `"mythic"` (not in `LOOT_RARITIES`), `orbs` being a non-array string, `MATCH_END.names` with a non-string value (`{0: 42}`), `MATCH_END.stats` with an incomplete stats object (`{roundWins: 1}` missing the other 3 fields), `ROUND_RESULT.result.status` being `"tied"` (not in `ROUND_RESULT_STATUSES`), and a `ROUND_RESULT.result` missing `damageTaken`. Also directly unit-tests each standalone validator (`validateRoundResult`, `validateMatchStatsMap`, `validateLootOrbs`, `validateNameMap`/`isValidName`) against both valid and a battery of invalid inputs (wrong type, missing field, non-object, non-array).
- **`"validateC2S"`**: covers every C2S message type's happy path (`QUEUE_JOIN`, `QUEUE_LEAVE`, `READY`, `PING` with `ts`, `CMD` wrapping a `BUY` command) plus failure modes (`PING` missing `ts` → null, `CMD` missing `cmd.type` → null, `CMD` missing `cmd` entirely → null, unknown top-level `type` → null, non-object input (`"hello"`/`null`/`42`) → null). The bulk of this block is dedicated to `USE_CONSUMABLE` CMD validation: `remove_item`/`reforge`-style consumables (`item_remover`, `reforger`) pass with OR without `targetItemId`, and if `targetItemId` IS present it's accepted (and effectively ignored downstream) rather than rejected; `radiant_enhancer` REQUIRES `targetItemId` (missing → null, present → passes); generic malformed-field cases (missing `consumableId`, missing `targetUnitId`, `targetUnitId` as a string instead of number, `targetItemId` as a number instead of string) all reject.
- **`"validateUseConsumableCmd"`**: a more focused unit-test of just that one validator function in isolation — same `remove_item`/`reforge`-vs-`radiant_enhancer` asymmetry, plus base-field malformation cases, mirroring (and slightly duplicating) the CMD-level tests above but calling the validator directly rather than through `validateC2S`.
- **`"decodeC2S envelope"`**: full round-trip through the envelope layer — a well-formed `QUEUE_JOIN` envelope decodes correctly; an envelope missing the `v` field → null; non-JSON garbage string → null; an envelope with an unknown payload `type` → null; a `CMD`/`USE_CONSUMABLE` envelope missing `targetItemId` for `radiant_enhancer` → null (proves the field-level USE_CONSUMABLE check fires even through the full envelope-decode path, not just at the `validateC2S`-direct-call level); the symmetric accept case for `reforger` without `targetItemId`.

**Invariants & constraints**
- This suite imports ONLY from the package's barrel (`../src/index.js`), never from `../src/messages.js` or `../src/envelope.js` directly — it's testing the package as consumers actually use it.
- The round-trip tests are the load-bearing guarantee that `encode` and `decodeS2C` are inverses for every S2C message shape currently defined; adding a new S2C message type to `messages.ts` without adding a round-trip case here would leave that type's encode/decode behavior unverified.
- The malformed-payload tests hand-build raw JSON strings rather than going through `encode`, specifically to simulate inputs `encode` would never itself produce (a real attacker or a buggy alternate server implementation) — this is intentional and necessary to test the REJECT path, since `encode` by construction only ever produces valid envelopes from typed messages.

**Depends on** — `packages/protocol/src/index.ts` (the public barrel) — the only import; transitively exercises every function/const in `messages.ts` and `envelope.ts`.

**Used by** — Run by `npm test` (vitest) as part of the protocol package's test suite; not imported by any other module.

**Notes** — The `USE_CONSUMABLE` validation logic gets disproportionate test coverage (3 separate describe blocks touch it: `validateC2S`, `validateUseConsumableCmd` directly, and `decodeC2S envelope`) reflecting that it's the one C2S command type with extra field-level validation beyond "has a string `type`" — this asymmetric complexity (one consumable kind needs a target item, others don't) is exactly the kind of subtle rule a regression suite needs to pin down precisely, and this file does so thoroughly.
