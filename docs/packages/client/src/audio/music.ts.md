# Path & purpose

`packages/client/src/audio/music.ts` -- generative, loopable background music engine: composes original (non-copyrighted) music from pure music-theory helpers (progressions, motif generation/variation, voice-leading) wrapped around a Web Audio lookahead scheduler.

# Responsibility

Owns the entire generative music system: per-mood configuration (tempo, scale, chord progression, layer gains), the pure music-theory building blocks (chord stacking, motif generation/variation, voice-leading, frequency math), the lookahead scheduling arithmetic, and the `GenerativeMusic` engine class that schedules and renders pad/bass/arpeggio/lead/percussion layers into real Web Audio nodes. Does NOT handle file-based music override or mixing with SFX -- that's `manager.ts`'s job; this file only produces the generative sound when asked.

# Exports

- `type MusicMood = "menu" | "planning" | "combat"` -- the three generative moods (results state reuses "menu" mood per `director.ts`'s `stateToMood`).
- `interface MoodConfig { bpm, scale, rootMidi, progression, arpGain, padGain, percGain, arpSubdiv, level }` -- per-mood tuning: `scale` is semitone offsets from root within one octave, `progression` is a list of chords (each a list of scale-degree indices), `percGain: 0` disables percussion entirely (menu mood), `level` is the mood's overall loudness trim (music intentionally underscores, never dominates).
- `const PROGRESSIONS: Record<MusicMood, number[]>` -- the chord-ROOT scale-degree sequence per mood, one chord per bar, 4 bars per mood: `menu: [0,5,2,6]` (i-VI-III-VII, Am-F-C-G, "reflective, falling"), `planning: [0,3,5,4]` (i-iv-VI-V, Am-Dm-F-E, "warm, turning home"), `combat: [0,6,5,4]` (i-VII-VI-v, Am-G-F-Em, "driving, modal descent").
- `function triad(rootDegree: number): number[]` -- stacks a diatonic triad `[root, root+2, root+4]` (root/third/fifth as scale-degree offsets, not semitones) from a scale-degree root.
- `function progressionFor(mood: MusicMood): number[][]` -- maps `PROGRESSIONS[mood]` through `triad` to get the full bar-by-bar chord-tone progression.
- `const MOODS: Record<MusicMood, MoodConfig>` -- the three fully-realized configs: menu (66bpm, A-minor, arpSubdiv 1, no percussion), planning (94bpm, A-dorian, arpSubdiv 2, light percussion), combat (134bpm, A-minor, arpSubdiv 2, heavier percussion+bass). All rooted at MIDI 57 (A3).
- `interface MixTable { lead, pad, bass, arp, perc }` -- per-layer relative balance multipliers, each in (0,1].
- `const MIX: Record<MusicMood, MixTable>` -- per-mood mix tables; lead consistently sits above pad, percussion never dominates (`perc: 0` for menu, rising through planning/combat).
- `function midiToFreq(midi: number): number` -- equal-tempered MIDI-to-Hz: `440 * 2^((midi-69)/12)`.
- `function degreeToMidi(rootMidi: number, scale: number[], degree: number): number` -- resolves a (possibly out-of-range, even negative) scale-degree to an absolute MIDI note, wrapping octaves via `Math.floor(degree/n)` and modulo-indexing into `scale`.
- `function generateMotif(seed: number, len: number): (number | null)[]` -- deterministic (mulberry32-seeded) generation of a short melodic phrase as scale-degree offsets above the chord; starts on a chord tone (0/2/4), then walks mostly stepwise (+-1, 55% chance) with occasional skips (+-2, 30%) or rare leaps to a new chord tone (15%), with an 18% chance per note (after the first) of a rest (`null`); clamps the contour to roughly one octave (`deg>6 -> -4`, `deg<-2 -> +4`) to keep it "singable."
- `function varyMotif(motif: (number|null)[], pass: number): (number|null)[]` -- deterministically (seeded by `pass`) ornaments a motif for loop iteration `pass`: `pass===0` returns it unchanged (states the theme plainly first loop); subsequent passes may fill a rest (25%), open a new rest (15%), apply a neighbor-tone ornament (+-1, 25% of remaining), lift an octave (+7, 10% of remaining), or leave the note as-is.
- `function voiceLead(prevMidi: number[], chordDegrees: number[], rootMidi: number, scale: number[]): number[]` -- for each target chord-tone degree, picks the octave (-1/0/+1 relative to its natural register) whose resulting MIDI note is closest to ANY note in `prevMidi`, producing smooth/shared voice motion between chords rather than parallel block-chord leaps; returns the new chord's MIDI notes sorted ascending. Empty `prevMidi` (first chord) just uses the natural-register note.
- `function dueSteps(nextTime: number, now: number, lookahead: number, step: number): { times: number[]; nextTime: number }` -- pure lookahead-scheduler arithmetic: returns every grid time in `(now, now+lookahead]` starting from `nextTime` (catching up to `now` if `nextTime` fell behind, e.g. a backgrounded tab), plus the advanced cursor; guards against runaway loops with a 256-iteration cap.
- `class GenerativeMusic` -- the engine. `constructor(ctx: AudioContext, out: GainNode)`. `get running(): boolean`. `setMood(mood: MusicMood): void` -- re-parameterizes WITHOUT restarting (regenerates the per-mood motif, resets loop-pass tracking, and clears `prevChord` so voice-leading re-anchors fresh in the new key); no-ops if mood is unchanged. `start(): void` -- begins scheduling (idempotent if already running); resets beat/motif/voicing state fresh. `stop(): void` -- clears the scheduler interval; lets already-scheduled voices ring out naturally (doesn't hard-cut audio).

# Key behavior

- **Musical structure**: each mood is a 4-bar progression (matching `PROGRESSIONS[mood].length`) that "moves and resolves" rather than looping on one static chord; chord tones are voice-led bar-to-bar for smooth motion. A short 8-note motif (`MOTIF_LEN=8`) spans exactly the 4-bar loop (one note every 2 beats, `motifStep = BEATS_PER_BAR/2`), generated once per mood from a stable per-mood seed (`MOTIF_SEED: {menu:77, planning:156, combat:183}`) so each state has its own recognizable, reproducible phrase, then re-varied (`varyMotif`) once per full loop pass so repeats aren't identical.
- **Scheduling** (`schedule`/`scheduleBeat`): a `setInterval` (`TICK_MS=25`ms) wakes the scheduler, which uses `dueSteps` to find every beat-grid time within a `LOOKAHEAD=0.12`s window and calls `scheduleBeat` for each, advancing an internal `beat` counter. Each beat resolves its bar/in-bar-position, looks up that bar's chord (`cfg.progression[bar % progLen]`), and on downbeats (`inBar===0`): re-derives the motif variation if entering a new loop pass, voice-leads the new chord from `prevChord`, and schedules pad notes for each voiced tone. Bass plays the chord root (1-2 octaves down, combat goes 2 octaves + an offbeat pulse for drive). The arpeggio walks chord tones at `cfg.arpSubdiv` notes/beat with a slow octave drift for evolution over time. The motif plays its current note (transposed onto the current chord root, +1 octave) every 2 beats, skipping nulls (rests). Percussion (gated by `cfg.percGain>0`) ticks every beat with a downbeat accent, alternating low/high filtered noise.
- **Voice rendering** (`pad`/`bass`/`lead`/`pluck`/`tick`, all private): each builds a small disposable node graph (2 detuned oscillators or 1 + filter(s) + gain envelope) per note, scheduled at an exact AudioContext time and auto-stopped after its duration -- no node reuse/pooling (acceptable since music notes are sparse relative to combat SFX). `pad` is explicitly high-passed (180Hz) to stay out of the bass's low range and the lead's midrange ("keeps the stack out of the mud" per inline comment). `bass` is low-passed at 320Hz for a "round, sub-y" tone. `lead` uses a brighter triangle+sine(2x) combo low-passed at 2600Hz so the motif reads clearly above the pad bed.
- **Determinism**: `generateMotif`/`varyMotif` use a local `mulberry32` PRNG (re-implemented in this file, NOT imported from `packages/sim/src/prng.ts` -- this is presentation-layer randomness, not gameplay-affecting, so the duplication is intentional/acceptable per the sim-purity invariant which only constrains gameplay code) -- same seed always produces the same note sequence, which is what makes the per-mood motifs stable and the per-pass variations reproducible/testable.

# Invariants & constraints

- `generateMotif`/`varyMotif`/`voiceLead`/`dueSteps`/`triad`/`progressionFor`/`midiToFreq`/`degreeToMidi` are all PURE and must stay that way -- explicitly called out in the file header as deliberately extracted/unit-tested so musical logic doesn't require a real Web Audio context to test.
- `MOTIF_LEN=8` must continue to evenly divide/align with `BEATS_PER_BAR * loopBars` (4 bars x 4 beats / 2 = 8) -- changing the loop length without updating this would desync the motif from the progression's loop seam.
- `PROGRESSIONS` arrays must stay length-4 (or whatever `loopBars` is derived from) since `bar % progLen` / `bar % loopBars` indexing throughout `scheduleBeat` assumes consistent loop length across pad/bass/motif.
- The local `mulberry32` here is a SEPARATE implementation from `packages/sim/src/prng.ts` -- do not assume they're the same function/import; this duplication is acceptable ONLY because music randomness is presentation, not simulation (would violate the sim-purity invariant if used the other way around).
- `stop()` clears the scheduler but deliberately does NOT stop in-flight scheduled oscillator/buffer nodes -- they ring out naturally. Don't "fix" this to hard-stop; it's intentional to avoid audio clicks.

# Depends on

Nothing internal to the audio system except implicitly being driven by `manager.ts` (which owns the `AudioContext`/`GainNode` passed into the constructor) and `director.ts` (`stateToMood` maps a `MusicState` to the `MusicMood` this file consumes). No imports from other modules at all (the file is entirely self-contained aside from the Web Audio global types).

# Notes

- The file header explicitly states "composed from scratch (no recognizable melodies)" -- a deliberate authorship/copyright consideration baked into the design rationale, not just a technical comment.
- `results` music state (from `director.ts`) maps to the `"menu"` mood, not a dedicated fourth mood -- there is no `results` entry in `MOODS`/`MIX`/`PROGRESSIONS`; any code trying to look up `MOODS.results` would fail at compile time (the type system prevents it) but it's worth knowing the match-over screen audibly reuses menu music by design, not by oversight.
- Octave-drift logic in the arpeggio (`drift = floor(seq / (chordDeg.length*2)) % 2 * scale.length`) is a slow, deterministic alternation rather than randomness -- arpeggio notes are NOT seeded/randomized, only the motif and its variations are.
