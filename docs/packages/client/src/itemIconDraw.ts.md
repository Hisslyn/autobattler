# Path & purpose

`packages/client/src/itemIconDraw.ts` -- the Pixi drawing half of the item-icon system: paints procedural emblem(s) for every item (single emblem for a component, two composed emblems inside a gilded frame for a completed item), handles the consumable-glyph fallback, and owns the runtime drop-in-PNG art cache. Pure resolution logic lives in `itemIcon.ts`; this module only paints.

# Responsibility

Owns the actual Pixi `Graphics`/`Sprite` construction for every item icon rendered anywhere in the client, the lazy-loading runtime texture cache for drop-in `public/items/<id>.png` art (mirroring `sprites.ts`'s unit-art cache), and the consumable-specific glyph fallback (since consumables have no recipe/emblem and would otherwise render nothing).

# Exports

- `function itemTextureLookup(path: string): PIXI.Texture | null` -- the synchronous cache-read function passed into `itemIcon.ts`'s generic `resolveItemTexture` at runtime; returns the cached texture, `null` if cached-as-absent, or `null` if never attempted (cache miss and lookup-miss are indistinguishable from this function's return value alone -- both return `null`).
- `function onItemArtReady(cb: () => void): () => void` -- subscribe to "an item texture finished loading" notifications (so a static, already-rendered view can redraw once art lands); returns an unsubscribe function.
- `function requestItemArt(itemId: string): void` -- fire-and-forget lazy loader: builds the art path via `itemArtPath`, returns immediately if already attempted (`cache.has(path)`), else marks it attempted up-front (`cache.set(path, null)` -- "dedupe concurrent renders," so two render calls in the same frame don't both kick off a duplicate `PIXI.Assets.load`), then asynchronously loads via `PIXI.Assets.load<PIXI.Texture>(path)`. On success, overwrites the cache entry with the real texture and notifies every `onItemArtReady` listener. On failure (404/decode error), the catch handler does nothing further -- the cache entry stays `null` from the up-front mark, so the procedural emblem fallback persists and the load is NEVER retried.
- `interface ItemIconOpts { radius?: number; rarity?: string; reducedMotion?: boolean; dimmed?: boolean }` -- icon draw options: `radius` (default 14, half-size in px), `rarity` (a loot rarity string driving frame/shine tint via `rarityColor`, defaults to the plain `itemFrame` theme color when absent), `reducedMotion` (skips the animated shine sweep, static frame remains), `dimmed` (halves alpha, e.g. for an eliminated/disabled context).
- `function drawItemIcon(parent: PIXI.Container, itemId: string, x: number, y: number, opts?: ItemIconOpts): number` -- the main entry point, documented below in Key behavior. Returns the resolved `frameTint` color number (so "callers can match other surfaces" -- e.g. a chip's own border color matching the icon's rarity tint).

# Key behavior

- **`drawItemIcon`'s resolution order**: (1) checks for a drop-in PNG via `resolveItemTexture(itemId, itemTextureLookup)` -- if found, draws a `PIXI.Sprite` clipped to a `radius*2` square centered at `(x,y)`, applies the dimmed alpha, and returns immediately (NO frame, NO emblem -- "a PNG always wins, clipped to a disc, no frame/emblem"). (2) If no texture yet, calls `requestItemArt(itemId)` to kick off a lazy load for NEXT time, then falls through to draw the procedural fallback for THIS render. (3) Resolves `itemIcon(itemId, gameData)` (the pure resolver from `itemIcon.ts`); if `null` (typically a consumable, which has no recipe/emblem), looks up the item's `consumableEffect` and maps it through the LOCAL `CONSUMABLE_GLYPH` table (`remove_item`->`bag`, `reforge`->`refresh`, `radiant_upgrade`->`sun`) to draw a fallback glyph via `glyphs.ts`'s `drawGlyph` -- "so the chip never renders empty"; if the id resolves to neither a real icon NOR a consumable effect (an unexpected/unknown id), draws literally nothing (explicitly noted as "should not happen for real items"). (4) If `icon.kind === "component"`, draws ONE emblem (`drawEmblem`) at `radius*1.5` size in the neutral `C.itemEmblem` tint, unframed. (5) If `icon.kind === "completed"`, draws the gilded frame (`drawCompletedFrame`) THEN both recipe emblems side-by-side inside it -- emblem 0 offset up-left (`x - r*0.34, y - r*0.1`) tinted `C.itemEmblem`, emblem 1 offset down-right (`x + r*0.34, y + r*0.14`) tinted `C.itemEmblemAlt` (a DIFFERENT tint specifically "so it reads as 'built from X + Y'" -- the two component sources stay visually distinguishable even when overlapping).
- **`drawEmblem`** (internal): one `switch` over `ItemEmblem`, each case a small hand-drawn vector shape using the same fill/stroke pattern as `glyphs.ts` -- `blade` (sword silhouette: blade polygon + crossguard line + grip line + pommel dot), `vest` (hexagonal torso-plate polygon + a collar-notch detail stroke in `C.tokenBg` -- punching a visual "hole" using the background color rather than true geometric subtraction), `crystal` (faceted gem polygon + a center facet-line detail stroke), `bow` (an arc + bowstring line + nocked-arrow line + arrowhead triangle), `cloak` (a bezier hooded-mantle silhouette + center seam line), `belt` (a strap rect + a square buckle rect + an inner buckle-hole rect in `C.tokenBg`), `rod` (a staff line + an orb-head circle + a faint halo ring), `glove` (a rounded-rect fist + a thumb rounded-rect + three knuckle-ridge tick lines), `flask` (a bezier rounded-vial body + a rectangular stopper). Every shape uses `glyphStrokeWeight(size)` (from `theme.ts`) for its primary stroke width, the SAME shared formula `glyphs.ts` uses -- explicitly so "item emblems and class glyphs are visually interchangeable at the same render size."
- **`drawCompletedFrame`** (internal, the gilded frame + shine): draws a filled+stroked rounded-rect frame (fill `C.itemCompleted`, stroke in the rarity `tint`), then a small corner "gilding tick" (an L-shaped stroke in the top-left corner) "so the frame reads as 'completed' even color-blind" (a deliberate accessibility redundancy beyond color alone). Always draws a STATIC subtle shine glint (a slanted parallelogram, alpha reduced further under `reducedMotion` -- 0.1 vs 0.18). Unless `reducedMotion`, additionally builds an ANIMATED shine sweep: a second parallelogram masked to the frame's rounded-rect bounds (via a `PIXI.Graphics` used as a `.mask`), driven by a `PIXI.Ticker.shared` callback that loops its x-position over a `periodMs=2600` cycle using wall-clock `performance.now()` modulo arithmetic (deterministic relative to real time, not frame-count -- consistent speed regardless of framerate). The ticker callback self-removes via `sweep.once("destroyed", ...)` -- whenever the sweep Graphics is destroyed (e.g. the icon's container is torn down), the ticker entry is cleaned up automatically, preventing a leaked per-icon ticker callback.
- **Cache semantics** (`cache: Map<string, PIXI.Texture|null>`): three possible states per path -- absent from map (never attempted), present with `null` (attempted, failed/pending), present with a real `Texture` (succeeded). `requestItemArt` transitions absent->null synchronously (before the async load even starts) specifically to dedupe concurrent calls within the same render pass; the eventual async resolution either leaves it at `null` (failure) or upgrades it to the real texture (success, also firing listeners).

# Invariants & constraints

- **A drop-in PNG always wins over the procedural icon, unconditionally** -- per the doc comment "a PNG always wins, clipped to a disc, no frame/emblem"; this is consistent with the project-wide art-drop-in-slot convention (mirrors `sprites.ts` for units) -- absence is a clean no-op fallback, presence is an unconditional override with no partial-composition (a dropped-in PNG never gets the gilded frame treatment).
- **A failed/absent art load is cached as a permanent `null` and NEVER retried** -- per `requestItemArt`'s comment, "404/decode failure caches null... never retried, exactly like the unit-art slot." If a PNG is later added to `public/items/` after a failed lookup already cached `null` for that path, the running client session would NOT pick it up without a page reload (the cache has no TTL/invalidation).
- **`CONSUMABLE_GLYPH` must stay in sync with `ConsumableEffect`'s full value set** -- it's a local (not exported, not shared with `itemIcon.ts`) lookup specifically for the three known consumable effects; adding a new `ConsumableEffect` value to the data/types without adding a corresponding glyph entry here would silently leave that consumable's icon completely blank (the `glyph` lookup would be `undefined` and the `if (glyph)` guard skips drawing anything).
- The animated shine sweep's ticker is added to the GLOBAL `PIXI.Ticker.shared` -- every simultaneously-visible completed-item icon (e.g. a full inventory bar of completed items) adds its OWN independent ticker callback; this is a per-icon cost, not a single shared animation loop, so a very large number of simultaneously rendered completed-item icons could accumulate many ticker callbacks (each individually cheap, just a position update, but still worth knowing for perf auditing under a busy inventory).
- `drawItemIcon`'s return value (`frameTint`) is always computed up-front and returned from EVERY branch (texture-found, consumable-glyph, component, completed) even though it's only visually USED in the completed-item branch (the frame's stroke color) -- callers relying on the returned tint to "match other surfaces" should be aware it's the rarity-or-default tint regardless of whether this particular icon actually rendered a frame.

# Depends on

`pixi.js` (PIXI namespace: `Graphics`, `Sprite`, `Container`, `Assets`, `Ticker`, `Texture`). `@autobattler/data` (`gameData` -- looked up for the consumable-effect fallback and passed into `itemIcon`). `./theme.js` (`C`, `rarityColor`, `glyphStrokeWeight`). `./itemIcon.js` (`itemIcon`, `resolveItemTexture`, `itemArtPath`, types `ItemEmblem`, `ItemIcon` -- the pure resolution layer this module paints from). `./glyphs.js` (`drawGlyph`, type `GlyphKind` -- the consumable-fallback glyph drawing).

# Used by

Any client surface rendering an item icon: the inventory bar chips and loot-orb reveal contents in `scenes/match.ts`, equipped-item icons on board/bench `UnitToken`s (via `unitToken.ts`), and the inspect panel's equipped-items row + item-detail modal (`inspectPanel.ts`'s `renderUnitInspect`/`renderItemDetail`/`renderItemPicker`, all of which call `drawItemIcon` directly).

# Notes

- The file's emblem-drawing approach (vector shapes via fill/stroke combos, shared stroke-weight formula) is explicitly "the same approach as glyphs.ts" -- a reader familiar with that file's conventions (half-extent `s = size/2`, relative offsets, `glyphStrokeWeight`) can apply the same mental model here.
- The corner "gilding tick" detail on the completed-item frame is a small, easy-to-miss accessibility feature (color-blind-safe completed/component distinction) worth preserving in any future frame-redesign work.
- `itemTextureLookup`'s `?? null` at the end (`return cache.get(path) ?? null`) is technically redundant for a `Map<string, PIXI.Texture|null>` (a present `null` value and an absent key both already coerce to `undefined`/`null` appropriately via `.get`), but makes the "never returns `undefined`" contract explicit for `resolveItemTexture`'s generic signature.
