# Path & purpose

`packages/client/src/driver.ts` -- defines the `IDriver` interface (the single abstraction the match scene programs against) and `LocalDriver`, the offline/Practice-mode implementation that runs the full match state machine locally in the browser using `packages/rules` directly, on its own timers.

# Responsibility

Owns the contract between the Pixi match scene and "whatever is actually running the match" (local simulation vs networked server), and provides the local/Practice implementation: an in-browser authoritative loop driving `MatchState` through `createMatch`/`advancePhase`/`applyCommand`/`applyAiCommands` from `@autobattler/rules`, emitting a `DriverEvent` stream the scene subscribes to, and owning the planning/resolution/playback-cap timers that pace a Practice match exactly like a server would pace an online one.

# Exports

- `type Outcome = "win" | "loss" | "draw"`.
- `interface IDriver` -- the full driver contract every match-scene consumer programs against (implemented by both `LocalDriver` here and `NetDriver` in `netDriver.ts`):
  - `readonly seatIndex: number` -- which seat this driver represents (always 0 for `LocalDriver`/the human in Practice; varies for `NetDriver`).
  - `on(listener): () => void` -- subscribe to `DriverEvent`s; returns an unsubscribe function.
  - `getState(): MatchState` -- the live `MatchState` snapshot (NOTE: returns the actual mutable object in `LocalDriver`, not a deep copy -- callers must treat it as read-only).
  - `startPlanning(): void` -- begins a planning phase (starts the local clock for `LocalDriver`).
  - `playerCommand(cmd): CommandResult` -- submits a player command (BUY/SELL/MOVE/etc, the exact type from `applyCommand`'s third parameter) and returns its `CommandResult` (never throws).
  - `ready(): void` -- present on the interface for compatibility; `LocalDriver`'s implementation is an explicit no-op (see Key behavior).
  - `getMyPairing(): {opponentId, isGhost, side} | null` -- this seat's current-round PvP pairing, or null (no pairing, e.g. PvE/bye rounds).
  - `getMyOpponentBoard(): (UnitInstance|null)[] | null` -- the opposing board for the current round (PvP opponent's board, or the PvE mob board on a PvE round).
  - `getMyCombatResult(): CombatResult | null` -- this seat's last combat result (PvP or PvE).
  - `getMyOutcome(): Outcome | null` -- win/loss/draw normalized to this seat.
  - `isPveRound(): boolean` -- true if the current round is a PvE creep round.
  - `getPveStageName(): string | null` -- the PvE stage's display name, or null if not PvE.
  - `getMyLootOrbs(): LootOrb[]` -- this seat's already-decided loot orbs for the round (empty array when none).
  - `getMyRoundResult(): RoundResult | null` -- this seat's resolution-screen result for the just-finished round.
  - `getMatchStats(): Record<number, MatchStats> | null` -- per-seat accumulated match stats (round W/L, total damage dealt/taken).
  - `getUpcomingPveBoard(): BoardState | null` -- the upcoming PvE mob board, ONLY valid during PLANNING for a PvE round; uses negative uids (display-only, must never be fed to the sim, per the doc comment -- it would collide with nothing real but isn't meant for combat).
  - `combatPlaybackDone(): void` -- scene-to-driver signal that combat-log playback has finished or was skipped; `LocalDriver` holds RESOLUTION until this fires (capped); `NetDriver` ignores it (server-paced).
  - `advanceFromResolution(): void` -- advance out of the resolution screen (normally fired by the Continue button or an auto-advance timer).
  - `getPlanningTimeLeft(): number` -- ms remaining in the current planning phase, 0 if not applicable.
  - `skipPlanning?(): void` -- OPTIONAL; playtest-only convenience that ends planning immediately via the exact same path the timer fires on expiry (no logic bypass). Present only on drivers that own the planning clock (`LocalDriver`); absent on `NetDriver` (the scene hides the Skip UI affordance when this is `undefined`).
  - `dispose(): void` -- tear down timers/sockets when leaving a match.
- `const HUMAN_PLAYER_ID = 0` -- the fixed seat id `LocalDriver` always treats as the human in Practice mode (not exported... actually it's a top-level `const`, not exported, used only internally).
- `const PLANNING_TIMER_MS = 30_000` -- EXPORTED specifically so "the HUD timer dial can derive its fill fraction from the same constant the local clock counts down" -- the authoritative planning-phase duration.
- `type DriverEvent = {type:"state";state} | {type:"phase_change";phase;round} | {type:"match_over";placements;mmr?;stats?}` -- the event stream shape; `match_over` optionally carries per-seat `mmr` before/after (populated by `NetDriver` from the server's `MATCH_END`, never by `LocalDriver` since Practice has no MMR) and per-seat `stats` (populated by both drivers).
- `class LocalDriver implements IDriver` -- documented fully below.

# Key behavior

- **Constructor**: seeds a fresh `MatchState` via `createMatch(seed, gameData)` (seed defaults to `Date.now()` -- this is the one place determinism is deliberately broken for variety in Practice, mirroring how online matches get a server-issued seed instead), then sets `name` on every player object directly (seat 0 = the passed `humanName` or `"You"`, the rest = `"Bot N"`) -- explicitly noted that "Names live only on the client view (not used by the sim)" (a side-channel mutation onto the rules' `PlayerState` objects purely for client display, harmless to game logic). Emits an initial `{type:"state"}` event.
- **Event emission**: `on`/`emit` is a simple synchronous listener-array pub/sub; no event queueing or deduplication.
- **PvE-aware accessors**: `getMyOpponentBoard`/`getMyCombatResult`/`getMyOutcome` all branch on `isPveRound()` first -- PvE rounds have no PvP pairing (`getPairingFor` would return null), so they read `state.lastOpponentBoards`/`lastCombatResults` directly keyed by `HUMAN_PLAYER_ID` instead. `getMyOutcome` for PvE computes win/draw/loss purely from `result.winner` (0=player win, since the player board is always side 0 against mobs) -- explicitly noted as "purely informational" since PvE never damages HP/streak.
- **`getMatchStats()`** builds a fresh `Record<number, MatchStats>` by reading the 4 accumulator fields (`roundWins`, `roundLosses`, `totalDamageTaken`, `totalDamageDealt`) straight off every player's live `PlayerState` -- not cached, recomputed on every call.
- **`getUpcomingPveBoard()`** is gated to ONLY return non-null during the PLANNING phase (`if (this.state.phase !== "PLANNING") return null`) -- explicitly because outside PLANNING the "upcoming" board concept doesn't apply (it's either already the live combat board during COMBAT, or stale during RESOLUTION); delegates to `previewPveStage(state, gameData)` from rules.
- **`startPlanning()`**: records `planningStartTime = Date.now()`, emits a `phase_change` event, and arms a single `setTimeout` for the FULL `PLANNING_TIMER_MS` -- "Planning always runs the full duration; the timer (not a manual ready) is the only thing that advances out of planning." This is a deliberate design choice: there is no early-ready-to-skip-planning path in Practice (unlike the online server, which per `CLAUDE.md` allows "READY from all human seats skips the 30s wait" -- `LocalDriver` does NOT implement that shortcut; `ready()` is an explicit no-op here).
- **`playerCommand(cmd)`**: rejects immediately with `{ok:false, error:"PHASE_INVALID"}` if not currently in PLANNING (no `applyCommand` call at all in that case); otherwise delegates straight to `applyCommand(state, HUMAN_PLAYER_ID, cmd, prng, gameData)` and, only on success, emits a `state` event (failed commands do NOT trigger a re-render event -- the caller must read the `CommandResult`'s `ok:false` to surface a toast/error itself).
- **`ready()`** is a documented no-op -- kept on the interface purely so any caller still holding a reference doesn't break; the actual advance-out-of-planning path is exclusively the timer.
- **`skipPlanning()`** (optional interface member, IMPLEMENTED here): a playtest convenience that simply calls the SAME private method (`_onPlanningTimerFired`) the real timer invokes on expiry -- "no sim bypass, no new logic," it just collapses the wait.
- **`_onPlanningTimerFired()`** (private): clears the timer handle, guards against a stale fire after the phase already changed (`if (this.state.phase !== "PLANNING") return`), then runs `_applyAllAiCommands()` followed by `_advanceToResolution()`.
- **`_applyAllAiCommands()`** (private): iterates every player except the human and skips dead ones, calling `applyAiCommands(state, player.id, prng, gameData)` per bot -- the seeded bot policy from `packages/rules/src/ai.ts`.
- **`_advanceToResolution()`** (private): calls `advancePhase(state, gameData)` ONCE, which per the rules state machine takes the match from PLANNING through COMBAT (running the actual sim) and lands it in RESOLUTION -- but this method emits a `phase_change: "COMBAT"` event immediately afterward (even though the state itself has already progressed past COMBAT into RESOLUTION by the time this runs) specifically so the scene starts COMBAT-phase rendering/playback at the right moment; then emits a `state` event with the post-combat state (carrying the full combat result/log for the scene to play back). It then arms `pendingResolution = true` and a `playbackCapTimerId` fallback timeout -- this cap exists so a missing/broken scene integration can never permanently stall the match (RESOLUTION would otherwise wait forever for `combatPlaybackDone()`). The cap duration is carefully computed: `oneXMs = ceil(result.ticks * 1000 / TICK_HZ)` (the real 1x playback duration at the canonical 30 Hz sim rate, imported from `@autobattler/sim/src/fixed.js`), then `capMs = ceil(oneXMs / SLOWEST_PLAYBACK_SPEED) + PLAYBACK_CAP_BUFFER_MS` (scaled by the slowest selectable speed, 0.25x, so the cap is 4x the 1x duration plus a 2-second buffer) -- this guarantees the cap timer can NEVER fire before genuine playback at the slowest speed setting would finish. Uses the PvE-aware `getMyCombatResult()` accessor specifically because a PvE round produces a real combat result/log too (no pairing, but still a result) -- using a pairing-only accessor here would wrongly compute `capMs=0` for PvE rounds and tear the mob board down on frame 1.
- **`combatPlaybackDone()`**: guarded by `pendingResolution` (so a stray/duplicate call from the scene, or the cap timer firing after the scene already called it, is a safe no-op); clears the cap timer; if the match is now over (`isMatchOver(state)`), emits `match_over` with `placements` + (if available) `stats`, and stops. Otherwise emits `phase_change: "RESOLUTION"` and a `state` event "while the state is actually in RESOLUTION" (i.e. this is the moment the resolution screen should actually render, distinct from the earlier COMBAT phase_change emission), then arms a `resolutionTimerId` auto-advance timeout of `gameData.economy.resolutionSeconds * 1000` -- the resolution screen's auto-advance-after-N-seconds behavior, matching the documented "VISIBLE auto-advance countdown" in `scenes/match.ts`.
- **`advanceFromResolution()`**: the Continue-button/auto-advance path out of RESOLUTION. Guards: must be in RESOLUTION (`!== "RESOLUTION"` -> no-op), must NOT still be mid-playback (`pendingResolution` true -> no-op, "playback still running"). Clears the resolution timer. If match-over, emits `match_over` (same shape as in `combatPlaybackDone`) and stops; otherwise calls `advancePhase` again (RESOLUTION -> PLANNING, applying income + shop refresh per the rules state machine) then `startPlanning()` (re-arming the full 30s timer) and emits a fresh `state` event.
- **`dispose()`**: clears all three timer handles (planning/resolution/playbackCap), nulls them, and empties the listener array -- the scene/app must call this when leaving a Practice match to avoid orphaned timers firing into a torn-down scene.

# Invariants & constraints

- **`LocalDriver` is the ONLY driver that owns real wall-clock timers** -- `NetDriver` is server-paced and reads server messages instead; this is why `skipPlanning` is optional on `IDriver` and intentionally absent from `NetDriver` (the server, not the client, owns the planning clock online).
- **Renderer-is-dumb / driver-mediates-everything**: per `CLAUDE.md`, ALL planning-phase actions must go through `driver.playerCommand` -> `applyCommand`; `LocalDriver.playerCommand` is the concrete enforcement point for Practice mode -- it never lets a command through outside PLANNING and never mutates `MatchState` itself (delegates entirely to `applyCommand`).
- `getState()` returns the live, mutable `MatchState` object directly, NOT a defensive copy -- any caller that mutates the returned object directly (rather than going through `playerCommand`) would corrupt the authoritative state with no validation; this is an implicit contract callers must respect (read-only consumption only).
- The `playbackCapTimerId` cap calculation MUST always be >= the real slowest-speed playback duration -- the inline comment is explicit that "it must never fire BEFORE genuine playback completes," and the formula is deliberately tied to `SLOWEST_PLAYBACK_SPEED` (0.25) rather than a flat constant, so that if the slowest selectable speed ever changes, the cap's safety margin tracks it (assuming `SLOWEST_PLAYBACK_SPEED` here is kept in sync with whatever speeds `combat/player.ts`'s `PlaybackSpeed` type actually allows).
- `_advanceToResolution`'s ordering of emitted events (COMBAT phase_change with the state already past COMBAT, THEN later a RESOLUTION phase_change once playback truly finishes) is intentional and must not be "simplified" by collapsing the two emissions -- the scene relies on receiving the COMBAT event to start playback of a state that already contains the full result, and the separate RESOLUTION event to know when to actually show the resolution screen.
- `getUpcomingPveBoard`'s negative-uid boards are explicitly DISPLAY-ONLY ("never feed to the sim") -- any code path that might pass this board into `simulateCombat` or any rules function expecting real combat uids would be a bug.

# Depends on

`@autobattler/data` (`gameData` -- the loaded content singleton, passed into every rules call). `@autobattler/rules/src/state.js` (type-only: `MatchState`, `RoundResult`). `@autobattler/sim/src/types.js` (type-only: `UnitInstance`, `CombatResult`, `BoardState`). `@autobattler/rules/src/loot.js` (type-only: `LootOrb`). `@autobattler/protocol` (type-only: `MatchStats`). `@autobattler/rules` (barrel: `createMatch`, `advancePhase`, `isMatchOver`). `@autobattler/rules/src/commands.js` (`applyCommand`). `@autobattler/rules/src/ai.js` (`applyAiCommands`). `@autobattler/rules/src/rounds.js` (`getPairingFor`, `isPveRound`, `pveStageForRound`, `previewPveStage`). `@autobattler/sim/src/prng.js` (`mulberry32` -- seeds this driver's local command/AI-decision PRNG stream, separate from the sim's own internal combat PRNG which is seeded per-combat).

# Used by

`packages/client/src/scenes/match.ts` (the match scene is constructed with an `IDriver` and is agnostic to whether it's a `LocalDriver` or `NetDriver`). Practice mode specifically instantiates `LocalDriver` (likely from `main.ts` or the mode-select UI flow) and passes the human's profile name in. `packages/client/src/netDriver.ts`'s `NetDriver` implements the same `IDriver` interface as the networked counterpart -- the scene/app code is written once against `IDriver` and works against either.

# Notes

- `HUMAN_PLAYER_ID` is hardcoded to `0` and not exported -- `LocalDriver.seatIndex` is ALSO hardcoded `0`, so in Practice mode the human is always seat 0 by construction (`createMatch`'s player array always places... actually this assumes the rules' `createMatch` always orders the human as players[0]; the comment "seat 0 is the human" confirms this is a deliberate, hardcoded Practice convention, not something derived from server seat assignment as it would be online).
- The `ready()` no-op combined with the always-full-duration planning timer means Practice mode currently has NO way to advance planning early via a "Ready" button, unlike the documented online behavior where "READY from all human seats skips the 30s wait" -- a reader implementing a Practice "ready up" UI feature would need to either wire it to `skipPlanning()` (the playtest convenience) or extend `LocalDriver` with real ready-tracking logic; today's `ready()` is purely a compatibility stub.
- `dispose()` clears timers but does NOT emit any final event (no "disposed" `DriverEvent`) -- callers must independently know to stop listening/rendering after calling `dispose()`.
