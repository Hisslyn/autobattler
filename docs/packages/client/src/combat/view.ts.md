# Path & purpose

`packages/client/src/combat/view.ts` -- the Pixi rendering layer for combat playback: renders strictly from `CombatPlayer` frames (reducer output + fx stream), animates HP/mana bars, particles, projectiles, ability auras, dissolves, screen shake, and the overtime tint -- never reads `MatchState` or game logic directly.

# Responsibility

Owns ALL visual presentation of an in-progress combat: drawing every alive unit's token (depth-sorted), animating one-shot fx (`CombatFx` from `player.ts`) into actual particles/projectiles/auras/floaters, smoothing HP/mana bars toward their target values, managing death dissolve animations, screen shake, and the overtime banner/edge tint. This is the terminal consumer of the combat pipeline: `engine.ts` (sim) -> `reducer.ts` (state fold) -> `player.ts` (clock + fx derivation) -> `view.ts` (this file, pure rendering).

# Exports

- `type HexToPixel = (hex: HexCoord) => { x: number; y: number }` -- the coordinate-mapping function callers must supply (typically composed with `boardProjection.ts`'s perspective warp).
- `class CombatView`:
  - `readonly container: PIXI.Container` -- the root display object to add to the scene graph; contains `unitLayer` (tokens) and `fxLayer` (all particles/projectiles/auras, drawn above units) as children, in that order.
  - `constructor(toPixel: HexToPixel, bannerPos: {x,y}, opts: { reducedMotion?, scale?, edge?: {w,h}, entityScale?: (hex)=>number })` -- `toPixel` is the hex-to-screen mapping; `bannerPos` positions the "OVERTIME" text; `opts.edge` sizes the overtime edge-tint rect (defaults to `2*bannerPos`); `opts.scale` is a uniform board render scale (landscape shrinks the whole board; portrait uses 1); `opts.entityScale` is a PER-HEX depth-scale function (smaller for farther-back hexes under the board's perspective tilt, defaults to constant 1) -- this is exactly `BoardProjection.scaleAt` plumbed through from the caller.
  - `renderFrame(frame: PlaybackFrame, dtMs: number): void` -- the single per-tick entry point: clears and redraws all alive unit tokens (depth-sorted), spawns any new one-shot fx (skipped entirely under reduced motion), advances projectiles/transients/dissolves/timers/shake, and renders the overtime banner/tint if active.
  - `destroy(): void` -- clears all internal arrays and destroys the Pixi container tree (with children).

# Key behavior

- **Depth sort**: every frame, units are sorted by their projected screen Y (`this.toPixel(hex).y` ascending) before drawing, so a nearer (lower-on-screen under the tilted board) unit visually draws in front of a farther one -- correct occlusion under the perspective projection.
- **Bar smoothing** (`drawUnit`): each unit's HP/mana track a per-unit `BarState {hp, hpChip, mana}` in the `bars` map, eased toward the frame's true `hp/maxHp`/`mana/maxMana` fractions via `ease(cur, tgt, rate, dtMs) = cur + (tgt-cur)*min(1, rate*dtMs)` (exponential smoothing). `hp` eases fast (`HP_RATE=0.012`), `mana` smoothly (`MANA_RATE=0.009`), and `hpChip` (the trailing "white damage chip" effect) eases even slower (`CHIP_RATE=0.004`) but is force-corrected (`hpChip = max(hp, hpTarget)`) whenever it would otherwise lag BEHIND the current hp (i.e. the chip only trails on damage, never lags on healing/recovery). Under reduced motion, all three snap instantly to target (no easing).
- **Per-unit container transform**: each token is drawn inside its own `PIXI.Container` so two transient effects can compose without fighting: a crit "scale-pop" (`pop` map, decaying `1 + 0.25*(remainingMs/POP_MS)` scale) and a hit "recoil bob" (`recoil` map, decaying `-3*(remainingMs/RECOIL_MS)` Y offset). Both are timers ticked down in `tickTimers`.
- **Token radius is depth-scaled but the token itself stays upright**: `r = UNIT_R * this.scale * this.entityScale(u.hex)` -- only the SIZE shrinks with apparent distance (per `entityScale`, i.e. `BoardProjection.scaleAt`), the token graphic itself is never actually skewed/rotated to match the board tilt ("Upright + screen-aligned token; only its radius is depth-scaled" per inline comment) -- this is a deliberate simplification: full perspective-correct token warping was not implemented, only scale.
- **fx spawning** (`spawnFx`, the switch over `CombatFx.kind`): `projectile` spawns a small traveling dot+ring graphic added to a `projectiles` array (tracked separately from `transients` since its position updates by interpolating a travel fraction, not by a TTL fade) -- capped at `MAX_PROJECTILES=24`, silently dropped past that. `contact` spawns a short directional streak line toward 70% of the attack vector (a quick "lunge" suggestion), TTL-based. `impact`/`floater`/`abilityHit`/`dissolve` each delegate to dedicated helper methods. `overtime` is a no-op here (the actual overtime banner/edge is driven by `frame.overtime`, checked every frame in `renderOvertime`, not by the one-shot fx event).
- **Projectile landing** (`advanceProjectiles`): computes travel fraction `t = (tick - startTick) / travelTicks`; once `t>=1` and not yet landed, marks `landed=true` and calls `spawnImpact` at the destination (this is where the view spawns the ranged-attack's landing impact, exactly as documented in `player.ts`: the emitter only sends the bolt, the view spawns its own landing effect once travel completes). Landed projectiles are removed from the layer and their Graphics recycled into the pool.
- **Impact** (`spawnImpact`): a stroked ring (bigger + colored `floatCrit` if `crit`, else `fxImpact`), sets a `recoil` timer on the target, and on crit additionally sets a `pop` timer plus calls `addShake(SHAKE_MAG_CRIT)`.
- **Floater** (`spawnFloater`): a `PIXI.Text` reading `-<amount>` (with a trailing `!` on crit), colored by crit/magic/physical (`floatCrit`/`floatMagic`/`floatPhys`), font size 13 (crit) vs 9 (normal), rising upward (`rise: 0.03` px/ms) over `FLOATER_TTL_MS=700`ms, fading out via alpha.
- **Ability aura** (`spawnAbilityAura`): per-`AbilityFxKind` visual via `abilityColor(kind)` (a simple lookup: `magic_damage`->`fxAbilityMagic`, `burn`->`fxAbilityBurn`, `shield`->`fxAbilityShield`, `buff`->`fxAbilityBuff`, `stealth`->`fxAbilityStealth`): shield draws a stroked ring + faint fill (a "bubble"), buff a faint filled disc, stealth a faint filled disc (smaller alpha, suggesting fading), magic_damage/burn a sharp bright stroked burst ring that also grows (`grow:0.006`) over its `AURA_TTL_MS=520`ms lifetime.
- **Death dissolve** (`startDissolve`/`advanceDissolves`): when a `dissolve` fx fires, the unit's LAST drawn identity/position/radius is snapshotted from the `lastUnit` cache (populated every `drawUnit` call) into the `dissolves` map -- this lets the view keep rendering a fading copy of the unit even after the reducer has already removed it from the frame's alive set (since `renderFrame`'s main loop only draws `frame.units` that are `alive`). Each dissolve tick draws a fresh token at `y + 8*k` (sinking), `scale 1-0.25*k` (shrinking), `alpha 1-k` (fading) where `k = ageMs/DISSOLVE_TTL_MS` (480ms); expires and is removed once `k>=1`. Also triggers `addShake(SHAKE_MAG_DEATH)` and clears the unit's bar/recoil/pop state.
- **Screen shake** (`addShake`/`applyShake`): `addShake(mag)` takes the MAX of the new magnitude and the current magnitude decayed by remaining time fraction (so a second hit during an active shake doesn't simply reset/stack additively, but also doesn't get ignored if it's a bigger hit) -- capped at `SHAKE_MAG_CAP=5`. `applyShake` offsets the WHOLE `container` position using a deterministic (non-random) two-frequency sine/cosine oscillation phased by remaining shake time (`shakeMs`), decaying linearly to zero over `SHAKE_MS=200`ms -- explicitly "no RNG" so shake is itself part of the deterministic-render guarantee.
- **Object pooling** (`acquireGfx`/`recycleGfx`/`gfxPool`): most one-shot Graphics nodes are drawn fresh each spawn but recycled back into a shared free-list (capped at `MAX_TRANSIENTS=48`, excess destroyed) rather than always calling `new PIXI.Graphics()` -- a mobile-perf optimization for busy full-board combats. `addTransient` additionally enforces the `MAX_TRANSIENTS` cap by evicting the OLDEST transient (FIFO) when a new one would exceed budget, rather than refusing the new spawn -- "newer, more relevant fx still show" per the inline comment.
- **Reduced motion**: `renderFrame` skips `spawnFx` ENTIRELY (`if (!this.reducedMotion) for (const fx of frame.fx) this.spawnFx(fx)`) -- no particles/projectiles/auras spawn at all under reduced motion (note this is a SEPARATE, more aggressive cut than `player.ts`'s reduced-motion downgrade, which still emits some fx like `impact`/`floater`/`abilityHit`/`dissolve` -- this view-layer gate drops literally everything in the fx stream, relying on bar/token rendering alone for combat readability). Bar easing also snaps instantly rather than smoothing. The overtime edge-tint graphic is also skipped under reduced motion (only the text banner remains).

# Invariants & constraints

- **Renderer-is-dumb**: this file must render STRICTLY from `PlaybackFrame` (positions/hp/mana/alive from the reducer, fx from the one-shot stream) -- "never reads MatchState," per the file header. No game-logic decisions belong here; if something needs new combat-affecting logic, it belongs in the sim/reducer/player layers, not here.
- **Determinism**: per the header, "the render is a pure function of (log, seed)" -- this is why screen shake explicitly avoids `Math.random()` (deterministic sine/cosine phase) and why every animation timer is driven by `dtMs` (caller-supplied) rather than wall-clock reads. Any new visual effect added here must preserve this: no `Math.random()`, no `Date.now()`.
- Reduced-motion gating exists in BOTH this file and `player.ts` (per `CLAUDE.md`: "Reduced-motion gating lives in BOTH layers") -- `player.ts` downgrades the emitted fx SET (so audio/readability survive), `view.ts` ADDITIONALLY refuses to spawn ANY fx regardless of what's emitted, plus disables bar easing and the overtime edge graphic. A reader modifying reduced-motion behavior must update both layers consistently, understanding they're not redundant -- they serve different purposes (emission control vs render-spawn control).
- `lastUnit` is the single source of identity for a unit that has just died -- if a `dissolve` fx fires for a uid never seen in `drawUnit` (shouldn't happen in practice since death always follows at least one alive frame), `startDissolve` silently no-ops (`if (!last) return`).
- The `MAX_TRANSIENTS`/`MAX_PROJECTILES` caps are deliberate mobile-performance budgets, not arbitrary -- per the inline comment "Particle budget for a full board on mobile: drop new spawns past the cap" (projectiles) and "Budget hit: evict the oldest" (transients). Raising these caps without performance testing on lower-end mobile hardware could reintroduce frame drops on a busy 8-unit-vs-8-unit board.
- Bar state (`bars` map) is deleted whenever a unit becomes not-alive in the current frame (`renderFrame`'s main loop: `else this.bars.delete(u.uid)`) -- a unit's bar state does NOT persist into its dissolve animation; dissolve draws a static last-known token without bars at all (`drawUnitToken(..., { radius: d.r, piece: {} })` -- no `bars` option passed).

# Depends on

`pixi.js` (PIXI namespace -- Containers, Graphics, Text). `@autobattler/data` (`gameData` -- used once per `drawUnit` call to look up a unit's `tier` from its `defId`, since `UnitFrame` doesn't carry tier directly). `@autobattler/sim/src/hex.js` (type-only: `HexCoord`). `./player.js` (type-only: `AbilityFxKind`, `CombatFx`, `PlaybackFrame`, `UnitFrame` -- the entire input contract this view renders). `../theme.js` (`C` -- the theme color palette: `fxAbilityMagic/Burn/Shield/Buff/Stealth`, `fxProjectile`, `fxProjTrail`, `fxAttackFlash`, `fxImpact`, `floatCrit/Magic/Phys`, `textOvertime`, `fxOvertimeEdge`). `../unitToken.js` (`drawUnitToken` -- the shared token-drawing primitive also used for board/bench tokens).

# Used by

`packages/client/src/scenes/match.ts` (constructs a `CombatView` for the active combat phase, feeds it `PlaybackFrame`s from a `CombatPlayer` every tick via `renderFrame`, tears it down via `destroy()` on phase change). `packages/client/src/audio/manager.ts` consumes the SAME `frame.fx` stream in parallel (via `handleCombatFx`) -- `view.ts` and the audio manager are siblings both reading from `player.ts`'s output, not from each other.

# Notes

- `gameData.units.find((d) => d.id === u.defId)` runs once per alive unit EVERY frame in `drawUnit` -- a linear scan over all unit defs per unit per frame; given the dataset is only 50 units and boards are small (<=8v8), this is presumably not a real perf concern, but a reader optimizing render perf might consider memoizing defId->tier once per combat rather than re-scanning every frame.
- The "checkers piece" comment (`piece: {}` passed to `drawUnitToken`) refers to a specific visual mode of the shared token component -- a "short volume" battlefield presentation distinct from how the same token might render on the bench/shop (worth checking `unitToken.ts`'s own doc for what `piece` options exist).
- Mob units (PvE) are rendered through this exact same pipeline with no special-casing visible in this file -- their distinct "neutral ring" tinting is handled inside `drawUnitToken`/`unitToken.ts` (a defId absent from `data.units` is detected there), not here; this file's `gameData.units.find(...)` for a mob defId would return `undefined` and fall back to `tier ?? 1`, meaning mob tokens render as tier-1-colored unless `unitToken.ts` separately overrides the ring color for mobs.
