# Path & purpose

`packages/client/src/scenes/match.ts` -- the single in-match Pixi scene (`MatchScene`): the entire visible/interactive surface for one match (planning UI, combat playback, resolution overlay, match-over overlay), rendered strictly from `MatchState` snapshots + `CombatEvent` logs delivered by an `IDriver`. This is the largest file in the client (~4300 lines) and the terminal consumer of nearly every other pure model/layout/theme module in `packages/client/src`.

# Responsibility

Owns: all Pixi rendering for the match (board, bench, shop, HUD, trait rail, item inventory bar, scout/inspect/trait-detail modals, loot reveal, combat playback wiring, resolution/match-over overlays), all pointer interaction (tap-to-buy, drag-to-move/sell/equip/combine, long-press-to-inspect), all phase-transition side effects (timers, music state, audio cues, layer teardown/rebuild), and translating every player intent into a `driver.playerCommand(...)` call. Does NOT own game logic, command validation, or simulation -- per the renderer-is-dumb invariant, it renders from snapshots/logs and never re-derives outcomes; every number shown (refunds, damage, MMR deltas) is read off data the driver/rules already computed.

# Exports

- `interface MatchSceneOptions { settings, audio, onLeave, onPause, layout }` -- constructor options: `settings` (`SettingsStore`), `audio` (`AudioManager`), `onLeave` (callback to return to the menu -- called from the pause panel or the match-over Main Menu button), `onPause` (opens the DOM pause panel; the ☰ button itself lives in the Pixi HUD so it scales with the viewport, but the modal itself is owned by the shell/`UiApp`), `layout` (the initial `MatchLayout`).
- `class MatchScene` -- the scene. Public surface: `readonly container: PIXI.Container` (the root, added to `app.stage` by the caller), `constructor(app, driver, opts)`, `onLayoutChange(layout: MatchLayout): void` (re-renders the current phase against a new layout -- called by `main.ts` on every resize/orientationchange), `destroy(): void` (tears down all timers/tickers/subscriptions and removes the container's children).

# Key behavior

**Construction**: builds 18 named `PIXI.Container` layers (`buildSceneLayers`) bound 1:1 by `zIndex` to the named constants from `combatLayout.ts` (`L0_BOARD_ENV` through `L8_TOAST`, plus several hand-picked "magic-ok" values for the Skip pill/shop panel/backdrop that sit between the documented layers) -- stacking is governed by `zIndex`, NOT `addChild` order; the container itself is `sortableChildren=true`. Creates an invisible full-screen `dragCatcher` Graphics (zIndex 900, `eventMode:"none"` until a drag starts) that captures `pointermove`/`pointerup`/`pointerupoutside` during any unit or item drag. Subscribes to `driver.on(...)`: a `"state"` event calls `render(state)`; a `"phase_change"` event dispatches to `onPlanningStart`/`onCombatPhase`/`onResolutionPhase` by phase name; a `"match_over"` event calls `onMatchOver(placements, mmr, stats)`. Also subscribes to `onUnitArtReady`/`onItemArtReady` (drop-in PNG art finishing its lazy load) to repaint the static planning board so glyphs swap to sprites without waiting for the next state push. Finally renders the current state once and calls `driver.startPlanning()`.

**`render(state)`**: the PLANNING-phase entry point, called on every `"state"` event regardless of phase (always renders the HUD; only renders board/bench/shop/trait-rail/item-panel content when `state.phase === "PLANNING"`). This is the function that keeps the scene a pure function of state -- no separate diffing, the relevant layers' children are torn down and rebuilt each call (the per-region `render*` methods each clear their own layer first).

**Phase transitions**:
- `onPlanningStart()`: tears down combat playback/resolution timers/loot reveal/press state, clears the combat layer, closes scout/inspect modals, resets the trait-rail pager, sets music state to `planning`, plays the `roundStart` SFX, plays `goldGain` if this seat's gold increased since last planning entry (voicing income -- no game logic, just comparing `me.gold` to a remembered `prevGold`), re-renders, and starts the visible planning countdown ticker (`startPlanningTimerTick`, ~4Hz, recolors to `hpLow` at ≤5s remaining).
- `onCombatPhase()`: clears the planning timer, hides the playtest Skip pill, sets music state to `combat`. Bridges the cut with a 120ms alpha cross-fade of the planning layers (skipped entirely under reduced-motion, which renders combat immediately) -- the fade's completion callback double-checks the phase is STILL `COMBAT` before calling `renderCombat` (a fast PvE round can already be in RESOLUTION by the time the 120ms fade finishes, which would otherwise stamp the combat header over the resolution overlay).
- `renderCombat(state)`: tears down any prior playback, clears combat/board/bench/shop/trait layers, draws the board background panel + arena tealight torches (gold meters, both sides), resolves the opponent's display name (PvE stage name+"Creeps", `Player N`, or "Ghost"/"Bye" variants), draws BOTH hex zones (enemy half tinted `mobZone` for PvE, `enemyHex` otherwise; own half `myHex`), then either starts real event-log playback (`startPlayback` + `renderPlaybackControls`) when `driver.getMyCombatResult()` has events, or renders a static own-board snapshot and immediately calls `driver.combatPlaybackDone()` (a bye/no-pairing round). A "COMBAT · vs X" header text is added LAST with an explicit `zIndex` (`Z_COMBAT_HEADER`) so the sortable `combatLayer` always draws it above the hex tiles regardless of insertion order -- PvE rounds render NO header text (the PvE identity is communicated only via the enemy-zone tint + the pre-combat board preview).
- `startPlayback(events, side)`: constructs a `CombatPlayer(events, ticksPerSec, gameData, {reducedMotion})` and a `CombatView(toPixel, bannerPos, {reducedMotion, scale, edge, entityScale})`, wiring a per-tick ticker function that calls `player.advance(deltaMS)` then `view.renderFrame(frame, deltaMS)`. `toBoard`/`toPixel`/`entityScale` are the THREE coordinate-conversion closures every combat visual routes through: `toBoard` maps a sim hex coordinate to BOARD-SPACE pixels via `toDisplayHex` (keeps "my" units on the bottom rows regardless of which sim `side` they're actually on) + `hexToPixel`; `toPixel` additionally runs that through the board's perspective `fwd()` (the projective homography from `boardProjection.ts`); `entityScale` reads the same board-space point's depth scale so entities (and floaters/particles) shrink correctly with board perspective.
- `onResolutionPhase()` / `renderResolution(state)`: shows the round-result detail (win "Dealt N damage" / loss "−N HP" / PvE/bye "no damage", numbers from `driver.getMyRoundResult()`), starts the loot reveal animation if this is a PvE round (`startLootReveal`), and starts both an actual auto-advance timer (`RESOLUTION_AUTO_ADVANCE_MS`, derived from `gameData.economy.resolutionSeconds`) and a visible draining-bar/seconds countdown ticker so the UI never silently drifts from the real auto-advance.
- `onMatchOver(placements, mmr?, stats?)`: sets music to `results`, plays `roundWin`/`roundLoss` SFX based on whether this seat placed top-4, draws a themed dim-scrim + rounded panel (sized from a CONTENT BUDGET that accounts for header + N placement rows + an optional MMR line + an optional personal-stats line + the Main Menu button, clamped into the safe area via `centeredModal` so 8 placements never overflow even on a short landscape viewport), lists every seat's placement (`#N name (You)` for this seat, resolved via `seatName` which falls back to `"Player N"` when no name is known), an MMR delta line (color-coded green/red) if `mmr[seat]` is present, a personal stats line (`Record W–L`, `Dealt X · Taken Y`) if `stats[seat]` is present -- ALL numbers come straight from the driver/rules, the renderer only formats and displays them -- and a "Main Menu" button wired to `opts.onLeave()`.

**Combat-layer z-stack discipline**: the constructor's `buildSceneLayers` doc comment is itself load-bearing: several containers intentionally share one `zIndex` (`L2_UNITS` for `benchLayer`+`planningFxLayer`, `L5_HUD` for `hudLayer`/`shopLayer`/`traitLayer`/`shopToggleLayer`/`combatLayer`/`lootLayer`, `L6_INSPECT` for `scoutLayer`+`inspectLayer`), and within a shared `zIndex` the STABLE TIE-BREAK is `addChild` insertion order -- so the file's `addChild` call sequence is not arbitrary, it encodes the intended sub-ordering (planning VFX over board/bench tokens; combat overlay + loot reveal over persistent HUD chrome; inspect over scout). The drop-down shop panel + its full-screen backdrop scrim get hand-picked `zIndex` values (870/871, marked `// magic-ok`) deliberately ABOVE the Skip pill (850) and HUD but BELOW the active drag sprite (999)/drag catcher (900), so an in-progress drag is never intercepted by an open shop panel.

**Drag/drop interaction model**: unit drags (`startDragBoard`/`startDragBench` → `onDragMove` → `onDragEnd`) create a floating `dragSprite` tracked via the global `dragCatcher`, resolving drop targets via `boardSlotAt`/`benchSlotAt`/`inSellZone` and issuing `MOVE`/`SELL` commands through `driver.playerCommand`. Item drags (`startDragItem` → `onItemDragEnd`) similarly resolve onto a unit (EQUIP, auto-combine surfaced via a flourish derived from the unit's item-slot-count delta) or onto another inventory item (COMBINE_ITEMS, with a live `combinePreview` hint rendered via `updateCombineHint` -- a NO_RECIPE preview shows an `itemCombineNo` ring + toast and sends NO command). Tapping (not dragging) a shop card buys; tapping a bench/board unit selects it (for a subsequent move/sell tap) or opens the sell zone; long-pressing any unit (shop/bench/board) opens the inspect panel (`armInspect`/`openUnitInspect`) via a `pressTimer` that's cancelled if a drag starts first.

**Drop-down shop panel** (`shopPanelOpen`/`shopPanelOffsetY`/`animateShopPanel`): the shop is hidden behind a money-sack toggle button (`renderShopToggle`/`drawMoneySack`) rather than always visible; opening slides the panel down from off-screen (animated via `shopPanelAnimFn`, a ticker closure) and shows a full-screen interactive backdrop (`shopBackdropLayer`) that swallows all other pointer events and dismisses the panel on an outside tap. `SHOP_PANEL_HIDE_MARGIN` (4px) ensures the closed panel's border fully clears the screen top rather than leaving a stray 1-2px gold rim visible.

**Arena tealight candles** (`drawArenaTorches`/`drawTorchPillar`, `TORCH_W/H/FRONT/BACK/MID` constants): a gold-economy visualization -- 5 candles per side, the player's (left) column spanning the board's front half and the opponent's (right) column the back half, each candle's screen size depth-scaled by the board projection so nearer candles read larger. `torchLit`/`TORCHES_PER_SIDE` come from `torchMeter.ts` (not yet read in this excerpt but referenced) -- the lit-candle count is presumably a function of the player's gold.

# Invariants & constraints

- **Renderer-is-dumb**: every `render*` method reads from `MatchState`/`CombatEvent`/the pure models (`inspectModel`, `traitStripModel`, `xpProgress`, `sellValue`, `itemModel`, `combinePreview`, `lootRevealModel`, etc.) -- never re-derives game outcomes. All mutating player actions funnel through `driver.playerCommand(cmd)`.
- **`zIndex` is the enforced stacking law**, not `addChild` order -- a reader adding a new layer must assign it a `zIndex` consistent with `combatLayout.ts`'s documented stack, and if it shares a zIndex with an existing layer, must insert it in the correct relative `addChild` position for the tie-break to resolve correctly.
- The combat header's `zIndex = Z_COMBAT_HEADER` exists specifically because `combatLayer` is `sortableChildren=true` and would otherwise render the header in raw insertion order (potentially behind later-added hex tiles) -- any new combat-layer child added AFTER the header in source order needs its own zIndex consideration or it could visually cover the header.
- `onCombatPhase`'s fade-completion re-check of `this.driver.getState().phase === "COMBAT"` is a deliberate race-condition guard against fast PvE rounds finishing before the 120ms cross-fade animation does -- removing this check would let a stale combat render stamp itself over an already-active resolution overlay.
- The resolution auto-advance timer (`RESOLUTION_AUTO_ADVANCE_MS`) is derived directly from `gameData.economy.resolutionSeconds` specifically so the visible countdown can never drift from the actual driver/server auto-advance -- a hardcoded duration here would be a footgun if `economy.json`'s value ever changes.
- Online play (`NetDriver`) auto-skips playback to the end if the server advances the phase first (per CLAUDE.md) -- this file's playback methods don't special-case that explicitly visible here, but `combatPlaybackDone()`/`skipPlayback()` are the convergence points; a reader investigating online-playback-skip behavior should look at how `onCombatPhase`/`renderCombat` react to a `phase_change` arriving mid-playback.
- The Skip pill (`renderSkipButton`, `skipLayer`) is explicitly PLAYTEST-ONLY (Practice/local mode) -- it calls `driver.skipPlanning?.()` which is `undefined` on `NetDriver` (per `driver.ts`'s `IDriver` doc), so this file must hide/disable it whenever the active driver lacks that optional method (consistent with CLAUDE.md's "the scene hides the Skip affordance when this is undefined").
- Every teardown path (`onPlanningStart`, `renderCombat`, `onResolutionPhase`, `onMatchOver`, `destroy`) repeats the same defensive sequence (`teardownPlayback`, `clearResolutionTimer`, `clearLootReveal`, `clearPress`, `closeScout`, `closeInspect`) -- any NEW per-phase transient state (a future overlay, a future ticker) must be added to ALL of these teardown call sites or it will leak across phase transitions / persist after `destroy()`.

# Notes

- This file is the terminal integration point for nearly all the "pure model" files documented elsewhere (`hudModel.ts`, `inspectModel.ts`, `traitDetailModel.ts`, `sellValue.ts`, `itemModel.ts`, `combinePreview.ts`, `lootReveal.ts`, `benchLayout.ts`, `layout.ts`) -- a reader wanting to understand any ONE piece of in-match UI behavior should expect to find its pure logic in one of those smaller files and its RENDERING/wiring here.
- The combat-layer zIndex sharing scheme (multiple containers at `L5_HUD`, tie-broken by `addChild` order) is the single most fragile invariant in this file from a "future edit breaks something subtly" perspective -- there's no automated enforcement beyond the `combatLayout.test.ts` regression guard mentioned in CLAUDE.md (header-outranks-tiles), so most of the OTHER orderings here (planning-fx-over-tokens, loot-reveal-over-combat-overlay, inspect-over-scout) are enforced only by code review / this doc, not by tests.
- `prevGold`'s income-voicing logic (`goldGain` SFX on planning entry if gold increased) is a small piece of presentation-only "no game logic" inference from two consecutive state snapshots -- worth flagging if it's ever wrongly assumed to BE economy logic; it's purely descriptive of an already-applied change.
- The drop-down shop panel + corner money-sack toggle is a LANDSCAPE-style interaction (per `layout.ts`'s notes that `regions.shop` means the toggle button in landscape, the actual card strip in portrait) -- this single file's `renderShop`/`renderShopToggle`/`renderShopPanel` triad has to handle BOTH orientations' differing meanings of the shop region, which is a meaningful source of complexity concentrated here.
