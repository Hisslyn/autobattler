# Path & purpose

`packages/client/src/layout.ts` -- the pure (no Pixi, no DOM, no I/O) orientation/scale-to-fit/region-rect resolver for the entire match scene. Given a viewport size + safe-area insets, produces a `MatchLayout` describing the design canvas size, CSS scale/offset, and every named UI region's rect in design space. This is the single source of truth for "where does each piece of match UI go" -- `scenes/match.ts` renders entirely from `this.layout.regions` rather than hardcoded positions.

# Responsibility

Owns: (1) orientation selection (landscape vs portrait) from viewport aspect ratio, (2) portrait's height-driven region budget algorithm (`portraitRegions`, detailed in `PORTRAIT_LAYOUT_SPEC.md`), (3) landscape's 4-fixed-edge-cluster + residual-board region architecture (`landscapeRegionsFor`/`landscapeClusterThickness`), (4) safe-area-inset-aware positioning so no interactive element sits under a device notch in either orientation, (5) a handful of small pure geometry helpers (`centeredModal`, bench slot hit-testing, opponent-rail tile geometry, shop panel slot/card-content layout, planning-region pointer hit-testing) shared by the renderer and (per their "unit-tested" framing) test suites.

# Exports

- `const LANDSCAPE_THRESHOLD = 1.1` -- the aspect-ratio (`viewportW/viewportH`) cutoff: `>= 1.1` is landscape, below is portrait. Deliberately above 1.0 so a square-ish device still reads as portrait; comfortably below both 16:9 (~1.78) and 19.5:9 (~2.17) landscape ratios.
- `const PORTRAIT_W = 390`, `const PORTRAIT_H = 844` -- portrait reference design dimensions (the height is nominal/legacy; actual portrait design height is VIEWPORT-DRIVEN, see Key behavior).
- `const LANDSCAPE_W = 1280`, `const LANDSCAPE_H = 592` -- the landscape reference design canvas (aspect ≈2.162:1), replacing a prior 844×390 design. Deliberately wider than a typical 16:9 phone so both 16:9 and 4:3 devices exercise the surplus/clamp logic.
- `interface Rect { x, y, w, h }` -- the universal design-space rectangle shape used throughout.
- `interface SafeInsets { top, right, bottom, left }` -- CSS-pixel safe-area insets (from `env(safe-area-inset-*)`).
- `interface MatchRegions` -- the full named-region contract every orientation must populate: `board`, `opponentRail`, `traitRail`, `traitTabBar` (landscape-only tab switcher between traits/items tabs; zeroed rect in portrait, which has no tab switcher), `shop`, `bench`, `hud`, `readyButton`, `statusRow`, `sellControl`.
- `interface LandscapeClusters { topBar, leftRail, rightRail, bottomBar, boardArea }` -- landscape-only exposure of the 4 fixed-edge clusters + the residual board-containing band, "for tests and future stages"; `undefined` in portrait.
- `interface MatchLayout { orientation, designW, designH, scale, canvasOffsetX, canvasOffsetY, regions, portraitDesignH?, clusters? }` -- the full resolved layout: `scale` converts design-space coords to CSS pixels: `cssPx = designPx * scale`; `canvasOffsetX/Y` is the CSS-pixel position of the canvas top-left in the viewport (incorporates centering + safe insets); `portraitDesignH` is set only in portrait (the actual usable-viewport-driven design height); `clusters` is set only in landscape.
- `interface ResolveLayoutInput { viewportW, viewportH, safe? }` -- the resolver's input; `safe` is `Partial<SafeInsets>`, defaults all to 0.
- `function portraitRegions(designH: number): MatchRegions` -- the pure portrait region-budget algorithm, callable independent of viewport/scale logic (purely a function of the target design height) -- "unit-testable without viewport / scale logic." See Key behavior for the full algorithm.
- `function landscapeClusterThickness(usableW, usableH): { topBarH, bottomBarH, leftRailW, rightRailW }` -- the pure landscape cluster-dimension solve, exposed standalone for tests; independent of POSITION (only thickness), see Key behavior.
- `function bottomCornerButtonTop(regionTop, regionBottom, regionW): number` -- computes the Y of the TOP edge of a circular bottom-corner medallion button (Buy-XP / shop-toggle) whose footprint spans `[regionTop, regionBottom]`, mirroring `match.ts`'s own `buyXpFootprint` radius-cap math (`LS_CORNER_BTN_MAX_R=56`, rim fraction 0.16) "so the rail-clearance math here stays in lockstep with the rendered button diameter." Used internally by `landscapeRegionsFor` to cap the rails' bottom edge above the button, and presumably also called directly by `match.ts` to position the buttons themselves.
- `function resolveLayout(input: ResolveLayoutInput): MatchLayout` -- THE public entry point. Computes `aspect = viewportW/viewportH`, dispatches to `landscapeLayout` or `portraitLayout` based on `LANDSCAPE_THRESHOLD`.
- `function centeredModal(layout: MatchLayout, contentW: number, contentH: number, pad = 20): Rect` -- centers a modal panel within the design bounds minus `pad` on every edge; clamps `w`/`h` down to fit if the requested content size exceeds the available space (caller should enable internal scrolling in that case, per the doc comment).
- `function landscapeBenchSlotCenter(i: number, bench: Rect): { x, y }` -- pixel center of bench slot `i` (0..8) in a single 1×9 horizontal row filling `bench`.
- `function landscapeBenchSlotAt(px, py, bench: Rect): number | null` -- inverse hit-test: maps a design-space pointer position to a bench slot index 0..8, or `null` if outside the bench rect.
- `type PlanningRegion = {zone:"board",slotIdx}|{zone:"bench",slotIdx}|{zone:"sell"}|{zone:"hud"}|{zone:"shop",cardIdx}|{zone:"readyButton"}|null` -- the named planning-phase pointer region union.
- `function planningRegionAt(px, py, layout, boardSlot, benchSlot): PlanningRegion` -- consolidated pointer hit-test across all planning regions, checked in priority order (board slot wins if the caller already determined a hex hit, then bench with ±7px forgiveness, then sell with ±6px forgiveness, then readyButton exact, then shop with a card-index sub-computation over 6 cards, then hud, else `null`). The caller is responsible for computing `boardSlot` itself (via hex geometry, e.g. `hexFromPointer`) and passing it in -- this function only folds that into the unified region-priority decision, it doesn't do hex math itself.
- `interface RailTile { col, row, tileX, tileY, cx, cy, tileW, tileH }` -- one opponent-rail grid tile's full geometry.
- `function opponentRailTile(seatIdx, cols, rows, rail: Rect): RailTile` -- pure grid-cell geometry for seat `seatIdx` in a `cols`×`rows` grid filling `rail` (row-major: `col = seatIdx % cols`, `row = floor(seatIdx/cols)`).
- `function shopPanelSlotRect(area: Rect, slot: number, gap = 4): Rect` -- the drop-down shop panel's slot grid: divides `area` into exactly 7 equal-width slots left-to-right (slot 0 = the refresh button, slots 1-6 = the 6 unit shop cards), each slot's rect shrunk by `gap` for visual separation.
- `interface ShopCardContentLayout { discY, discR, nameY, traitY, tierY }` -- a shop card's internal vertical content offsets (relative to the card's own top), all pure ratios of `cardH`.
- `function shopCardContentLayout(cardH: number): ShopCardContentLayout` -- computes those offsets proportionally so a card renders correctly across its full variable height range (64-84px per the comment); the tier/cost row is pinned near the bottom (`cardH - 9`) and the trait line is clamped to always clear it by at least 10px (`traitY = min(cardH*0.71, tierY - 12)`).

# Key behavior

**Orientation selection**: `resolveLayout` computes `aspect = viewportW/viewportH` and picks landscape at `aspect >= 1.1`, else portrait. This is the ONLY branch point; everything downstream is orientation-specific.

**Portrait (`portraitLayout` + `portraitRegions`)**: height-driven. `usableH = viewportH - safe.top - safe.bottom`; the design height EQUALS this usable height exactly (`dH = usableH`), and `scale = usableW / PORTRAIT_W` (width-only scale) -- so the canvas always fills the full usable vertical space with no letterboxing top/bottom, while width scales proportionally and may letterbox left/right. `portraitRegions(dH)` then runs the budget algorithm (see `PORTRAIT_LAYOUT_SPEC.md` for the full spec, already documented separately): a static top band (status row + opponent rail, fixed 54px total) is reserved first, then `board` claims a fraction (`P_BOARD_FRAC ≈ 0.4265`) of the remaining height clamped to `[280, 392]`, then `traitRail` claims a fraction of what's left clamped to `[32,44]`, then a 7-gap solve distributes the remainder with `hud`/`bench`/`shop`/`readyButton` starting at fixed minimums (32/32/72-or-64/44) -- any leftover SURPLUS (when the viewport is taller than the bare-minimum stack) inflates regions toward their design maxima in a fixed PRIORITY ORDER: board first, then shop, then (skipping the intentionally-fixed 44px `readyButton`) hud, then bench, then traitRail, and finally the inter-region gap itself widens (up to 8px) absorbing any remainder. Below `P_FLOOR_THRESHOLD=680` usable height, the shop minimum drops to a tighter floor (64 instead of 72) and the gap minimum drops to 3 (instead of 4) so very short viewports (down to the documented 360×640 CSS floor) still avoid clipping.

**Landscape (`landscapeLayout` + `landscapeRegionsFor` + `landscapeClusterThickness`)**: a fixed 1280×592 reference canvas, scaled UNIFORMLY (`scale = min(usableW/1280, usableH/592)`, preserving aspect, unlike portrait's width-only scale) and centered with safe-inset-aware offsets. The region solve runs entirely in the FIXED 1280×592 design space (never the raw viewport), with device safe-insets converted from CSS px to design units (divided by `scale`) before being used as cluster-position floors. Architecture: 4 fixed-thickness edge clusters (`topBar`, `leftRail`, `rightRail`, `bottomBar`) sized by `landscapeClusterThickness` (each has its own REFERENCE thickness clamped to a `[min,max]` -- topBar 30-56 around a 40 reference, bottomBar 52-72 around 60, leftRail 96-160 around 120, rightRail 80-130 around 120 -- these are FIXED dimensions per the architecture, never stretched by viewport surplus; a defensive floor additionally shrinks rails/bars toward their MIN, never below it, if the viewport is so tight the residual board would go negative relative to `LS_BOARD_MIN_W=320`/`LS_BOARD_MIN_H=240`). Each cluster's leading edge is positioned at `max(safeInsetDesign, LS_BASE_MARGIN=12)` so nothing sits under a notch on either short landscape edge. The board+bench occupy a residual vertical band between the top bar and a bottom controls row (`LS_BOARD_BENCH_FRAC=0.88` target split), with the bench BUTTING the board's bottom edge directly (no gap -- "board + bench read as one continuous play area"); the board's width targets `LS_BOARD_NEAR_FRAC=0.75` of the design width but is capped to clear the wider of the two rails symmetrically on both sides (`halfCap`), and the board is centered on the SAFE-AREA horizontal center (`safeCenterX`), not the screen's raw center. The opponent rail (`rightRail`) is a single 1×8 vertical seat column whose TOP starts at the top bar's bottom edge (reclaiming headroom so "Player 1 reads at the top corner") but whose BOTTOM is capped above the bottom-right shop-toggle button's top (via `bottomCornerButtonTop` + `LS_CORNER_GAP=6`) so the last seat tile never sits under that button. The left rail holds a `traitTabBar` (top strip, fixed 20px) + `traitRail` (the remainder) -- the tab switcher toggles which content (traits vs items) fills the rail below it. The bottom controls row holds `hud` (econ display, left-anchored, width clamped `[200,250]` as ~18% of the row), `sellControl` (64px wide, immediately left of the shop button), and `shop` (the shop-TOGGLE button rect, 96px wide, anchored to the absolute bottom-right corner -- "the money-sack shop toggle owns the bottom-rightmost corner... the shop cards themselves remain drop-down-only," meaning `regions.shop` in landscape is NOT the 6-card strip itself but the button that opens a drop-down panel, whose own 7-slot grid is computed separately via `shopPanelSlotRect`). `readyButton` is explicitly ZEROED in landscape ("READY is a DOM control" in that orientation, not a Pixi-rendered region).

# Invariants & constraints

- **Portrait is explicitly preserved byte-for-byte** across the landscape stage-1 rewrite -- the file's own comments state this twice ("NOT TOUCHED by the stage-1 landscape rewrite"); any future landscape architecture change should NOT need to touch `portraitRegions`/`portraitLayout`, and a diff touching both should be treated with suspicion.
- The 844px portrait design-height case is the explicitly preserved baseline ("the 844px design case stays visually equivalent to the prior hardcoded values") -- `portraitRegions(844)` is effectively a regression-test anchor.
- Edge clusters in landscape are FIXED reference dimensions that NEVER stretch to absorb viewport surplus -- all surplus space in landscape goes to the board exclusively (unlike portrait, where surplus is distributed across multiple regions in priority order). This is a deliberate architectural difference between the two orientations worth not conflating.
- The board in landscape centers on the SAFE-AREA center (`safeCenterX`, accounting for asymmetric left/right insets), not the raw screen/canvas center -- a notched device with an inset only on one side will NOT render the board dead-center of the canvas.
- `shop` region's meaning is ORIENTATION-DEPENDENT: in portrait it is (implicitly, per the wider system) the actual 5/6-card shop strip rect; in landscape it is explicitly just the toggle BUTTON, with the actual card grid computed on-demand via `shopPanelSlotRect` against a separately-computed drop-down panel area (`innerArea`, built by `match.ts`, not by this file). A reader must not assume `regions.shop` always means "the rendered shop cards."
- `MatchRegions.traitTabBar` only has real (nonzero) content in landscape; portrait always zeroes it (`{x:0,y:0,w:0,h:0}`) since portrait shares one rail for both tabs without a switcher.
- `planningRegionAt`'s priority order is significant and intentional (board > bench > sell > readyButton > shop > hud > null) -- it returns the FIRST matching region in that fixed order, so overlapping forgiving-bounds regions resolve deterministically rather than ambiguously.

# Depends on

No imports -- this file is entirely self-contained (only uses its own internal `clamp` helper and standard `Math`).

# Used by

`packages/client/src/scenes/match.ts` -- the primary and almost certainly sole consumer; confirmed direct imports include `centeredModal`, `opponentRailTile`, `shopCardContentLayout`, `shopPanelSlotRect` (and by extension, given `CLAUDE.md`'s description of "match.ts renders entirely from this.layout.regions," every other export is very likely consumed there too: `resolveLayout` itself, `landscapeBenchSlotCenter`/`landscapeBenchSlotAt` for landscape bench hit-testing, `planningRegionAt` for drag/drop region resolution). `packages/client/src/main.ts` calls `resolveLayout` on resize/orientationchange per `CLAUDE.md`. `packages/client/src/coachmarks.ts` reads `MatchLayout.portraitDesignH` (per `CLAUDE.md`'s coachmark height-awareness note) to scale ring placement on short viewports.

# Notes

- **This file has materially diverged from `CLAUDE.md`'s documented landscape description.** `CLAUDE.md` describes landscape as "a three-column design (844×390)... the trait rail stacked vertically down the left, the 9-slot bench as a 3×3 grid + sell + item bar in the left column, and the opponent rail (4×2 grid) + HUD cluster + shop (5 wide cards) + Ready in the right thumb zone." The ACTUAL code in this file implements a substantially different, later architecture: a 1280×592 reference canvas, a 4-edge-cluster + residual-board model, a 1×9 single-row bench (not 3×3) butting the board directly, a 1×8 single-column opponent rail (not 4×2) sharing the right rail with NOTHING else, a drop-down shop panel opened via a corner toggle button (not 5 always-visible wide cards), a trait/items TAB SWITCHER in the left rail (not items-in-the-left-column-as-a-separate-bar), and NO Ready button region at all (it's a DOM control in landscape, not Pixi). Any reader relying on `CLAUDE.md`'s landscape description for this part of the client should treat the CODE in this file (and presumably a newer, undocumented "design-bible rewrite stage 1" effort referenced in the file's own header comments) as authoritative, and `CLAUDE.md`'s landscape section as STALE for this specific area.
- The header comment explicitly calls this a "stage 1 of the design-bible rewrite" -- implying further stages were planned/expected; a reader investigating layout behavior that seems incomplete or in-flux (e.g. placeholder-feeling constants, the `LandscapeClusters` type being described as "exposed... for tests and future stages") should expect more landscape-architecture changes to land in this file over time.
- `bottomCornerButtonTop`'s radius math (`LS_CORNER_BTN_MAX_R=56`, rim fraction 0.16) is declared as a deliberate MIRROR of `match.ts`'s own `buyXpGeom`/`buyXpFootprint` button-radius computation -- if that button's sizing formula in `match.ts` or `hudModel.ts` ever changes, this file's constants must be updated in lockstep or the rail-bottom clearance will silently drift out of sync with the actual rendered button (a footgun for any future HUD-button resize work).
- `shop`'s drop-down-panel framing (button vs. always-visible cards) is a meaningfully different INTERACTION MODEL than portrait's (presumably) always-visible shop strip -- worth flagging to a UX reviewer if cross-orientation parity is ever audited, since the user-facing shop-browsing flow differs by orientation in a way `CLAUDE.md` doesn't currently capture.
