# Path & purpose

`packages/client/src/netDriver.ts` -- `NetDriver`, the `IDriver` implementation for Online-mode matches: translates the `NetClient`'s server message stream into the same `MatchState`/`DriverEvent` shape `LocalDriver` produces, runs a LOCAL combat sim for immediate visual playback (reconciled against the server's authoritative `COMBAT_RESULT`), and forwards player actions (`CMD`/`READY`) over the wire.

# Responsibility

Owns the entire server-driven side of `IDriver`: reconstructing a client-side `MatchState` from `STATE_SNAPSHOT`/`STATE_DELTA` messages (since the server never sends a full `MatchState` -- only public player views + this seat's own private view), tracking this seat's pairing/opponent board/combat result/round result/loot orbs from their respective private messages, running a same-seed local sim at `COMBAT_START` so the scene can render combat the instant it starts (without waiting for the slower authoritative result), and logging (not correcting) any mismatch against the server's `COMBAT_RESULT` when it later arrives. The renderer is presentation-only; this driver is the seam between "what the server says happened" and "what `scenes/match.ts` reads."

# Exports

- `class NetDriver implements IDriver` -- the sole export.
  - `constructor(url: string, authToken?: string)` -- constructs its own `NetClient(url)` (no token passed to `NetClient`'s constructor; reconnection seat-tokens are tracked separately, see Key behavior) and subscribes to its event stream; sends `QUEUE_JOIN` (with `authToken` if provided) on a FRESH connection (when `net.seatToken` is falsy -- a reconnect sends its own `RECONNECT` from within `NetClient` and this constructor's logic does nothing extra in that case).
  - All other methods are the `IDriver` interface implementation (see `driver.ts` for the contract): `get seatIndex`, `on`, `getState`, `startPlanning` (no-op -- server drives phases), `playerCommand` (sends `CMD` over the wire, returns an optimistic `{ok:true}` immediately -- the server's eventual delta or `ERROR` is the real outcome), `ready` (sends `READY`), `getMyPairing`, `getMyOpponentBoard`, `getMyCombatResult`, `getMyOutcome`, `isPveRound`, `getPveStageName`, `getMyLootOrbs`, `getMyRoundResult`, `getMatchStats`, `getUpcomingPveBoard`, `combatPlaybackDone` (no-op -- server paces phases, this is purely visual locally), `advanceFromResolution` (no-op -- server auto-advances), `getPlanningTimeLeft`, `dispose` (stops the `NetClient`, clears listeners -- this is the ONLINE leave-match path: a disconnect, which the server's AFK-handling treats as a forfeit since there's no SURRENDER message).
  - `get phaseEndsAt(): number` / `get clockOffset(): number` -- extra accessors NOT on `IDriver` (NetDriver-specific): `phaseEndsAt` is the server-reported absolute timestamp the current phase ends; `clockOffset` is `serverTs - (clientSendTs + rtt/2)` computed from PONG round-trips, used by `getPlanningTimeLeft` to convert the server's absolute deadline into a locally-meaningful countdown.

# Key behavior

**Connection bootstrap**: on the underlying `NetClient`'s `"connected"` event, if there's no `seatToken` yet (fresh connection, never matched before), sends `QUEUE_JOIN` (with `authToken` for an authenticated seat, or bare for a guest/anonymous attempt -- though per CLAUDE.md `QUEUE_JOIN` without a valid token gets rejected server-side with `UNAUTHENTICATED`). If a `seatToken` IS already held, `NetClient` itself already sent `RECONNECT` on `onopen`, so this constructor does nothing extra.

**State reconstruction is the core complexity**: the server NEVER sends a full `MatchState`. Instead:
- `STATE_SNAPSHOT` carries `{round, phase, me (full private state for THIS seat), players (public views for all 8), lastPairings}`. `_buildStateFromSnapshot` (first snapshot) constructs a full local `MatchState` by using `snap.me` verbatim for this seat (attaching the public `name` field, which `me` itself lacks) and filling EVERY OTHER seat with placeholder private fields (`gold:0, items:[], shop:[null×5], lastBoard:null` -- since those are genuinely unknown for opponents) layered under their public view. `_applySnapshot` (subsequent snapshots) does the same merge but mutates the existing `_state` object in place instead of rebuilding it.
- `STATE_DELTA` carries `{changedSeat, players (public views), me? (full private state, present ONLY when changedSeat === mySeat)}`. The handler updates `_state.players[mySeat]` wholesale from `delta.me` only when this seat is the one that changed; for EVERY seat (including itself) it always re-syncs the public fields (`name, hp, alive, placement, level, xp, winStreak, loseStreak`) from the delta's public view; and for seats OTHER than itself, also re-syncs `board`/`bench` (since opponent board/bench are public-visible, unlike gold/items/shop which stay seat-private and are never touched here for other seats).
- Every `STATE_SNAPSHOT`/`STATE_DELTA` handler ends by emitting a synthetic `{type:"state", state: this._state}` `DriverEvent` so the scene re-renders -- the driver is the ONLY place that decides when a state mutation is "complete enough" to broadcast to listeners.

**Combat start -- dual-path local sim**: on `COMBAT_START`, the driver first clears per-round transient state (`lastLootOrbs`, `_myRoundResult`, `_myPairing`, `_myOpponentBoard`, `_myCombatResult`) -- explicitly noted as "fresh each round." Then branches:
- **PvE round** (`isPveRound(round, gameData)` true): there is NO pairing for PvE (server's `lastPairings` is empty), so the mob board never arrives via `opponentSnapshots`. The driver builds the SAME mob board locally that the server's `runPveRound` would build (`buildMobBoard` + `pveStageForRound`, both from `rules/rounds.ts`, deterministic from `_state` + the stage), runs `simulateCombat` itself with a per-seat-derived seed (`derivePairingSeed(e.roundSeed, mySeat)`), and stores the result as `_myCombatResult` immediately -- so the mob board renders the INSTANT combat starts, without waiting on the server's slower authoritative `COMBAT_RESULT` (which still overwrites this later, see below). Failures are swallowed (`try/catch`, "local sim failed; wait for COMBAT_RESULT").
- **PvP round**: finds this seat's pairing entry in `e.pairings`, determines `side` (0 if this seat is the pairing's first id, else 1) and `isGhost` (an unpaired seat on an odd player count gets a synthetic "ghost" opponent, `side===0 && bId<0`). Builds `boardA`/`boardB` from `boardToCombatState`/`ghostToCombatState` (mirroring the server's exact board construction via the same shared `rules/rounds.ts` helpers) and runs `simulateCombat` locally with `derivePairingSeed(e.roundSeed, pairingIndex)` -- again purely for IMMEDIATE rendering, again failure-swallowed.

**Combat result reconciliation**: when the authoritative `COMBAT_RESULT` later arrives, if a local result was already computed AND its `events` array differs (compared via `JSON.stringify` equality) from the server's, a `console.warn` logs the mismatch (winner values from both) -- this is diagnostic ONLY, never corrective beyond what happens next: `_myCombatResult` is unconditionally overwritten with the server's result (`// Server result is canon`). The scene, having already started playing back the LOCAL result's events, may visually jump/diverge if a mismatch occurred; this driver does not attempt to reconcile mid-playback, only logs.

**Other private per-seat messages**: `LOOT` stores `e.orbs` into `_state.lastLootOrbs.set(mySeat, orbs)` (already-decided orbs; this client never re-derives them, only animates). `ROUND_RESULT` stores `e.result` into both `_myRoundResult` and `_state.lastRoundResult.set(mySeat, result)`. `MATCH_END` merges the public `e.names` map onto `_state.players[seat].name` (so the match-over overlay shows real names instead of "Player N"), stores `e.stats` into `_matchStats`, and emits a `match_over` `DriverEvent` carrying `placements` + optional `mmr`/`stats`. `PONG` updates `_clockOffset`.

**`getPlanningTimeLeft`**: returns `max(0, _phaseEndsAt - (Date.now() + _clockOffset))` -- converts the server's absolute deadline (`_phaseEndsAt`, set from `PHASE_CHANGE.endsAt`) into a client-clock-adjusted remaining-ms value using the PONG-derived `_clockOffset`.

# Invariants & constraints

- **Server result is canon** (explicit comment + CLAUDE.md invariant): local sim exists purely so the scene has SOMETHING to render immediately at combat start; the moment `COMBAT_RESULT` arrives, it unconditionally wins, mismatch or not.
- The local sim at `COMBAT_START` MUST construct boards using the exact same shared helpers (`boardToCombatState`/`ghostToCombatState`/`buildMobBoard`/`derivePairingSeed`) the server's `rounds.ts` uses -- any divergence in board construction (not just sim logic) would cause spurious mismatches even with identical sim code, since the seed AND inputs must match bit-for-bit.
- `playerCommand` is OPTIMISTIC -- it returns `{ok:true}` immediately without waiting for the server's actual validation result. A command that the server later rejects produces an `ERROR` message (not modeled in this file's switch -- presumably handled elsewhere or silently dropped) but the caller already received an `ok:true` from this call. A reader relying on `playerCommand`'s return value to gate UI feedback must understand it does NOT reflect server-side legality.
- `getState()` returns a fully-populated STUB `MatchState` (`players:[]`, all maps empty, `round:1`, `phase:"PLANNING"`) when `_state` is still null (before the first `STATE_SNAPSHOT` arrives) -- callers must tolerate an empty-players state during the brief connect/queue window.
- `_buildStateFromSnapshot`/`_applySnapshot` placeholder fields for opponents (`gold:0, items:[], shop:[null×5]`) are NOT real data -- they exist only because `MatchState["players"][i]`'s TYPE requires those fields to be present; a reader must never read `otherSeat.gold`/`.items`/`.shop` as meaningful (they're always the placeholder, since the server never sends another seat's private economy).
- This file relies on the seat index being known (`this.mySeat`) BEFORE any `STATE_SNAPSHOT`/`STATE_DELTA` arrives meaningfully -- `mySeat` is set from `MATCH_FOUND.seatIndex`, which must arrive first in the message ordering (a reasonable assumption given matchmaking always precedes match state).

# Used by

`packages/client/src/main.ts` -- `startMatch("online")` constructs `new NetDriver(SERVER_URL, ui.auth?.token)` and passes it to `MatchScene`. `packages/client/src/scenes/match.ts` consumes it exclusively through the `IDriver` interface (per the renderer-is-dumb invariant, never reaching into `NetDriver`-specific fields beyond what `IDriver` exposes, except possibly `phaseEndsAt`/`clockOffset` if the scene needs raw timer math -- though `getPlanningTimeLeft` already encapsulates that).

# Notes

- The constructor passes `url` but NOT `authToken` to `new NetClient(url)` -- `NetClient`'s own constructor accepts an optional `token` parameter (used for RECONNECT), but `NetDriver` never supplies one at construction; the seat token instead arrives later via `MATCH_FOUND` and is tracked INSIDE `NetClient` itself (`this.token = msg.token` in `net.ts`'s `onmessage`), not by `NetDriver`. `authToken` here is a different concept entirely -- the player's account auth token, sent once in the `QUEUE_JOIN` payload, never reused for reconnection.
- The "local sim at combat start, reconciled at result" pattern is a deliberate latency hide: without it, the client would show a static board for however long it takes the server to actually run+broadcast the authoritative sim. The reconciliation `console.warn` is the ONLY signal a developer gets if client/server sim ever diverge -- there's no telemetry/reporting hook visible here, just a browser console log, which is easy to miss in production.
- `previewPveStage`'s "negative uids, display-only, never fed to the sim" caveat is inherited unchanged from `LocalDriver`'s identical comment -- both drivers share this exact constraint since `getUpcomingPveBoard` calls the same rules helper.
