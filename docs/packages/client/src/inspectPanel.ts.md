# Path & purpose

`packages/client/src/inspectPanel.ts` -- Pixi renderers for ALL in-match modal inspect/detail overlays: unit inspect, item detail, the radiant-enhancer item picker, and trait detail. Kept in the Pixi world (not DOM) to match the existing scout overlay convention; renders strictly from pure models (`InspectModel`, `ItemModel`, `TraitDetailModel`), no game logic.

# Responsibility

Owns the visual construction of every "tap/long-press to learn more" modal panel in the match scene: shared chrome (scrim, rounded panel box, close button, scale-in reveal animation, word-wrap text helper, the trait diamond+glyph chip motif) plus four panel-specific render functions. Each render function fully replaces the target layer's children on every call (no incremental diffing) and is purely a function of its model input.

# Exports

- `function renderUnitInspect(layer, m: InspectModel, onClose, layout?, reducedMotion=false): void` -- renders the unit/mob inspect panel: token + name + tier/cost header, origin+classes trait chip row (empty for mobs), an optional ability block (omitted entirely when `m.ability` is null, e.g. an abilityless mob), a 2-column stat grid, and an optional equipped-items chip row (omitted when `m.items` is empty). Panel height is computed dynamically from `m.items.length`/`m.ability` presence so it never collides regardless of which optional sections render.
- `function renderItemDetail(layer, m: ItemModel, onClose, reducedMotion=false, layout?): void` -- renders a single item's detail panel: icon disc + name + kind label header, a stat-bundle row per `m.stats` entry, and EITHER a "Passive" block (when `m.passive` is set -- completed/radiant items) OR an "Effect" block (when `m.consumable && m.consumableEffect` -- the three consumables) -- these two blocks are mutually exclusive in practice since `ItemModel` never sets both, though the code doesn't explicitly enforce that, it just checks each condition independently.
- `function renderItemPicker(layer, items: ItemModel[], onPick: (itemId)=>void, onCancel, layout?, reducedMotion=false): void` -- renders the radiant_enhancer item-picker overlay: a header ("Choose an item to upgrade") plus one tappable row per item in the list (icon + name); tapping a row calls `onPick(item.id)`; the scrim/close-button both call `onCancel`. Pure presentation -- "the caller decides what picking/cancelling means."
- `function renderTraitDetail(layer, m: TraitDetailModel, onClose, layout?, reducedMotion=false): void` -- renders the trait-detail panel: a header (trait diamond + name + "Origin/Class · N fielded"), then one row per breakpoint (`m.rows`) showing a count badge (with a `▸` prefix when `active`), the breakpoint's effect text, dimmed when `!reached` (alpha 0.55) -- "so meaning isn't color-only" (an accessibility note: active/reached state is ALSO conveyed by the `▸` marker and alpha, not solely by color).

All four functions share the SAME modal pattern: `layer.removeChildren()` -> `scrim(layer, onClose/onCancel, layout)` -> `makePanelContainer` (scale-in wrapper) -> compute `panelRect` -> `panelBox` (chrome) -> `closeButton` -> panel-specific content.

# Key behavior

- **`dims(layout?)`**: returns the active layout's `{designW, designH}`, or hardcoded portrait fallback `{w:390, h:844}` when no layout is supplied (explicitly "used when no layout is supplied, e.g. tests").
- **`panelRect(layout, contentW, contentH, portraitY)`**: when a `layout` is supplied, caps the panel height to `min(contentH, layout.designH - 40)` and delegates to `centeredModal(layout, contentW, cap, 16)` (from `layout.ts`) -- "so short landscape never overflows." Without a layout, falls back to a hardcoded portrait-centered rect using the literal `PORTRAIT_W`/`PORTRAIT_H` constants and the caller-supplied `portraitY` -- "preserve the original placement" (a deliberate backward-compatibility path predating layout-awareness).
- **`makePanelContainer(layer, reducedMotion)`**: creates a child `PIXI.Container`, and unless `reducedMotion`, animates it scale `0.88 -> 1.0` + alpha `0 -> 1` over 100ms via a self-driving `requestAnimationFrame` loop (NOT a Pixi ticker -- explicitly "needs no Pixi ticker," so it keeps animating even if the Pixi render loop is paused, though it self-terminates once `k>=1` or the container is destroyed mid-animation via the `c.destroyed` guard). Under `reducedMotion`, snaps directly to the final state (`scale=1, alpha=1`) with no animation.
- **`scrim(layer, onClose, layout)`**: a full-design-area `bgScrim`-colored rect at 0.6 alpha, interactive (`pointerdown` -> `onClose`) -- the dismiss-by-tapping-outside affordance.
- **`panelBox(layer, x,y,w,h, accent)`**: three layered rounded-rect draws -- the fill (`bgInspect` @ 0.98 alpha), an accent-colored 1.5px outer border, and a 1px `surfaceFloat`-colored inset highlight rim 1px in from the edges -- "so the panel reads as a raised modal, not a flat overlay rectangle." Sets `eventMode = "static"` specifically so taps on the panel body "swallow taps so they don't dismiss via the scrim" (event propagation containment, not bubbling prevention by other means).
- **`closeButton(layer, x, y, onClose)`**: draws a small 26x22px rounded rect with a "✕" glyph, but explicitly expands its `hitArea` to ~62x44px ("min touch target") via a manually-constructed `PIXI.Rectangle` offset leftward/up-and-down from the visual bounds, "without changing the 26x22px visual; the leftward expansion stays inside the panel interior" (so the larger hit area doesn't spill outside the panel's own bounds and accidentally overlap the scrim/other content).
- **`text(...)` / `wrapText(...)`**: shared monospace-font text helpers; `wrapText` does naive word-wrap using `charW = size*0.6` as a per-character width estimate (not actual text measurement) to compute `perLine` character budget, then greedily packs words into lines, flushing a line via the shared `text()` helper whenever the NEXT word would exceed the budget.
- **`traitDiamond(layer, traitId, cx, cy, dr=8)`**: the shared rotated-diamond + glyph trait motif (also used by the HUD trait strip, per the comment "shared trait motif") -- draws a diamond polygon filled at 0.5 alpha in the trait's color plus a full-alpha stroke outline, then overlays the trait's class glyph (`drawGlyph` + `glyphForTraits([traitId])`) scaled to `dr*1.2`.
- **`renderUnitInspect`'s dynamic sizing**: `itemRowH = m.items.length > 0 ? 56 : 0`, `abilityBlockH = m.ability ? 66 : 0` (66 = 52px block + 14px gap, per the inline comment), `contentH = 274 + abilityBlockH + itemRowH` -- the panel's base height (274) is a fixed budget for the header/traits/stat-grid sections, with the two optional blocks added on top only when present. The stat grid's `gridY` start position ALSO shifts (`y+104` with no ability vs `y+170` with one) so the grid never overlaps a present ability block.
- **`renderUnitInspect`'s items row** (when present): draws an "ITEMS" label then one chip per equipped item -- chip width is text-length-derived (`18 + name.length*5.0 + 12`), filled in the item's own `color`, with an EXTRA gilded inner-rim stroke specifically for non-component (completed+) items (`if (!item.component)`) -- "Completed items get the gilded inner rim (shared with the item frame motif)." Each chip draws its procedural/art icon via `drawItemIcon` with `reducedMotion: true` HARDCODED (not the panel's own `reducedMotion` parameter -- small static icons inside an already-reduced-detail inspect chip are always drawn in their simplest, non-animated form regardless of the caller's actual motion setting).
- **`itemKindLabel`/`itemKindAccent`** (internal): exhaustive switches over `ItemModel.tier` producing a display label ("Component"/"Completed Item"/"Radiant Item"/"Artifact"/"Mythical Item"/"Consumable") and an accent color (component->`itemComponent`, completed->`accentGold`, radiant->`radiantBadge`, artifact/mythical->`accentGold` (reusing the completed accent, explicitly noted as "No dedicated artifact/mythical accent yet"), consumable->`itemConsumableRim`).
- **`renderItemDetail`'s sizing**: `contentH = 96 + statLines*24 + passiveH + effectH + 16` where `passiveH = m.passive ? 44 : 0` and `effectH = m.consumable ? 48 : 0` -- again fully dynamic per which optional block is present.
- **`renderItemPicker`'s rows**: each row is a tappable `PIXI.Graphics` with `eventMode:"static"`, explicit `hitArea` matching its visual rect, and a `pointerdown` handler calling `onPick(item.id)` directly -- a flat list, no scrolling/virtualization (assumes the picker's item count is always small enough to fit, consistent with it only ever listing a single unit's equipped tier-2 items, capped by `MAX_ITEMS_PER_UNIT`).
- **`renderTraitDetail`'s rows**: each breakpoint row's fill/stroke/alpha depend on BOTH `r.active` (currently-satisfied breakpoint -- affects fill color choice and border color/alpha) and `r.reached` (this breakpoint count has been hit at all, which for a single-pass breakpoint list is identical to `active` in most cases but the two are tracked as distinct flags on the model -- the row's OVERALL alpha is `r.reached ? 1 : 0.55`, dimming any not-yet-reached breakpoint regardless of the active styling).

# Invariants & constraints

- **Pure-presentation-from-model invariant**: every function here takes an already-fully-decided pure model (`InspectModel`/`ItemModel`/`TraitDetailModel`) and NEVER computes game logic itself -- per the file header, "Both panels render strictly from the pure models... no game logic here." Any new data needed by a panel must be added to the relevant model file (`inspectModel.ts`/`itemModel.ts`/`traitDetailModel.ts`), not derived inline here.
- **Every render call clears and rebuilds the ENTIRE layer's children** (`layer.removeChildren()` at the top of all four functions) -- there is no incremental update path; calling any of these functions repeatedly (e.g. every frame) would be wasteful, they are meant to be called ONCE per panel-open event, not per render frame.
- The close button's expanded hit area must stay "inside the panel interior" -- a future change to button position/panel padding must re-verify the `x-18` leftward expansion doesn't spill past the panel's left edge or outside the panel's own bounds into the scrim's tap-to-dismiss zone (which would create a confusing double-dismiss-target overlap).
- `wrapText`'s character-width estimate (`size*0.6`) is an APPROXIMATION, not real glyph measurement -- for the monospace font family used throughout this file this is reasonably accurate, but if any caller ever passes a proportional/non-monospace font through `wrapText`, line-wrap widths would be wrong; the function is implicitly coupled to the monospace assumption baked into the shared `text()` helper.
- `renderItemDetail`'s passive-vs-effect block logic checks `m.passive` and `m.consumable && m.consumableEffect` as two SEPARATE independent `if`s, not an exhaustive either/or switch -- relies on `ItemModel`'s own construction invariant (in `itemModel.ts`) that these are never both simultaneously truthy for one item; this file does not itself guard against both rendering at the same y-offset if that invariant were ever violated upstream.
- Reduced motion is honored both for the panel's own reveal animation (`makePanelContainer`) and is passed through to `drawItemIcon` in `renderItemDetail`'s header icon and `renderItemPicker`'s row icons -- but `renderUnitInspect`'s equipped-item-chip icons HARDCODE `reducedMotion: true` regardless of the caller's actual setting (see Key behavior) -- a deliberate, not accidental, asymmetry worth knowing if a reader expects uniform reduced-motion propagation.

# Depends on

`pixi.js` (PIXI namespace). `./theme.js` (`C`, `tierColor`, `traitColor` -- all panel colors route through theme, no hex literals here). `./unitToken.js` (`drawUnitToken` -- the unit-inspect header token). `./glyphs.js` (`drawGlyph`, `glyphForTraits` -- trait diamond glyphs + the mana/coin icons). `./itemIconDraw.js` (`drawItemIcon` -- item icons in the items row, item-detail header, and picker rows). `./inspectModel.js` (type-only: `InspectModel`). `./traitDetailModel.js` (type-only: `TraitDetailModel`). `./itemModel.js` (`consumableDescription`; type `ItemModel`). `./layout.js` (`centeredModal`; type `MatchLayout`).

# Used by

`packages/client/src/scenes/match.ts` -- the sole consumer: long-press on any shop/bench/board unit calls `renderUnitInspect`; tapping a trait-strip chip calls `renderTraitDetail`; tapping a loose inventory item chip calls `renderItemDetail`; dropping a `radiant_enhancer` consumable onto a unit with 1+ eligible items (per `consumablePicker.ts`'s `radiantDropRoute`) calls `renderItemPicker`. All four are torn down "on every phase change" per `CLAUDE.md`'s inspect-panel description.

# Notes

- The file's comment at `renderItemDetail`'s icon header explicitly flags `drawItemIcon`'s consumable fallback behavior as "specced FOR UI-DESIGNER below" -- this is an inline note directing future visual-design work to the icon-drawing module (`itemIconDraw.ts`), not something resolved within this file.
- `PORTRAIT_W`/`PORTRAIT_H` (390x844) are hardcoded fallback dimensions duplicated from the broader layout system's default design resolution -- if the project's canonical design resolution ever changes, this file's fallback constants would need updating in lockstep (no shared import ties them to a single source, by the file's own admission they exist "used when no layout is supplied, e.g. tests").
- `renderTraitDetail`'s `▸` badge prefix and the overall use of shape+alpha (not solely hue) to distinguish active/reached states reflects a deliberate accessibility consideration carried through from the broader design system (`CLAUDE.md`'s inspect-panel summary: "marking reached/active (shape+arrow, not color-only)").
