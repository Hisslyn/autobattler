# Path & purpose

`packages/client/src/itemModel.ts` -- pure (no Pixi, no game logic) display-model derivations for the in-match item system: turns raw item ids / a player's inventory / a unit's equipped slots into glanceable, fixed-point-safe view models. Originally a "phase 10b" file, now also covers later item-tier additions (artifact/mythical/radiant/consumable).

# Responsibility

The single read-model layer between `items.json` (via the data loader) and every client UI surface that displays an item -- name, tint, stat lines, passive/consumable description, tier classification. The AUTHORITATIVE inventory and equip/combine/consume logic all live in `packages/rules`; this file only DESCRIBES what's already there, never decides or mutates anything.

# Exports

- `interface ItemStatLine { label: string; value: string }` -- one formatted stat row (e.g. `{label:"AD", value:"+10"}`).
- `type ItemTier = "component" | "completed" | "radiant" | "artifact" | "mythical" | "consumable"` -- the client's full display-tier classification. Per the doc comment, this "mirrors rules' itemKind/itemTier but flattens 'completed vs radiant' into one tier axis for display" -- i.e. data's `ItemKind` (`component`/`completed`/`artifact`/`mythical`/`consumable`) does NOT distinguish a radiant item from its base completed item (both are `kind: "completed"` per the loader), but this client-side `ItemTier` does split them out via an id-prefix check, giving the UI a richer tier signal than the raw data layer provides.
- `interface ItemModel { id, name, component, color, stats, passive, tier, consumable, consumableEffect }` -- the full per-item display model. `component: boolean` is a legacy convenience flag kept "for existing call sites" (`=== tier === "component"`), `color` is the display tint number from `theme.ts`, `stats` is the fixed-point-safe `ItemStatLine[]`, `passive` is a human-readable one-line description or `null` for stat-only items, `consumable`/`consumableEffect` flag the three special consumable items.
- `function passiveDescription(passive: ItemPassiveData): string` -- exhaustive switch over a completed item's passive kind: `burn` -> `"On hit: burn the target for {value} over {duration} ticks."`, `shield` -> `"Start of combat: gain a {value} shield for {duration} ticks."`.
- `function consumableDescription(effect: ConsumableEffect): string` -- exhaustive switch over the 3 `ConsumableEffect` values: `remove_item` -> "Removes all items from a unit back to your inventory.", `reforge` -> "Reforges each of a unit's items into a different item of the same tier.", `radiant_upgrade` -> "Upgrades one completed item on a unit into its radiant (tier-4) version."
- `function itemStatLines(stats: ItemDataDef["stats"]): ItemStatLine[]` -- converts a raw stat-bundle record into display lines: filters out `null`/`undefined` stat keys, maps each remaining key through `STAT_LABEL` (hp->"HP", ad->"AD", as->"Attack Speed", armor->"Armor", mr->"Magic Resist", mana->"Mana", abilityDamage->"Ability Power"; an unmapped key falls back to its raw key string) and formats its value via `formatStatDelta` from `statFormat.ts` (this is the line that keeps fixed-point `as` values from leaking as raw integers).
- `function itemModel(itemId: string, data: GameData): ItemModel | null` -- the main per-item resolver: looks up `itemId` in `data.items`, returns `null` if unknown, else classifies tier via internal `tierOf`, derives `color` via internal `colorForTier`, builds `stats` via `itemStatLines`, builds `passive` via `passiveDescription` if `def.passive` exists, and sets `consumableEffect` only when `tier === "consumable"`.
- `interface InventoryEntry extends ItemModel { index: number }` -- an `ItemModel` plus its position in the player's raw `items` array.
- `function inventoryModel(items: readonly string[], data: GameData): InventoryEntry[]` -- maps a player's loose-item id list into `InventoryEntry[]`, preserving array order (critical: "an index maps straight back to a command target" -- i.e. the returned `index` is exactly the array index the rules layer expects for an EQUIP/COMBINE_ITEMS/USE_CONSUMABLE command argument); unknown ids are silently dropped (not nulled-and-kept -- the output array can be SHORTER than the input if any id fails to resolve, though in practice every persisted inventory id should resolve).
- `interface EquippedSlots { items: ItemModel[]; max: number; free: number; full: boolean }` -- a unit's equipped-item slot summary.
- `function equippedSlots(unit: UnitInstance, max: number, data: GameData): EquippedSlots` -- builds the slot summary for one unit: resolves each equipped item id to its `ItemModel` (dropping unknowns), and computes `free = max(0, max - unit.items.length)` / `full = unit.items.length >= max`. `max` is passed in by the caller (mirroring the rules' `MAX_ITEMS_PER_UNIT` constant) specifically "so this module stays free of the rules import" -- this file never imports `packages/rules` directly.
- `function tier2EquippedItems(unit: UnitInstance, data: GameData): ItemModel[]` -- returns only the unit's equipped items whose tier is exactly `"completed"` (i.e. tier-2 completed-but-not-radiant items) -- "the exact set radiant_enhancer is allowed to target." Explicitly documented as a PURE UI-ROUTING helper used ONLY to decide whether to open an item-picker overlay (1+ results -> show picker) or send the consumable command with no target and let the server reject it (0 results -> server-side typed error) -- "Never used to block/branch the command itself," i.e. this function's result must never gate whether the command is SENT, only which UI path renders.

# Key behavior

- **`tierOf(def)`** (internal): calls `itemKind(def)` (from `@autobattler/data`) first; if it returns `"consumable"`, `"component"`, `"artifact"`, or `"mythical"`, that's the tier directly. The remaining case is data's `"completed"` kind, which this file further splits: if `def.id.startsWith("radiant_")` the client tier is `"radiant"`, else `"completed"` -- per the comment, the data loader gives radiant variants the SAME `kind: "completed"` as their base item, so "the id prefix is the only signal distinguishing them" at this layer.
- **`colorForTier(tier)`** (internal): maps tier to a `theme.ts` color constant -- `component`->`C.itemComponent`, `completed`->`C.itemCompleted`, `radiant`->`C.itemFrame` ("the gilded item-frame gold... already the 'completed item, upgraded' motif via itemIconDraw's frame"), `consumable`->`C.itemConsumable`. `artifact` and `mythical` currently have NO dedicated theme color and explicitly REUSE `C.itemCompleted` as a documented placeholder ("so they read as equippable" rather than looking broken/unstyled) -- a future ui-designer task would add dedicated `itemArtifact`/`itemMythical` theme keys and update this switch.

# Invariants & constraints

- Must stay free of Pixi imports and free of any `packages/rules` import (the `max` slot-cap parameter is threaded in by the caller specifically to preserve this boundary).
- Raw fixed-point stat values must NEVER leak past this module unconverted -- `itemStatLines` always routes through `formatStatDelta`, per the file header's explicit invariant statement.
- `inventoryModel`'s returned `index` field is load-bearing for command dispatch -- any caller using `InventoryEntry.index` as a command argument depends on `inventoryModel` preserving the EXACT same ordering as the player's raw `items` array (a stable, 1:1-skipping-unknowns mapping, not a reindexed/sorted one).
- `tier2EquippedItems` must remain a pure read used for UI ROUTING ONLY -- the comment is explicit and emphatic that legality enforcement for `radiant_upgrade` happens server-side (consistent with `CLAUDE.md`'s "Command legality is enforced server-side in packages/rules" invariant); this function existing with zero results is a valid normal case (it triggers the "send and let server reject" path), not an error condition to special-case here.

# Depends on

`@autobattler/data` (types `GameData`, `ItemDataDef`, `ItemPassiveData`, `ConsumableEffect`; function `itemKind`). `@autobattler/sim/src/types.js` (type-only: `UnitInstance`, for `equippedSlots`/`tier2EquippedItems`'s unit parameter). `./theme.js` (`C`, the item-tier color constants). `./statFormat.js` (`formatStatDelta`, the fixed-point-safe stat formatter).

# Used by

Per `CLAUDE.md`'s phase-10b section, `scenes/match.ts`'s item inventory bar (renders chips via `inventoryModel`) and `inspectPanel.ts`'s equipped-items rows / item-detail modal both consume this module's models. `combinePreview.ts` and `consumablePicker.ts` (already documented) are sibling pure-model files in the same family but are independent of this one (they call data's `recipeResult` directly rather than going through `itemModel`).

# Notes

- This file has visibly grown beyond its original "phase 10b" scope (the header comment) to also cover `artifact`/`mythical`/`radiant`/`consumable` tiers -- these read as a LATER item-tier-expansion feature not mentioned by name in `CLAUDE.md`'s `packages/data` section as of this doc's writing, but the code is unambiguous: `ItemKind` in the data loader already has 5 variants (`component`/`completed`/`artifact`/`mythical`/`consumable`), `radiantItemId`/a radiant-upgrade builder exist in the loader, and `ConsumableEffect` has exactly 3 values (`remove_item`/`reforge`/`radiant_upgrade`). A reader auditing CLAUDE.md against the actual data/rules layers for completeness should expect to find this same tier-expansion surfaced in `packages/data`'s loader and probably a `USE_CONSUMABLE` command in `packages/rules/src/commands.ts` (not yet directly confirmed in this documentation pass, but strongly implied by `tier2EquippedItems`'s comment referencing exactly that command name).
- The `artifact`/`mythical` placeholder-color decision (reusing `itemCompleted`) is a known, explicitly-commented gap -- worth flagging to a ui-designer agent if a future task touches item visuals, since the code already documents the intended fix (dedicated theme colors).
