# Path & purpose

`packages/client/tests/hudModel.test.ts` -- unit tests for `hudModel.ts`'s three pure derivations: `traitStripModel` (board -> trait-strip chips with breakpoint/tier state), `xpProgress` (xp/level/thresholds -> fill fraction), and `buyXpGeom` (a region rect -> the full circular Buy-XP button geometry).

# Responsibility

Owns: regression coverage for the trait-strip chip model's counting/breakpoint/tier/sort logic (using REAL `gameData.units`/`gameData.traits` content, not synthetic fixtures), the xp-progress bar's fraction math against both a hand-built threshold table and the real `economy.json` thresholds, and the Buy-XP button's pure geometry staying internally coherent (positive sizes, correct anchoring, arc/badge placement) across both portrait and landscape HUD region shapes including the documented short-viewport floor.

# Exports

None (a Vitest test file). Defines `unit(defId)` (a minimal `UnitInstance` stub cast via `as UnitInstance`, carrying only `defId` -- since `traitStripModel` only reads `.defId` off each board entry) and `unitsWithTrait(traitId)` (filters `gameData.units` down to those carrying a given trait id, returning their ids -- used to pull REAL units of a known trait for board construction, rather than inventing fixture units).

# Key behavior

**`traitStripModel` suite**:
- "returns an empty model for an empty board": `traitStripModel([null,null], ...)` returns `[]` -- an all-null board produces zero chips.
- "splits active and inactive traits and computes next breakpoints": pulls 2 real knight units (knight's breakpoints are 2/4/6 per CLAUDE.md), confirms the resulting chip has `count:2`, `activeBreakpoint:2`, `nextBreakpoint:4`, `color` matching `traitColor("knight")` directly, `activeTier:1` (1-based: the FIRST breakpoint tier satisfied), `tierCount:3` (knight has 3 breakpoints total: 2/4/6). Then with only 1 knight: `activeBreakpoint:null`, `count:1`, `nextBreakpoint:2`, `activeTier:0` (no tier reached), `tierCount:3` still. Then mixes 2 knights (active) + 1 sorcerer (presumably inactive at count 1) and confirms ACTIVE-FIRST ordering: every active chip's index precedes every inactive chip's index in the array.
- "counts unique defIds (duplicate copies count once)": a board with the SAME knight defId appearing TWICE plus a different knight once still counts as `2` (not 3) for the knight chip -- confirms `traitStripModel` counts unique DEFINITIONS contributing to a trait, not raw unit-instance copies (consistent with how trait breakpoints work in the actual game: stacking duplicate copies of the same unit doesn't multiply trait count).
- "reports activeTier / tierCount and the maxed denominator": 4 distinct summoners (summoner breakpoints are 2/4, so 4 maxes it) produce `count:4`, `tierCount:2`, `activeTier:2` (both breakpoints reached -> top tier), `activeBreakpoint:4`, `nextBreakpoint:null` (maxed, no further breakpoint), and the explicit maxed-detection idiom `activeTier === tierCount` (e.g. a UI rendering "4/4" can detect "fully maxed" this way without a separate boolean).
- "sorts by activation tier reached, not raw count (a maxed trait outranks a higher-count lower-tier one)": the most elaborate test in the suite, explicitly proving the chip SORT ORDER prioritizes `activeTier` over raw `count`. Constructs a board with 4 summoners (`activeTier:2`, maxed) and 3 knights (`activeTier:1`, NOT maxed despite a count of 3) -- confirms the summoner chip sorts BEFORE the knight chip even though the knight's raw count (3) could naively seem comparable, because tier (2 vs 1) wins. Then as a "sanity" cross-check, with 4 summoners (`activeTier:2`,`count:4`) vs 5 knights (`activeTier:2`,`count:5`) -- EQUAL tiers -- the higher RAW COUNT (knight, 5) now sorts first, confirming count is the correct TIEBREAKER only when tiers are equal, matching the documented sort priority in `hudModel.ts`'s source (active-first -> tier desc -> count desc -> id).

**`xpProgress` suite** (using a hand-built `t=[0,2,6,12,20,32,50,76,110]` threshold table, matching the structure of the real one):
- "computes the fraction within the current level": `xpProgress(0,1,t)` -> `{inLevel:0, needed:2, frac:0, maxed:false}` (level 1 spans xp 0-2, zero progress at xp 0); `xpProgress(1,1,t)` -> `{inLevel:1, needed:2, frac:0.5}` (halfway through level 1's 2-xp span); `xpProgress(4,2,t)` -> `{inLevel:2, needed:4, frac:0.5}` (level 2 starts at xp 2, spans to xp 6 i.e. 4 wide, and 4 xp total means 2 xp INTO level 2, half of 4).
- "reports a full, maxed bar at the top level": `xpProgress(110,9,t)` (the 9th/last threshold, `t.length===9`) returns `{frac:1, maxed:true, needed:0}` -- at or past the max level, no further progress is meaningful.
- "clamps the fraction to 1": `xpProgress(5,1,t).frac` is `1` even though `xp=5` exceeds level 1's nominal 2-xp span -- confirms the fraction never overshoots 1.0 even with an inconsistent (xp,level) pair, a defensive clamp.
- "matches the real economy.json thresholds at known boundaries": FIRST asserts `gameData.economy.levelXpThresholds` literally equals the SAME `[0,2,6,12,20,32,50,76,110]` array used as the synthetic `t` above (a content-drift guard: if the real thresholds ever change, this specific assertion fails immediately, flagging that the rest of this describe block's hand-built-table tests may need re-deriving). Then exercises `xpProgress` against the REAL thresholds at two boundary points: `xpProgress(8,3,real)` -> `{level:3, inLevel:2, needed:6, frac:1/3}` (level 3 starts at xp 6, spans to xp 12, 8xp means 2 in, 2/6=1/3) and `xpProgress(20,5,real)` -> `{level:5, inLevel:0, frac:0}` (exactly AT a level-boundary threshold -> zero progress into the new level, not negative or rolled-over).

**`buyXpGeom` suite**: builds 4 representative HUD-region rects by calling the REAL `resolveLayout` at 4 viewport configurations -- portrait tall (390x844), portrait short floor (390x606, the documented supported minimum), landscape sub-column (844x390, taking the RIGHT 96px-wide slice of the landscape hud region), and a smaller landscape (640x360, same 96px slice) -- and for EACH region runs the same coherence assertions (parameterized via a `for` loop building one `it(...)` per region, with the region's height embedded in the test name for traceability): all of `r`/`rimW`/`badgeR`/`arcW` are `>0`, `fontSize >= 6`; the circle is anchored with its LEFT edge clamped inside the region (`cx >= region.x + g.r`) and VERTICALLY CENTERED (`cy` close to `region.y + region.h/2`); the progress arc's radius is `>= r` (sits OUTSIDE the disc) and spans exactly `Math.PI/2` (a quarter circle / 90°, matching the documented "fills bottom -> top" design); the level badge's center is BOTH right-of and below the main circle's center (`badgeCx > cx`, `badgeCy > cy`, matching its documented bottom-right overlap position); and the floating xp-fraction text sits ABOVE the button, fully outside its radius (`fracY < cy - r`).

# Invariants & constraints

- `traitStripModel`'s sort order is, per the source comment mirrored by this test suite: active-first (any active beats any inactive) -> `activeTier` descending -> `count` descending -> `traitId` lexicographic (for full determinism on ties). A reader changing this priority order would break the elaborate "sorts by activation tier" test, which exists specifically to lock this exact 4-level priority in place.
- `unit(defId)`'s cast (`{defId} as UnitInstance`) is a deliberately minimal stub -- it works ONLY because `traitStripModel` reads nothing off a board entry besides `.defId` (looking up the rest, like trait membership, via the separately-passed `units`/`traits` data arrays); a reader extending `traitStripModel` to read any OTHER field off a `UnitInstance` would need to update this test file's stub helper too, or it would silently read `undefined` for that field.
- The `xpProgress` content-drift guard (asserting `gameData.economy.levelXpThresholds` equals the synthetic `t` table) means if a designer changes the REAL thresholds in `economy.json`, this specific test fails FIRST (cheaply, with a clear diff) before any of the boundary-math assertions further down would even run against stale expectations -- a reader seeing this specific assertion fail should treat it as "the data changed, the test's literal needs updating," not a logic bug in `xpProgress` itself.
- `buyXpGeom`'s radius formula (`Math.max(16, Math.min(region.h/2-1, region.w*0.34, 56))`, per the source) caps out at `56` -- intentionally documented in the source as having been "lifted from 30 -> 56" specifically so the landscape econ cluster (a taller region) can render a visibly bigger button; this test's 4 parameterized regions are chosen specifically to span both the THIN portrait HUD band (height-constrained) and the TALLER landscape sub-column (width-constrained via the `region.w*0.34` term, since the test slices a fixed 96px-wide sub-column) -- a reader adding a 5th viewport shape to this test should keep that "spans both binding-constraint regimes" intent in mind rather than just picking arbitrary new dimensions.

# Notes

- `buyXpGeom` is NOT mentioned by name in CLAUDE.md's `hudModel.ts` bullet (which only calls out `traitStripModel`/`xpProgress` explicitly as "Both unit-tested") -- this function and its test coverage appear to be a later addition to the HUD model beyond what the architecture doc's `hudModel.ts` summary currently describes; a reader relying solely on CLAUDE.md's prose would miss this function's existence and its fairly intricate pure-geometry test coverage.
- The `activeTier`/`tierCount` fields and their "maxed iff activeTier===tierCount" idiom are a clean, test-documented pattern worth reusing elsewhere in the codebase if a similar "how many of N tiers have I reached" UI affordance is ever needed (e.g. for a different progression mechanic) -- this test file is the best concrete reference for that idiom's exact semantics.
