# Path & purpose

`packages/client/tests/driver.test.ts` -- the largest behavioral test suite for `LocalDriver` (and a few `NetDriver` cross-checks): a headless full-match AI smoke test, exhaustive `LocalDriver` phase-flow/timer tests (many explicitly framed as regressions for specific previously-shipped bugs), the PvE-round driver surface (combat result, loot orbs, opponent/mob board, upcoming-stage preview), and the item drag-to-EQUIP command path shared by two different UI entry points.

# Responsibility

Owns: end-to-end confidence that `LocalDriver`'s phase state machine (PLANNING -> COMBAT -> RESOLUTION -> PLANNING) advances correctly under real (faked) timers, that RESOLUTION is correctly HELD until the scene signals playback completion (not advanced automatically), that PvE rounds expose a coherent combat result + mob board + loot orbs (deterministically, by seed), that the inline "upcoming creep" planning preview is exposed/hidden at exactly the right rounds for both `LocalDriver` and `NetDriver`, and that the EQUIP command is idempotent/non-double-applying regardless of which UI surface issued it.

# Exports

None (a Vitest test file). Defines a local `FakeWebSocket` class (minimal stub: `OPEN=1` static, inert `readyState=0`, no-op `send`/`close`, never fires `onopen` -- used so a `NetDriver` can be constructed in tests without a real network connection, with its internal `_state` injected directly via an `as unknown as {_state}` cast) and, in the item-drag suite, a `benchUnit(uid)` helper building a full `UnitInstance` object literal from `gameData.units[0]`'s def.

# Key behavior

**"headless AI match"**: drives a COMPLETE match to conclusion using ONLY the pure rules layer directly (`createMatch`, `advancePhase`, `applyAiCommands`, `isMatchOver` -- no driver involved at all) with a `safeguard` loop cap of `10000` iterations to prevent an infinite-loop test hang. Confirms the match eventually ends (`isMatchOver===true`), at most 1 player remains alive at the end, and the safeguard never actually triggered (`safeguard < 10000` -- i.e. the match genuinely concluded rather than being forcibly cut off). This is a baseline rules-layer sanity check that exists in this file (rather than a rules-package test) likely because it shares fixtures/imports with the driver tests below.

**"LocalDriver phase flow"** (uses `vi.useFakeTimers()`/`vi.useRealTimers()` around every test):
- The headline 3-round test confirms the EXACT event sequence `["PLANNING","COMBAT","RESOLUTION"] × 3 + "PLANNING"` is emitted, and -- critically -- that each `phase_change` EVENT is emitted while `driver.getState().phase` ALREADY reflects that new phase (no event-before-state-update ordering bug): captured via `phaseAtEmit.push(driver.getState().phase)` inside the listener itself, then cross-checked after the fact. Also walks through the exact timer semantics per round: planning starts with `>29_000`ms left, advancing the full 30s flips to RESOLUTION immediately (the COMBAT phase is run synchronously inside the 30s timer fire, with COMBAT's own event slipped into the phases array but not separately timed), `combatPlaybackDone()` must be called before the resolution PAUSE timer can elapse, advancing to `resolutionMs - 1` stays in RESOLUTION, and the FINAL 1ms tick flips to PLANNING for `round+1`.
- "resolution overlay teardown path runs": confirms (via the EVENT log alone, not state direct-checks) that across 3 rounds using the Continue-button path (`combatPlaybackDone()` then `advanceFromResolution()`), EVERY `RESOLUTION` event is immediately followed by a `PLANNING` event in the emitted sequence -- the test's comment explains WHY this matters: `MatchScene.onPlanningStart` destroys the resolution overlay, and that teardown is driven SOLELY by receiving the `PLANNING` event, so this is really testing "the overlay will always get torn down."
- "Continue (advanceFromResolution) skips the pause and restarts planning": confirms calling `advanceFromResolution()` after `combatPlaybackDone()` immediately flips to PLANNING round 2 with a fresh `>29_000`ms timer, AND that the CANCELLED resolution pause timer doesn't double-fire later (advancing another 60s afterward never re-enters COMBAT -- the stale timer was properly cleared, not just ignored).
- "PvE round (round 1)": confirms `isPveRound(1)===true` (ground truth from rules), `driver.isPveRound()` mirrors it, `driver.getPveStageName()` is a string, after the 30s elapse the driver is in RESOLUTION with `getMyPairing()===null` (no PvP pairing) but `getMyCombatResult()` non-null (a real combat happened, vs the mob board), `getMyLootOrbs().length > 0`, and -- the determinism check -- a SECOND `LocalDriver` constructed with the SAME seed (`21`) produces BYTE-IDENTICAL loot orbs (`JSON.stringify` equality) after the same advance sequence.
- "PvE round: playback is HELD" -- an explicitly-labeled regression test for a specific shipped bug: "PvE mob board flashes and vanishes in Practice." The bug's root cause is described precisely in the comment: the OLD playback-duration CAP computed itself from `getPairingFor(...).result`, which is `null` for a PvE round (no pairing) → `capMs=0` → the cap fired on the very next macrotask, tearing down the combat view before it ever rendered a frame. The fix makes the cap PvE-aware. The test proves the fix: after the 30s planning elapses, only `["PLANNING","COMBAT"]` have been emitted (NOT yet `"RESOLUTION"`), the combat result has real events, and crucially advancing time by `0` (flushing pending microtasks/macrotasks) does NOT produce a `"RESOLUTION"` event -- only an explicit `combatPlaybackDone()` call releases it.
- "PvE round: mob board reaching the driver is non-empty and renders" -- another explicitly-labeled regression test ("PvE mobs not rendering"). Confirms `getMyOpponentBoard()` is non-null with at least one non-null slot, the combat result's tick-0 state contains side-1 (mob) units, and -- a data-integrity cross-check -- every mob unit's `defId` is ABSENT from `gameData.units` (confirming the mob-vs-real-unit distinguishing heuristic used throughout the renderer, e.g. in `unitToken.ts`'s ring-color logic) but PRESENT in `gameData.mobs.mobs` (confirming mob ids are real, just in a separate catalog).
- "PvE round: NetDriver-equivalent local derivation (COMBAT_START)": iterates rounds `[1,2,3,4,7,10]`, skipping any that aren't PvE per `isPveRound`, and for each PvE round directly exercises the EXACT shared rules helpers `NetDriver` itself uses online (`pveStageForRound`, `buildMobBoard`, `boardToCombatState`, `derivePairingSeed`, then a raw `simulateCombat` call) -- confirming the mob board is always non-empty and the resulting combat's tick-0 state always contains side-1 units, for EVERY currently-configured PvE stage, not just round 1. This is a forward-looking completeness check protecting against a future stage/round being silently mis-configured.
- "planning duration is a full 30s for every round, regardless of adjacent PvE status": an explicitly-labeled regression test ("user reported never observing Planning before a PvE round"). Drives rounds 1-5 (PvE rounds 1-3, then PvP 4-5) and confirms EVERY round's planning phase starts with EXACTLY `30_000`ms left (not shortened for PvE rounds or the round immediately following one), confirms the timer counts down normally (`20_000` left after 10s elapse), and that the recorded PvE-ness sequence is `[true,true,true,false,false,false]` matching rounds 1-6's actual PvE/PvP split -- ground-truthing that the bug report's premise (short-circuited planning around PvE rounds) does NOT occur.
- "ready() mid-planning is a no-op; only the full 30s timer reaches RESOLUTION": an explicitly-labeled regression test ("READY no longer skips planning"). Confirms calling `driver.ready()` at multiple points during planning (immediately, at 15s, at 29.999s) never advances the phase -- ONLY the timer firing at the full 30,000ms mark does.
- "RESOLUTION is held until combatPlaybackDone; duplicate calls are no-ops": confirms COMBAT runs (synchronously, on the 30s timer) and the state IS already `"RESOLUTION"` even though the EMITTED event sequence stops at `["PLANNING","COMBAT"]` (the state and the public event stream can diverge -- state reflects RESOLUTION immediately, but the `phase_change:RESOLUTION` EVENT is deliberately delayed until playback signals done); confirms `advanceFromResolution()` called BEFORE `combatPlaybackDone()` is a no-op (round stays 1); confirms `combatPlaybackDone()` THEN emits RESOLUTION exactly once even if called a SECOND time (idempotent, no duplicate RESOLUTION events).

**"getUpcomingPveBoard (PvE planning preview)"** (stubs the global `WebSocket` with `FakeWebSocket` so `NetDriver` can be constructed):
- Confirms `LocalDriver.getUpcomingPveBoard()` during a PvE round's planning returns a non-null board with `>0` units, EVERY unit's `uid < 0` (the documented "disposable negative preview uid" convention -- guaranteed never to collide with a real combat uid), and the result is BYTE-IDENTICAL (via `JSON.stringify`) to calling the pure rules accessor `previewPveStage(driver.getState(), gameData)` directly -- i.e. `LocalDriver`'s method is a thin pass-through with no additional client-side logic.
- "tracks the round being planned for, not the previous round": an explicitly-labeled off-by-one regression guard. Walks rounds 1-3 (PvE) confirming the preview chip is visible during EACH one's OWN planning (not the previous round's), then confirms round 4 (PvP, immediately after the PvE rounds) correctly shows NO preview -- explicitly ruling out an off-by-one bug where the chip might check the JUST-FINISHED round's PvE-ness instead of the CURRENT one.
- "reuses the pure previewPveStage (idempotent, no state mutation, no new rules logic)": confirms calling `previewPveStage` twice produces identical output, the driver's own accessor matches, AND -- crucially -- the ENTIRE match state's JSON serialization is byte-identical before and after these calls (proving the preview truly never mutates state, consistent with its disposable-negative-uid design avoiding any pool/namespace interaction).
- "LocalDriver returns null once out of the planning phase": confirms after the 30s elapses into COMBAT, the preview accessor returns `null` (the inline creep preview is a PLANNING-phase-only affordance).
- "NetDriver derives the same preview client-side (no protocol round-trip)": constructs a real `NetDriver("ws://test")`, injects its private `_state` field directly (cast via `as unknown as {_state}` since this is normally managed internally) to `{round:1, phase:"PLANNING"}`, and confirms its `getUpcomingPveBoard()` output is byte-identical to `LocalDriver`'s -- proving both drivers funnel through the SAME pure rules accessor (`previewPveStage`) without any server round-trip needed for this purely-cosmetic preview.
- "NetDriver returns null on a non-PvE round and outside planning": three injected `_state` scenarios (`{round:4,phase:"PLANNING"}` PvP round, `{round:1,phase:"COMBAT"}` PvE round but wrong phase, `null` no state at all) ALL correctly return `null` -- confirming `NetDriver`'s preview accessor degrades gracefully under every non-applicable condition, including before any state has even arrived from the server.

**"item drag EQUIP command path"**: an explicitly-labeled shared-path test, per its header comment: "Both the main item bar and the landscape tab-2 browse render the player's inventory via `inventoryModel` and drag a chip -> `startDragItem` -> `tryEquip` -> `driver.playerCommand({type:"EQUIP",...})`. The scene is Pixi/WebGL and not headless-testable, so we exercise the one command path both sources funnel into."
- "an EQUIP issued the way both sources issue it moves the item onto the unit": manually injects a bench unit and an `"iron_sword"` component into the player's state, resolves it through the REAL `inventoryModel` (confirming the inventory entry the UI would actually drag is findable), then issues the EXACT `EQUIP` command shape (`unitUid`+`itemId` from the resolved `InventoryEntry`) and confirms the command succeeds, the unit now holds the item, and the player's loose inventory no longer contains it.
- "the same item is consumed once — a second identical EQUIP fails (no double-handling)": issues the SAME `EQUIP` command TWICE in a row; the first succeeds, the second is rejected with the EXACT typed error `{ok:false, error:"ITEM_NOT_FOUND"}` (the item is already consumed, gone from inventory) -- confirming neither UI entry point could ever cause a double-equip even if both somehow fired the same drag gesture's command twice.

# Invariants & constraints

- This file's tests are HEAVILY regression-framed -- the majority carry an explicit "Regression for: ..." comment describing a SPECIFIC previously-shipped bug. A reader fixing a NEW driver/timing bug should both check whether an existing test here already covers it, and follow the established pattern (detailed root-cause comment + a test that would have failed under the old buggy behavior) when adding a new one.
- The RESOLUTION-hold mechanism (state flips to RESOLUTION immediately on the timer firing COMBAT synchronously, but the PUBLIC `phase_change` EVENT for RESOLUTION is delayed until `combatPlaybackDone()`) is a subtle but load-bearing distinction throughout this file -- a reader must track BOTH `driver.getState().phase` (immediate) and the EMITTED event stream (delayed) separately; conflating them would misread several of these tests' intent.
- `vi.useFakeTimers()`/`vi.useRealTimers()` wrap EVERY test in the "LocalDriver phase flow" describe block via `beforeEach`/`afterEach`, but the later "getUpcomingPveBoard" suite manages fake timers manually PER-TEST (`vi.useFakeTimers()`/`vi.useRealTimers()` inside a `try/finally`) rather than via the describe-level hooks -- a maintainer adding a new test to that second suite needing fake timers must remember to wrap it in its own try/finally, since the surrounding describe block does NOT do this automatically (unlike the first suite).
- The `FakeWebSocket` stub intentionally NEVER calls `onopen` -- this is what keeps the constructed `NetDriver` "inert" (it never believes it's connected), which is WHY the tests must inject `_state` directly via a type-unsafe cast rather than relying on any real message-driven state population.
- The item-drag suite's framing ("the one command path both sources funnel into") establishes that THIS test file is the canonical place to verify EQUIP-from-drag correctness for BOTH UI entry points (item bar AND landscape tab-2 browse) -- since the actual Pixi/WebGL drag gesture code itself is untestable headlessly, a maintainer changing either UI's drag-to-equip wiring should verify it still funnels through `driver.playerCommand({type:"EQUIP",...})` with this exact shape, since that's the only point these tests can observe.

# Notes

- This is arguably the single most VALUABLE regression-test file in the client package for anyone touching `LocalDriver`'s timer/phase-transition logic, given the density of explicitly-documented historical bugs it guards against (PvE flash-and-vanish, empty PvE mob board, planning-skipped-near-PvE, ready()-skips-planning) -- any future driver refactor should run this file FIRST and treat every regression comment as a hard constraint, not just a nice-to-have.
- The "headless AI match" test's presence in this file (rather than in a rules-package test) is a minor organizational quirk -- it tests ONLY the pure rules layer with no driver involvement, but likely lives here because it shares the `gameData`/`mulberry32`/`createMatch` imports with the rest of the suite and serves as a quick sanity baseline before the driver-specific tests run.
