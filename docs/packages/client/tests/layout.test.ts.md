# Path & purpose

`packages/client/tests/layout.test.ts` -- the largest and most exhaustive test file in the client package: full geometric regression coverage for `layout.ts`'s `resolveLayout` (orientation detection, scale-to-fit, safe-area insets, the landscape 4-cluster + residual-board architecture, all named regions in both orientations, the portrait height-driven stack) plus its smaller pure geometry helpers (`centeredModal`, `landscapeBenchSlotCenter`/`landscapeBenchSlotAt`, `planningRegionAt`, `opponentRailTile`, `shopCardContentLayout`, `shopPanelSlotRect`, `landscapeClusterThickness`, `bottomCornerButtonTop`), `hudControlsLayout.ts`'s `landscapeHudControls`, and `combatLayout.ts`'s full 9-layer scene z-stack ordering.

# Responsibility

Owns: the master geometric correctness contract for the ENTIRE match-scene layout system across both orientations and the full supported viewport range (the 360x640 floor up to large/ultrawide screens) -- non-overlap of every named region, in-bounds containment, touch-target minimums, safe-area-inset handling (notches), the landscape "4 fixed-edge clusters + one residual board area" architecture (top bar / left rail / right rail / bottom bar, with the board occupying whatever's left), the portrait region budget that reproduces the prior hardcoded layout exactly at the canonical 844 design height while degrading gracefully down to the 606px floor, and the full 9-named-layer scene z-stack ordering used throughout `scenes/match.ts`.

# Exports

None (a Vitest test file). Defines reusable geometry helpers: `rectsOverlap(a,b)` (axis-aligned overlap test, exclusive right/bottom edge), `pairs(arr)` (all unique unordered pairs, used to exhaustively check every region against every other), `withinBounds(r,maxW,maxH)` (containment in `[0,maxW)x[0,maxH)`), `coreRegions(layout)` (returns `[board,traitRail,opponentRail,shop,bench,hud]` -- the canonical "core regions" set checked for bounds/non-overlap, deliberately excluding `traitTabBar` since it intentionally sits WITHIN the `traitRail` column rather than being a disjoint sibling).

# Key behavior

**"orientation detection"**: confirms an 844x390 viewport resolves `orientation:"landscape"` with `designW/H` equal to the `LANDSCAPE_W`/`LANDSCAPE_H` constants; a 390x844 viewport resolves `"portrait"` with `PORTRAIT_W`/`PORTRAIT_H`; the orientation flips exactly AT `LANDSCAPE_THRESHOLD` (aspect ratio 1.1 -- just at/above it is landscape, just below is portrait); a perfectly SQUARE 600x600 viewport stays portrait (aspect 1.0 < threshold); confirms the landscape design canvas is the FIXED `1280x592` reference (aspect ratio ~2.162) regardless of the actual device's aspect ratio -- i.e. the design canvas itself never changes shape, only the SCALE applied to fit it varies by viewport.

**"scale-to-fit"**: confirms `designW*scale <= viewportW` and `designH*scale <= viewportH` (with a small epsilon) hold in BOTH orientations -- the design canvas is scaled to fit entirely within the viewport, never overflowing; confirms scale grows proportionally on a LARGER screen (a 780-wide portrait viewport gives `scale ~= 780/PORTRAIT_W` exactly, and similarly for an extra-wide landscape viewport against `LANDSCAPE_H`).

**"safe-area insets"**: confirms left/right insets in landscape shrink the USABLE width (scale becomes `<=` the no-insets scale) and shift the canvas offset RIGHTWARD (away from a notch); confirms top/bottom insets in portrait shrink usable height and shift the offset DOWNWARD; confirms the canvas's offset+scaled-size always stays fully within the safe-area-bounded viewport rectangle (never rendering INTO a notch); and two more nuanced tests on the landscape 4-cluster system's "leading edge" placement: with ZERO device inset, every cluster's leading edge sits at EXACTLY `LS_BASE_MARGIN` (documented as "the floor for the no-notch case," since the region solve runs in fixed DESIGN space and a real CSS-px inset doesn't directly enter that solve); with a LARGE device inset (sized so the usable viewport area exactly equals the 1280x592 design, making `scale===1` and CSS-px insets map 1:1 to design-space), the cluster's leading edge visibly grows PAST the bare `LS_BASE_MARGIN` to accommodate the inset -- proving `max(safeInsetDesign, LS_BASE_MARGIN)` is genuinely applied, not just a fixed margin regardless of device.

**"landscape clusters"**: confirms `layout.clusters` is DEFINED in landscape (with all 5 properties: `topBar`/`leftRail`/`rightRail`/`bottomBar`/`boardArea`) but UNDEFINED in portrait (clusters are "a landscape-only concept," per the test name) -- a reader checking `layout.clusters` must always guard for portrait. Confirms the 4 edge clusters are pairwise non-overlapping and within design bounds. Confirms `boardArea` is exactly the RESIDUAL rectangle left after the 4 edge clusters are placed (its edges align flush against the inner edges of `leftRail`/`rightRail`/`topBar`/`bottomBar`, and it overlaps none of them). Confirms the actual RENDERED `board` region fits fully inside `boardArea` and reads as "a wide, shallow slab" (`board.w/board.h > 1.5`, explicitly NOT hex-aspect) -- because, per the comment, the board is rendered as "a wide perspective slab (near edge spans most of the width, laid back via BOARD_TILT)" rather than a plain rectangular hex grid; confirms the board's near (bottom) edge targets `~75%` of the design width and the board horizontally CENTERS on the safe-area center.

**"landscapeClusterThickness clamp behavior"**: directly exercises `landscapeClusterThickness(w,h)` (the pure function computing each edge cluster's thickness from the design/usable dimensions) across several scenarios: at the reference 1280x592 design, all 4 thicknesses land within documented min/max bands (`topBarH` 30-56, `bottomBarH` 52-72, `leftRailW` 96-160, `rightRailW` 80-130); at 2x the reference size (simulated ultrawide), every thickness STILL caps at its documented maximum (never grows unbounded); at a much NARROWER usable width (500 vs the 1280 reference, "simulating a squeezed 4:3-ish landscape splitscreen"), the rails compress toward their minimum rather than ever eating into the board's own minimum floor (`LS_BOARD_MIN_W=320`, confirmed via `leftRailW+rightRailW <= 500-320+0.5`); confirms a real 16:9 device (1920x1080, scaled down) and a 4:3-equivalent device (800x600) BOTH resolve to the SAME fixed 1280x592 design with all 4 clusters still valid (positive size, non-overlapping with `boardArea`) -- the design canvas's shape is independent of the device's actual aspect ratio, only the SCALE differs.

**"bottomCornerButtonTop"**: tests the pure `bottomCornerButtonTop(y, h, w)` -- computing the TOP of a circular corner button's footprint (used for collision/clearance checks against other regions) given a region's `y`/height, mirroring the same radius-capping formula as `buyXpGeom`/`hudModel.ts` (radius capped at `h/2-1` and at an absolute max of `56`, rim width `radius*0.16`). Three precise hand-computed numeric checks (e.g. `h=110 -> r=54, rim=8.64, cy=255, top=255-62.64`) plus a qualitative check that the footprint genuinely OVERHANGS above the region's nominal top edge (since the circle is taller than a typical bench row).

# Invariants & constraints

- **Landscape is now a fixed 1280x592 design canvas regardless of device aspect ratio** -- this is a SUBSTANTIALLY different architecture than CLAUDE.md's documented "three-column design (844x390)" description; the real current landscape system uses a 4-fixed-edge-cluster (`topBar`/`leftRail`/`rightRail`/`bottomBar`) + one residual `boardArea` architecture, with the rendered `board` itself being a "wide perspective slab" (tilted via `BOARD_TILT`, not a plain rectangle) that targets ~75% of the design width on its near edge. A reader relying on CLAUDE.md's landscape-layout prose alone would be working from a STALE/superseded description; `layout.ts`'s actual source and this test file are the ground truth for the CURRENT landscape architecture.
- `layout.clusters` is `undefined` in portrait and ALWAYS defined in landscape -- any code reading `layout.clusters!` (non-null-asserted, as several of this file's own helper functions do) MUST first branch on `layout.orientation==="landscape"`, or it will throw/misbehave in portrait.
- Portrait is explicitly called out (in TWO separate describe-block header comments) as "UNTOUCHED -- portrait is out of scope" for whatever refactor introduced the landscape 4-cluster system -- meaning the portrait-region tests in this file are primarily REGRESSION guards preserving pre-existing behavior (including byte-identical legacy position checks at specific x/y/w/h values), not documentation of NEW portrait work.
- The portrait height-driven design means `designH`/`portraitDesignH` vary with the VIEWPORT while `scale` stays fixed at a WIDTH-only ratio (`viewportW/PORTRAIT_W`) -- this is the inverse of the landscape model (fixed design, varying scale) and is the reason coachmark ring placement (per `coachmarks.test.ts`, documented earlier in this pass) must read the LIVE `portraitDesignH` rather than assume the canonical 844.
- The `SCENE_LAYER_ORDER`/`SCENE_LAYER_NAMES` sort-by-VALUE-not-by-NAME-SUFFIX caveat is explicitly and repeatedly flagged in this file's comments as a trap for future maintainers -- adding a new layer constant with a "logical" name suffix does NOT guarantee its position in these arrays matches that suffix; the arrays must be read/maintained by ACTUAL VALUE, always.
- Several tests REPRODUCE production code's geometry formulas inline as local helper functions within the test file itself (`buyXpFootprintLs()` mirroring `match.ts`'s `buyXpFootprint()`/`hudModel.buyXpGeom`'s exact radius-capping math) rather than importing and calling the real function -- this is a DELIBERATE choice (likely because the real footprint function lives inside a Pixi scene class method, not as a standalone pure export) but it creates a manual-sync risk: if the real geometry formula changes, this test's local mirror must be updated in lockstep or it will silently test against STALE geometry while still passing.
