# Path & purpose

`packages/client/tests/inspectModel.test.ts` -- unit tests for `inspectModel.ts`'s `inspectModel(defId, instance, data)` (assembles a unit-inspect panel's full identity/stat/ability model from data + an optional live instance) and `abilityDescription(name, effect, abilityDamage)` (renders a one-line human-readable description per ability-effect kind).

# Responsibility

Owns: regression coverage that `inspectModel` correctly resolves a real unit def's identity (name/tier/cost/origin/classes), renders ALL stat rows with fixed-point-safe formatting (especially attack speed), switches between base-stat (shop preview) and live-instance (owned unit) values for hp/mana/star, falls back to the PvE mob catalog (`data.mobs.mobs`) for a defId absent from `data.units` (with the mob-specific reduced shape: no traits, no shop cost, no items), and that `abilityDescription` produces the right prose for every one of the engine-supported ability-effect kinds, including the fixed-point-aware rendering of an attack-speed buff's delta.

# Exports

None (a Vitest test file).

# Key behavior

**`inspectModel` suite**:
- "returns null for an unknown unit": `inspectModel("__nope__", null, gameData)` is `null` -- an id absent from BOTH `data.units` and `data.mobs.mobs` resolves to nothing, never throws.
- "assembles identity, traits, cost and the full stat block from data": using the FIRST real unit def (`gameData.units[0]`) with `instance=null` (a shop-preview scenario): `name`/`tier` match the def directly, `cost` equals `tier` (units' buy cost IS their tier, per the source), `star` defaults to `1` (no live instance to read a star from), `origin.id` matches `def.origin`, `classes` mapped ids match `def.classes` exactly. Confirms ALL 8 expected stat labels (`HP, AD, AS, Armor, MR, Range, Mana, Ability`) are present in the stats array. Confirms the SHOP-PREVIEW-specific value shapes: `HP` shows just the base value (`"${def.hp}"`, no `current/max` split since there's no live instance), `Mana` shows `"${def.manaStart}/${def.mana}"` (start/max, even without an instance -- a preview of the unit's natural mana-start position). Confirms `AS` (attack speed, fixed-point scale 1000) is formatted as `(def.as/1000).toFixed(2)` -- e.g. raw `750` renders as `"0.75"` -- NEVER the raw integer, consistent with the project-wide `statFormat.ts` convention.
- "uses the live instance's current hp/mana/star when owned": builds a partial `UnitInstance` (cast `as UnitInstance`) with `star:2`, `hp:7` (vs `maxHp:def.hp`), `mana:3` (vs `maxMana:def.mana`), and the def's own `ad`/`armor`/`mr` (left unmodified) -- confirms `inspectModel` reports `star:2` (read from the instance, not defaulted to 1) and the `HP`/`Mana` stat VALUES switch to the live `current/max` format (`"7/${def.hp}"`, `"3/${def.mana}"`) instead of the shop-preview's single-value/start-max format -- this is the owned-unit (board/bench) inspect scenario.
- "resolves a PvE mob defId via the mob fallback (no traits, no cost)": uses the FIRST real mob def (`gameData.mobs.mobs[0]`) with `instance=null`. Confirms the fallback resolves successfully (`m` not null), `name`/`tier` match the mob def, and the MOB-SPECIFIC reduced shape: `origin` is `null` (mobs carry no origin trait), `classes` is `[]` (no class traits), `cost` is `0` (mobs are never shop-purchasable, this is the sentinel value the panel renders as "tier-only, no cost" per the source comment), `items` is `[]` (mobs never hold equippable items, per the source comment "Mobs never hold items"). Confirms the CORE stat rows (`HP, AD, AS, Armor, MR, Range, Mana` -- notably NOT necessarily `Ability`, since that label is conditionally included only `...(mob.ability ? [...] : [])`) are still present and `HP`'s value matches the mob's base hp. The final assertion (`m.ability === null || typeof m.ability.name === "string"`) is a permissive either/or check acknowledging mobs MAY or MAY NOT define an ability (data-dependent on which specific mob happened to be first in the array), rather than asserting a specific outcome.
- "describes every ability effect kind": exercises `abilityDescription` directly for ALL 5 ability-effect kinds the engine supports (per CLAUDE.md: `magic_damage`, `burn`, `shield`, `buff`, `stealth`), each via a substring/regex match (`toMatch`) rather than an exact full-string match (tolerant of incidental prose wording changes as long as the key numbers/terms appear): `magic_damage` -> matches `/100 magic/` (the passed `abilityDamage` of 100 appears with "magic"); `burn` (`{burn:20, duration:40}`, `abilityDamage:80`) -> matches `/burns for 20/` (the burn-DoT amount, not the upfront `abilityDamage`); `shield` (`{amount:200, duration:60}`) -> matches `/200 shield/`; `buff` (`{stat:"ad", value:30, duration:80}`) -> matches `/ad by \+30/` (a non-fixed-point stat's delta renders as the raw signed value); `buff` again with `{stat:"as", value:280, duration:80}` -> matches `/as by \+0\.28/` -- explicitly confirming that an ATTACK-SPEED buff's delta is rendered through the FIXED-POINT-AWARE `formatStatDelta` (raw `280` -> `"+0.28"`), not the raw integer, distinguishing it from the plain `ad` case just above; `stealth` (`{duration:20}`) -> matches `/Untargetable/`.

# Invariants & constraints

- The mob-fallback path in `inspectModel` is keyed PURELY on id lookup order: it first checks `data.units`, and ONLY IF that lookup misses does it check `data.mobs.mobs` -- this is the same "defId absent from data.units => it's a mob" heuristic used throughout the renderer (e.g. `unitToken.ts`'s ring-color logic, per other test files in this suite); a reader adding any NEW non-unit, non-mob entity type that might share this lookup path would need to extend `inspectModel`'s fallback chain (currently exactly two tries, then `null`).
- The shop-preview vs owned-unit value-format distinction (`"${def.hp}"` single value vs `"${instance.hp}/${instance.maxHp}"` current/max split) is determined SOLELY by whether `instance` is `null` or not -- there is no separate explicit "isPreview" flag; a reader must remember that passing a non-null instance ALWAYS switches to the live-value rendering mode, even a deliberately partial/stub instance (as the second test's hand-built instance demonstrates).
- `abilityDescription`'s `buff` case formatting depends entirely on `formatStatDelta(effect.stat, effect.value)` from `statFormat.ts` -- ONLY the `as` (attack speed) stat is fixed-point-scaled (divide by 1000) when rendering a delta; all other stats render their raw signed value directly. A reader adding a new fixed-point stat in the future would need to update `statFormat.ts`'s scale-aware list, and this test's `ad` (raw) vs `as` (scaled) pairing is the canonical example distinguishing the two formatting paths.
- The mob stats array's `Ability` row is CONDITIONALLY present (spread via `...(mob.ability ? [...] : [])`) -- unlike the unit stats array where `Ability` is unconditionally always the 8th row. A reader inspecting a mob's stat-row COUNT should not assume it always matches a unit's; it varies by whether that specific mob def happens to carry an ability.

# Notes

- This test file does NOT exercise the `items` field's non-empty case for a real owned unit (the live-instance test's hand-built instance has no `items` field set at all, so `inspect.items` would resolve via `(instance?.items ?? [])` to an empty array implicitly) -- coverage of the equipped-items row rendering through `inspectModel` specifically is left to whatever test exercises `itemModel.ts`/`inspectPanel.ts` directly, if any; a reader investigating the equipped-items display path should look elsewhere first.
- The mob fallback test deliberately picks `gameData.mobs.mobs[0]` (whichever mob happens to be first in the real data) rather than a specific named mob id -- this makes the test content-order-dependent in the same spirit as several other files in this suite (e.g. `combinePreview.test.ts`'s "first item with a recipe" pattern), trading a small amount of specificity for automatic tracking of real data without needing updates when mob ids are added/removed/reordered.
