# packages/sim/tests/pairPassive.test.ts

**Path & purpose** — `packages/sim/tests/pairPassive.test.ts`. Dedicated behavioral tests for `engine.ts`'s `pairPassive` mechanic (the co-equip-gated bonus effect that activates only when BOTH items of a specific named pair are equipped on the SAME unit) — exercises two REAL "artifact" item pairs from actual content data: `voidstaff`+`voidmantle` (start-of-combat shield) and `warblade`+`warplate` (on-hit burn).

**Responsibility** — This is the file that resolves the gap flagged when documenting `types.ts`'s local `ItemDef` (which lacked a `pairPassive` field despite `engine.ts` reading `itemDef.pairPassive` at runtime): this test file PROVES `pairPassive` is a real, live field on actual item content (`gameData.items.find(i => i.id === "voidstaff")!.pairPassive` is accessed directly and asserted non-trivial), confirming `@autobattler/data`'s real item type DOES carry this field even though `sim`'s own local `ItemDef` interface in `types.ts` doesn't declare it — exactly the discrepancy predicted in that file's documentation.

**Key behavior — test setup**
- Reuses the SAME `mk`/`board`/`hpEvents` helper pattern as `abilities.test.ts` (identical implementations, not shared/imported — duplicated across both files).
- `HIDE`: a permanent-stealth `AbilityDef` (`duration: 1_000_000`) used to neutralize a unit so it never moves/attacks/is targeted — "isolates the observable under test (mirrors abilities.test.ts)" per its own comment.
- `VOIDSTAFF`, `VOIDMANTLE`, `WARBLADE`, `WARPLATE`: looked up directly from `gameData.items` by id — REAL content data, not synthetic test fixtures, meaning these tests' correctness is coupled to these 4 specific items continuing to exist in `items.json` with their current pair-passive configuration.
- The file's own top-of-file comment explains the testing philosophy precisely: "The Phase 3 artifacts carry a `pairPassive` that fires only when BOTH paired items are equipped on the SAME unit. There are two observable manifestations: voidstaff + voidmantle → a start-of-combat `shield`...; warblade + warplate → an on-hit `burn` DoT... These tests assert on those real event-log observables, never a guessed one." — i.e. deliberately avoiding asserting on INTERNAL state (like reading `unit.shield` directly, which isn't exposed) and instead inferring the passive's effect purely from externally-observable event-log behavior (hp readings), mirroring how a real consumer (client renderer, or another test like `room.test.ts`'s resimulation) would have to verify things too.

**Key behavior — test by test**
1. **"both paired items on the same unit grant the shield (early hits absorbed)"**: equips BOTH `voidstaff` AND `voidmantle` on a defender (0 ad/armor, high hp) against a moderate-ad attacker; reads the defender's post-setup `maxHp` from the `init` event (capturing whatever base stat bonuses the two items granted, e.g. the comment notes "mr/hp from voidmantle, etc." apply as base stats regardless); asserts the VERY FIRST `hp` event for the defender STILL equals `maxHp` exactly (the shield fully absorbed the first hit, with zero hp loss showing in the event stream) — and separately asserts the pair passive's configured `shieldVal` (read directly off `VOIDSTAFF.pairPassive!.effect.value`) is greater than 0 (a basic content sanity check, confirming this isn't accidentally testing a zero-value no-op passive).
2. **"either item ALONE does not grant the pair shield (first hit lands)"**: runs the SAME setup but with only ONE of the two items at a time (`["voidstaff"]` then `["voidmantle"]`, via a loop) and a STRONGER attacker (`ad: 200`, to ensure even a base-stat-boosted, single-item-equipped defender's hp visibly drops) — asserts the FIRST hp reading is LESS than `maxHp` in BOTH single-item cases, i.e. NO shield protected the first hit when the partner item is missing — the comment clarifies why a stronger attacker is needed here: "No shield (mr mitigates magic, not physical AD), so the first hit lands" — meaning even though `voidmantle`/`voidstaff` ALONE might still grant SOME base stat (e.g. `mr`, magic resist) that partially mitigates damage, that's irrelevant against a PHYSICAL attack (`ad`-based), so the lack of the SHIELD specifically is what's truly being measured here, not residual base-stat tankiness.
3. **"both paired items on the same unit apply a burn DoT to the struck target"**: equips both `warblade` and `warplate` on a deliberately SLOW attacker (`as: 50`, "Attacker hits once then idles (very slow as), isolating the burn ticks" — same isolation technique as `abilities.test.ts`'s analogous on-hit-burn item-passive test) targeting a high-hp dummy; scans the target's hp-event deltas for ANY consecutive pair whose drop equals EXACTLY `burnVal` (read off `WARBLADE.pairPassive!.effect.value`) — confirming a pure burn-only tick (no attack damage mixed in) occurred, i.e. the on-hit burn DoT from the PAIR passive genuinely fired and ticks independently.
4. **"either item ALONE applies NO on-hit burn (only the attack hp drop occurs)"**: same setup with only one item at a time, this time ALSO giving the TARGET a permanent `HIDE` stealth ability (an additional isolation technique not used in test #3 — presumably because without the pair-burn DoT muddying things, the attacker's own regular attack damage could otherwise be mistaken for "no burn ever happened," so making the target untargetable... wait, re-reading: actually the target has `ability: HIDE` which makes IT untargetable, meaning the SLOW attacker would never be able to land its initial attack either — but the test still expects to observe events; this detail is worth a careful re-read by a future agent verifying the exact mechanics, as the comment doesn't fully reconcile why HIDE is added here but not in test #3). Asserts NO consecutive hp delta ever equals exactly `burnVal` — confirming the single-item case never triggers the pair-gated burn passive.
5. **"warblade's base ad/as bundle applies with or without warplate equipped"** (describe: "artifact base stats apply regardless of partner presence"): directly tests the OTHER half of the pairPassive contract — that the item's OWN base `stats` bundle (e.g. `warblade`'s `ad` bonus) is NEVER gated by the partner's presence, only the EXTRA pair-passive bonus is. Measures a clean (non-crit) first-attack damage with `warblade` ALONE vs. `warblade`+`warplate` TOGETHER, and asserts BOTH cases deal IDENTICAL damage (`50 + wbAd`, base 50 ad + warblade's own ad stat) — since `warplate` grants armor/hp (not ad), pairing it shouldn't change the attacker's OWN damage output at all; this directly confirms `engine.ts`'s `applyItems` (base stats, unconditional) is entirely separate from `applyStartOfCombat`'s `pairPassive` gating (extra effect, conditional).
6. **"same seed + same paired items yields an identical event log across two runs"** (describe: "pair passive determinism"): runs the SAME attacker/target setup (one side with `warblade`+`warplate`, the other with `voidstaff`+`voidmantle`) TWICE with the same seed, JSON-stringifies the full `CombatResult` each time, and asserts byte-identical output — confirming `pairPassive` resolution introduces NO non-determinism (e.g. no accidental reliance on object key iteration order when checking `unit.items.includes(partnerId)`, which in practice is a simple array `.includes()` call and thus inherently order-independent and deterministic, but this test locks that in empirically regardless).

**Invariants & constraints**
- This file is content-coupled to 4 SPECIFIC real items (`voidstaff`, `voidmantle`, `warblade`, `warplate`) continuing to exist in `items.json` with a `pairPassive` referencing each other as partners and the specific effect kinds (`shield`/`burn`) — if `items.json` were ever restructured to remove these exact items or change their pair configuration, this entire test file would need updating (or would start failing with non-null-assertion crashes on the `!.pairPassive!`/`.effect` chains).
- **This file is the definitive, authoritative confirmation that `pairPassive` is real, live, runtime-functioning content data** — any future agent investigating the `ItemDef`/`pairPassive` type-vs-usage gap noted in `types.ts`'s documentation should treat THIS test file (rather than `types.ts`'s local interface) as the ground truth for what a real item def's shape includes.
- The asymmetry between test #4 (uses `HIDE` on the target) and test #3 (doesn't) is a genuine area worth a careful second look if a future agent is debugging a `pairPassive`-related issue — it's not necessarily a bug (the tests both pass as written, presumably), but the REASON for the asymmetry isn't self-evident from the comments alone and deserves closer inspection before relying on it as a template for new tests.

**Depends on** — `vitest`; `../src/engine.js` (`simulateCombat`); `@autobattler/data` (`gameData`, for the 4 real items); `../src/types.js` (`AbilityDef`, `BoardState`, `UnitInstance` types).

**Used by** — Run as part of `npm test`; not imported elsewhere.

**Notes** — Together with `abilities.test.ts`, this file completes the full behavioral specification for `engine.ts`'s item-passive system: `abilities.test.ts` covers the UNCONDITIONAL per-item passive (`passive` field), while this file covers the CONDITIONAL co-equip passive (`pairPassive` field) — a future agent should consult BOTH files for complete item-passive coverage, not just one.
