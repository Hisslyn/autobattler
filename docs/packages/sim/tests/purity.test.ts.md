# packages/sim/tests/purity.test.ts

**Path & purpose** — `packages/sim/tests/purity.test.ts`. A static-analysis test (not a behavioral test) that scans EVERY `.ts` file under `packages/sim/src/` (recursively) for the literal substrings `Math.random`, `Date.now`, and `parseFloat`, failing if any are found.

**Responsibility** — This is the direct, literal, automated enforcement of CLAUDE.md's hard invariant "packages/sim is pure: no I/O, no Math.random, no Date, no floats" — specifically the randomness-source and wall-clock-time halves of that rule (plus a `parseFloat` check, presumably as a proxy/guard against accidentally introducing float-parsing logic that would violate the "no floats" half). Any future code change to `packages/sim/src/*.ts` that introduces a call to any of these three APIs will fail this test immediately, with the exact offending file path named in the assertion message.

**Key behavior**
- `getAllTsFiles(dir)`: a simple recursive directory walker (`readdirSync` with `withFileTypes: true`) that collects every file ending in `.ts` anywhere under the given root, descending into subdirectories. Run once against `SIM_SRC = join(import.meta.dirname, "../src")` — i.e. `packages/sim/src/` relative to this test file's own location, using `import.meta.dirname` (a modern Node ESM API) rather than `__dirname` (which isn't available in ESM modules) — confirms this test file itself runs as an ES module.
- Three tests, each independently re-reading EVERY file's full source text (`readFileSync(f, "utf8")`) and running a regex (`/Math\.random/g`, `/Date\.now/g`, `/parseFloat/g`) against it, asserting the match result `toBeNull()` (i.e. zero occurrences) with a custom failure message naming the specific file that violated the rule.
- This is PURE TEXT/STRING scanning — it does NOT parse the TypeScript AST, so it would also flag these substrings if they appeared inside a STRING LITERAL or COMMENT (a false positive risk) but conversely would MISS equivalent-but-differently-spelled violations like `globalThis.Math.random` or destructured/aliased imports (e.g. `const { random } = Math; random()` would NOT be caught by the `Math\.random` regex). This is a deliberate trade-off for simplicity over precision — the test favors a cheap, fast, obviously-correct-for-the-common-case check over a more robust but complex AST-based one.

**Invariants & constraints**
- This test ONLY scans `packages/sim/src/` — it does NOT scan `packages/sim/tests/` itself (this file and its sibling test files ARE permitted to do things like use `mulberry32` as a fuzz-test seed generator, or theoretically even call `Math.random()` if a test ever needed to, without tripping this guard) — purity is enforced on the SHIPPED ENGINE CODE only, not on the test harness.
- This test does NOT check for "no I/O" generally (e.g. it wouldn't catch a hypothetical `fetch()` or `fs.readFileSync()` call accidentally added to `engine.ts`) or "no floats" in the broader arithmetic sense (it only catches the specific `parseFloat` API call, not e.g. a stray `3.14` floating-point literal or a `/` division that produces a non-integer result at runtime) — it's a narrow, targeted guard against exactly 3 specific API surface violations, not a comprehensive purity verifier. The BROADER purity guarantees (integer-only arithmetic via `fixed.ts`'s helpers, no other I/O) are enforced by convention/code-review and the OTHER test files' behavioral assertions (e.g. `determinism.test.ts` would likely fail if `Math.random` were somehow used despite evading this textual scan, since outputs would stop being seed-deterministic) rather than by this file specifically.
- A future agent adding a new file under `packages/sim/src/` should be aware this test will automatically include it in its scan (no manual registration needed — the recursive directory walk picks up any `.ts` file present at test-run time).

**Depends on** — `vitest`; Node's `fs` (`readFileSync`, `readdirSync`) and `path` (`join`) — this is the ONLY file in `packages/sim` (source or test) that performs direct filesystem I/O, but since it's a TEST file (not part of the shipped `src/` engine), it doesn't violate the "sim is pure" rule itself (the rule applies to the engine's own runtime code, not its test harness's tooling).

**Used by** — Run as part of `npm test`; not imported elsewhere.

**Notes** — This file is conceptually similar in spirit to the theme.ts hex-literal forbidding test mentioned in CLAUDE.md for the client package ("theme test forbids `0x` outside theme.ts") — both are lightweight regex/text-based guards enforcing an architectural rule that would otherwise rely purely on developer discipline/code review. Given the narrow regex-based approach, a determined or careless future change COULD still technically violate the spirit of "no I/O / no non-determinism" while evading this specific test (e.g. via `globalThis.Math["random"]`) — this test is a useful tripwire for the common case, not an absolute guarantee.
