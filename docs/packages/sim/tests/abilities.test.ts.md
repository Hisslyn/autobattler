# packages/sim/tests/abilities.test.ts

**Path & purpose** — `packages/sim/tests/abilities.test.ts`. Behavioral tests of `engine.ts`'s ability/status/item-passive resolution — `shield`, `burn` DoT, `buff`, `stealth`, and the two item passive kinds (`shield` start-of-combat, `burn` on-hit) — each isolated via carefully hand-built unit instances designed to eliminate confounding interactions.

**Responsibility** — Verifies each supported `AbilityEffect`/passive kind produces EXACTLY the numeric/event behavior described in `engine.ts`'s logic and CLAUDE.md's ability list, using minimal hand-crafted 1v1 boards rather than realistic full-board combat (so each mechanic can be isolated and asserted exactly).

**Key behavior — test helpers**
- `mk(uid, team, q, r, overrides)`: builds a full `UnitInstance` with sensible defaults (`defId: "warrior"`, `hp/maxHp: 1000`, `ad: 50`, `as: 700`, `armor/mr: 0`, `range: 1`, `mana: 0`, `maxMana: 100`, `abilityDamage: 100`) at the given hex position, then spreads `overrides` on top — every test in this file calls this with task-specific overrides (e.g. enormous `hp` for a "dummy" unit that should never die, custom `ability` defs, pre-seeded `statusEffects`/`items`).
- `board(...units)`: wraps a units list into a `BoardState`.
- `hpEvents(events, uid)`: filters the event log down to just `{type:"hp", uid}` events for one specific unit — the primary tool every test uses to inspect a unit's hp trajectory tick-by-tick.

**Key behavior — test by test**
1. **"a cast shield fully absorbs damage until it expires"** (shield ability): a defender with an absurdly high `mana`/`maxMana` (both `10000`, so it starts AT full mana and casts on tick 0) and a `Barrier` ability (`shield amount: 100000, duration: 40`) faces an attacker with enormous `hp` (so it never dies and keeps attacking). Asserts: a self-targeted `cast` event with `dmg: 0` exists for the defender; EVERY `hp` event for the defender BEFORE tick 35 reads exactly `1000` (full hp — the absurdly large shield absorbs everything, so the defender's `hp` literally never moves while the shield is active, even though it's being hit repeatedly — only the `shield` POOL is being drawn down, invisibly to the `hp` event stream); and the VERY LAST `hp` event overall is LESS than `1000` (confirming that once the 40-tick shield duration expires — zeroing `unit.shield` per `engine.ts`'s status-effect-expiry logic — the next hit finally lands on real hp).
2. **"a pre-seeded burn deals exactly value per tick for its duration"** (burn DoT in isolation): seeds BOTH units with a permanent-ish `stealth` ability (`duration: 100000`) so NEITHER can ever be targeted by the other — eliminating attacks/casts entirely, isolating the burn status effect's pure tick-by-tick behavior. The "burned" unit starts with a hand-placed `{type:"burn", value:50, duration:10}` status effect already in its `statusEffects` array (bypassing the normal apply-burn-via-cast/on-hit path entirely — testing the STATUS TICK mechanic in isolation). Asserts: EXACTLY 10 `hp` events occur within `tick <= 10` (one per tick of the burn's duration); the LAST one reads `5000 - 500` (10 ticks × 50 damage = 500 total); EVERY one of the 10 events matches the exact expected linear decay `5000 - 50*(i+1)`; and NO further hp events occur between tick 10 and tick 20 (confirming the burn fully expired and nothing else is damaging this unit, since both are mutually un-targetable).
3. **"a cast buff raises the unit's AD by exactly its value"** (buff ability): a caster with `ad: 60`, `mana: 0`/`maxMana: 40`, and an `Empower` ability (`buff stat:"ad" value:100 duration:200`) faces a zero-AD "dummy" with enormous hp (so it survives the whole test and never threatens the caster). Finds the `cast` event's tick, then finds the caster's `attack` events EXCLUDING crits (`crit === false`, to avoid the crit-multiplier confound) both BEFORE and AFTER the cast tick. Asserts: pre-cast clean hits deal `60` damage (`mitigate(60, armor=0)` against a 0-armor target — with 0 resist, mitigation is a no-op, so raw `ad` passes through unchanged); post-cast clean hits deal `160` (`mitigate(60+100, 0)` — the EXACT buffed AD, confirming the buff's `addStat` grant applies the full `value` additively with no partial/scaled effect).
4. **"a stealthed unit cannot be targeted until its stealth ends"**: a stealthed unit (`Vanish`, `duration: 30`) with high `ad` faces a high-hp attacker. Asserts: EVERY `attack`/`cast` event whose `targetUid === 0` (i.e. every hit landed ON the stealthed unit) occurs at `tick >= 30` (confirming the attacker genuinely cannot target the stealthed unit before its stealth window elapses — `getEnemies`'s `untargetableUntil` filter is doing its job). ALSO asserts the stealthed unit ITSELF still attacks the (fully targetable) enemy during the stealth window (`tick < 30`) — confirming stealth makes a unit UNTARGETABLE, not also unable to act itself; it can freely attack out from stealth.
5. **"a start-of-combat shield passive absorbs the first hit"** (item passive — shield): looks up a REAL item from `gameData.items` whose `passive.kind === "shield"` (rather than hand-rolling a fake item def — exercising the actual content data), equips it on a defender, faces a high-hp attacker with small `ad`. Reads the defender's POST-setup `maxHp` from the `init` event snapshot (capturing whatever the item's stat bundle bumped it to, if anything) and asserts the FIRST `hp` event for the defender equals that `maxHp` EXACTLY — i.e. even the very first hit (worst case including a crit, per the comment "First hit (<= 15 with crit) is well under the shield value") gets FULLY absorbed by the item's start-of-combat shield, with zero hp loss showing up in the event stream.
6. **"an on-hit burn passive applies a DoT to the struck target"** (item passive — burn): looks up a real item with `passive.kind === "burn"`, equips it on a SLOW attacker (`as: 50`, deliberately low attack speed "Attacker hits once then idles (very slow)" per the comment, isolating individual attack-then-burn-tick sequences rather than overlapping rapid hits), and reads the target's hp event stream. Scans CONSECUTIVE pairs of hp readings and checks whether ANY consecutive drop equals EXACTLY the item's burn-per-tick value (`burnVal`, read directly off the real item's `passive.value`) — proving at least one observed hp drop was a PURE burn tick (no attack damage mixed in that same reading), confirming the on-hit burn DoT genuinely applies and ticks independently of the triggering attack itself.

**Invariants & constraints**
- Every test in this file uses EXTREME stat values (enormous hp on one side, near-instant full mana via huge starting `mana`/`maxMana`, near-permanent stealth/shield durations) specifically to ISOLATE one mechanic at a time and avoid combinatorial interference from other systems (death, multiple overlapping status effects, mana regeneration timing) muddying the assertion — this is a deliberate, recurring test-construction pattern worth recognizing when reading or extending this file.
- Tests 5 and 6 deliberately use REAL content data (`gameData.items.find(...)`) rather than synthetic item defs — meaning these two tests' pass/fail status depends on at least one item in `items.json` actually having a `shield`/`burn` passive (if `items.json` were ever changed to remove all such passives, these tests would fail with a `.find()` returning `undefined` and a subsequent crash on `!.id`/`!.kind`, not a graceful skip) — a content-coupling point worth knowing about.
- These tests do NOT hand-construct an `ItemDef` with a `pairPassive` field anywhere — meaning the `pairPassive` mechanic (gated co-equip passive bonus) flagged as having a type-definition gap when documenting `types.ts` is NOT exercised by this file at all; a future agent verifying `pairPassive`'s correctness would need to look elsewhere (or write a new test) since this file's coverage stops at the BASE per-item passive only.

**Depends on** — `vitest`; `../src/engine.js` (`simulateCombat`, the unit under test); `@autobattler/data` (`gameData`, for real item-passive lookups); `../src/types.js` (`AbilityDef`, `BoardState`, `StatusEffect`, `UnitInstance` types).

**Used by** — Run as part of `npm test`; not imported elsewhere.

**Notes** — This file complements `purity.test.ts`/`determinism.test.ts` (which presumably check broader invariants) by going deep on EACH individual ability/passive's exact numeric behavior — together with `pairPassive.test.ts` (the next file in manifest order, likely covering the co-equip gating this file skips) these form the complete behavioral specification for everything `engine.ts`'s ability/passive system can do.
