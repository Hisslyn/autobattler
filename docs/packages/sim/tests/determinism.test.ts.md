# packages/sim/tests/determinism.test.ts

**Path & purpose** — `packages/sim/tests/determinism.test.ts`. The core determinism-guarantee test suite for `engine.ts` — proves repeated runs with the same seed are byte-identical, different seeds diverge, the overtime ramping-damage failsafe resolves stalemates correctly, and that trait breakpoints count UNIQUE unit defIds rather than raw copies.

**Responsibility** — Directly enforces CLAUDE.md's "Every change must keep determinism tests green" hard invariant — this IS that test (or one of the primary ones). Also locks in the overtime/hard-cap tie-breaking behavior and the unique-defId trait-counting rule, two subtle behavioral details that would be easy to regress silently.

**Key behavior — test helpers**
- `makeBoard(team)`: builds a 3-unit board from the FIRST 3 entries of `gameData.units` (real content data, not synthetic units), placed along one edge row (`r=0` for team 0, `r=7` for team 1), each at base stats (1-star, no items, `manaStart`/`mana` from the def).
- `makeTankBoard(team, hp)` (inside the "overtime" describe): builds 1 or 2 copies of `gameData.units[0]` with deliberately crippled `ad: 1`/`armor: 0` (the comment explains: "Unkillable by normal damage within 1200 ticks (ad 1, armor 0 → 1 dmg/hit)") and a caller-supplied huge `hp` — engineered so normal combat damage alone could never finish the fight before the hard cap, forcing the overtime mechanic to be the actual deciding factor.
- `knightUnit(uid, defId, team)` (inside the "trait breakpoints" describe): a generic unit builder parameterized by `defId`, used to construct specific named units (`"warrior"`, `"paladin"`, `"archer"`) for the knight-trait tests.
- `firstAttackDamageBy(boardA)`: places a SINGLE fixed ranged attacker (`"archer"`, team 1, positioned at `{0,1}`) against `boardA`, runs combat, and returns the damage of the first NON-CRIT `attack` event targeting unit 100 (the attacker) — wait, re-reading: actually `e.uid === 100` filters for events WHERE THE ATTACKER IS uid 100 (the archer) — so this measures damage the FIXED ARCHER deals TO one of `boardA`'s knight units, which (since the archer's own stats are fixed/known) directly reveals whether the KNIGHT side's armor buff is active (more armor → less damage taken → lower measured `dmg`). The function's own comment clariffies this exact reasoning: "we instead measure damage TAKEN by a knight from a fixed attacker, which reflects whether the armor buff activated."

**Key behavior — test by test**
1. **"100 runs produce identical JSON output"**: runs `simulateCombat` with the SAME boards+seed 100 times, JSON-stringifies each FULL `CombatResult` (not just events — winner, ticks, survivingUnits too), and asserts every single one of the 100 runs produces a STRING-IDENTICAL result to the first — the most direct, brute-force determinism proof in the whole test suite.
2. **"event log hash is stable for fixed seed"**: hashes just the `events` array (sha256 of its JSON) for one specific seed (`12345`) and asserts the hash `toMatchSnapshot()` — a Vitest SNAPSHOT test, meaning the EXPECTED hash value is stored in a separate `__snapshots__` file (not inline in this test) and compared against on every run; if `engine.ts`'s behavior for this exact board/seed combination ever changes (even a tiny, seemingly-unrelated tweak), this snapshot would FAIL and require explicit `--update` (re-approval) — making this test a tripwire for ANY combat-output-affecting change, not just deliberate determinism regressions. (The actual snapshot file itself was not read as part of this documentation pass; it lives alongside this test under a `__snapshots__` directory per Vitest convention.)
3. **"different seeds produce different event-log hashes; same seed identical"**: the comment explains the test's design intent: "Melee-heavy boards trade many attacks, so the 15% crit roll is guaranteed to fire (and differ) across seeds" — i.e. this specific board composition (the first 3 `gameData.units`) is chosen because it produces enough attacks for crit-roll variance to actually matter (a board with zero attacks ever happening couldn't demonstrate seed-sensitivity via this mechanism). Hashes seed `1` and seed `2`'s event logs and asserts they DIFFER; independently confirms at least one CRIT actually occurred for seed 1 (proving the PRNG had a real opportunity to influence the outcome, not just an untested theoretical difference); then re-hashes BOTH seeds again and asserts each matches its FIRST hash (re-confirming same-seed-repeatability as a secondary check alongside the cross-seed-difference check, in the same test).
4. **"unkillable boards enter overtime and resolve before the hard cap"** (overtime describe): pits a 2-unit "tank" team (120,000 hp each... actually per-unit hp = the passed value, so 2 units × 120,000 hp = effectively more total hp) against a 1-unit tank team (100,000 hp) — comment: "Team 0 units have more per-unit HP, so the ramp kills team 1 first." Asserts: EXACTLY ONE `overtime_start` event fires (confirming the one-time-only emission gate in `engine.ts`); it fires at EXACTLY `tick: 1200` (the literal `overtimeStartTick` constant from `gameData.gameplay`, confirming this test is pinned to the CURRENT value of that tuning constant — a coupling point if it's ever retuned); the final `result.ticks` is GREATER than 1200 (combat continued past the overtime threshold) but LESS than `gameData.economy.overtimeHardCapTicks` (resolved before hitting the absolute ceiling — the ramping damage successfully finished the fight); and `winner === 0` (team 0, the higher-total-hp side, wins — confirming the ramping damage applied EQUALLY to both teams' UNITS, so the side with more aggregate hp simply survives longer and wins, exactly matching the "ramping true damage" overtime mechanic description).
5. **"hard cap decides by total remaining HP, never an endless draw"**: uses astronomically large hp (10 million each side) specifically so even the RAMPING overtime damage can't kill anyone before the absolute `overtimeHardCapTicks` ceiling is reached. Asserts `result.ticks === overtimeHardCapTicks` EXACTLY (confirming the simulation genuinely runs to the hard cap rather than looping forever or erroring) and that the winner is decided by TOTAL REMAINING HP across survivors — team 0 (2 units) beats team 1 (1 unit) since their AGGREGATE remaining hp is higher even though both sides are still "alive" at the cutoff — this is the concrete proof of `engine.ts`'s tie-break-by-total-hp fallback, and crucially demonstrates the engine NEVER returns an unresolved/looping state — there's always a winner (or an exact-tie draw) once the hard cap is hit.
6. **"2 copies of one knight do NOT activate knight(2)"** (trait breakpoints describe): builds a board of 2 `"warrior"` units (same defId, same trait set) and measures the archer's damage against them — asserts the damage equals the UNBUFFED mitigation value (`55 - trunc(55*40/140) = 40`, the comment's own worked-out arithmetic using warrior's 40 armor) — i.e. having 2 COPIES of the identical unit defId does NOT trigger the `knight` trait's 2-count breakpoint, directly proving `applyTraits`'s unique-defId counting (`uniqueDefIds.add(u.defId)` — a `Set`, so duplicate defIds collapse to 1) rather than counting raw unit instances.
7. **"2 distinct knights DO activate knight(2)"**: same setup but with one `"warrior"` and one `"paladin"` (two DIFFERENT defIds, presumably both carrying the `knight` trait) — asserts the measured damage is NOW less than the unbuffed `40` value, confirming the `knight(2)` breakpoint's `+200` armor bonus (per CLAUDE.md's "knight keeps its armor curve (+200/+500/+800)") DID activate this time, since 2 genuinely distinct knight-trait units are present.

**Invariants & constraints**
- This file is THE canonical regression guard for `packages/sim`'s "deterministic given the same seed" hard invariant — any future change to `engine.ts`'s logic that introduces even subtle non-determinism (e.g. relying on object key iteration order where it isn't guaranteed, or accidentally introducing a `Math.random()` call) would very likely be caught by test #1 or #2 here.
- Test #2's reliance on a Vitest snapshot means this test's pass/fail behavior is NOT fully self-contained within this file — a reader/maintainer must also be aware of the corresponding `__snapshots__/determinism.test.ts.snap` file (not read in this documentation pass) as the actual source of the expected value being compared against.
- Tests #4/#5 pin the EXACT current values of `overtimeStartTick` (1200, hardcoded as a literal in the test) and rely on `overtimeHardCapTicks`/`overtimeBaseDamage`/`overtimeRampPerTick` (read live from `gameData.economy`, not hardcoded) — meaning a tuning change to `overtimeStartTick` specifically (unlike the other overtime constants) WOULD require updating this test's literal `1200` assertion, while a change to the OTHER overtime constants would automatically be picked up since those are read live from `gameData`.
- The unique-defId trait counting behavior (tests #6/#7) is one of the more subtle, easy-to-regress rules in the whole engine — a future agent modifying `applyTraits` should treat these two tests as the definitive spec for "what counts toward a trait breakpoint."

**Depends on** — `vitest`; `node:crypto` (`createHash`, aliased import style `import { createHash } from "crypto"` — note this uses the BARE `"crypto"` specifier rather than the `node:crypto` prefix used elsewhere in the codebase, a minor stylistic inconsistency, though both resolve identically in Node); `../src/engine.js` (`simulateCombat`); `@autobattler/data` (`gameData`, for real unit defs + economy/gameplay constants); `../src/types.js` (`BoardState` type).

**Used by** — Run as part of `npm test`; not imported elsewhere.

**Notes** — This file's `firstAttackDamageBy` doc comment in the source is slightly confusing on first read (it says "team 0 knights are the ones being measured via their armor" then immediately clarifies the ACTUAL mechanism is measuring damage taken FROM a fixed attacker) — worth being precise about this when reasoning about the test, as initially documented above: the measured quantity is damage the archer deals TO whichever knight unit it attacks, which is a proxy for "how much armor does the target effectively have," which is the actual thing being verified.
