# packages/sim/src/hex.ts

**Path & purpose** — `packages/sim/src/hex.ts`. The axial hex-grid geometry module — defines the 7×8 board's coordinate system, distance metric, neighbor enumeration, and an A* pathfinder, all used by the combat engine for targeting/movement.

**Responsibility** — Owns ALL hex-grid math: coordinate representation, in-bounds checking, distance calculation, neighbor adjacency, and shortest-path pathfinding around obstacles. No combat logic lives here — purely geometric/graph primitives.

**Exports**
- `interface HexCoord { q: number; r: number }` — axial hex coordinates (the two-axis system; no redundant third `s` axis stored, though it's implicitly `-q-r` if ever needed).
- `COLS = 7`, `ROWS = 8` — the board's fixed dimensions (per CLAUDE.md: "axial hex grid 7×8"), used both for bounds-checking and for `engine.ts`'s own local `hexKey` position-encoding (`r * COLS + q`).
- `hexDistance(a: HexCoord, b: HexCoord): number` — axial hex distance: `(|dq| + |dr| + |dq+dr|) / 2` (using `>> 1` for the integer division, equivalent to `Math.floor` for the always-non-negative sum here) — the standard axial-coordinate hex distance formula, returning the number of hex steps between two cells.
- `hexNeighbors(h: HexCoord): HexCoord[]` — returns the (up to 6) adjacent hexes to `h`, using the 6 standard axial direction vectors (`{1,0},{-1,0},{0,1},{0,-1},{1,-1},{-1,1}`), FILTERED to only those that satisfy `inBounds` — a hex on the board's edge/corner will return FEWER than 6 neighbors (never includes off-board coordinates).
- `inBounds(h: HexCoord): boolean` — true iff `0 <= q < COLS` AND `0 <= r < ROWS`.
- `hexAstar(start: HexCoord, goal: HexCoord, blocked: Set<number>): HexCoord[]` — standard A* shortest-path search from `start` to `goal`, treating every step between adjacent hexes as cost 1 (no terrain cost variation). `blocked` is a Set of `hexKey`-encoded positions (using THIS file's own internal `hexKey = r*COLS+q`, matching `engine.ts`'s separately-defined-but-identical local helper of the same formula) that are NOT traversable — EXCEPT the `goal` hex itself is always allowed even if present in `blocked` (`if (blocked.has(nbKey) && nbKey !== goalKey) continue` — meaning you can always path ONTO an occupied goal cell, e.g. to stand adjacent to/attack a unit occupying it, even though you couldn't path THROUGH an occupied cell elsewhere). Returns the path as an array of `HexCoord` EXCLUDING the start cell (so `path[0]` is the first step to take, matching how `engine.ts`'s movement logic consumes it: `unit.pos = path[0]`). Returns an EMPTY array if `start === goal` (same encoded key) OR if no path exists (open set exhausts without reaching the goal).

**Key behavior**
- `hexAstar`'s open-set selection is a NAIVE LINEAR SCAN for the lowest `fScore` (`for (const k of open) { if (f < bestF) ... }`) rather than a priority queue/heap — fine for this board's small size (7×8=56 cells max) but NOT an asymptotically optimal A* implementation; for THIS use case (tiny fixed grid, called once per unit per tick when out of range) this is an intentional simplicity-over-asymptotic-efficiency tradeoff, not a bug.
- The heuristic used for `fScore` is `hexDistance` itself (the EXACT, not just admissible/approximate, hex distance) — since hex distance with cost-1 steps is itself a perfectly admissible AND consistent heuristic (it never overestimates the true remaining cost, since every step can reduce hex distance by at most 1), this A* will always find a TRUE shortest path, behaving effectively identically to a BFS/Dijkstra in this specific metric space, just with the heuristic helping it explore more directly toward the goal first.
- Movement cost between any two adjacent hexes is uniformly 1 (`tentativeG = gScore[current] + 1`) — no varying terrain costs anywhere in this grid (the board has no concept of slow/fast terrain).
- `keyToCoord` map seeds the `start`/`goal` coordinates up front and is incrementally populated as neighbors are discovered — used to reconstruct the final coordinate-level path from the integer-keyed `cameFrom` backtrack chain.

**Invariants & constraints**
- **Pure**: no I/O, no randomness, no mutation of its inputs (`blocked` is read-only; `start`/`goal` are never mutated) — fully deterministic given the same inputs, consistent with the rest of `packages/sim`'s purity requirement.
- The `hexKey` encoding (`r * COLS + q`) used internally by THIS file for the A* open/closed-set bookkeeping is DUPLICATED (not imported/shared) in `engine.ts` as its own private top-level `hexKey` function with the IDENTICAL formula — this is intentional file-local encapsulation (this file doesn't export its `hexKey`, so `engine.ts` necessarily defines its own copy to build the `blocked` Set it passes in) rather than an inconsistency; both formulas MUST stay in agreement for `hexAstar`'s blocked-set lookups to correctly match the keys `engine.ts` constructs, so a future refactor changing one must change the other identically (a coupling risk worth flagging, though currently consistent).
- `hexNeighbors`'s 6 direction vectors define the AXIAL adjacency for a FLAT-TOP-rendered hex grid (matching the client's flat-top tile rendering per CLAUDE.md's `unitToken.ts` note: "hex board renders flat-top tiles") — the specific 6 vectors chosen (`{1,0},{-1,0},{0,1},{0,-1},{1,-1},{-1,1}`) are the standard axial-coordinate neighbor set; a reader reasoning about board topology should treat these AS the canonical adjacency, not re-derive it differently.

**Depends on** — Nothing (no imports) — entirely self-contained geometry/graph code.

**Used by** — `engine.ts` (`hexDistance` for targeting + range checks, `hexAstar` for movement pathing, `COLS` for its own local `hexKey`); presumably `packages/rules`' board-placement/validation logic (bounds-checking a unit's placement) and `packages/client`'s board-rendering math (hex-to-screen-space conversion, though the client likely has its OWN screen-space layout math separate from this pure axial-coordinate module) — exact cross-package usage to be confirmed when those files are documented.

**Notes** — `HexCoord` deliberately has NO third cube-coordinate field (`s = -q-r`) stored — it's always derivable on demand if ever needed, keeping the type minimal; nothing in this file currently needs it.
