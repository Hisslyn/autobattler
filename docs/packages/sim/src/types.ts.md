# packages/sim/src/types.ts

**Path & purpose** — `packages/sim/src/types.ts`. The complete type-only definitions for everything the combat engine consumes and produces: content-shape types (`UnitDef`, `TraitDef`, `ItemDef`, `AbilityDef`/`AbilityEffect`), runtime instance types (`UnitInstance`, `StatusEffect`, `BoardState`), and the combat-output types (`CombatEvent`, `CombatResult`, `InitUnitSnapshot`).

**Responsibility** — Owns the canonical shape of every data structure that flows into or out of `engine.ts`. This is the single place to look up exactly what fields a `UnitInstance` carries, what an ability effect can be, or what a `CombatEvent` variant's payload looks like — no other file in `packages/sim` redefines these shapes.

**Exports**
- `type AbilityEffect` — a discriminated union (tag: `kind`) of every supported ability behavior:
  - `{kind: "magic_damage"}` — single-target nuke, no extra fields (damage amount comes from the caster's `abilityDamage` stat, not from the effect itself).
  - `{kind: "burn"; burn: number; duration: number}` — magic damage PLUS a DoT: `burn` is the per-tick DoT damage value, `duration` is how many ticks it lasts.
  - `{kind: "shield"; amount: number; duration: number}` — self-targeted absorb shield: `amount` is the absorb pool size, `duration` is how many ticks before it expires (and zeroes any remaining absorb).
  - `{kind: "buff"; stat: "ad"|"as"|"armor"|"mr"|"abilityDamage"; value: number; duration: number}` — self-targeted temporary stat boost; note the `stat` field is TYPE-RESTRICTED to exactly these 5 named stats (NOT an open string — unlike `TraitDef`'s/`ItemDef`'s effect/stats fields which use a plain `string` key, `AbilityEffect`'s buff variant is more tightly typed at the TYPE level, though `engine.ts`'s `addStat` helper itself still does an unchecked dynamic write regardless of this narrower type).
  - `{kind: "stealth"; duration: number}` — start-of-combat-only untargetability window; NEVER triggers via the normal mana-cast dispatch (see `engine.ts`'s explicit `castKind !== "stealth"` exclusion).
- `interface AbilityDef { name: string; manaCost: number; effect: AbilityEffect }` — a unit's full ability definition; `manaCost` corresponds 1:1 to the unit's `mana` (max mana) stat per CLAUDE.md ("ability {name, manaCost=mana, effect}").
- `interface UnitDef { id, name, tier, hp, ad, as, armor, mr, range, mana, manaStart, abilityDamage, traits: string[], ability?: AbilityDef }` — the CONTENT-LAYER unit definition (as loaded from `units.json`/`mobs.json` via `@autobattler/data`) — base stats at 1-star, no items, no live combat state. `traits` is the flattened list of origin+class trait ids (CLAUDE.md: "each has one origin + 1-2 classes (flattened into traits)"). `ability` is OPTIONAL — a unit def without one simply never gets a real ability (though `engine.ts`'s cast dispatch would still default an ability-less unit to `"magic_damage"` if it ever reached full mana, per that file's `castKind = unit.ability?.effect.kind ?? "magic_damage"` fallback — though in practice no real unit def should be missing an ability if CLAUDE.md's "every unit) has...an ability" content guarantee holds).
- `interface TraitDef { id, name, breakpoints: Array<{count, effect: {stat, value}}> }` — a trait's full breakpoint table (e.g. 2/4/6 unit-count thresholds, each with its own stat bonus); `effect.stat` is a PLAIN STRING here (not type-restricted to a closed set, unlike `AbilityEffect`'s buff variant) — meaning the type system alone won't catch a typo'd stat name in trait content data.
- `type ItemPassive = {kind:"burn";value;duration} | {kind:"shield";value;duration}` — the two passive behaviors a COMPLETED item can carry (per CLAUDE.md: "at most one passive (burn on-hit or shield start-of-combat)").
- `interface ItemDef { id, name, stats: Partial<Record<string,number>>, component?: boolean, recipe?: [string,string], passive?: ItemPassive }` — note this file's `ItemDef` does NOT include a `pairPassive` field, even though `engine.ts`'s `applyStartOfCombat` function actively reads `itemDef.pairPassive` (`const pairPassive = itemDef.pairPassive`) — **this is a real type-vs-usage gap**: either `ItemDef` here is STALE relative to what `engine.ts` actually expects on an item def at runtime (meaning the REAL item-def shape, presumably defined/widened in `@autobattler/data`'s own types and passed through as `GameData`'s `items` array, carries an additional `pairPassive` field this LOCAL `ItemDef` type doesn't declare), or `engine.ts`'s `data.items` array element type doesn't actually use THIS interface at type-check time (TypeScript's structural typing would only flag this if `GameData.items`'s declared element type were strictly THIS `ItemDef` — if `@autobattler/data` defines a separately-widened item type for `GameData`, no type error would surface even though this file's `ItemDef` looks incomplete by comparison). A reader implementing item-passive-related logic should treat `@autobattral/data`'s OWN item type (to be confirmed when that package is documented) as authoritative over this local `ItemDef`, since this file's copy is evidently not the one actually constraining `engine.ts`'s `data.items` parameter.
- `interface UnitInstance { uid, defId, tier, star: 1|2|3, team: 0|1, pos: HexCoord, hp, maxHp, ad, as, armor, mr, range, mana, maxMana, abilityDamage, attackCooldown, statusEffects: StatusEffect[], items: string[], ability?, shield?, untargetableUntil?, onHitBurn? }` — the LIVE, mutable, per-combat unit state: a snapshot derived from a `UnitDef` (post tier/star resolution) PLUS live combat fields (`pos`, `attackCooldown`, `statusEffects`, `shield`, `untargetableUntil`, `onHitBurn`) that only exist/matter during an active simulation. `star` and `team` are tightly typed to their valid literal sets (`1|2|3` and `0|1` respectively). `shield`/`untargetableUntil`/`onHitBurn` are all OPTIONAL fields populated only when relevant (no shield → field absent/undefined, not zero) — their doc comments inline explain their purpose: `shield` = "Remaining absorb pool from shields (ability cast or item passive)"; `untargetableUntil` = "Untargetable while combat tick < this value (start-of-combat stealth)"; `onHitBurn` = "On-hit burn applied by an item passive."
- `interface StatusEffect { type: string, duration: number, value: number, stat?: string }` — a generic status-effect record; `type` is a plain string (covers `"burn"`/`"buff"`/`"shield"` as observed in `engine.ts`, though NOT a closed union at the type level); `stat` is documented inline as "For 'buff' effects: the stat to revert on expiry" — only meaningful when `type === "buff"`.
- `interface BoardState { units: UnitInstance[] }` — the entire input shape `simulateCombat` expects for each side — just a flat list of unit instances (their `team`/`pos` fields already encode which side and where, rather than the board being organized as some 2D grid structure at this type level).
- `interface InitUnitSnapshot { uid, side: 0|1, defId, star, hex: HexCoord, hp, maxHp, mana, maxMana, items: string[] }` — explicitly documented inline as "Per-unit snapshot at combat start, after star/item/trait application" — this is exactly the per-unit payload shape carried in the `"init"` `CombatEvent` variant.
- `type CombatEvent` — the discriminated union (tag: `type`) of every event the engine can emit: `init` (full per-unit snapshot array), `move` (`uid, from, to`), `attack` (`uid, targetUid, dmg, crit`), `cast` (`uid, targetUid, dmg` — no crit field, abilities never crit), `mana` (`uid, value` — absolute post-change value), `hp` (`uid, value` — absolute post-change value, clamped non-negative per `emitHp`'s `Math.max(0, ...)`), `death` (`uid`), `overtime_start` (no extra fields), `end` (`winnerSide: 0|1|"draw", survivingUids: number[]`). Its own inline doc comment states: "The event log fully describes combat: a consumer can reconstruct positions/hp/mana/alive per uid at any tick by folding events in order, without re-running game logic. 'hp' and 'mana' carry absolute values (post-change, hp clamped at 0) and are emitted only on change." — this is the EXACT property the client's pure `reducer.ts` (per CLAUDE.md) relies on, and what `room.test.ts`'s resimulation/hash comparison validates server-side.
- `type CombatEventType = CombatEvent["type"]` — a convenience alias extracting just the union of `type` tag string literals (`"init"|"move"|"attack"|...`), useful for code that needs to reference/compare event-type strings without the full event shape.
- **Trace types (opt-in combat instrumentation, behavior-neutral)** — added for the dev trace harness + QA invariant suite (see `docs/combat-testing.md`); none of these affect default combat output:
  - `type RetargetReason` = `"acquired_no_target" | "switched_target_dead" | "switched_target_untargetable" | "switched_target_out_of_range" | "switched_forced" | "retarget_recomputed"` — why a unit's resolved target changed between ticks. A CHARACTERIZATION of the current stateless nearest-recompute targeting (so `retarget_recomputed` off a still-valid target is EXPECTED). `switched_forced` is reserved for a taunt-style forcing effect the engine doesn't have, so it is never emitted (kept for spec completeness).
  - `interface TraceUnitRecord { uid; side: 0|1; defId; hex: HexCoord; hp; mana; targetUid: number|null; action: "move"|"attack"|"cast"|"idle"; damageDealt: number }` — one alive unit's end-of-tick state for the trace; `action` is a single label by precedence cast > attack > move > idle; `damageDealt` is post-mitigation/post-shield attack+cast damage this unit dealt this tick.
  - `interface TraceRetarget { uid; fromUid: number|null; toUid: number|null; reason: RetargetReason }` — a single target change for one unit on one tick.
  - `interface TraceTick { tick: number; units: TraceUnitRecord[]; retargets: TraceRetarget[] }` — per-tick frame (every alive unit + any target changes, units in uid-ascending order).
  - `interface CombatTrace { ticks: TraceTick[] }` — the full opt-in trace (one frame per simulated tick).
- `interface CombatResult { winner: 0|1|"draw", ticks: number, survivingUnits: UnitInstance[], events: CombatEvent[], trace?: CombatTrace }` — the complete output of `simulateCombat`; `trace` is OPTIONAL and populated ONLY when called with `{ trace: true }` (absent on the default path, so existing consumers are unaffected): who won (or drew), how many ticks it took, the FINAL state of all surviving units (post-combat, including any unexpired buffs/shields — though as noted in `engine.ts`'s doc, downstream consumers mostly care about `tier`/`star` from these rather than live combat stats), and the full replayable event log.

**Key behavior** — None (this file is purely type declarations, no runtime code, no executable logic of any kind).

**Invariants & constraints**
- This file is the canonical shape reference for the combat engine's entire input/output surface — any change to `engine.ts`'s behavior that adds/removes a field from `UnitInstance` or a `CombatEvent` variant MUST be reflected here first (types-then-implementation, or at minimum kept in lockstep), since downstream consumers (the client's `reducer.ts`, the server's serialization logic, `room.test.ts`'s resimulation) all type against these exact shapes.
- The `ItemDef`/`pairPassive` discrepancy (see Exports above) is the one concrete inconsistency found in this file relative to actual `engine.ts` usage — flagged for a future agent's awareness rather than fixed here (documentation-only scope).
- `AbilityEffect`'s `kind` discriminated union is the EXHAUSTIVE list of ability behaviors the engine can execute, matching CLAUDE.md's "Abilities at full mana...magic_damage (single-target), burn (magic dmg + DoT), shield (self absorb), buff (self stat buff, reverted on expiry); stealth resolves at combat start" — any new ability behavior would require adding a new union member here AND a new dispatch branch in `engine.ts`.
- `HexCoord` (used by `UnitInstance.pos` and `InitUnitSnapshot.hex`) is imported from `./hex.js`, not redefined here — this file relies on `hex.ts`'s coordinate type rather than duplicating it.

**Depends on** — `./hex.js` (`HexCoord` type only).

**Used by** — `engine.ts` (every type in this file, as documented above); virtually every other `packages/sim` file that touches combat state; downstream consumers across `rules`/`server`/`client`/`balance` that import these types directly via the package's exported deep path `@autobattler/sim/src/types.js` (confirmed used by `room.test.ts`: `import type { UnitInstance, CombatResult } from "@autobattler/sim/src/types.js"`).

**Notes** — The flagged `ItemDef`/`pairPassive` gap is worth a dedicated look if a future agent is doing item-passive work: confirm whether `@autobattler/data`'s actual item type includes `pairPassive` (likely, since `engine.ts` reads it successfully at runtime against real content data) and whether this `sim`-local `ItemDef` is simply an intentionally-narrower/stale local copy that happens not to matter because `GameData.items`'s real element type comes from `@autobattler/data` rather than this file.
