# packages/sim/src/engine.ts

**Path & purpose** — `packages/sim/src/engine.ts`. THE pure deterministic combat engine — the single function (`simulateCombat`) that, given two boards, a seed, and game content data, runs an entire combat from start to finish and returns the outcome plus a fully-replayable event log.

**Responsibility** — Owns ALL combat resolution logic: stat application (star multipliers, items, traits), the per-tick simulation loop (status effects → mana/cast → movement → attacks → death cleanup), targeting, pathing, damage mitigation, ability resolution, and the overtime ramping-damage failsafe. This is the single source of truth for "what happens when two boards fight," used identically by both the authoritative server and any client-side resimulation.

**Exports**
- `simulateCombat(boardA: BoardState, boardB: BoardState, seed: number, data: GameData): CombatResult` — the ONLY public export. Runs a complete deterministic combat between `boardA` (team 0) and `boardB` (team 1) using `seed` to drive the internal `mulberry32` PRNG, and `data` (the loaded `@autobattler/data` content) for all unit/trait/item/ability/gameplay/economy constants. Returns a `CombatResult` (`{winner: 0|1|"draw", ticks, survivingUnits, events}`). Does not mutate its `boardA`/`boardB` inputs (both are deep-cloned via `cloneBoard` before any combat math runs).

**Key behavior**
- **Setup phase** (before any ticks run): clones both boards into one combined `allUnits` array (team 0 from `boardA`, team 1 from `boardB`); for EACH unit applies `applyStarMultiplier` (multiplies `hp`/`maxHp`/`ad`/`abilityDamage` by `data.gameplay.starMultipliers[String(unit.star)]`, a fixed-point multiplier keyed by star count as a STRING — e.g. `"2"` → 1800/1000=1.8x per CLAUDE.md, `"3"` → 3240/1000=3.24x; a missing/1-star entry defaults to `SCALE` i.e. no change, and the function short-circuits entirely if `mult === SCALE`) then `applyItems` (sums each equipped item's `stats` bundle additively onto the unit's own stats via `addStat`, which ALSO special-cases `stat === "hp"` to keep `maxHp` in sync with `hp` whenever hp itself is the stat being added). THEN calls `applyTraits` once across the WHOLE combined unit list (not per-unit) — `applyTraits` iterates EACH team separately, and for each `data.traits` entry counts UNIQUE `defId`s (not raw unit copies — 3 copies of the same unit count once) among that team's units that have the trait, finds the HIGHEST breakpoint whose `count` threshold is met-or-exceeded, and applies that single breakpoint's `{stat, value}` effect additively to every team member that carries the trait (lower breakpoints are NOT stacked — only the highest active one applies, consistent with how trait breakpoints work in CLAUDE.md's "12 traits...breakpoints at 2/4/6" description). FINALLY calls `applyStartOfCombat` per unit, which handles: item passive application (a `passive` on an item def — `shield` adds to `unit.shield` + refreshes a `shield` status-effect duration, `burn` sets `unit.onHitBurn`, ALWAYS applied if present), a `pairPassive` (an ADDITIONAL effect that activates ONLY if `unit.items` also includes `pairPassive.partnerId` — gated on co-equipping a specific partner item; the function's own doc comment explains: "The item's own base stat bundle...is never gated — base stats always apply regardless. A unit that equips only one item from a pair receives the base stats but NOT the pair passive. A missing partner id...silently skips the pairPassive; no error"), and start-of-combat `stealth` (sets `unit.untargetableUntil = ability.effect.duration` if the unit's ability effect kind is `"stealth"` — this is the ONLY ability kind resolved here rather than on cast, per CLAUDE.md).
- **The init event**: immediately after setup, pushes ONE `{type: "init", tick: 0, units: [...]}` event capturing every unit's POST-setup snapshot (`uid, side, defId, star, hex, hp, maxHp, mana, maxMana, items`) — this is the baseline a consumer folds all subsequent events onto to reconstruct full combat state at any point.
- **Fixed timestep / 30 Hz**: the sim runs a fixed timestep at the canonical `TICK_HZ` (30 ticks/s of game time, the single constant in `fixed.ts`). One tick = `1/30` s of game time on every device; the loop advances by whole integer ticks only and never reads wall-clock time/`Date`/FPS. All durations are AUTHORED IN SECONDS in `packages/data` (fixed-point, scale 1000) and converted to integer ticks via `secondsToTicks(secondsFixed) = trunc(secondsFixed * TICK_HZ / SCALE)`. So `overtimeStartTick = secondsToTicks(data.gameplay.overtimeStartSeconds)` (60000 → 1800) and `hardCap = secondsToTicks(data.economy.overtimeHardCapSeconds)` (90000 → 2700). The former raw fields `ticksPerSec`/`overtimeStartTick`/`overtimeHardCapTicks` no longer exist.
- **The main tick loop** (`while (tick < hardCap)`, `hardCap = secondsToTicks(data.economy.overtimeHardCapSeconds)`):
  - Breaks immediately if either team has zero alive units (checked BEFORE doing any per-tick work).
  - **Overtime check**: once `tick >= overtimeStartTick` (= `secondsToTicks(data.gameplay.overtimeStartSeconds)`), emits a ONE-TIME `overtime_start` event (on the first qualifying tick only, gated by the local `overtime` boolean) then applies a RAMPING TRUE-DAMAGE tick to EVERY alive unit on EVERY subsequent tick: `trueDmg = overtimeBaseDamage + overtimeRampPerTick * (tick - overtimeStartTick)` — this damage bypasses armor/mr entirely (subtracted straight from `hp`), emits an `hp` event per unit, and emits a `death` event for anyone who drops to/below 0 — if this true-damage tick alone wipes out an entire team, the loop breaks immediately (incrementing `tick` first) without running the rest of that tick's normal unit-action loop.
  - **Per-unit action order within a tick** (units processed in a DETERMINISTIC `uid`-ascending sort each tick, `getAlive().slice().sort((a,b) => a.uid - b.uid)` — NOT in original board order or any randomized order, guaranteeing pure determinism for a given seed):
    1. **Status effects tick**: each of the unit's `statusEffects` is processed — a `"burn"` status deals its stored `value` as TRUE damage (bypasses mitigation) to its OWN holder (note: burn is stored as a status effect on the BURNED unit, so this damages `unit` itself, not a target) and emits an `hp` event + (if lethal) a `death` event, setting a local `burnedToDeath` flag; EVERY status effect's `duration` is decremented regardless of type; effects with `duration > 0` survive into a fresh `survivors` array, while EXPIRING effects get type-specific cleanup — an expiring `"buff"` REVERTS its stat grant (`addStat(unit, se.stat, -se.value)`, i.e. buffs are NOT permanent, they're temporary boosts undone on expiry) and an expiring `"shield"` zeroes `unit.shield` entirely (any remaining absorb pool is lost, not banked). If the unit burned to death (or was already at ≤0 hp from a previous tick's effect, defensively checked), the rest of this unit's turn is skipped via `continue`.
    2. **Mana / ability cast**: finds enemies via `getEnemies` (alive units on the other team whose `untargetableUntil` has either not been set or has already elapsed — i.e. stealth-expired or never-stealthed units only); if no enemies remain, skips the unit's turn entirely (still alive, just nothing to act against). If `unit.mana >= unit.maxMana` AND the unit's ability isn't a `"stealth"` kind (stealth never triggers a cast — it already resolved at combat start), dispatches by `ability.effect.kind` (defaulting to `"magic_damage"` if the unit has no `ability` at all, meaning EVERY unit that reaches full mana attempts SOME cast, even a default single-target nuke if it has no real ability defined — though in practice every real unit def has an `ability`):
       - `"magic_damage"` / `"burn"`: finds the nearest valid target via `findTarget`, mitigates `unit.abilityDamage` against the target's `mr` via `mitigate`, routes the damage through `damageThroughShield` (absorbs from `target.shield` first, THEN spills into `hp`), emits a `cast` event (`{uid, targetUid, dmg}` — note `cast` events do NOT carry a `crit` flag, abilities never crit), emits the target's new `hp`, ZEROES the caster's mana (emitting a `mana` event only if it WAS non-zero, since events are change-gated), and if the cast kind is specifically `"burn"` AND the target survived, applies a fresh `applyBurn` DoT status effect to the TARGET (overwriting any pre-existing burn's value/duration rather than stacking); if the target died from the magic damage itself, emits `death` instead of granting `manaPerDamageTaken`; otherwise the target gains `manaPerDamageTaken` mana (clamped to its max via `gainMana`).
       - `"shield"`: adds `eff.amount` to the CASTER's OWN `shield` pool (additively, stacking with any pre-existing remaining shield) and refreshes the shield status's duration via `refreshStatus`, emits a `cast` event targeting itself (`targetUid: unit.uid, dmg: 0`), zeroes mana.
       - `"buff"`: adds `eff.value` to the caster's own `eff.stat` (one of `ad/as/armor/mr/abilityDamage`) via `addStat`, ALSO pushes a NEW `"buff"` status effect (carrying the SAME stat/value/duration) so the boost is correctly REVERTED when that status later expires (the status-effect tick logic above handles the revert), emits a self-targeted `cast` event, zeroes mana.
       - All three cast branches `continue` immediately after casting — a unit that casts on a given tick does NOT also move or attack that same tick.
    3. **Movement**: if no cast happened, re-finds the nearest target via `findTarget` (skips turn if none); if `hexDistance(unit.pos, target.pos) > unit.range`, computes the SET of currently-occupied hexes (every OTHER alive unit's position, encoded via `hexKey = r*COLS+q`) and calls `hexAstar(unit.pos, target.pos, occupied)` to get a path; if a path exists, moves the unit ONE STEP (`path[0]`) toward the target and emits a `move` event (`{from, to}`) — units move at most one hex per tick, never multiple, regardless of computed path length.
    4. **Attack**: re-measures `hexDistance` AFTER the (possible) move this tick; if now within `unit.range`: if `unit.attackCooldown <= 0`, performs an attack — rolls a crit via the SIM'S OWN SEEDED PRNG (`prng() % SCALE < data.economy.critChance`, i.e. a fixed-point percentage roll against `SCALE=1000`), computes raw AD (crit-multiplied via `fmul` if the roll succeeded), mitigates against `target.armor`, routes through the target's shield, emits an `attack` event (`{targetUid, dmg, crit}` — THIS event DOES carry the crit flag, distinguishing it from `cast`), emits the target's new hp, grants the ATTACKER `manaPerAttack` mana (gainMana, always — note this happens regardless of whether the target died, unlike the defender's `manaPerDamageTaken` grant which only happens if `target.hp > 0`), and if the target died emits `death`, else grants the target `manaPerDamageTaken` mana AND (if the attacker carries an `onHitBurn`, i.e. an equipped item with a `burn` passive) applies a fresh burn DoT to the target via `applyBurn`. Resets `attackCooldown` to `Math.trunc(TICK_HZ * SCALE / unit.as)` (fixed-point attack-speed-derived cooldown at the canonical 30 Hz rate — higher `as` means a SHORTER cooldown, in ticks). If `attackCooldown > 0` instead (still recovering), simply decrements it by 1 and does nothing else this tick — this ALSO applies in the "not yet in range" `else` branch (cooldown still ticks down even while out of range and just having moved, so attack-speed timing isn't reset by movement).
  - `tick++` at the end of every loop iteration (including overtime-only ticks).
- **Combat end / winner determination**: after the loop exits (either by early break or hitting `hardCap`), counts each team's final alive units; if exactly one team has survivors, that team wins; if BOTH teams are wiped simultaneously (can happen via the SAME overtime tick killing both sides' last units at once), the result is `"draw"`; if `hardCap` was reached with BOTH teams still having survivors (overtime damage wasn't enough, in principle — though in practice the ramping damage virtually guarantees a resolution well before `hardCap`), the winner is decided by comparing TOTAL REMAINING HP across each team's survivors (higher total wins; exactly equal HP totals → `"draw"`). Pushes a final `{type: "end", tick, winnerSide, survivingUids}` event, then returns `{winner, ticks: tick, survivingUnits: finalAlive.map(cloneUnit), events}` — `survivingUnits` is a FRESH deep clone of the final alive units (post-combat stat state, including any still-active buffs/shields that hadn't expired — though by combat end these don't matter for anything downstream since the units' tier+star are what `calcPlayerDamage`-style logic reads per CLAUDE.md, not their live combat stats).

**Invariants & constraints**
- **Pure and deterministic**: no `Math.random`, no `Date`, no I/O — the ONLY randomness source is the `mulberry32(seed)` PRNG instantiated once at the top of `simulateCombat`, consumed ONLY for crit rolls (`prng() % SCALE < critChance`). Given the same `boardA`/`boardB`/`seed`/`data`, this function ALWAYS produces byte-identical output — this is the property `room.test.ts`'s client-resimulation test directly exercises and verifies.
- **All arithmetic stays integer/fixed-point**: `fmul`/`fdiv` (from `fixed.ts`, `SCALE=1000`) are used for any multiplicative stat scaling (star multipliers, crit multiplier, attack-cooldown-from-attack-speed); raw addition (`addStat`) is plain integer arithmetic since trait/item/buff bonuses are already expressed in the same fixed-point units as the base stats they're added to. No floating-point division ever appears outside `Math.trunc(...)`-wrapped fixed-point helpers.
- **Inputs are never mutated**: `boardA`/`boardB` are deep-cloned (`cloneBoard`/`cloneUnit`, which clones `pos`, `statusEffects` array+contents, and `items` array) before any combat math runs — callers can safely reuse the SAME `BoardState` object across multiple `simulateCombat` calls (e.g. running several PvE matchups against the same mob board) without any cross-call contamination.
- **Event log fully describes combat** (per its own type-level doc comment in `types.ts`): a consumer (renderer or client resimulator) can reconstruct positions/hp/mana/alive-state at any tick purely by folding `events` in order — `hp`/`mana` events carry ABSOLUTE post-change values (not deltas) and are emitted ONLY when the value actually changes (change-gated, e.g. `gainMana` returns early if the computed `next` mana equals the current mana, suppressing a redundant event).
- **Targeting** is nearest-enemy with a deterministic uid-ascending tiebreak (`findTarget`'s `d === bestDist && e.uid < best.uid` comparison) — never random, never order-dependent on array iteration order beyond the explicit tiebreak.
- **Stealth is the one ability resolved OUTSIDE the normal cast dispatch** — it sets `untargetableUntil` once at combat start (`applyStartOfCombat`) and is explicitly EXCLUDED from the mana-cast dispatch's eligibility check (`castKind !== "stealth"`), so a stealth unit's mana can still fill up over time but will never actually "cast" mid-combat — its only effect is the one-time start-of-combat untargetability window.
- **PvE mobs reuse this same function unchanged** per CLAUDE.md — nothing in `simulateCombat` itself special-cases mobs; the engine has no concept of "is this a mob," it only sees `UnitInstance`s and a `GameData`. The mob-exclusion-from-trait-bonuses behavior described in CLAUDE.md happens NATURALLY here: `applyTraits` looks up each unit's `defId` in `data.units` (`data.units.find(d => d.id === u.defId)`) — a mob's `defId` is ABSENT from `data.units` (mobs live in `data.mobs` instead), so `def` is `undefined` and `def?.traits.includes(...)` short-circuits to `undefined`/falsy, meaning mob units are silently never counted toward (or granted) any trait breakpoint, on EITHER team.
- **Damage always routes through shield before hp** (`damageThroughShield`, used by both ability-cast magic damage and physical attacks) — shield is a flat absorb pool, decremented first, with any OVERFLOW (`remaining`) spilling into actual `hp` loss; the function's return value (`remaining`, the hp actually lost) is computed but NOT currently used by any caller in this file (both call sites ignore the return value) — a minor dead-code-adjacent observation, not a bug, since the side effect (mutating `target.hp`/`target.shield`) is what matters.
- **Mitigation formula floors at 1 damage minimum**: `mitigate` computes `max(1, rawDamage - trunc(rawDamage * resist / (mitigationBase + resist)))` — even an enormous resist stat can never reduce an attack/ability hit below 1 damage, guaranteeing combat always progresses (no permanent stalemate from over-stacked resistances).

**Depends on** — `./prng.js` (`mulberry32`, the seeded RNG — crit rolls only); `./fixed.js` (`fmul`, `SCALE`, `TICK_HZ` — the canonical 30 Hz rate, `secondsToTicks` — seconds→ticks conversion for the overtime thresholds); `./hex.js` (`hexDistance`, `hexAstar`, `COLS` — targeting distance + pathing + the local `hexKey` encoding); `./types.js` (`BoardState`, `CombatEvent`, `CombatResult`, `StatusEffect`, `UnitInstance` — all type-only); `@autobattler/data` (`GameData` type only, for the `data` parameter's shape — `data.gameplay`, `data.economy`, `data.traits`, `data.units`, `data.items` are all read).

**Used by** — `packages/rules` (`rounds.ts`'s `runCombatPhase`/`runPveRound` call this to resolve each pairing/PvE matchup server-side); `packages/server` (indirectly, via `rules`, and directly in `room.test.ts`'s resimulation helper which calls `simulateCombat` straight from a test); `packages/balance` (`runner.ts`'s `runMatchup` calls this N times per seed to gather win-rate statistics); `packages/client` (`netDriver.ts` resimulates locally from the received seed for rendering/reconciliation, per CLAUDE.md).

**Trace instrumentation (opt-in, behavior-neutral)** — `simulateCombat` now takes an OPTIONAL 5th param `options?: { trace?: boolean }` (exported as `SimulateOptions`). When `options.trace` is true it collects a `CombatTrace` and attaches it to the returned `CombatResult` as `result.trace`; otherwise the function does EXACTLY what it did before (no allocations on the default path, no behavior change — verified: same winner/ticks/events with and without the flag, and the default call has no `trace` field). The trace NEVER consumes the PRNG, never changes iteration order, and never mutates any unit field or the cloning logic — it only OBSERVES already-computed values inside the existing per-unit action loop and assembles one `TraceTick` per simulated tick (after the action loop, before `tick++`), iterating end-of-tick alive units in the engine's existing uid-ascending order. Per alive unit it records end-of-tick hex/hp/mana, the resolved target uid the engine actually used this tick (the `findTarget` result for move/attack and magic_damage/burn casts; the caster itself for self-target shield/buff casts; null if idle/no-enemies), a single `action` label by precedence cast > attack > move > idle, and `damageDealt` (post-mitigation/post-shield attack+cast damage; overtime true-damage is environmental and NOT attributed). It maintains a transient `Map<uid, number|null>` of previous-tick targets (NOT a field on `UnitInstance`) and emits a `TraceRetarget` whenever a unit's resolved target uid differs from its previous tick's, inferring the `RetargetReason` from observable end-of-tick state (prev null → acquired_no_target; prev dead → switched_target_dead; prev untargetable → switched_target_untargetable; prev out of range → switched_target_out_of_range; prev still valid but a different enemy chosen → retarget_recomputed, which is EXPECTED given the stateless nearest-recompute). `switched_forced` is reserved for a taunt-style forcing effect that the engine doesn't have, so it's never emitted. The harness (`scripts/trace.ts`) + fixtures (`packages/sim/tests/fixtures/`) + the QA invariant suite consume this; see `docs/combat-testing.md`.

**Notes** — The `addStat` helper does an unchecked dynamic property write (`(unit as unknown as Record<string, number>)[stat] = ...`) — this means ANY string can in principle be passed as a `stat` name from trait/item/buff data (there's no compile-time or runtime validation that `stat` is one of the unit's real numeric fields), so a malformed `traits.json`/`items.json` entry with a typo'd stat name would silently create a NEW, unused property on the unit object rather than erroring — a content-data correctness responsibility that falls on `packages/data`'s schema validation (if any) rather than this engine. The special-case `if (stat === "hp") unit.maxHp = unit.hp` inside `addStat` means any stat bonus targeting `"hp"` (e.g. a hypothetical HP-granting item) would treat it as a MAX-HP increase that also tops up current hp to the new max, not a heal-only or max-only effect — worth knowing precisely if reasoning about a future hp-granting trait/item's exact behavior.
