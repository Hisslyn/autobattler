# packages/rules/src/pool.ts

**Path & purpose** — `packages/rules/src/pool.ts`. The shared unit pool: initial pool construction from `economy.json`'s per-tier counts, and draw/return primitives without replacement. This is the entire mechanism behind the project's "pool conservation" hard invariant (total unit copies across pool + all player benches/boards is constant).

**Responsibility** — Owns the pool's data structure (`Map<defId, remainingCount>`) and every operation that changes a count: building it fresh at match start, decrementing on draw, incrementing on return (single unit or a batch of units, e.g. on player elimination). Does NOT decide WHEN to draw/return — that's the callers' job (`match.ts` for the starting unit, `shop.ts` for shop rolls, `commands.ts` for BUY/SELL/REROLL, `rounds.ts` for elimination cleanup).

**Exports**
- `buildInitialPool(data: GameData): Map<string, number>` — constructs the match's starting pool: one entry per unit def id (`data.units`), each initialized to `data.economy.poolCounts[String(unit.tier)]` (looked up by tier as a STRING key, since `poolCounts` is a JSON object — `Record<string, number>`), defaulting to 0 if a tier has no configured count (shouldn't happen given all 5 tiers are covered in `economy.json`). Every unit of the same tier starts with the SAME pool count (e.g. all 13 tier-1 units each get 29 copies, not a shared pool of 29 split across them — the pool is per-DEFID, not per-tier).
- `drawFromPool(pool, defId): boolean` — attempts to draw one copy of `defId`: returns `false` immediately if the current count is `<= 0` (or the id isn't in the map at all, via `?? 0`), else decrements by 1 and returns `true`. The boolean return is the caller's signal for "did this draw actually happen" — callers (shop rolling, starting-unit grant) must check it rather than assume success.
- `returnToPool(pool, defId): void` — increments `defId`'s count by 1 (`?? 0` default if absent, so returning a never-drawn id still works, defensively). No upper-bound check — nothing in this file prevents a count from exceeding its original starting value (callers are trusted to only return what was actually drawn elsewhere).
- `returnUnitsToPool(state, units, data): void` — batch return for a LIST of `UnitInstance`s: for each unit, looks up `gameplay.copiesPerStar[String(u.star)]` (defaulting to 1 if absent) and calls `returnToPool` that many times — a star-2 unit returns multiple copies (matching how it absorbed multiple pool draws when it was created via auto-merge), a star-1 unit returns exactly 1.

**Key behavior**
- The pool is keyed by exact unit DEFID (e.g. `"frost_knight"`), not by tier — so depleting one tier-1 unit's pool (e.g. everyone buying all the `peasant` copies) does NOT affect any OTHER tier-1 unit's availability; each def id has its own independent counter.
- `drawFromPool`'s no-replacement semantics mean a fully-depleted unit (count 0) simply can never be drawn again until something returns a copy (a SELL, a player elimination, a reroll returning shop slots) — the shop-rolling logic (`shop.ts`, documented separately) must itself skip/avoid offering depleted units, since `drawFromPool` only refuses the draw, it doesn't reroll or substitute.
- `returnUnitsToPool`'s `copiesPerStar` lookup mirrors the EXACT same lookup used in `commands.ts`'s SELL handler (`data.gameplay.copiesPerStar[String(unit.star)] ?? 1`) — both paths (an explicit player SELL and a bulk elimination-cleanup return) must stay consistent with each other for pool conservation to hold; if `copiesPerStar` values are e.g. `{"1":1, "2":3, "3":9}` (1/3/9 base copies absorbed per star), a star-3 unit returns 9 pool copies when sold or when its owning player is eliminated.

**Invariants & constraints**
- This file is the foundation of pool conservation: every unit a player can ever hold ultimately traces back to a `drawFromPool` call (shop roll, starting unit), and every unit that leaves play (SELL, REROLL returning old shop slots, player elimination) must trace forward to a `returnToPool`/`returnUnitsToPool` call — any code path that creates/destroys a `UnitInstance` WITHOUT a matching pool operation breaks the invariant and must be treated as a bug.
- PvE mobs are explicitly OUTSIDE this system — mob defIds don't appear in `data.units`, so they're never represented in the pool map at all; `buildMobBoard` (in `rounds.ts`) never calls any function in this file.
- Items are also outside this system entirely — nothing in `pool.ts` is ever invoked for item ids; items have no pool/conservation concept.
- The Map is mutated in place by all three mutating functions (`drawFromPool`/`returnToPool`/`returnUnitsToPool`) — callers pass `state.pool` directly, there's no copy-on-write or transaction semantics; a failed/rejected command elsewhere (e.g. `commands.ts` returning an error before reaching a pool call) must not have already mutated the pool, which holds today because every pool mutation in `commands.ts` happens AFTER all legality checks pass.
- `buildInitialPool` always returns the SAME initial distribution for the SAME `data` (purely a function of `economy.poolCounts` + `data.units`'s tiers) — deterministic, no randomness involved in pool SIZE (only in WHO draws what, via the seeded shop-roll PRNG elsewhere).

**Depends on** — `@autobattler/data`'s `GameData` type (`data.units`, `data.economy.poolCounts`, `data.gameplay.copiesPerStar`); `./state.js`'s `MatchState` type (only used by `returnUnitsToPool`'s signature, for `state.pool`); a local dynamic import type reference to `@autobattler/sim/src/types.js`'s `UnitInstance` (inlined as `import("@autobattler/sim/src/types.js").UnitInstance[]` directly in the parameter type rather than a top-level type import — a stylistic choice, functionally identical to a regular import).

**Used by**
- `packages/rules/src/match.ts` — `createMatch` calls `buildInitialPool` once, and `drawFromPool` for every player's starting unit.
- `packages/rules/src/shop.ts` — `rollShop` calls `drawFromPool` for each rolled slot (and presumably skips defIds with zero availability — see `shop.ts`'s own doc).
- `packages/rules/src/commands.ts` — `returnToPool` in SELL (per-copy) and REROLL (returning the old shop); implicitly, BUY does NOT call `drawFromPool` again since the unit was already drawn at shop-roll time.
- `packages/rules/src/rounds.ts` — `returnUnitsToPool` is called when a player is eliminated, returning their entire bench+board back to the shared pool so other players' future draws reflect the now-empty seats' former holdings.
- `packages/rules/tests/pool.test.ts` — direct unit coverage of draw/return/batch-return and conservation arithmetic.

**Notes** — None of the three mutating functions validate that `defId` is even a real unit id — `drawFromPool`/`returnToPool` will happily operate on an unrecognized key (creating a fresh Map entry via the `?? 0` defaults). This is harmless in practice since callers always pass real, validated def ids, but it means this file provides no defense against a caller bug that passes a malformed id — any such bug would silently create a stray pool entry rather than erroring.
