# packages/rules/src/rounds.ts

**Path & purpose** — `packages/rules/src/rounds.ts`. The largest and most complex file in `packages/rules`: derives the stage/PvE schedule from the round number, builds PvP pairings (avoiding repeat opponents, handling ghosts on odd counts), constructs PvE mob boards, runs the actual combat phase (calling into `@autobattler/sim`), applies HP/streak/placement/elimination consequences, and distributes per-round income. This is the file that turns "round N starts" into "round N's outcome is fully resolved."

**Responsibility** — Owns the entire round-resolution pipeline: (1) the stage/PvE schedule formula, (2) PvE mob board construction (live and preview), (3) pairing seed derivation, (4) PvP opponent pairing with anti-repeat backtracking + ghost handling, (5) running PvE rounds (every alive player vs the shared mob board, gold+loot, no player HP damage), (6) running PvP rounds (combat, HP damage, streaks, per-round/match stat accumulators, elimination + placement + pool return), and (7) per-round income distribution. Does NOT own the actual unit-vs-unit combat simulation itself (delegates to `@autobattler/sim`'s `simulateCombat`) or the loot table resolution (delegates to `./loot.js`).

**Exports**

*Stage/PvE schedule:*
- `stageForRound(round): {stage, roundInStage}` — pure stage formula: round 1-3 = stage 1 (`roundInStage` = round itself); round 4+ maps into stage 2, 3, 4... each `STAGE_LEN=7` rounds long, with `roundInStage` cycling 1-7. Round 4 → stage 2 RIS 1; round 10 → stage 2 RIS 7; round 11 → stage 3 RIS 1.
- `isPveRound(round, _data?)` — pure: true for ALL of stage 1 (rounds 1-3), and for `roundInStage` 4 or 7 within stage 2+ (so each 7-round stage cycle is PvP/PvP/PvP/PvE/PvP/PvP/PvE). The second parameter is accepted but ignored — kept only for call-site backward compatibility with an older signature; do not rely on it for anything.
- `pveStageForRound(round, data): MobStageDef | null` — returns the matched `data.mobs.stages[]` entry (by `{stage, roundInStage}`) for a PvE round, or `null` for a non-PvE round (or if no stage data matches, which shouldn't occur given `mobs.json` covers the schedule).

*Mob board construction:*
- `buildMobBoard(state, stage, data): BoardState` — builds the LIVE PvE creep board for an actual combat: allocates mob uids from the match's REAL, MUTATING `state.nextUid` counter (same namespace as player units — shared across the whole match) and converts to a side-1 `BoardState` via `boardToCombatState`.
- `previewPveStage(state, data): BoardState | null` — a SIDE-EFFECT-FREE preview of the upcoming PvE encounter (for UI display before combat actually runs): returns `null` if the current round isn't PvE; otherwise builds an IDENTICAL board (same units/placement) but with NEGATIVE, LOCALLY-SCOPED preview uids (starting at -1, decrementing) that never touch `state.nextUid` and never collide with real combat uids. Calling this any number of times is fully idempotent. Explicitly documented as "display-only... must never be used for actual combat resolution."

*Seeding:*
- `derivePairingSeed(roundSeed, pairingIndex): number` — a 4-step avalanche hash mix (XOR + `Math.imul` with fixed hex constants, all `magic-ok`-annotated as non-tuning hash constants) that derives one pairing's combat seed from the round's seed + its index. MUST be used IDENTICALLY by rules, the server, and any client re-simulating from `COMBAT_START` — any deviation breaks server/client combat reconciliation.

*Board conversion:*
- `boardToCombatState(board, side): BoardState` — converts a slot array (`(UnitInstance|null)[]`, indexed by board position) into a sim `BoardState`: assigns `team = side`, computes each unit's hex `pos` from its slot index (`q = i % COLS`, `r` computed so side 0 occupies the bottom rows and side 1 the top rows mirrored — `ROWS - 1 - floor(i/COLS)` for side 1), and filters out null slots.
- `ghostToCombatState(units): BoardState` — converts a dense (no nulls, positions already baked in) unit array into a side-1 `BoardState` by just stamping `team: 1` — used for ghost ("bye fight against a snapshotted board") pairings where positions are already meaningful from when the board was originally captured.

*Pairing introspection:*
- `PairingView` interface — `{opponentId, isGhost, side, result, opponentBoard, outcome}` — a normalized, perspective-specific view of one player's last pairing.
- `getPairingFor(state, playerId): PairingView | null` — scans `state.lastPairings` for the entry involving `playerId`, normalizes side (0 if they were "A", 1 if "B"), detects ghost (`isGhost = side===0 && bId<0`), and derives `outcome` (win/loss/draw) from `state.lastCombatResults.get(playerId)`'s `winner` relative to the player's side.
- `buildPairings(state, prng): [number,number][]` — the full opponent-pairing algorithm (see Key behavior).

*Round execution:*
- `runPveRound(state, prng, data): void` — runs a full PvE round for every alive player.
- `runCombatPhase(state, data): void` — the single entry point `match.ts`'s `advancePhase` calls; dispatches to `runPveRound` or runs the full PvP pairing+combat+consequences pipeline.
- `distributeIncome(state, data): void` — adds `calcIncome(player, data)` gold to every alive player; called once per round during RESOLUTION→PLANNING.

**Key behavior**
- **Stage formula constants** (`STAGE1_LEN=3`, `STAGE2_START=4`, `STAGE_LEN=7`, `PVE_RIS_A=4`, `PVE_RIS_B=7`) are explicitly documented as STRUCTURAL, not tuning numbers — they define the literal shape of the match's round schedule (3 PvE rounds, then repeating 7-round stages with PvE at positions 4 and 7), and changing them would be a structural game-design change requiring code review, distinct from a `gameplay.json`/`economy.json` content tweak.
- **`buildPairings`'s anti-repeat algorithm**: shuffles alive players via Fisher-Yates using the round's `prng`. For an EVEN player count, runs a recursive backtracking `matchWithBudget` that tries to find a perfect matching whose TOTAL meet-count sum is `<= budget`, starting at `budget=0` (all-unmet pairing) and incrementing until a valid matching is found (worst case, the sum of all pairwise meet-counts, guaranteeing termination) — so it always PREFERS pairs who've never met, falling back to the LEAST-total-repeat matching only when a fully-fresh pairing is impossible. For an ODD player count, uses a GREEDY approach instead (no backtracking): each unpaired player in shuffle order picks the first never-met opponent found, or failing that the least-met (tiebreak: lowest seat id) — the leftover unpaired player gets a GHOST match instead of a real opponent.
- **Ghost matches**: the one player left unpaired (odd alive count) fights a snapshotted board from a RANDOMLY chosen (via `prng`) ELIMINATED player's `lastBoard` (their board at the moment of elimination) rather than a live opponent — encoded in `lastPairings` as `[ghostFighterId, -(ghostSource.id + 1)]` (negative id encodes "this is a ghost, decode the real source id by negating and subtracting 1"). If there are NO eliminated players yet (very early game with an odd headcount — shouldn't normally happen with 8 starting players since 8 is even, but could occur after odd numbers of eliminations), the unpaired player gets NO pairing at all that round (silently skipped — they neither fight nor get a result entry beyond the bye default).
- **Pairing history**: `state.pairingHistory` is a `Map<playerId, Map<opponentId, meetCount>>`, updated symmetrically (both directions) for every REAL pairing (ghost "pairings" do NOT update history, since `bId` is negative and the loop only records `[a,b]` pairs from the matching/greedy result — ghost assignments happen in a separate loop afterward and are never added to `pairingHistory`).
- **`runPveRound`**: clears all `last*` round-scoped state maps, draws TWO prng values upfront (`roundSeed` for combat, `lootSeed` for loot — drawn in that fixed order, so determinism depends on this exact sequence), builds ONE shared mob board for the round (reused across every player's fight — same mobs, same stats, same starting positions for everyone), then for each alive player: runs `simulateCombat(playerBoard, mobBoard, derivePairingSeed(roundSeed, player.id), data)` (per-player deterministic seed derived from the shared round seed by player id as the "pairing index"), stores the result + a board snapshot, sets `lastRoundResult` to `{status:"pve", damageTaken:0, damageDealt:0}` (NEVER counts as W/L, no player HP change, no streak change — matches CLAUDE.md exactly), grants `economy.pveBaseGold` flat gold, and generates+applies seeded loot via a PER-PLAYER loot prng (`mulberry32(derivePairingSeed(lootSeed, player.id))` — note: reuses `derivePairingSeed`, originally named for PAIRING seeds, as a general avalanche-mix-by-index utility here for loot too). The already-decided orbs are stored in `lastLootOrbs` for the client to animate without re-deriving.
- **`runCombatPhase`'s PvP path**: re-seeds a `prng` from `state.prngState` and immediately advances it (storing the NEW state) — this happens BEFORE the PvE/PvP branch check, so PvE rounds also consume this one prng draw even though `runPveRound` then draws its OWN `roundSeed`/`lootSeed` from the SAME prng instance passed in (continuing the same stream, not a fresh one). For PvP: calls `buildPairings`, draws `roundSeed` from the continuing prng stream, defaults EVERY alive player's `lastRoundResult` to `{status:"bye", damageTaken:0, damageDealt:0}` (so anyone NOT covered by a real pairing this round — e.g. the rare odd-count-with-no-ghost-source case — correctly shows a bye), then iterates every pairing:
  - For a ghost pairing (`bId < 0`): builds `boardB` from the snapshotted ghost source's `lastBoard`, sets `isGhost=true`.
  - For a real pairing: builds both boards, snapshots each side's opponent board for the OTHER player's `lastOpponentBoards` entry (`opponentBoardSnapshot` for A's view of B, AND a direct `state.lastOpponentBoards.set(bId, [...playerA.board])` for B's view of A — done inline, asymmetric code shape but symmetric result).
  - Runs `simulateCombat(boardA, boardB, derivePairingSeed(roundSeed, pairingIndex), data)` — note the seed here is derived by PAIRING INDEX (position in the `pairings` array), NOT by player id (unlike `runPveRound`, which derives by player id) — this is a meaningful asymmetry worth knowing if ever debugging seed reproduction.
  - Both real players get the SAME `CombatResult` object reference stored in `lastCombatResults` (not copies) — a ghost fighter's opponent does NOT get a result entry (the ghost source might be a long-eliminated player who isn't even in `state.players` as alive, so there's nothing to update).
  - **Win/loss consequences**: winner's `winStreak++`/`loseStreak=0`; loser's `loseStreak++`/`winStreak=0`. Damage is computed ONCE via `calcPlayerDamage` (survivor-count + tier-weight + star formula, PLUS `damageBase + floor(round/damageRoundDivisor)` — so damage scales up gradually as the match progresses) using the WINNER's surviving units, and the SAME number both reduces the loser's `hp` AND is recorded as the winner's `damageDealt`/loser's `damageTaken` — `recordRoundWin`/`recordRoundLoss` are the only places these accumulators (`PlayerState.roundWins`/`roundLosses`/`totalDamageDealt`/`totalDamageTaken`) and the per-round `lastRoundResult` are touched. A DRAW (`result.winner` is neither 0 nor 1) does NOTHING — no HP, no streak change, no win/loss recorded (both players keep their already-defaulted bye-shaped 0/0 result, with `status` still "bye" even though they DID fight — a draw is treated identically to a bye for `lastRoundResult` purposes, just no win/loss credited).
  - A GHOST FIGHT'S WINNER still gets full win consequences (streak/placement-relevant), but the LOSING SIDE consequences only apply to the real fighter — there's no real "playerB" to update when `isGhost` is true (the `if (!isGhost)` guards skip updating a second player).
  - Both real players' `lastBoard` is re-snapshotted AFTER combat (`playerA.board.filter(...)` — note this re-reads the CURRENT board, which combat doesn't mutate, so this is really just "snapshot the board state as of this round" for potential future ghost-fight use if this player gets eliminated later).
- **Elimination**: after all pairings resolve, computes the NEXT placement number as `(count of already-!alive players) + 1` (placements assigned in REVERSE finishing order — the FIRST player eliminated gets the HIGHEST placement number, i.e. last place; placement counts UP as more players get eliminated, so the very last elimination(s) get LOW placement numbers, i.e. near-winner). Collects all players with `hp <= 0` this round, sorts them by `hp` ascending (lowest HP first = "more thoroughly defeated" = eliminated "first" in tiebreak order if multiple hit 0 in the same round) — ties on exact HP aren't further broken (stable sort preserves original array order, i.e. player id order, as the final tiebreak). For each eliminated player: flips `alive=false`, assigns `placement`, pushes to `state.placements`, returns ALL their bench+board units to the pool (`returnUnitsToPool`) AND any remaining shop slot units (`returnToPool` per slot), then WIPES their bench/board/shop to empty (so an eliminated player's `PlayerState` no longer holds any units/shop, keeping pool accounting clean and preventing any further interaction with stale holdings).
- **`distributeIncome`**: trivial — one `calcIncome` call + gold add per alive player, no other side effects.

**Invariants & constraints**
- Pool conservation is enforced exactly at elimination time: `returnUnitsToPool` + per-shot-slot `returnToPool` ensures an eliminated player's ENTIRE holdings (bench, board, undrafted shop copies) flow back to the shared pool — no units are ever silently destroyed.
- `derivePairingSeed` MUST stay byte-identical across rules/server/client — this is the literal mechanism enabling client-side combat re-simulation from a server-sent seed to reconcile against the server's authoritative `COMBAT_RESULT`.
- PvE rounds NEVER touch player `hp`, `winStreak`/`loseStreak`, or count toward `roundWins`/`roundLosses` — confirmed directly in `runPveRound`'s body (no `hp`/streak mutation anywhere in that function).
- A draw's `lastRoundResult` stays at the pre-set "bye" default (0/0, no win/loss) — this is a deliberate simplification: a draw is NOT its own distinct `RoundResultStatus` value (the type only has `won`/`lost`/`bye`/`pve`), so a draw is INDISTINGUISHABLE from a true bye at the `lastRoundResult` level, even though `getPairingFor`'s `outcome` field DOES expose `"draw"` separately (derived independently from `CombatResult.winner`, not from `lastRoundResult`).
- `runCombatPhase`'s single prng draw-and-advance happens UNCONDITIONALLY before the PvE/PvP branch — both code paths consume exactly one shared "entry" draw from the continuing stream before doing their own further draws, preserving a consistent prng-consumption shape across round types for determinism purposes.
- The PvP seed-by-pairing-INDEX vs PvE seed-by-PLAYER-ID asymmetry (noted above) is intentional per their different needs (PvP pairings are positionally ordered in an array each round; PvE has no "pairing" concept, just one player vs the shared mob board, so player id is the natural per-player discriminator) but is a real, non-obvious asymmetry worth flagging to anyone trying to verify a specific seed by hand.
- `previewPveStage`'s negative uid counter is PURELY LOCAL AND DISPOSABLE per call — never persisted, never compared across calls, exists only so a UI can render "here's what you'll fight" before the round actually executes, with zero risk of corrupting the real uid namespace.

**Notes** — The recursive `matchWithBudget` backtracking search is worst-case exponential in player count, but bounded in practice by the project's fixed 8-player lobby size (small enough search space) and by the budget-incrementing outer loop terminating as soon as ANY valid matching is found at the current budget (not searching for the OPTIMAL one beyond "is there a matching at this budget," then immediately accepting it) — this is a deliberate "good enough, terminates fast" approach rather than a true minimum-cost perfect matching. The asymmetric code shape in the real-pairing branch (one inline `state.lastOpponentBoards.set(bId, ...)` call sitting next to the loop's general `opponentBoardSnapshot` variable used for `aId`) reflects that B's opponent-board entry is fully known immediately (it's just A's CURRENT board) while A's needs the snapshot variable to also serve the ghost branch.
