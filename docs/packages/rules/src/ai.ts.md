# packages/rules/src/ai.ts

**Path & purpose** — `packages/rules/src/ai.ts`. Pure, seeded bot policy for the planning phase: given a player's current state, decides and applies a sequence of BUY_XP/BUY/MOVE commands to simulate a reasonable (not optimal) human player's shopping/board-management behavior.

**Responsibility** — Owns the entire AI decision-making logic for bot-controlled seats. This is the ONLY place bot "intelligence" lives; the server's room logic and the client's LocalDriver both call into this same function rather than each implementing their own bot behavior, guaranteeing Practice (LocalDriver) and Online (server-side bot seats) bots behave identically given the same seed/state.

**Exports** — `applyAiCommands(state: MatchState, playerId: number, prng: Prng, data: GameData): void` — the sole export. Runs one full decision pass for one player: an XP-buy check, a shop-buying pass (using a trait-overlap heuristic), and a board-filling pass (moving bench units onto empty board slots up to the level cap). Mutates `state` in place via repeated calls to `applyCommand` (each call is independently legality-checked and typed-error-safe, per `commands.ts`'s contract — `ai.ts` doesn't need its own legality checks since `applyCommand` already enforces them and silently no-ops/returns a `CommandResult` the AI ignores).

**Key behavior**
1. **No-op guard**: if the player doesn't exist or `!player.alive`, returns immediately — dead/missing players never get AI commands.
2. **XP buy**: if `player.gold > data.gameplay.aiXpGoldThreshold` (a single tunable constant from `gameplay.json`), issues one `BUY_XP` command. This is a flat threshold check, not a more nuanced gold-management curve — bots buy XP greedily once they're sitting on "enough" surplus gold, every planning phase the threshold is exceeded.
3. **Shop scoring + buying pass**: builds `boardDefIds` (the unit def ids currently on the player's board, skipping null/empty slots). For each shop slot the player can currently AFFORD (`player.gold >= def.tier`, using tier as both cost AND the variable name — flat tier=cost assumption baked into this file, distinct from any more nuanced cost table that might exist in `economy.json`), computes `traitOverlapScore(slot.defId, boardDefIds, data)` (counts, for every unit already on the board, how many of ITS traits the candidate shares — summed across all board units, so a unit overlapping traits with 3 board units scores higher than one overlapping with 1) and a final desirability `score = overlap * aiTraitOverlapWeight - cost` (another `gameplay.json` tunable weight balances trait-synergy preference against raw gold cost). Slots are sorted descending by score and then bought greedily in that order, with TWO additional guards re-checked at buy time (state may have changed from earlier buys in this same pass): the slot/def must still resolve, gold must still cover the cost, AND — unless the unit has SOME trait overlap with the (possibly now-changed) board — the AI won't buy if doing so would drop gold below `aiInterestReserve` (another `gameplay.json` constant), i.e. bots preserve interest-earning gold UNLESS the buy has real synergy value.
4. **Board-filling pass**: re-fetches the player state fresh (`p2`, since earlier commands mutated `state`) and walks the bench, moving each occupied bench slot onto the board (first empty board slot, or appending after the current board count if none is null) until `boardCount >= p2.level` (the level-derived board-slot cap) — fills the board up to capacity, doesn't rearrange existing board units, doesn't sell or bench anything.

**Invariants & constraints**
- Pure (no I/O), takes its randomness only through the passed-in seeded `Prng` (mulberry32, forwarded straight through to `applyCommand`, which itself may or may not consume it depending on the command type) — determinism-safe per the project's hard invariant.
- All AI tuning constants (`aiXpGoldThreshold`, `aiTraitOverlapWeight`, `aiInterestReserve`) live in `packages/data/src/gameplay.json`, NOT hardcoded here — consistent with the "all tuning numbers live in packages/data" invariant.
- Every state mutation goes through `applyCommand` — `ai.ts` never mutates `MatchState` fields directly, so bot actions are subject to exactly the same legality rules as a human's commands (can't overspend, can't exceed bench/board caps, etc.).
- `traitOverlapScore` does a fresh `data.units.find(...)` scan per candidate per board-unit (O(n·m) per call, called repeatedly across shop slots and again per buy-time re-check) — not optimized, but the unit/board counts are small (max 5 shop slots, max ~9 board units), so this is a non-issue at the current scale; would matter if those bounds ever grew significantly.
- The "first empty board slot, else append after boardCount" logic in the board-filling pass assumes `board` is a fixed-length array of nullable slots (`(UnitInstance | null)[]`) — `indexOf(null)` returning -1 means no null slot remains, so the fallback appends at `boardCount` (which only makes sense if `boardCount < board.length`, implicitly guaranteed by the `boardCount >= p2.level` break condition combined with level never exceeding board length elsewhere in the system).
- This logic is deliberately simple/heuristic, not game-theoretically optimal — it exists to give bots plausible, predictable behavior, not to be a strong opponent; CLAUDE.md describes it as a "seeded bot policy," not an optimization-driven AI.

**Depends on**
- `@autobattler/data`'s `GameData` type (reads `data.units`, `data.gameplay.aiXpGoldThreshold`/`aiTraitOverlapWeight`/`aiInterestReserve`).
- `./state.js` (`MatchState` type — reads `state.players[playerId]`'s `gold`/`alive`/`board`/`bench`/`shop`/`level`).
- `@autobattler/sim/src/prng.js` (`Prng` type — passed through to `applyCommand`, never directly invoked here).
- `./commands.js` (`applyCommand` — the only mutation path).

**Used by**
- `packages/server/src/room.ts` — backfills empty seats with bots at matchmaking time and calls `applyAiCommands` for each bot seat during the planning phase (likely near phase-end, mirroring how `rounds.ts`'s `runCombatPhase` is invoked once pairings are locked).
- `packages/client/src/driver.ts` (`LocalDriver`) — applies AI commands for all non-human seats (seat 0 is the real player in Practice mode, seats 1-7 are bots) at planning-phase end, per CLAUDE.md ("AI commands applied at phase end").

**Notes** — The shop-buying pass re-derives `boardDefIds` ONCE before the loop (from the pre-buy board state) but the buy-time interest-reserve check re-derives trait overlap fresh against the CURRENT board (post any earlier buys in the same pass that already moved to board — though buys only add to bench/shop interactions, not directly to board, so in practice this recompute is mostly equivalent to the original `boardDefIds` unless a MOVE happened mid-pass, which it doesn't within this function). The two-pass structure (score-and-sort once, then re-validate-and-buy in a loop) means the desirability ORDER is fixed from the initial board snapshot even though individual buy decisions re-check gold/overlap against the live state.
