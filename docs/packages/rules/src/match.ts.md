# packages/rules/src/match.ts

**Path & purpose** — `packages/rules/src/match.ts`. The top-level match lifecycle orchestrator: creates a fresh `MatchState` from a seed, advances the phase state machine (PLANNING → COMBAT → RESOLUTION → next round's PLANNING), and answers match-over/winner queries. This is the entry point both the server (`room.ts`) and any headless test/sim driver (e.g. `runMatchToEnd`) use to drive a match forward.

**Responsibility** — Owns match SETUP (initial pool, players, starting unit, first shop rolls) and PHASE TRANSITIONS (the three-phase cycle, income distribution between rounds, shop refresh). Does NOT own the actual combat resolution logic (delegates to `rounds.ts`'s `runCombatPhase`) or income math (delegates to `rounds.ts`'s `distributeIncome`) — this file is the glue/sequencing layer, not the rules themselves.

**Exports**
- `createMatch(seed: number, data: GameData): MatchState` — builds a brand-new match: seeds a mulberry32 `prng` from `seed`, builds the initial unit pool (`buildInitialPool`), constructs `gameplay.playerCount` `PlayerState`s (id 0..N-1) with starting `hp`/`gold` from `gameplay.json`, empty bench, a `board` array of `gameplay.boardSlots` nulls, empty `items`, a `shop` array of `economy.shopSlots` nulls, zeroed streaks, `alive: true`, `placement: null`, and all four match-stat accumulators (`roundWins`/`roundLosses`/`totalDamageTaken`/`totalDamageDealt`) at 0. Assembles the `MatchState` shell (`round: 1`, `phase: "PLANNING"`, `nextUid` starting at 10000, empty `pairingHistory`/`placements`/`lastPairings`/`lastCombatResults`/`lastOpponentBoards`/`lastLootOrbs`/`lastRoundResult` maps/arrays). Gives every player ONE copy of `gameplay.startingUnitId` on bench slot 0 (drawn from the pool, so pool conservation holds from turn zero), then rolls every player's initial 5-slot shop using a freshly re-seeded sub-prng (`mulberry32(state.prngState)`).
- `advancePhase(state: MatchState, data: GameData): void` — the phase state machine's single transition function:
  - From `PLANNING`: sets phase to `COMBAT`, calls `runCombatPhase(state, data)` (which runs ALL the round's pairings/PvE and fully resolves them synchronously — by the time this call returns, all combat for the round is done), then sets phase to `RESOLUTION`.
  - From `RESOLUTION`: calls `distributeIncome(state, data)` (grants gold to every alive player), increments `state.round`, sets phase back to `PLANNING`, then refreshes EVERY alive player's shop (returns their current shop slots to the pool first, then rolls a fresh 5 using a freshly re-seeded sub-prng derived from `state.prngState`).
  - No `case` for `COMBAT` — `advancePhase` is never expected to be called while phase is `COMBAT` (that phase is entered and exited entirely within the `PLANNING`-branch call, never independently advanced).
- `isMatchOver(state: MatchState): boolean` — true when `<= 1` player has `alive === true` (covers both "exactly one survivor" and the degenerate "all eliminated" edge case as also "over").
- `getWinner(state: MatchState): PlayerState | null` — returns the SINGLE alive player if there's exactly one, else `null` (covers both "match not over yet" and the degenerate "everyone eliminated, no winner" case — both return `null`).
- `runMatchToEnd(seed, data, setup?): MatchState` — headless full-match runner: `createMatch`s a state, optionally lets a caller-supplied `setup` callback mutate it (e.g. tests pre-seeding boards), then loops `advancePhase` until `isMatchOver` or a 10000-iteration safeguard trips (a runaway-loop guard, not a tuning number — commented `magic-ok`). Used by tests and the balance package's batch sweeps to play out full matches without a server/client in the loop.

**Key behavior**
- PRNG threading: `createMatch` seeds ONE mulberry32 generator from the input `seed`, calls it ONCE (`prngState: prng()`) to get the state stored on `MatchState.prngState`, then re-seeds a SECOND mulberry32 instance FROM that stored state for the initial shop rolls (and calls IT once more to advance `prngState` again before the rolling loop even starts). This "store state, then re-derive a generator from it, then advance again" pattern is repeated in `advancePhase`'s RESOLUTION→PLANNING branch for shop refreshes — `state.prngState` is the single persisted PRNG checkpoint between rounds, and EVERY phase-driven step that needs randomness re-seeds a fresh mulberry32 instance from it then immediately advances it. This is the mechanism that gives the project's "shared deterministic simulation core" its single PRNG-state-per-match property: the WHOLE match's randomness (shop rolls, combat, AI, loot) descends from one evolving `prngState` value, never from independent unseeded sources.
- Starting unit grant has a defensive exhaustion guard: if `drawFromPool` ever fails (returns falsy — pool has no more copies of `startingUnitId`), the loop `break`s early, leaving any LATER players in the loop without a starting unit. The comment explicitly notes this "should never fire in a normal 8-player game" since the pool has far more tier-1 copies (29) than players (8) — this is purely defensive code for a configuration that shouldn't occur given `economy.json`'s pool counts.
- `runCombatPhase` (called from `advancePhase`'s PLANNING branch) is where `placements` actually gets populated (player ids pushed in elimination order) and `alive` gets flipped false — `match.ts` itself never touches `placement`/`alive` directly; it only triggers the call into `rounds.ts` that does.
- The RESOLUTION→PLANNING shop refresh ONLY touches ALIVE players (`state.players.filter(p => p.alive)`) — eliminated players keep whatever shop state they had at elimination (irrelevant since they're out of the match, but worth knowing if any code ever inspects a dead player's `shop` field).

**Invariants & constraints**
- `createMatch`/`advancePhase`/`runMatchToEnd` are deterministic given the same `seed` + `data` + (for `runMatchToEnd`) the same `setup` callback — no `Math.random`, no `Date`, no I/O, satisfying the project's purity invariant for `packages/rules`.
- `nextUid` starts at 10000 specifically so unit uids never collide with the mob-board uid namespace used by PvE rounds (`buildMobBoard` in `rounds.ts` allocates from the SAME match's `nextUid` counter, per `rounds.ts`'s own docs — both player units and PvE mobs share one monotonic counter scoped to the match).
- Pool conservation holds across `createMatch`'s starting-unit grant: every starting unit is drawn via `drawFromPool` (decrementing the pool), never conjured outside the pool-accounting system.
- `advancePhase` assumes it's called in the correct sequence (PLANNING then RESOLUTION then PLANNING...) — calling it while `phase === "COMBAT"` is a no-op (falls through both `if`/`else if` branches silently) since combat is resolved synchronously within the PLANNING branch's own call, never left half-advanced.
- `runMatchToEnd`'s 10000-iteration safeguard is a hard cap, not a tuning number — if a match somehow never reaches `isMatchOver` (a logic bug elsewhere), this function returns a NOT-fully-resolved `MatchState` rather than hanging forever; callers should be aware a returned state from `runMatchToEnd` is not unconditionally guaranteed to satisfy `isMatchOver`.

**Depends on**
- `@autobattler/data` — `GameData` (the typed content bundle: `gameplay`, `economy`, `units`, etc.).
- `./state.js` — `MatchState`/`PlayerState` types (the structures being built/mutated).
- `@autobattler/sim/src/types.js` — `UnitInstance` type (the starting unit's shape).
- `@autobattler/sim/src/prng.js` — `mulberry32` (the seeded PRNG constructor — the ONLY randomness source used anywhere in this file).
- `./pool.js` — `buildInitialPool`/`drawFromPool`/`returnToPool` (pool lifecycle for the starting unit grant and shop refresh).
- `./shop.js` — `rollShop` (shop slot generation, called at match creation and every round's PLANNING entry).
- `./rounds.js` — `runCombatPhase`/`distributeIncome` (the actual combat resolution and income logic this file sequences).

**Used by**
- `packages/server/src/room.ts` — calls `createMatch` once per matchmade lobby, then `advancePhase` to drive the match through its phases (likely gated by the 30s planning timer / all-ready signal), checks `isMatchOver`/`getWinner` for match-end handling and `MATCH_END` broadcast.
- `packages/client/src/driver.ts` (`LocalDriver`) — drives a local Practice match through the same `createMatch`/`advancePhase` calls (no server in the loop).
- `packages/balance/src/...` — batch sweeps likely use `runMatchToEnd` (or the lower-level functions directly) to play out many seeded matches headlessly for win-rate analysis. (Balance's own primary matchup tooling — `runMatchup`/`runSweep` — operates at the `sim` combat level directly via prebuilt boards rather than through full matches, per CLAUDE.md; `runMatchToEnd` is more likely used by `rules` tests and any future full-match balance tooling.)
- `packages/rules/tests/match.test.ts` — direct test coverage of the full lifecycle.

**Notes** — The double mulberry32-reseed pattern (`prng()` once to capture state, then `mulberry32(state.prngState)` to build a fresh generator, then `prng()` again to advance past the just-used draw) appears twice (once in `createMatch`, once in `advancePhase`'s RESOLUTION branch) and is the canonical pattern for "persist a PRNG checkpoint across an async/serialized boundary" in this codebase — any new code needing a PRNG mid-match should follow this exact same store/reseed/advance shape rather than inventing a new pattern, to keep `prngState` the single source of truth for resumability (e.g. server restart/reconnect rebuilding state from a snapshot).
