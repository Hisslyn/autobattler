# packages/rules/src/loot.ts

**Path & purpose** — `packages/rules/src/loot.ts`. Pure, seeded PvE loot generation and application: builds a round's reward orbs from `loot.json` using the match's shared PRNG, and folds a resolved orb's reward into a player's gold/inventory. This is the entire implementation of the "PvE loot is seeded-deterministic" hard invariant.

**Responsibility** — Owns two things: (1) deciding WHAT a PvE round's loot orbs contain (`generateLoot`), and (2) APPLYING one already-decided orb's reward to a player (`applyLootOrb`). Both are pure — no I/O, no direct `Math.random`, only the passed-in seeded `Prng`. The orbs returned by `generateLoot` are also the literal payload stored in `MatchState.lastLootOrbs` and sent to clients as the private `LOOT` message, so this file's output IS the canonical reveal data — the client only animates it, never re-derives it.

**Exports**
- `LootReward` — discriminated union: `{kind:"gold", amount}` | `{kind:"component", id}` | `{kind:"item", id}` — a single resolved reward (gold folds straight into player gold; component/item ids land in the player's loose inventory, same array as everything else in `player.items`).
- `LootOrb` — `{rarity: LootRarity, reward: LootReward}` — one revealed orb: its rarity (carried through purely for the client's reveal animation — ascending-rarity ordering, distinct color/shape per rarity) plus its resolved reward.
- `generateLoot(round: number, prng: Prng, data: GameData): LootOrb[]` — the round's full loot generation: looks up `data.loot.roundDrops[String(round)]` (a list of `{rarity, count}` — how many orbs of each rarity drop on this specific PvE round number), and for each `{rarity, count}` entry, draws `count` orbs from `data.loot.tables[rarity]` via `pickEntry` (weighted random pick consuming the shared `prng`). Returns `[]` immediately for a round with no configured drops (no entry in `roundDrops`) and silently skips a rarity whose table is empty (defensive, shouldn't occur given the data integrity tests).
- `applyLootOrb(player: PlayerState, orb: LootOrb): void` — applies one already-resolved orb's reward: gold rewards add to `player.gold`; component/item rewards push the id onto `player.items` (the SAME unequipped-inventory array used by EQUIP/UNEQUIP/COMBINE_ITEMS in `commands.ts` — loot-won items behave identically to shop-bought/sold items once they land in inventory).

**Key behavior**
- `pickEntry(entries, prng)` (internal helper, not exported): computes `total = sum of all entries' weight`, draws `roll = prng() % total`, then linear-scans `entries` subtracting each entry's weight from `roll` until `roll < entry.weight` — standard weighted-random-pick-by-cumulative-subtraction. Falls back to the LAST entry if the loop somehow exhausts without finding (defensive; mathematically shouldn't trigger given `roll < total` always).
- `entryToReward(entry)` (internal helper): converts a `LootEntry` (data-layer, carries `weight`) into a `LootReward` (rules-layer, weight stripped — the weight was only relevant for the random pick, not the resulting reward).
- `generateLoot`'s round-to-drops lookup is by STRING key (`String(round)`) since `roundDrops` is a `Record<string, RoundDrop[]>` in JSON (object keys are always strings) — callers must pass the numeric round, the string conversion is internal.
- The orbs are returned in `roundDrops` array order (each `{rarity,count}` entry's orbs pushed contiguously, in the same order count iterates) — NOT pre-sorted by rarity; any rarity-ordering for display (e.g. ascending-rarity reveal sequence) is done downstream by the client's `lootRevealModel`, not here.

**Invariants & constraints**
- Fully deterministic: same `round` + same `prng` stream position + same `data.loot` content → byte-identical `LootOrb[]` output, every time — this is the literal mechanism behind "PvE loot is seeded-deterministic... same seed + config → identical orbs/rewards."
- Every `prng()` call advances the shared match PRNG stream — `generateLoot` consumes exactly one `prng()` call per orb (via `pickEntry`), so the NUMBER of orbs generated (driven by `roundDrops[round]`'s total count) directly determines how many PRNG draws happen, which affects what randomness is available to anything called AFTER it in the same round. Callers (`rounds.ts`'s `runPveRound`) must call this at a fixed point in the round's PRNG consumption order for cross-run determinism to hold.
- Loot is NEVER pooled — `applyLootOrb` pushing an item/component id to `player.items` does NOT touch `state.pool` (unlike SELL/REROLL in `commands.ts`, which do return units to the pool). This matches the documented invariant "items are NOT pooled... and items are NOT pooled (neither affects conservation)."
- `generateLoot` itself does not mutate `PlayerState` — it only DECIDES the orbs; `applyLootOrb` is a SEPARATE explicit step the caller must invoke per player per orb to actually grant the reward. This split exists so the orbs can be stored (`MatchState.lastLootOrbs`) and sent to the client for reveal BEFORE/independently of when the gold or inventory mutation is applied — though in practice `runPveRound` likely calls both together per player.
- Component/item rewards are pushed to `player.items` as a flat array of ids — duplicates are allowed (pushing the same component id twice from two orbs just adds two entries), consistent with how `commands.ts` treats the inventory as a flat multiset of ids, not a deduplicated set.

**Depends on**
- `@autobattler/data` — `GameData` (for `data.loot.tables`/`data.loot.roundDrops`), `LootEntry`/`LootRarity` types (the data-layer shapes this file consumes and reshapes into its own `LootReward`/`LootOrb`).
- `./state.js` — `PlayerState` type (the `gold`/`items` fields `applyLootOrb` mutates).
- `@autobattler/sim/src/prng.js` — `Prng` type (the seeded randomness source, never `Math.random`).

**Used by**
- `packages/rules/src/rounds.ts` — `runPveRound` calls `generateLoot` once per PvE round (using a deterministic per-round/per-match seed derivation) to produce the orbs stored in `MatchState.lastLootOrbs`, then calls `applyLootOrb` per player per orb to actually grant gold/items.
- `packages/server/src/room.ts` — reads `state.lastLootOrbs` to build the private per-seat `LOOT` message sent before the RESOLUTION `PHASE_CHANGE`.
- `packages/client/src/...` (`driver.ts`/`netDriver.ts`) — `getMyLootOrbs()` accessors surface the already-decided orbs (from `lastLootOrbs` locally, or from the private `LOOT` message over the network) for the client's `lootRevealModel`/`startLootReveal` animation — the client never calls `generateLoot` itself; it only displays what the server/rules layer already decided.

**Notes** — The weighted-pick's `prng() % total` assumes `prng()` returns a non-negative integer with enough range relative to `total` for a roughly uniform distribution — this is the same mulberry32 convention used throughout `packages/sim`/`packages/rules` (an unsigned 32-bit-ish integer stream), not a special case introduced here. There is no per-entry zero-weight guard explicitly, but a zero-total `entries` array would produce a `roll = prng() % 0` which is `NaN` in JS — this can't happen in practice only because `loot.json`'s tables are guaranteed non-empty and weight-positive by the data integrity tests (`integrity.test.ts`), not by any defensive code in this file itself.
