# packages/rules/src/commands.ts

**Path & purpose** — `packages/rules/src/commands.ts`. The single legality-and-mutation gate for every player action during the planning phase: validates a `Command` against the current `MatchState` and either applies it (mutating state) or returns a typed `CommandError` — never throws. This is the entire "what can a player do right now" rulebook.

**Responsibility** — Owns command legality enforcement and the resulting state mutation for BUY, SELL, REROLL, BUY_XP, MOVE, EQUIP, UNEQUIP, COMBINE_ITEMS, and USE_CONSUMABLE. Also owns the auto-merge cascade (3 same-star copies → next star) that fires automatically after any BUY. This is the ONLY place that mutates `PlayerState.gold`/`bench`/`board`/`shop`/`items`/`xp`/`level` in response to a player action — the server and client (LocalDriver) both call this exact function so behavior is identical online/offline.

**Exports**
- `MAX_ITEMS_PER_UNIT = 3` — the per-unit equipped-item slot cap (constant, not content-tunable — explicitly commented `magic-ok: slot-cap union bound, not tuning`, meaning it's exempt from the "no magic numbers" convention because it's a structural bound, not a balance number).
- `Command` — discriminated union of the 9 command shapes: `BUY {shopSlotIndex}`, `SELL {unitUid}`, `REROLL {}`, `BUY_XP {}`, `MOVE {unitUid, toBench, toIndex}`, `EQUIP {unitUid, itemId}`, `UNEQUIP {unitUid, itemId}`, `COMBINE_ITEMS {itemIdA, itemIdB}`, `USE_CONSUMABLE {consumableId, targetUnitId, targetItemId?}`.
- `CommandError` — union of 15 typed error codes: `INSUFFICIENT_GOLD`, `BENCH_FULL`, `BOARD_FULL`, `EMPTY_SLOT`, `UNIT_NOT_FOUND`, `ITEM_NOT_FOUND`, `ITEM_SLOTS_FULL`, `NO_RECIPE`, `INVALID_POSITION` (declared but not seen thrown in this file's current logic — reserved/future), `PHASE_INVALID`, `CONSUMABLE_NOT_FOUND`, `ITEM_NOT_EQUIPPED`, `NO_ALTERNATIVE_ITEM`, `NOT_TIER_2_ITEM`, `NO_TIER_2_ITEMS_EQUIPPED`.
- `CommandResult` — `{ok: true} | {ok: false, error: CommandError}` — every `applyCommand` call returns this; callers (AI, server, rules tests) branch on `.ok` rather than try/catch.
- `recipeResult` — re-exported straight from `@autobattler/data` (pure, "data is the source" per the inline comment) so rules consumers don't need a second import path.
- `tryAutoMerge(state, playerId, defId, data): void` — recursive auto-merge: finds 3 copies of `defId` at star 1 (bench+board combined) and fuses them into one star-2 unit, OR 3 copies at star 2 into one star-3 unit; called automatically after every successful BUY (never called standalone by other commands — SELL/MOVE never trigger merges since they don't create new copies).
- `applyCommand(state, playerId, cmd, prng, data): CommandResult` — the main entry point; the one function every consumer (server room, client LocalDriver, AI) calls for every player action.

**Key behavior**
- **Phase gate**: `applyCommand`'s FIRST check is `state.phase !== "PLANNING"` → `PHASE_INVALID` for every command type — no command is ever legal outside planning (combat/resolution phases reject everything).
- **BUY**: resolves the shop slot's unit def, cost = `def.tier` (flat tier-as-gold-cost, no separate price table). Bench-full handling has a special carve-out: if the bench is full BUT this purchase would be the 3rd same-star-1 copy (completing an immediate merge, net bench growth ≤ 0), the buy is still allowed — `BENCH_FULL` only rejects when the bench is full AND the purchase wouldn't immediately reduce via merge. The shop copy was already drawn from the pool at ROLL time (`rollShop`), so BUY itself does no further pool draw — it only deducts gold, clears the shop slot, constructs a fresh star-1 `UnitInstance` (stats copied straight from the unit def, `mana: def.manaStart`, empty `items: []`), pushes it to the bench, then calls `tryAutoMerge`.
- **`tryAutoMerge`**: checks star 1 then star 2 (in that order, via `for (const targetStar of [1,2])`) — collects ALL bench+board units of that `defId`+star; if `< 3`, skips to the next star tier. If `>= 3`: removes exactly 3 copies (bench last-to-first, then board last-to-first if still needed), collecting each removed unit's `items` into `sourceItems` (so an upgrade preserves equipped items, capped — see below). Builds a new `UnitInstance` at `newStar = targetStar+1` with base stats from the unit def (NOT yet star-multiplied — star multipliers are applied by the SIM at combat start, not stored on the persisted `UnitInstance`), `items: sourceItems.slice(0,3)` (first 3 of the combined source items, since the new unit still has the same `MAX_ITEMS_PER_UNIT` cap), and pushes any OVERFLOW items (`sourceItems.slice(3)`) back to `player.items` (the inventory) rather than discarding them. Places the new unit on the bench if there's room, else the first empty board slot (if doing so wouldn't exceed `player.level`'s board cap), else falls back to bench (which may exceed `benchMax` momentarily — net bench delta from a merge is always -2, so this is safe even at a full bench). Finally recurses (`tryAutoMerge(state, playerId, defId, data)` again) to cascade — a 3rd star-1 merge into a star-2 might combine with 2 EXISTING star-2 copies into a star-3, handled by the recursive call re-scanning from star 1 again (which will find none left, then check star 2 and find the cascade).
- **SELL**: finds the unit (bench or board) by uid, returns its equipped items to inventory UNCHANGED/opaque (never decomposed back into source components — same convention as UNEQUIP), removes the unit from its slot, returns `copies = gameplay.copiesPerStar[star]` pool copies (since a star-2/3 unit "contains" multiple pool copies), and refunds `def.tier * copies * sellValueMultiplier` gold (a 2-star unit, which absorbed 3 pool copies, refunds 3x the per-copy value; matches `sellValue.ts`'s client-side preview formula).
- **REROLL**: requires `gold >= economy.rerollCost`; deducts cost, returns every CURRENT shop slot's unit back to the pool, then calls `rollShop` (draws 5 fresh slots using the seeded `prng`).
- **BUY_XP**: requires `gold >= economy.xpBuyCost`; deducts cost, adds `economy.xpBuyAmount` to `xp`, recomputes `level` via `levelForXp`.
- **MOVE**: handles both bench↔board and board-reposition in one command. `toBench: true` requires bench room (`BENCH_FULL` otherwise), removes from source, clamps `toIndex` into `[0, bench.length]`, splices in. `toBench: false` (moving to a board slot) clamps `toIndex` into `[0, boardSlots-1]`, checks the target slot's current occupant: if EMPTY and the move is bench→board, enforces the level-derived board cap (`BOARD_FULL` if already at cap — board→board repositioning is exempt from this check since it doesn't change board count); removes the moving unit from its source, and if the target was occupied, SWAPS the occupant back into the moving unit's old slot (a true swap, not an overwrite-and-lose) — this is how the planning UI's drag-to-swap behavior is implemented at the rules layer.
- **EQUIP**: looks up the item in the player's loose inventory (`player.items`), resolves the target unit. **Auto-combine check FIRST**: if any item the unit ALREADY holds forms a valid recipe pair with the incoming item (`recipeResult(held, cmd.itemId, ...)`), the held item slot is overwritten in-place with the completed result and the incoming item is removed from inventory — net equipped-slot count is UNCHANGED, so this path is allowed even if the unit is already at `MAX_ITEMS_PER_UNIT` (this is the documented "auto-combines in place... allowed even at the cap" behavior from CLAUDE.md). Only if no auto-combine match exists does the function check the slot cap (`ITEM_SLOTS_FULL` if full) and otherwise simply append the item to the unit's `items` array and remove it from inventory.
- **UNEQUIP**: removes one instance of `itemId` from the unit's `items` (by id match, first occurrence — `ITEM_NOT_FOUND` if absent) and pushes it back to the player's loose inventory.
- **COMBINE_ITEMS**: requires both `itemIdA` and `itemIdB` to be FINDABLE AS DISTINCT INVENTORY ENTRIES (the second lookup explicitly excludes the first's index: `findIndex((id, i) => i !== idxA && id === itemIdB)` — this matters for combining two copies of the SAME component id, e.g. two `iron_sword`s, which is a legal recipe input if such a recipe exists). Resolves `recipeResult(itemIdA, itemIdB, data.items)` — `NO_RECIPE` if null. Removes both entries (by index, higher index first to avoid shifting issues) and pushes the resulting completed item id to inventory.
- **USE_CONSUMABLE**: resolves the consumable in inventory and validates it's actually `itemKind() === "consumable"`; resolves the target unit. Branches on `consumableDef.consumableEffect`:
  - `"remove_item"`: strips ALL items off the target unit back to inventory, consumes the consumable. Zero equipped items is a NORMAL no-op success (`{ok:true}`) that does NOT consume the consumable — an explicit design choice so using it on an empty unit doesn't waste the item.
  - `"reforge"`: independently re-rolls EACH equipped item to a DIFFERENT random item of the SAME tier (excluding consumables and the current id itself), one `prng()` draw per item in equipped-slot order (deterministic given the shared match prng) — `NO_ALTERNATIVE_ITEM` if no same-tier alternative exists for some slot (partial failure: earlier slots in the loop may have already been mutated before hitting this error — NOT transactional/rolled-back, a footgun if iteration order matters to a caller). Zero items is also a no-op success without consuming.
  - `"radiant_upgrade"`: requires the target unit to have at least one non-radiant, base completed (tier-2) item equipped (`NO_TIER_2_ITEMS_EQUIPPED` otherwise); requires `targetItemId` to name a specific equipped item (`ITEM_NOT_EQUIPPED` if not found on the unit); that item must itself be a base completed item, not a component/consumable/already-radiant variant (`NOT_TIER_2_ITEM` otherwise); calls `getOrCreateRadiantItem` (lazily materializes/caches the radiant variant) and swaps the equipped slot to the radiant id, consumes the consumable.
  - `default` (unrecognized `consumableEffect`): `CONSUMABLE_NOT_FOUND`.

**Invariants & constraints**
- Every branch returns a `CommandResult`, NEVER throws — this is the literal embodiment of CLAUDE.md's "Command legality is enforced server-side... invalid commands return typed errors, never throw."
- All commands require `state.phase === "PLANNING"` — no exceptions; combat-phase or resolution-phase command attempts are uniformly rejected.
- Pool conservation is maintained here: BUY draws nothing new from the pool (already drawn at shop-roll time), SELL returns `copiesPerStar[star]` copies, REROLL returns the OLD shop's units before rolling a new one — every gold-for-unit/unit-for-gold exchange is pool-balanced.
- Items are NEVER pooled — EQUIP/UNEQUIP/COMBINE_ITEMS/consumable effects only move item ids between `player.items` (loose inventory) and `unit.items` (equipped), never touch `state.pool`.
- `reforge`'s per-item independent mutation is NOT transactional — a mid-loop `NO_ALTERNATIVE_ITEM` failure leaves earlier slots already reforged while later slots remain unchanged, returning an error result even though SOME state was mutated. Any caller treating a `{ok:false}` result as "nothing happened" for this command would be wrong; this is a real edge-case footgun worth flagging to anyone touching this command.
- `tryAutoMerge`'s base-stat construction for the new star unit does NOT apply star multipliers (1.8x/3.24x) — those live purely in the sim layer (`engine.ts`'s `applyStarMultipliers`), applied transiently at combat start; the persisted `UnitInstance.hp`/`ad`/etc. always reflect base (1-star-equivalent) stats regardless of the unit's actual `star` field.
- `MAX_ITEMS_PER_UNIT` is checked everywhere items are added to a unit EXCEPT the auto-combine path in EQUIP (by design — net slots unchanged) and the new-unit-from-merge path in `tryAutoMerge` (which explicitly slices to 3, discarding nothing — overflow goes to inventory, never lost).
- `findUnitAnywhere` searches bench first, then board — if a uid somehow existed in both (shouldn't happen, invariant violation elsewhere), bench wins silently.

**Depends on**
- `@autobattler/data` — `GameData` type, `recipeResult`/`itemKind`/`itemTier`/`getOrCreateRadiantItem` (all pure helpers from `loader.ts`).
- `./state.js` — `MatchState` type (the structure being mutated).
- `@autobattler/sim/src/types.js` — `UnitInstance` type (the shape of bench/board entries).
- `@autobattler/sim/src/prng.js` — `Prng` type (threaded through to `rollShop` and the `reforge` consumable's random picks).
- `./pool.js` — `returnToPool` (SELL, REROLL).
- `./shop.js` — `rollShop` (REROLL).
- `./economy.js` — `levelForXp` (BUY_XP).

**Used by**
- `packages/server/src/room.ts` — every validated C2S `CMD` message is forwarded to `applyCommand`; rejections become typed `ERROR`/`COMMAND_REJECTED` responses (or are simply not broadcast as a state delta).
- `packages/client/src/driver.ts` (`LocalDriver`) — `playerCommand` calls `applyCommand` directly for Practice mode (no network round-trip).
- `packages/rules/src/ai.ts` — `applyAiCommands` issues BUY_XP/BUY/MOVE via this same function.
- `packages/rules/tests/commands.test.ts`, `combine.test.ts`, `items.test.ts`, `consumables.test.ts`, `magicNumbers.test.ts` — direct test coverage of every command branch.

**Notes** — The `tryAutoMerge` recursion has no explicit depth guard, but is naturally bounded: each call either returns immediately (no 3+ copies found at either star) or removes exactly 3 units and adds 1, monotonically reducing the population at the current star while only ever creating star 2 or star 3 (never higher — there's no cascading beyond 3-star since the function only checks `targetStar` 1 and 2, never 3, so a 3-star unit can never auto-merge further; this matches the project's documented star cap). The `INVALID_POSITION` error code exists in the `CommandError` union but no current code path in this file returns it — it's reserved, possibly for a future stricter MOVE validation.
