# packages/rules/src/shop.ts

**Path & purpose** â€” `packages/rules/src/shop.ts`. Rolls a player's shop: picks a tier per slot using the level-based `shopOdds` table, then picks a specific unit def of that tier weighted by pool availability, drawing each rolled unit from the pool immediately (not at purchase time).

**Responsibility** â€” Owns the entire shop-roll algorithm: tier selection (level-gated odds), unit selection within a tier (uniform among pool-available candidates), and the pool draw that happens AT ROLL TIME (not at BUY time â€” `commands.ts`'s BUY never calls `drawFromPool` itself, relying entirely on this file having already drawn the copy when the shop was rolled).

**Exports** â€” `rollShop(state, playerId, prng, data): void` â€” the sole export. Replaces `player.shop` entirely with a freshly rolled set of `data.economy.shopSlots` slots (5 by default), each either a `ShopSlot {defId, tier}` or `null` (if the roll's tier/pool combination yields nothing available).

**Key behavior**
- `rollTier(prng, level, data)` (internal): looks up `data.economy.shopOdds[level-1]` (clamped: `rowIdx = min(level-1, odds.length-1)`, so a level beyond the configured table just reuses the LAST row â€” there's no "level too high" error, it gracefully plateaus), where each row is an array of weights, ONE WEIGHT PER TIER (tier 1 = `row[0]`, tier 2 = `row[1]`, etc.). Sums the row, draws `roll = prng() % rowSum`, then linear-scans accumulating weights until `roll < cumulative`, returning that tier (1-based: `t+1`). Falls back to `row.length` (the last/highest tier) if the loop somehow exhausts (defensive, shouldn't trigger since `roll < rowSum` always holds).
- `pickDefIdForTier(prng, tier, pool, data)` (internal): filters `data.units` to those of the EXACT rolled `tier` AND with `pool.get(id) > 0` (still available) â€” candidates with zero pool count are EXCLUDED from the random pick entirely (not picked-then-rejected; genuinely never considered), so the shop's effective per-unit odds within a tier shift dynamically as the match progresses and units deplete. Returns `null` if NO unit of that tier has any pool copies left (extremely rare in practice â€” would require an entire tier being exhausted across the whole lobby). Picks uniformly at random among the remaining candidates (`prng() % candidates.length`) â€” NOT weighted further by remaining count, so a unit with 1 copy left and a unit with 20 copies left are equally likely to be picked once both qualify as candidates (the pool-availability filter is binary: "any left at all," not proportional).
- `rollShop`'s main loop: for each of the 5 slots, rolls a tier, then picks a defId for that tier, then IMMEDIATELY calls `drawFromPool(state.pool, defId)` â€” if the draw SUCCEEDS (pool had a copy, which it should given `pickDefIdForTier` already filtered for availability â€” but there's a subtle race within the SAME roll: two slots in the SAME call could both pick the SAME defId if its pool count is `>= 2` at pick-time but the FIRST slot's draw then reduces it; the SECOND slot's draw call re-checks fresh), the slot becomes `{defId, tier}`. If the draw fails (pool now empty for that id, e.g. it had exactly 1 copy and a PRIOR slot in this same roll already claimed it), the slot becomes `null` instead â€” so a single `rollShop` call CAN produce fewer than 5 actual unit slots if pool scarcity bites mid-roll, even though `pickDefIdForTier` itself always re-filters fresh per slot.
- Every drawn unit's pool copy is taken out IMMEDIATELY at roll time â€” `state.pool` reflects "what's NOT currently sitting in someone's shop" at all times between rolls. Unsold shop contents are returned to the pool only when something explicitly triggers it: `commands.ts`'s REROLL (returns old shop before rolling new), `match.ts`'s RESOLUTIONâ†’PLANNING shop refresh (same pattern), or `rounds.ts`'s elimination cleanup (returns a dead player's leftover shop slots).

**Invariants & constraints**
- Pool conservation depends on this file's draw-at-roll-time behavior pairing correctly with EVERY caller's return-before-reroll behavior â€” if any future caller ever calls `rollShop` WITHOUT first returning the player's current shop to the pool, units would be double-counted as "out" and the pool would shrink incorrectly (a real footgun for anyone adding a new shop-refresh call site).
- A `null` shop slot (from a failed pool draw, either at `pickDefIdForTier`'s top-level "no candidates" check or the rarer same-roll race) is a legitimate, displayable state â€” the client must handle an empty/null shop slot gracefully (BUY against a null slot returns `EMPTY_SLOT` per `commands.ts`).
- All randomness flows through the passed-in seeded `Prng` â€” no `Math.random` â€” preserving determinism for the SAME match/round/player-call-order.
- `rollTier`'s odds table is PURELY a function of player LEVEL, not round number or any other state â€” two players at the same level always draw tiers from the identical weight distribution (though the ACTUAL rolled outcome differs per player/slot since each draws from its own position in the shared prng stream).

**Depends on** â€” `@autobattler/data`'s `GameData` (`data.economy.shopOdds`/`shopSlots`, `data.units`); `./state.js`'s `MatchState`/`ShopSlot` types; `@autobattler/sim/src/prng.js`'s `Prng` type; `./pool.js`'s `drawFromPool`.

**Used by**
- `packages/rules/src/match.ts` â€” `createMatch` (initial shop rolls for all players) and `advancePhase`'s RESOLUTIONâ†’PLANNING branch (shop refresh for alive players each round).
- `packages/rules/src/commands.ts` â€” REROLL command handler.
- `packages/rules/tests/shop.test.ts` â€” direct coverage of tier odds, pool-availability filtering, and the draw-at-roll-time behavior.

**Notes** â€” The within-roll same-defId race (two slots in one `rollShop` call both wanting the same scarce unit, with only the first succeeding) is real but benign â€” it simply means a heavily-depleted tier can produce fewer non-null slots than usual near the end of a match where many copies have been bought up, which is the intended scarcity pressure of a shared pool, not a bug.
