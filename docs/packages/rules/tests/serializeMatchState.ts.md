# packages/rules/tests/serializeMatchState.ts

**Path & purpose** — `packages/rules/tests/serializeMatchState.ts`. A shared TEST HELPER module (not itself a `.test.ts` suite — no `describe`/`it` blocks) providing `serializeMatchState`, a `Map`/`Set`-aware JSON serializer for `MatchState`, used by determinism tests across the `rules` test suite to produce comparable string snapshots of full match state.

**Responsibility** — Solves a specific gotcha: `JSON.stringify` silently DROPS `Map` and `Set` fields (serializing them as `{}`), so any naive `JSON.stringify(state)` comparison would be BLIND to differences in `MatchState`'s several Map-typed fields (`pool`, `pairingHistory`, `lastCombatResults`, `lastOpponentBoards`, `lastRoundResult`, `lastLootOrbs`) — two states that differ ONLY in, say, `pairingHistory` contents would falsely compare as equal. This module exists purely to make full-state determinism comparisons actually exhaustive.

**Exports**
- `serializeMatchState(state: MatchState): string` — `JSON.stringify(state, mapAwareReplacer)`. Produces a deterministic-order JSON string representation of a full `MatchState`, with every `Map` field converted to `{__map: [...entries()]}` and every `Set` field to `{__set: [...values()]}` before serialization, so their contents participate in string equality comparisons.

**Key behavior** — `mapAwareReplacer(_key, value)` is a `JSON.stringify` REPLACER function: for any value that's `instanceof Map`, returns a wrapper object `{__map: [[k1,v1],[k2,v2],...]}` (the Map's entries as an array of pairs, in iteration order); for any `instanceof Set`, returns `{__set: [v1,v2,...]}` (values in iteration order); for everything else, returns the value unchanged (default `JSON.stringify` behavior applies recursively).

**Invariants & constraints** — Because Map/Set ITERATION ORDER matters for this serialization (two Maps with the same entries inserted in a different order would serialize differently, and thus compare as "different" even though they're semantically equal as sets of key-value pairs), this helper is appropriate ONLY for determinism tests where the goal is "does running the exact same operation twice produce byte-identical internal state" (where insertion order should ALSO be deterministic if the underlying logic is truly deterministic) — it is NOT a general-purpose semantic-equality helper for comparing two states that are expected to be equal but built via different code paths with potentially different iteration orders. This distinction matters: it's correctly used in `match.test.ts`/`pve.test.ts` to compare two runs of the SAME deterministic operation, never to compare two states that should be equal "in spirit" but built differently.

**Depends on** — `../src/state.js` (`MatchState` type, for the parameter's type annotation only — no runtime import beyond the type).

**Used by** — `packages/rules/tests/match.test.ts` (the 50-run and AI-driven-match determinism tests, imported as `serializeState`); referenced conceptually by `pve.test.ts`'s determinism test (which instead does its own targeted `JSON.stringify` on individual fields like `lastCombatResults.get(i)` rather than the whole state — so not every determinism test in the suite uses this particular helper, only the full-state ones in `match.test.ts`).

**Notes** — This file lives in `tests/` and is NAMED like a test file but contains NO test cases itself (no `describe`/`it`) — it's a pure helper module that happens to be colocated with its only consumers rather than living in `src/` (a deliberate choice since it's a TEST-comparison concern, not a production serialization need — `MatchState` itself has no "official" serialization format defined in `src/`, e.g. for network transport that's handled entirely separately by `packages/protocol`). A future agent searching for "where does production code serialize MatchState for the wire" should NOT look here — this is test-only tooling.
