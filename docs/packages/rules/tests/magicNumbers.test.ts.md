# packages/rules/tests/magicNumbers.test.ts

**Path & purpose** — `packages/rules/tests/magicNumbers.test.ts`. A STATIC-ANALYSIS test (not a behavioral test): scans every `.ts` source file in `packages/rules/src` AND `packages/sim/src` for numeric literals, failing if any literal outside a small whitelist appears without an explicit `magic-ok` justification comment on that line. This is the automated enforcement of CLAUDE.md's "all tuning numbers live in packages/data" hard invariant.

**Responsibility** — Owns the project-wide guarantee that gameplay-tunable numbers can NEVER be hardcoded directly into `sim`/`rules` source — every such number must instead live in a `packages/data` JSON file and be read via `data.economy`/`data.gameplay`/etc. This test is a structural/architectural guard, not a correctness check of any particular function's output.

**Exports** — None (test file; pure side-effecting test runner).

**Key behavior**
- `SRC_DIRS` — scans exactly two directories: `packages/rules/src` and `packages/sim/src` (resolved relative to this test file's own location via `import.meta.dirname`) — notably does NOT scan `packages/protocol`, `packages/server`, `packages/client`, or `packages/balance` source, since the "no magic numbers" invariant is specifically scoped to the two PURE deterministic-core packages (`sim` and `rules`) where every tuning number must be data-driven; other packages (UI layout constants, server ports, etc.) are explicitly out of scope.
- `SKIP_FILES = {"prng.ts", "fixed.ts"}` — these two `sim` files are EXEMPTED ENTIRELY from scanning, not just whitelisted for specific values — `prng.ts` (the mulberry32 algorithm's magic constants) and `fixed.ts` (the fixed-point `SCALE = 1000` definition itself) are considered structural/foundational, not gameplay tuning, and reviewing them line-by-line for "violations" wouldn't be meaningful.
- `EXTRA_ALLOWED = {"hex.ts": [7, 8]}` — `hex.ts` gets a PER-FILE whitelist extension allowing literals `7` and `8` (the structural 7×8 axial hex grid dimensions — board column/row counts, considered structural geometry, not balance tuning).
- `ALLOWED = {0, 1, 2, 3}` — the GLOBAL whitelist applied to every scanned file: these four small integers are permitted ANYWHERE (loop indices, team ids 0/1, star-type union members 1/2/3, `+1`/`-1` math, array bounds) without needing a `magic-ok` comment, since they're so structurally pervasive that requiring justification on every occurrence would be noise rather than signal.
- `magic-ok` ESCAPE HATCH: ANY line containing the literal substring `"magic-ok"` anywhere on it is SKIPPED ENTIRELY (not just the flagged number — the WHOLE LINE is exempted) — this is the mechanism behind every `// magic-ok: <justification>` comment seen throughout `sim`/`rules` source (e.g. `rounds.ts`'s stage-formula constants, `commands.ts`'s `MAX_ITEMS_PER_UNIT`, `match.ts`'s `nextUid` start value and `runMatchToEnd`'s safeguard, `rounds.ts`'s hash-mix constants in `derivePairingSeed`) — every one of those comments exists SPECIFICALLY to satisfy this test, not just for human readability.
- **`findViolations(file)` algorithm**: reads the file, strips BLOCK comments (`/* ... */`, replacing their content with spaces to preserve line numbers — so a numeric literal INSIDE a block comment is never flagged) via `stripBlockComments`, then processes line-by-line: skips any line containing `magic-ok`; strips trailing LINE comments (`//...`) and STRING LITERALS (single/double/backtick-quoted, replaced with `""` so a number appearing inside a string — e.g. an error message — is never flagged) from the code portion; then regex-scans for numeric literals (`0x[hex]` or decimal, optionally with a fractional part) using a NEGATIVE LOOKBEHIND `(?<![\w$.])` to avoid matching numbers that are part of an identifier (e.g. avoiding false positives on something like `v2` — though that's a letter-digit pattern, not a real concern, the lookbehind primarily guards against matching the fractional part of a larger number or a suffix of an identifier ending in digits) or a property-access-like context. Each matched literal is compared against the per-file `allowed` set (global `ALLOWED` ∪ that file's `EXTRA_ALLOWED` entry if any); anything not in that set is recorded as a violation string `"<file>:<lineNum> literal <value> in: <trimmed line>"`.
- **Test generation**: dynamically `readdirSync`s both `SRC_DIRS`, filters to `.ts` files, skips `SKIP_FILES`, and generates ONE `it(...)` test PER REMAINING FILE — so adding a new file to `sim/src` or `rules/src` automatically gets its own magic-number check with zero changes needed to this test file itself.

**Invariants & constraints**
- This test enforces a REAL architectural discipline, not just a style preference: any future contributor adding a new combat constant, economy formula coefficient, or structural threshold directly into `sim`/`rules` code (rather than `packages/data`'s JSON) will have this test fail immediately, forcing the choice between (a) moving the number into data where it belongs, or (b) adding a `magic-ok` comment with justification if it's genuinely structural/non-tuning (a deliberate, reviewable escape valve, not a way to silently bypass the rule).
- The whitelist is DELIBERATELY narrow (`0,1,2,3` plus two fully-exempted files plus one file's two extra values) — this means even seemingly innocuous numbers like `4`, `5`, `100`, `1000` (the fixed-point scale, outside `fixed.ts`) are FLAGGED if they appear unjustified — contributors should expect to need a `magic-ok` comment for almost any number beyond the smallest indices.
- Since `magic-ok` exempts the ENTIRE LINE (not the specific literal), a line with a `magic-ok` comment could theoretically hide an UNRELATED unjustified number elsewhere on the same line — this is a known, accepted looseness in the check (per-line granularity, not per-literal), trading a small blind spot for simplicity.
- The regex-based string/comment stripping is a heuristic, not a full TypeScript parser — it could theoretically mishandle unusual edge cases (e.g. escaped quotes inside template literals with nested expressions), but in practice the codebase's actual style doesn't hit such edge cases, so this lightweight approach suffices without needing a real AST.

**Depends on** — `vitest`; Node's `fs` (`readFileSync`, `readdirSync`) and `path` (`join`, `basename`) — this is one of the few `rules`-package test files using Node I/O APIs directly (acceptable since TEST code, unlike `sim`/`rules` SOURCE code, isn't bound by the "pure, no I/O" invariant — only the deterministic core itself must avoid I/O, not its test harness).

**Used by** — Not imported elsewhere; runs under `npm test` as a project-wide architectural gate alongside the functional tests.

**Notes** — This is effectively a lightweight, custom-built lint rule implemented as a test rather than an ESLint plugin — a pragmatic choice given the narrow, project-specific nature of the rule (most generic "no-magic-numbers" ESLint rules don't support this level of per-file/per-token nuance without significant configuration). Any reader auditing `sim`/`rules` source for hardcoded tuning values can trust that if `npm test` passes, this specific invariant already holds across every file in those two packages — no separate manual audit needed.
