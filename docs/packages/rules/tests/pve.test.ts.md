# packages/rules/tests/pve.test.ts

**Path & purpose** — `packages/rules/tests/pve.test.ts`. The comprehensive suite for `rounds.ts`'s PvE subsystem (`stageForRound`, `isPveRound`, `pveStageForRound`, `buildMobBoard`, `previewPveStage`) plus `loot.ts`'s `generateLoot`/`applyLootOrb` — covers the stage/round-numbering formula exhaustively, seeded-deterministic loot generation, mob-board construction's pool-non-interaction, full PvE-round execution via `advancePhase`, and the side-effect-free preview API used for client-side scouting.

**Responsibility** — Owns the correctness proof for the ENTIRE PvE round-scheduling formula (which absolute round numbers are PvE vs PvP, across stages 1-5+) and the seeded-determinism guarantee for PvE loot (CLAUDE.md: "PvE loot is seeded-deterministic... same seed + config → identical orbs/rewards"). Also proves `previewPveStage` is truly side-effect-free (doesn't advance `nextUid`, doesn't touch the pool, returns structurally-identical results across repeated calls) — critical since it's designed to be called speculatively/repeatedly by a client wanting to show a scouting preview without committing any state.

**Exports** — None (test file). Helpers `makeUnit`/`fieldBoards` (same pattern as other PvE-adjacent test files: gives every player a `warrior` at board slot 0 and an `archer` at board slot 7, level 3). Module-level `KNOWN_PVE_ROUNDS = [1,2,3,7,10,14,17,21,24]` / `KNOWN_PVP_ROUNDS = [4,5,6,8,9,11,12,13,15,16,18,19,20,22,23]` — the canonical, exhaustively-derived round classification for rounds 1-24, reusable by any other test needing real PvE/PvP round numbers (already reused by `previewPveStage` tests in this same file).

**Key behavior / test coverage**

**`describe("stage formula")`** — pins the EXACT stage/round-numbering scheme:
- Stage 1 = rounds 1-3 (`roundInStage` 1-3, all PvE — `STAGE1_LEN=3` per `rounds.ts`).
- Stage 2 = rounds 4-10 (`roundInStage` 1-7); stage 3 = rounds 11-17; stage 4 = rounds 18-24; stage 5 = rounds 25-31 — confirms EVERY stage from 2 onward is exactly 7 rounds long (`STAGE_LEN=7`), and that stage 5 continues the SAME formula (no special-casing beyond stage 4 — the formula generalizes indefinitely).
- `isPveRound`: within stage 1, ALL 3 rounds are PvE; within stages 2+, ONLY `roundInStage` 4 and 7 are PvE (2 PvE rounds per 7-round stage) — confirmed both via the `KNOWN_PVE_ROUNDS`/`KNOWN_PVP_ROUNDS` membership tests AND via a fully parameterized loop over stages 2-4 checking every `roundInStage` 1-7 individually against the expected `ris===4 || ris===7` pattern.
- **"exhaustive check rounds 1-25"**: a literal boolean array for EVERY round 1-25, the strongest possible pin against any accidental change to the stage-boundary or PvE-position formula — this single test would catch virtually any regression in `stageForRound`/`isPveRound`.

**`describe("loot generation")`** — `generateLoot`/`applyLootOrb`:
- **Determinism**: `generateLoot(round, mulberry32(seed), data)` called twice with the SAME fresh prng instance (a NEW `mulberry32(0xabc)` each time, not a shared/advanced one) for rounds 1-3 produces byte-identical (`JSON.stringify` equal) results — the core seeded-determinism proof.
- **Orb count/rarity matches config**: for each of rounds 1-3, the number of orbs returned by `generateLoot` exactly matches the SUM of `count` across that round's `loot.roundDrops[round]` entries, and every orb's `rarity` is one of the rarities listed in that round's config (no orb materializes a rarity not configured for that round).
- **Reward validity**: over 500 iterations × 3 rounds (1500 total `generateLoot` calls sharing ONE continuously-advancing prng), every resolved gold reward has `amount > 0` and every item/component reward's `id` is a real id in `gameData.items` — a large-sample fuzz check that the weighted-table resolution never produces a malformed reward.
- **Rarity weighting sanity**: takes `gameData.loot.tables.common`, finds the single heaviest-weighted entry, then samples 4000 independent `generateLoot(1, ...)` calls (taking only the FIRST orb each time, since round 1 produces "2 common orbs" per the comment) and tallies which reward each one resolved to — confirms the heaviest-weight entry's observed count clearly exceeds any single weight-3 component entry's count, a statistical (not exact) proof that the weighted-pick logic actually respects relative weights rather than e.g. picking uniformly.
- **Coverage**: confirms `loot.roundDrops` has a non-empty entry for ALL of rounds 1,2,3,7,10,14,17,21,24 (every currently-defined PvE round through stage 4) — catches a missing loot config for any PvE round before it would surface as a runtime gap.
- **`applyLootOrb`**: feeding one gold orb (+4), one "item" orb (`iron_sword__sorcerer_rod`), one "component" orb (`tear_flask`) confirms gold accumulates numerically (`player.gold === 4`) while BOTH item and component reward kinds land in the SAME flat `player.items` array, in call order — confirms `applyLootOrb` doesn't distinguish "item" vs "component" reward kinds when applying to inventory (both are just pushed as ids), only "gold" is treated specially.

**`describe("PvE round")`** — mob/board mechanics + full round execution:
- **"mobs are not unit-pool defs and carry no traits"**: EVERY `gameData.mobs.mobs` entry has an `id` ABSENT from `gameData.units` (mobs are a disjoint id-space from pooled units — the mechanism by which `applyTraits` never counts mobs, per the sim's documented behavior) and an empty `traits: []` array.
- **"every defined PvE stage references real mobs"**: every `gameData.mobs.stages[].units[].mobId` resolves to a real entry in `gameData.mobs.mobs` — a referential-integrity check on the mob content data.
- **"pveStageForRound returns correct stage for all defined stage entries"**: for every stage definition in `mobs.json`, independently recomputes its absolute round number (mirroring the SAME formula under test elsewhere — `stage===1 ? roundInStage : 4+(stage-2)*7+roundInStage-1`) and confirms `pveStageForRound(round, data)` resolves back to a stage with the matching `name` — a round-trip consistency check between the formula and the actual stored stage list.
- **"pveStageForRound returns null on PvP rounds"**: rounds 4,5,6,8,9 (real PvP rounds) have no stage.
- **"builds the mob board on the enemy side without touching the unit pool"**: `buildMobBoard` produces a board with one unit per stage-defined slot, every unit's `team === 1` (the enemy/mob side), and — critically — the unit POOL's total count is UNCHANGED before/after the call (compared against the POST-`createMatch` pool total, accounting for the match's own starting-unit draws) — direct proof of "PvE mobs are NOT drawn from the unit pool" (CLAUDE.md).
- **"runs mob combat for every alive player, awards base gold + loot, and deals no HP damage"**: fields real boards for all 8 players, advances ONE phase (PLANNING→RESOLUTION, running the full PvE round), then confirms: `state.lastPairings` is EMPTY (no PvP pairings recorded during a PvE round — `rounds.ts` keeps it `[]`); every player's HP is UNCHANGED from before the round (PvE never damages player HP); every player has a `lastCombatResults` entry with a non-empty event log (real combat happened, even though there's no HP stake); every player's gold increased by at least `pveBaseGold`.
- **"is fully deterministic: same seed reproduces gold, inventory, and combat"**: running the identical `createMatch`+`fieldBoards`+`advancePhase` sequence TWICE with the SAME seed (2024) produces byte-identical gold, inventory, AND serialized combat results for every player — the PvE-specific analog of the broader full-match determinism suite in `match.test.ts`.

**`describe("previewPveStage")`** — the side-effect-free scouting API (the most rigorously tested function in this file):
- Returns `null` on a non-PvE round (tested both for round 4 specifically and exhaustively across all of `KNOWN_PVP_ROUNDS`).
- **Structural match against the REAL board**: builds the authoritative `buildMobBoard` result first (capturing `nextUid` before/after to confirm it DOES mutate/advance the uid counter, as expected for a real board), then calls `previewPveStage` and confirms `nextUid` is UNCHANGED by the preview call — then sorts both boards' units by position and asserts every stat field (`defId,tier,star,pos,hp,maxHp,ad,as,armor,mr,range,mana,maxMana,abilityDamage`) matches exactly between preview and real, EXCEPT `uid` (explicitly excluded — "uid differs by design, preview uses negative disposable ids").
- **"preview uids are all negative"**: every unit in a preview board has `uid < 0` — the disposable-id convention that lets a preview be discarded without polluting the real uid namespace.
- **"CRITICAL no-side-effect" test** (explicitly flagged as critical in its own name): calls `previewPveStage` THREE times in a row on the same state, confirms `state.nextUid` is identical to its pre-call snapshot after ALL three calls (not just unchanged between calls — literally never moved), and confirms all three returned boards are STRUCTURALLY IDENTICAL (same units/positions, JSON-compared after position-sorting) — proving repeated preview calls are not just side-effect-free individually but also mutually idempotent (same inputs always produce the same preview, since uid generation for the preview must itself be deterministic/reproducible rather than consuming any incrementing counter).
- **"preview is consistent across all defined PvE rounds with a known stage"**: loops every `KNOWN_PVE_ROUNDS` entry; if `mobs.json` has a stage defined for that round, `previewPveStage` returns a non-null board with the matching unit count; if NOT (a PvE round formula-wise, but lacking actual stage content — e.g. possibly round 24 or some later-stage round not yet authored), preview correctly returns `null` rather than erroring or returning an empty/malformed board.

**Invariants & constraints** — This file is the canonical reference for: (1) the EXACT PvE round numbers through round 31, (2) proof that PvE loot is genuinely seeded-deterministic at the per-orb level, (3) proof that mobs never touch the unit pool, (4) the full no-side-effect contract for `previewPveStage` (uid counter untouched, repeated calls idempotent, negative disposable uids) — any client/server code relying on calling `previewPveStage` speculatively (e.g. a scouting UI calling it on every frame without re-render-thrash concerns) can trust these guarantees are test-enforced.

**Notes** — The rarity-weighting test is STATISTICAL (not exact-value), so it carries a theoretical (extremely small in practice with N=4000 against a 40-vs-3 weight ratio) chance of flaking if the PRNG sequence happens to produce an outlier sample — accepted as a pragmatic tradeoff for testing weighted-random behavior without re-implementing the weighting math as a separate pure function to unit-test directly. The "preview is consistent" test's else-branch (PvE round with NO stage entry → preview null) implies `mobs.json` may legitimately NOT define a stage for every numerically-PvE round (e.g. if content for a later stage hasn't been authored yet) — `previewPveStage`'s null-on-missing-stage behavior is therefore a deliberate, tested fallback, not an oversight.
