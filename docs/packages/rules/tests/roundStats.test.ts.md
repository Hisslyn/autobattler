# packages/rules/tests/roundStats.test.ts

**Path & purpose** — `packages/rules/tests/roundStats.test.ts`. The dedicated suite for `rounds.ts`'s per-round result tracking (`MatchState.lastRoundResult`) and per-match accumulated stats (`PlayerState.roundWins`/`roundLosses`/`totalDamageDealt`/`totalDamageTaken`) — covers win/loss/draw/bye/PvE result classification, the winner-damageDealt-equals-loser-damageTaken invariant, accumulation across multiple rounds, and the "rebuilt fresh each round" no-staleness guarantee.

**Responsibility** — Verifies the entire per-round/per-match stats system described in CLAUDE.md's `rounds.ts` internals note: that `runCombatPhase` defaults every alive player to a bye, that winner/loser pairings overwrite that default correctly with matching damage terms, that draws and the ghost-source's "third wheel" case correctly STAY at the bye default (no W/L credited), that PvE rounds mark every alive player `pve` with zeroed damage and no accumulator changes, and that the per-match accumulators (`roundWins` etc.) only grow on real PvP wins/losses, never on byes/draws/PvE.

**Exports** — None (test file). Helper `makeUnit(uid, defId, team, q, r)` (parameterized position, unlike most other test files' fixed `{q:0,r:0}` — needed here since boards are explicitly multi-unit). Key helper `twoPlayerWin(seed, a, b)`: creates a match, forces `round = 4` (the first real PvP round — stage 2, roundInStage 1, since stage 1's rounds 1-3 are all PvE), sets ONLY players `a` and `b` alive (so `buildPairings` is forced to pair exactly those two together), gives player `a` a level-5 3-unit board (warrior+archer+paladin) and player `b` a COMPLETELY EMPTY board (`new Array(28).fill(null)`) — guaranteeing `a` wins deterministically regardless of seed, since an empty board can't fight back. This is a clean, reusable pattern for any test needing a deterministic win/loss outcome without depending on combat RNG.

**Key behavior / test coverage**

**`describe("per-round combat stats")`**:
- **"records a win for the winner and a loss for the loser with matching damage"**: after `runCombatPhase`, the winner's `lastRoundResult` has `status:"won", damageTaken:0, damageDealt>0`; the loser's has `status:"lost", damageDealt:0, damageTaken>0`; and CRITICALLY `winner.damageDealt === loser.damageTaken` — the same survivor-based damage term used on both sides of the pairing (per `rounds.ts`'s `calcPlayerDamage`).
- **"applies the loser's damageTaken to their HP"**: confirms the loser's ACTUAL HP delta (`hpBefore - hpAfter`) equals exactly `loser.damageTaken` as recorded in `lastRoundResult` — proves the stats-tracking number isn't just a cosmetic display value but the literal amount subtracted from HP.
- **"works regardless of which side (A or B) wins"**: pairs seats 5 and 3 with player 5 holding the stacked board (so 5, not necessarily the "A-side", wins) — confirms result classification is correctly attributed by PLAYER ID, not by which side of the pairing tuple happened to win (a sanity check against an A/B-side mixup bug).

**`describe("match-stat accumulation")`**:
- **"accumulates roundWins/roundLosses and total damage across rounds"**: runs `runCombatPhase` TWICE (round 4, then round 5, same two-player setup each time) and confirms `roundWins`/`roundLosses` increment to 2 each (not reset to 1) and `totalDamageDealt`/`totalDamageTaken` STRICTLY INCREASE (not reset) — proving these are running accumulators across the whole match, not per-round snapshots. Also reconfirms the winner/loser damage-term equality holds independently on the second round.
- **"starts every player at zeroed accumulators"**: fresh `createMatch` gives every player `roundWins=roundLosses=totalDamageTaken=totalDamageDealt=0` — a basic initialization sanity check.

**`describe("lastRoundResult correctness")`** — the most behaviorally subtle sub-suite, covering the various NON-win/loss classification paths:
- **"marks an unpaired alive player (ghost-side real player) as a bye, 0/0"**: with exactly 3 alive players (forcing `buildPairings` to create one real pairing + one ghost fight, leaving one alive player serving ONLY as the ghost's "source" — i.e. their `lastBoard` is borrowed for the ghost fight but THEY are not actually a participant this round), confirms at least one of the three players' results is `status:"bye"` with `damageTaken:0, damageDealt:0` — the bye default was never overwritten for that player since they weren't a real participant.
- **"a draw leaves both players at the bye-equivalent 0/0 (no W/L credited)"**: forces both paired players to have COMPLETELY EMPTY boards (guaranteeing a draw — no survivors on either side) — confirms both ends up with `status:"bye"` (NOT a distinct "draw" status at the `lastRoundResult`/`PlayerState` level — the test comment explicitly notes "the default, never overwritten on a draw"), zeroed damage, and zero `roundWins`/`roundLosses` change for either — confirms a draw is functionally indistinguishable from a bye at the per-round-result/accumulator level (though `getPairingFor`'s `outcome` field, tested separately in `pairing.test.ts`, DOES expose `"draw"` distinctly — that's a different, view-layer-only distinction).
- **"PvE marks every alive player as pve with 0/0 and no W/L"**: calls `runPveRound` directly (not via `advancePhase`) with a fresh `mulberry32(state.prngState)` — confirms every player's result is `status:"pve"`, zeroed damage, AND zero accumulator change (`roundWins`/`roundLosses`/`totalDamageTaken`/`totalDamageDealt` all stay 0) — the strongest confirmation that PvE rounds are completely excluded from match-stat accounting, matching CLAUDE.md's explicit note "bye/PvE never count as a W/L or accrue damage."
- **"lastRoundResult is rebuilt fresh each round (no stale entries)"**: runs a real PvP round (round 4, getting `status:"won"` for player 0), then switches to round 7 (a PvE round — stage 2, roundInStage 4) and runs `runPveRound` — confirms BOTH players' `lastRoundResult` entries are now `"pve"`, i.e. the PREVIOUS round's "won" status doesn't linger/leak into the new round's map — `lastRoundResult` is a fresh `Map` each round, not incrementally patched.

**Invariants & constraints** — This file is the authoritative behavioral proof for the `RoundResult`/per-match-accumulator system described in `rounds.ts`'s and `state.ts`'s CLAUDE.md notes. Key non-obvious facts pinned here: (1) draws and "bye via ghost-source" are LITERALLY the same status value (`"bye"`) at this layer — there is no separate "draw" `RoundResultStatus`; (2) PvE rounds use a DIFFERENT direct call path (`runPveRound`, not `runCombatPhase`) and a reader must call the right one depending on whether the round is PvE (`isPveRound`) — the tests model this exactly by manually choosing which function to call based on the round number; (3) `winner.damageDealt === loser.damageTaken` is enforced as an invariant tested from multiple angles (direct equality check, HP-delta cross-check, and re-verified on a second round).

**Depends on** — `vitest`; `@autobattler/data` (`gameData`); `../src/match.js` (`createMatch`); `../src/rounds.js` (`runCombatPhase`, `runPveRound`); `@autobattler/sim/src/prng.js` (`mulberry32`); `../src/state.js` (`MatchState` type); `@autobattler/sim/src/types.js` (`UnitInstance` type).

**Used by** — Not imported elsewhere; runs under `npm test`.

**Notes** — The PvE tests construct their own `mulberry32(state.prngState)` rather than letting `advancePhase` manage the PRNG checkpoint internally — this directly calls `runPveRound`, bypassing the normal `match.ts` phase-transition flow entirely; a reader adapting this pattern should remember that in PRODUCTION code (`advancePhase`), the PRNG-checkpoint-and-reseed pattern documented in `match.ts`'s own doc is what actually manages this, and these tests are intentionally reaching one layer deeper than normal calling code would.
