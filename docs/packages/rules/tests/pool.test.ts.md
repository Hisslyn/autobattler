# packages/rules/tests/pool.test.ts

**Path & purpose** ŌĆö `packages/rules/tests/pool.test.ts`. The dedicated suite proving CLAUDE.md's "pool conservation" hard invariant ("total unit copies across pool + all player benches/boards is constant") holds across every code path that moves units: initial match setup, BUY/SELL/REROLL, per-round shop refresh, player elimination, a full AI-driven match, and the 3-copy merge cascade (including item preservation through merges).

**Responsibility** ŌĆö Owns the project's correctness guarantee that units are never duplicated or destroyed by accident anywhere in the simulated economy ŌĆö every unit copy is either sitting in the shared pool, sitting in a shop slot (drawn at roll time), or sitting on some player's bench/board (counted by star-weighted "copies" ŌĆö 1/3/9 for star 1/2/3, reflecting that a 2-star consumed 3 base copies and a 3-star consumed 9).

**Exports** ŌĆö None (test file). Internal helpers:
- `totalInPool(pool)` ŌĆö sums all values in a `Map<string, number>` (defId ŌåÆ count).
- `totalInMatch(state)` ŌĆö the FULL conservation accounting function: pool total + for every player, every bench/board unit weighted by `star === 1 ? 1 : star === 2 ? 3 : 9` (the copies-per-star cost) + every non-null shop slot counted as `+1` (since a shop slot already drew its copy from the pool at roll time, per the comment "Shop slots ARE drawn from the pool at roll time"). This is the single function used by EVERY conservation assertion in this file ŌĆö any future change to the conservation accounting (e.g. a new place a unit copy can "live") would need this helper updated.
- `initialTotal()` ŌĆö sums `gameData.economy.poolCounts[tier]` across all `gameData.units` (i.e. recomputes what the starting total SHOULD be directly from data, independent of `buildInitialPool`'s own logic ŌĆö a true independent cross-check).

**Key behavior / test coverage**
- **"initial total matches data counts"**: `buildInitialPool`'s total equals the independently-computed `initialTotal()`.
- **"draw+return is conservative"**: one `drawFromPool`+`returnToPool` round-trip returns the pool to its exact starting total.
- **"initial shop rolls and starting bench units draw from the pool, total stays constant"**: at match creation, the pool is reduced by exactly (shop draws + starting bench draws), but `totalInMatch` (which counts shop+bench+pool together) still equals `initialTotal()` ŌĆö proves shop/bench draws don't create or destroy copies, only relocate them.
- **"every player starts with exactly 1 bench unit of the configured startingUnitId"**: confirms `gameplay.startingUnitId` (data-driven, not hardcoded) is what every player's single starting bench unit resolves to, at star 1.
- **"pool conservation holds at match start"**: a second confirmation of the conservation total immediately after `createMatch`, with a different seed (42) ŌĆö a basic regression pin independent of the more detailed test above.
- **"buy/sell preserves total unit count"**: BUY (drawing a shop slot, already counted) then SELL (the same unit, returning it to the pool) round-trips `totalInMatch` back to its pre-action value at EACH step (not just net) ŌĆö proves neither BUY nor SELL alone breaks conservation.
- **"REROLL preserves the total"**: 10 consecutive REROLLs (returning the old shop to pool, drawing a fresh one) each individually preserve `totalInMatch` ŌĆö stresses the reroll path specifically, since it's the highest-churn pool operation in normal play.
- **"per-round shop refresh preserves the total"**: across one full PLANNINGŌåÆRESOLUTIONŌåÆPLANNING round cycle (two `advancePhase` calls), `totalInMatch` is checked after EACH phase transition ŌĆö covers the automatic shop-refresh-on-RESOLUTIONŌåÆPLANNING path (not a player-triggered REROLL, but the same underlying pool mechanics).
- **"elimination returns bench, board, and undrafted shop to the pool"**: manually fields a `shadowblade` board unit for every player (explicitly drawn from the pool via `drawFromPool` to keep accounting correct, not just constructed in-place ŌĆö the test is careful to maintain the invariant even while setting up its OWN fixture), sets every player except player 0 to 1 HP, then advances phases in a loop until match-over (capped at 200 iterations), asserting `totalInMatch` is preserved after EVERY single `advancePhase` call (the most granular per-step check in the file) ŌĆö and at the end confirms every eliminated player's `bench`/`board`/`shop` are all fully emptied (proving `returnUnitsToPool`, called at elimination per `rounds.ts`, actually clears the player's holdings, not just updates the pool counts independently ŌĆö a consistency check between the two sides of the same operation).
- **"invariant holds across a full AI-driven match"**: runs a complete `runAiMatch`-style loop (seed 1234, `applyAiCommands` for every alive player each PLANNING phase, `advancePhase` otherwise) and asserts `totalInMatch` is preserved after EVERY individual `applyAiCommands` call AND after every `advancePhase` call ŌĆö the broadest possible fuzz-style check, since the real AI policy can issue any legal command sequence (multiple buys, sells, moves, rerolls) and this proves the invariant survives ALL of them in combination, across a full match to completion.
- **`describe("merge cascade")` sub-suite** (3 tests, using locally-defined `makeUnit` helpers mirroring the pattern from other test files):
  - **"3x 1-star -> 2-star on bench"**: clears the player's starting bench unit (carefully RETURNING its pool copies first via `returnToPool`, scaled by `copiesPerStar[star]`, to keep the test's OWN setup conservation-clean), force-stocks the shop with 3 identical-defId slots (treated as pre-drawn, bypassing `rollShop`), then BUYs all three sequentially ŌĆö confirms the result is exactly ONE 2-star unit and ZERO leftover 1-stars (the merge fully consumes all three copies, no partial/duplicate state).
  - **"3x 2-star -> 3-star (cascade from bench+board mix)"**: starts with TWO 2-star units already on bench and ONE 2-star already on board (a mix of bench+board, not all in one place), then buys 3 MORE 1-star copies ŌĆö confirms `tryAutoMerge`'s cascade correctly recognizes that 3├Ś(2-star) is reachable (after the 3 new 1-stars first merge into a 4th 2-star) and produces exactly one 3-star, proving the merge logic scans BOTH bench and board together, not just one location.
  - **"merge preserves items: 3 sources with 2 items each -> merged has 3, inventory has 3"**: three 1-star source units (2 bench + 1 board) each hold 2 equipped items (6 total, deliberately overlapping ids: 2x `iron_sword`, 2x `chain_vest`, 2x `mana_crystal`), then calls `tryAutoMerge` DIRECTLY (not via BUY ŌĆö confirming `tryAutoMerge` is independently callable/testable apart from the BUY command flow) ŌĆö confirms the merge result keeps exactly 3 of the 6 items ON the merged unit and overflows the other 3 to the player's loose inventory (`player.items`), with the exact multiset of all 6 original item ids preserved across the merged-unit+inventory split (sorted-array equality check) ŌĆö this is the authoritative behavioral proof of `commands.ts`'s documented "items overflow to inventory beyond the 3-slot cap" merge rule, with the EXACT distribution (3 stay, 3 overflow) pinned for a 3-source merge.

**Invariants & constraints** ŌĆö This file is the most rigorous practical proof of the pool-conservation hard invariant from CLAUDE.md; a reader wanting CONCRETE confidence that a change hasn't broken conservation should run this suite specifically. The star-weighted "copies" accounting (1/3/9) is a manually-mirrored constant in `totalInMatch` ŌĆö if `gameplay.copiesPerStar` in `economy.json`/`gameplay.json` ever changes from the implicit 1/3/9 (3-merge) scheme, this test helper would need a matching update (it's NOT derived from data here, unlike the bench-clearing code in the merge-cascade test which DOES read `gameData.gameplay.copiesPerStar` dynamically ŌĆö an inconsistency between the conservation-total helper, which hardcodes `1/3/9`, and the merge-cascade setup code, which reads the same scheme from data).

**Depends on** ŌĆö `vitest`; `@autobattler/data` (`gameData`); `../src/pool.js` (`buildInitialPool`, `returnToPool`, `drawFromPool`); `../src/match.js` (`createMatch`, `advancePhase`, `isMatchOver`); `../src/commands.js` (`applyCommand`, `tryAutoMerge` ŌĆö note `tryAutoMerge` is imported and called DIRECTLY here, confirming it's exported from `commands.ts` for testability beyond just being an internal helper of the BUY/MOVE paths); `../src/ai.js` (`applyAiCommands`); `@autobattler/sim/src/prng.js` (`mulberry32`).

**Used by** ŌĆö Not imported elsewhere; runs under `npm test`.

**Notes** ŌĆö The elimination test's safeguard of 200 iterations (much smaller than other tests' 10000) reflects that with most players forced to 1 HP, the match should end in very few rounds ŌĆö a TIGHTER safeguard here actually serves as an implicit assertion that the engineered scenario produces eliminations promptly, not just an arbitrary infinite-loop guard. The merge-cascade tests bypass `rollShop` entirely by directly assigning `player.shop` arrays with raw `{defId, tier}` shop-slot objects ŌĆö a reusable pattern for any test needing deterministic, guaranteed-available shop contents without fighting the RNG-driven `rollShop`/pool-availability logic.
