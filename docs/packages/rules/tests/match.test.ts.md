# packages/rules/tests/match.test.ts

**Path & purpose** — `packages/rules/tests/match.test.ts`. The top-level, end-to-end determinism suite for a FULL match: drives the entire `createMatch`→`advancePhase`(×N)→match-over lifecycle (both via the convenience `runMatchToEnd` helper and via a hand-rolled AI-driven loop) and asserts byte-identical outcomes across repeated runs of the same seed, plus correct final-state invariants (exactly one survivor, exactly seven eliminated).

**Responsibility** — Owns the project's single most important regression guard: "same seed -> identical placement order" across a COMPLETE match (potentially dozens of rounds, all PvE/PvP pairing decisions, all combats, all AI decisions) — this is the test that would catch ANY accidental nondeterminism introduced anywhere in `rules` or `sim` (a stray `Math.random`, an unstable Map/Set iteration order, a floating-point operation, a Date.now() call, etc.), not just within one function.

**Exports** — None (test file). Internal helpers:
- `makeUnit(uid, defId, team)` — builds a fully-populated `UnitInstance` at 1-star from a `units.json` def (mirrors the same hand-construction pattern seen in other test files; copies `hp/ad/as/armor/mr/range/manaStart→mana/mana→maxMana/abilityDamage` directly off the def, zeroes `attackCooldown`/`statusEffects`/`items`).
- `setupBoards(state)` — a `BoardSetupFn` (matches `runMatchToEnd`'s optional third parameter's shape) that gives EVERY player a level of 3 and a single `shadowblade` (tier-3, described in-line as "kills fast") unit on their board — ensures every PvP combat in the resulting match produces a clear winner quickly rather than dragging into overtime/draws, keeping the test's combat outcomes decisive and the match short enough to run 50 times in one test.
- `runAiMatch(seed)` — an alternative match-driving loop NOT using `runMatchToEnd`: manually loops `while (!isMatchOver(state) && safeguard < 10000)`, and on every `PLANNING` phase calls `applyAiCommands` for every alive player (using a SINGLE shared `mulberry32(seed)` prng instance threaded across the whole match, advancing its internal state call-by-call) before calling `advancePhase`. This exercises the REAL AI decision path (full shop/buy/equip/move logic) rather than the trivial fixed `setupBoards`, at the cost of not controlling combat outcomes as tightly.
- Both helpers depend on `serializeMatchState` (from the sibling `serializeMatchState.ts` test helper, not yet documented in this pass) to produce a comparable string/JSON snapshot of the final `MatchState`.

**Key behavior / test coverage**
- **"same seed -> identical placement order, 50 runs"**: runs `runMatchToEnd(0xc0ffee, gameData, setupBoards)` once to get a baseline serialized string, then runs it 49 MORE times with the SAME seed and setup function, asserting EVERY one matches the baseline exactly — by far the strongest determinism check in the suite (50 independent full-match runs, not just 2).
- **"two same-seed AI-driven matches in one process are byte-identical (uids included)"**: runs `runAiMatch(0xfeed)` twice and compares serialized output — exercises the FULL AI policy + planning-phase command pipeline (not just combat/pairing), and explicitly calls out that UIDs are included in the comparison (i.e. `nextUid` allocation order must also be deterministic, not just final placements).
- **"different seeds produce different outcomes"**: sanity check that seeds 1 and 2 (via `runAiMatch`) produce DIFFERENT serialized results — guards against a degenerate bug where the simulation ignores the seed entirely and always produces the same outcome (which would make the "same seed" tests vacuously true).
- **"match ends with exactly 1 alive player"**: `runMatchToEnd(42, gameData, setupBoards)` → exactly 1 of 8 players has `alive === true` at the end.
- **"all 8 players get placements assigned or one is winner"**: `runMatchToEnd(7, gameData, setupBoards)` → exactly 7 players are eliminated (`!alive`) and `isMatchOver(state)` is true.

**Invariants & constraints** — This suite is the litmus test for the project's core promise ("shared deterministic simulation core") at the FULL MATCH granularity, not just unit/combat granularity — any change anywhere in `sim`/`rules` that could introduce nondeterminism should be expected to surface here even if it's missed by more narrowly-scoped tests. The 50-iteration first test is deliberately expensive/thorough; a reader adding a new source of randomness or iteration-order-dependent logic anywhere in the simulated path should expect this test to be the final, most sensitive check.

**Depends on** — `vitest`; `@autobattler/data` (`gameData`); `../src/match.js` (`createMatch`, `advancePhase`, `runMatchToEnd`, `isMatchOver`); `../src/ai.js` (`applyAiCommands`); `@autobattler/sim/src/prng.js` (`mulberry32`); `../src/state.js` (`MatchState` type); `@autobattler/sim/src/types.js` (`UnitInstance` type); `./serializeMatchState.js` (`serializeMatchState`, imported as `serializeState` — the shared serialization helper used to produce comparable snapshots, NOT itself a `.test.ts` file despite living in `tests/`).

**Used by** — Not imported elsewhere; runs under `npm test`.

**Notes** — `runAiMatch`'s safeguard (`safeguard < 10000`) is a defensive infinite-loop guard, not a meaningful gameplay limit — a real match should end in far fewer than 10000 phase-advances; hitting that cap would itself indicate a bug (a match that never reaches game-over) rather than legitimate long play. The two different match-driving strategies (`runMatchToEnd` + fixed `setupBoards` vs. the hand-rolled `runAiMatch` loop) deliberately give this suite coverage of both the "happy path" convenience helper AND the lower-level phase-by-phase API that a server `room.ts` would actually use in production.
