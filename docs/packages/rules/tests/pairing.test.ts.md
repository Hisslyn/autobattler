# packages/rules/tests/pairing.test.ts

**Path & purpose** — `packages/rules/tests/pairing.test.ts`. Targeted Vitest suite for `rounds.ts`'s `buildPairings` (anti-repeat opponent matching) and `getPairingFor` (per-player normalized view of a completed pairing) — the two most algorithmically subtle exports in `rounds.ts`.

**Responsibility** — Verifies the pairing algorithm's structural correctness (no self-pairs, no duplicate appearances, ghost-fight generation on odd counts), its anti-repeat GUARANTEE over a full round-robin-equivalent span (7 rounds among 8 players → exactly 7 distinct opponents, each met exactly once), its deterministic fallback tie-breaking when forced repeats are unavoidable, and `getPairingFor`'s perspective-normalization logic (win/loss/draw from either side of a stored pairing, ghost detection, null for an unpaired player).

**Exports** — None (test file).

**Key behavior / test coverage**
- **"no player is paired with themselves"** / **"each player appears at most once in pairings"**: basic structural sanity over one `buildPairings` call at match creation (8 players, no history yet) — every pairing tuple `[a, b]` has `a !== b`, and accumulating a `Set` of seen ids across all pairs never sees a repeat (covers both the `a` slot and, when `b >= 0` i.e. not a ghost marker, the `b` slot too).
- **"over 7 PvP rounds with 8 alive players, each player faces 7 distinct opponents"**: calls `buildPairings` 7 TIMES IN SEQUENCE on the same live `state` (confirming `buildPairings` itself mutates/accumulates `state.pairingHistory` as a side effect — the test comment explicitly notes "buildPairings records pairing history (meet counts) itself"), asserting each call returns exactly 4 pairs (8 players / 2), then asserts EVERY player's `pairingHistory` entry has exactly 7 distinct opponent keys (all 7 other players) each met with count exactly 1 — this is the strongest possible proof that with 8 players and 7 rounds, the algorithm achieves a PERFECT round-robin (everyone meets everyone exactly once, no repeats, no misses) — confirming the anti-repeat design works at its theoretical best case.
- **"fallback picks the least-met opponent, tiebreak lowest seat"** (two sub-scenarios):
  - 2-alive-players scenario: with only players 0 and 1 alive and a SEEDED prior history showing they've already met twice (`pairingHistory.get(0).get(1) === 2`), `buildPairings` is FORCED to re-pair them (no other choice) — confirms the meet COUNT increments correctly afterward (now 3) even on a forced repeat — i.e. `buildPairings` doesn't refuse to pair previously-met players when there's no alternative; it just prefers fresher matchups when there IS a choice.
  - 4-alive-players scenario (players 4,5,6,7): seeds asymmetric history where player 4 has met player 5 three times but players 6 and 7 only once each — confirms `buildPairings`'s fallback logic picks player 4's LEAST-met available opponent (NOT player 5, which it's met far more) — directly tests the "prefer unmet/least-met pairs" preference described in `rounds.ts`'s backtracking algorithm.
- **"ghost pairing created for odd-player-count"**: eliminates player 7 (setting `alive=false` AND `lastBoard={units:[]}` — the ghost source needs a `lastBoard` to fight against per `rounds.ts`'s ghost mechanism) to leave 7 alive players (odd), then confirms exactly ONE of the resulting pairs has a negative `b` value (the ghost marker convention — a negative id represents "vs the ghost of an eliminated player's last board," not a real live opponent).
- **`getPairingFor` sub-suite** (uses a local `makeResult(winner)` building a minimal stub `CombatResult` with empty `survivingUnits`/`events`):
  - **"normalizes the A-side perspective"**: with `lastPairings = [[2, 5]]` and a stored result where `winner === 0` (the A-side, i.e. player 2, won), `getPairingFor(state, 2)` returns `{side: 0, opponentId: 5, isGhost: false, outcome: "win"}` — confirms side-0 callers see their own side's numeric winner code translated directly to "win".
  - **"normalizes the B-side perspective"**: same pairing/result, but `getPairingFor(state, 5)` (the B-side player) sees `{side: 1, opponentId: 2, outcome: "loss"}` — confirms winner-code 0 (A-side win) is correctly INVERTED to "loss" from the B-side's perspective; then flips the stored result to `winner === 1` and re-checks both sides see the opposite outcome (5 → "win", 2 → "loss") — exhaustively proves the perspective-normalization logic handles both winner codes from both sides correctly (4 combinations total across the two tests).
  - **"handles draws and ghosts"**: `lastPairings = [[3, -2]]` (player 3 vs a ghost, the negative value being some encoded reference to the eliminated source — the exact encoding, e.g. `-2` perhaps relating to player 1's id, isn't asserted on, only that it's negative) with a `"draw"` result — confirms `isGhost: true`, `side: 0` (the real player is always the A-side in a ghost pairing), `outcome: "draw"` (the draw winner value passes through unchanged, not flipped, since draws have no side-dependent inversion). Also confirms `getPairingFor(state, 6)` — a player NOT present in `lastPairings` at all — returns `null` (not an error, not a default object).

**Invariants & constraints** — This suite is the canonical proof that the anti-repeat pairing system actually delivers its promised guarantee (a full round-robin in exactly N-1 rounds for N players) rather than just approximating it — any future change to `buildPairings`'s backtracking/fallback logic should be re-validated against the 7-rounds/7-distinct-opponents test specifically, since that's the test with zero tolerance for algorithmic regression (any repeat or miss within that specific 7-round window fails it).

**Depends on** — `vitest`; `@autobattler/data` (`gameData`); `../src/match.js` (`createMatch`); `../src/rounds.js` (`buildPairings`, `getPairingFor`, the two functions under test); `@autobattler/sim/src/prng.js` (`mulberry32`); `@autobattler/sim/src/types.js` (`CombatResult` type, for the local `makeResult` stub).

**Used by** — Not imported elsewhere; runs under `npm test`.

**Notes** — The fallback tests directly mutate `state.pairingHistory` (a `Map<number, Map<number, number>>`) via raw `.set()` calls rather than going through any public API — confirming `pairingHistory`'s shape is exactly "player id → Map of opponent id → meet count" and that `buildPairings` reads (and updates) this structure directly off `MatchState`, not via a separate side-channel. The ghost test's `lastBoard: { units: [] }` (an empty board) is sufficient to satisfy the ghost-pairing precondition — `buildPairings` doesn't require the eliminated player's last board to be non-empty to generate a ghost fight, it only checks that a `lastBoard` value EXISTS.
