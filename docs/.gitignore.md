# .gitignore

**Path & purpose** — `/.gitignore` (repo root). The single, shared git ignore-rules file for the entire monorepo — there is no per-package `.gitignore` anywhere in `packages/*`.

**Responsibility** — Tells git which files/directories to never track: build artifacts, dependency trees, OS cruft, local secrets, and editor-local settings.

**Exports** — N/A (not a code file; a plain ignore-pattern list).

**Key behavior — full content**
```
node_modules/
dist/
*.log
.DS_Store
.env
.env.local
.cache
.vscode/settings.json
*.tsbuildinfo
```
- `node_modules/` — the npm workspace dependency tree (root + every package's hoisted deps).
- `dist/` — any package's compiled output directory (relevant if any package ever emits build artifacts; `packages/client`'s Vite build output and any future `tsc` emit target would land here).
- `*.log` — any log file anywhere in the tree.
- `.DS_Store` — macOS Finder metadata cruft (consistent with this being developed on a Mac, per the environment's `Darwin` platform).
- `.env` / `.env.local` — local environment variable files, e.g. where a developer would set `DATABASE_URL`/`AUTH_SECRET`/`PORT` overrides per CLAUDE.md's documented env vars, without committing secrets.
- `.cache` — generic local tool cache directory (could be Vite's or another tool's).
- `.vscode/settings.json` — IDE-local workspace settings excluded so editor preferences don't leak into the shared repo, while still allowing OTHER `.vscode/` files (e.g. `launch.json`, `extensions.json`) to be committed if present (this is a narrowly-scoped exclusion, not a blanket `.vscode/` ignore).
- `*.tsbuildinfo` — TypeScript's incremental build cache file, produced by `tsc --build` (per `tsconfig.json`'s usage via `npm run typecheck`) — a local perf cache, not a build artifact meant to be shared/committed.

**Invariants & constraints**
- This file does NOT ignore `balance-report.md`/`balance-report.json` (the generated output of `npm run balance`) — both are confirmed present and TRACKED in git history (first added in an early commit per `git log`), meaning these specific generated files are deliberately committed as a snapshot/reference rather than excluded — a notable exception to the general "generated files aren't tracked" pattern, worth knowing if a future agent expects `npm run balance`'s output to be gitignored and finds it isn't.
- Does not ignore `package-lock.json` (correctly so — lockfiles should be committed for reproducible installs, just excluded from THIS documentation effort's scope per the standing "skip... lockfiles" rule, which is a documentation-process rule, not a git-ignore rule).
- A future agent adding a new package or tool that produces local-only artifacts (caches, logs, build output) should extend this single shared file rather than adding a package-specific `.gitignore`, consistent with the monorepo's broader "one shared root config" pattern also seen in `tsconfig.json` and `vitest.config.ts`.

**Depends on** — Nothing.

**Used by** — git itself (read automatically); not referenced by any source file or script.

**Notes** — This file was discovered during the documentation manifest's final re-scan pass (a new/previously-untracked-by-the-manifest file) and added as a new entry, alongside `references/stage bar/stage-bar-spec.md`. The deliberate non-ignoring of `balance-report.md`/`.json` (see Invariants) is the one detail here worth flagging prominently for any future agent reasoning about what's safe to assume is build-artifact-and-therefore-disposable versus intentionally version-controlled output.
